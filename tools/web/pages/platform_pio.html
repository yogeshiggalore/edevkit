PIO — RP2350's Programmable I/O subsystem

A focused reference on the Programmable I/O (PIO) subsystem that gives the RP2040 / RP2350 their disproportionate I/O power. PIO is what lets a $1 chip with no dedicated SWD, JTAG, SWO, QSPI-master, WS2812, DPI-video, or Ethernet-MAC peripheral nevertheless implement all of them simultaneously — three small CPUs that share registers with the M33 cores and run independent custom programs against the GPIO pads, every cycle, deterministically. This document covers the hardware model (PIO blocks, state machines, FIFOs, side-set, IRQ, DMA chaining), the four-instruction ISA, the pioasm toolchain, the pre-shipped Pico-SDK PIO programs, the timing model for sub-MHz to multi-hundred-MHz protocols, and how edevkit1 partitions its 12 state machines across SWD, JTAG, SWO, I²C, SPI/QSPI, and WS2812.

Audience. A firmware engineer who has poked at SWD/JTAG/SPI on a "real" SoC with dedicated peripherals, and now needs to understand how a PIO state machine implements those same protocols, when to write a new .pio program, and which design choices the RP2350 PIO model makes that you should plan around. By the end you should be able to read any .pio source file end-to-end, understand why side-set and delay matter, lay out FIFOs and DMA chains for a new protocol, and partition a new feature across edevkit1's already-busy PIO budget.

Sister docs. Every protocol doc on the hub builds on this one. Once PIO is clear, read: debug_swd.html and debug_jtag.html for the ARM-debug PIO programs; debug_swo.html for the trace-receive PIO; proto_i2c.html / proto_spi.html / proto_qspi.html for the protocol testers; edevkit1.html §7 for the v1 PIO budget.

Why a "tiny CPU near the pads" beats a fixed peripheral

Most microcontrollers contain a fixed menu of I/O peripherals: 2 UARTs, 1 SPI, 1 I²C, maybe a quadrature decoder. If your application needs a third UART, or a CAN controller on a part that doesn't have one, or a custom protocol that isn't in the catalog at all, you're stuck — bit-banging it from CPU is too slow and too jittery, and a hardware redesign means a different chip.

The RP2040 (2021) and RP2350 (2024) take a fundamentally different approach: instead of more fixed peripherals, they include a small array of programmable peripherals. Each one is a 5-instruction-deep, 32-register, 9-instruction-wide tiny CPU sitting directly next to the GPIO pads, with deterministic per-cycle behaviour. You write a few dozen lines of assembly that implements your protocol (UART at 7.5 megabaud, DPI video at 33 MHz, WS2812 timing-perfect, SWD master, anything), and it runs autonomously — pulling work from a FIFO, driving the pins, signalling completion through DMA or IRQ.

The trade-off, compared to a fixed UART: you have to write the UART. The win: that UART runs at any baud rate you can describe, supports parity any way you want, and there can be eight of them — limited only by the number of state machines, not by what the chip vendor decided to silicon-bake. PIO is "FPGA fabric for the rest of us" — far less flexible than a real FPGA, but in the price and power class of an MCU.

The PIO promise PIO is what makes RP2040 / RP2350 unusually well-suited for tools like edevkit1 — devices that don't know in advance what protocols they will be asked to speak. CMSIS-DAP needs SWD; some users need JTAG; some need SWO; some need I²C, SPI, QSPI, UART; some need WS2812. A traditional MCU would need every one of these as a separate hardware peripheral. PIO lets the same silicon implement all of them, swap mid-session, and still leave headroom.

The headline upgrade is 3 PIO blocks instead of 2, giving 12 state machines instead of 8. For edevkit1, this is the difference between "I can fit SWD + I²C + SPI + WS2812 simultaneously" and "I can fit all of those plus QSPI and SWO and JTAG with room to spare". The ISA is a strict superset of RP2040's PIO; existing .pio programs assemble unchanged.

Backwards compatibility Every .pio program written for RP2040 — including the entire body of community PIO code on GitHub, the Pico-SDK examples, and the rp2040-pio-emulator's reference implementations — runs on RP2350 with no modifications. The pioasm toolchain accepts the same source and emits the same bytecode; you only need to recompile if you want to use the new IRQ index range.

PIO blocks & state machines

The unit of independence in PIO is the state machine (SM). RP2350 has 12 of them, organised into 3 blocks of 4. Within a block, the four SMs share:

  • One 32-instruction instruction memory (so all four SMs in a block run programs out of the same RAM, but each starts at a different program-counter address).
  • One set of 8 IRQ flags that they can raise or wait on (PIO0 IRQ flags don't reach PIO1 by default — they're block-local).
  • A common GPIO mapping window — every SM in PIO0 can address GPIOs 0..31, every SM in PIO1 can address its own 0..31 window, etc. (RP2350 extends this to 0..47.)

Across blocks: SMs are fully independent. PIO0 SM0 cannot stall PIO1 SM3 by hogging the instruction memory. PIO0's instruction memory and PIO1's instruction memory are physically separate. The system bus is shared (so DMA channels feeding PIOs from RAM compete for bandwidth), but most other resources are per-block.

The 32-instruction instruction memory limit is one of the key constraints when designing PIO programs. A complex protocol (say a full SWD master with line-recovery sequence, parity, ack, and turnaround) might take 25 instructions, leaving only 7 for the next program in the same block. You partition by block, not just by SM.

Inside a state machine — registers, scratch, OSR/ISR, FIFOs

Each SM contains:

ElementWidthRole
X32-bitGeneral-purpose scratch register. Loop counters, captured pin values, intermediate computations.
Y32-bitSecond scratch register. Same role as X. Most loop bodies use one of X/Y as the loop counter.
OSR (Output Shift Register)32-bitHolds the bits being shifted out to pins. Each OUT instruction shifts N bits out of the OSR onto the pin pads. Refilled from the TX FIFO via PULL.
ISR (Input Shift Register)32-bitMirror of OSR for input direction. Each IN instruction shifts N bits from pins into the ISR. When full, can be pushed into the RX FIFO via PUSH.
TX FIFO4 × 32-bit (or 8 × 32-bit if joined)Holds host-supplied data the SM hasn't yet shifted out. CPU or DMA writes; SM consumes via PULL.
RX FIFO4 × 32-bit (or 8 × 32-bit if joined)Holds SM-captured data the host hasn't yet read. SM produces via PUSH; CPU or DMA consumes.
PC5-bitProgram counter — index into the block's 32-word instruction memory.
Status flags—"TX FIFO not full", "RX FIFO not empty", auto-pull threshold, auto-push threshold.

The OSR/ISR are where most of the cleverness lives. A 32-bit OSR can be configured to auto-pull a new word from TX FIFO whenever it's been emptied past a threshold (so a streaming protocol never has to explicitly issue PULL); it can shift either MSB-first or LSB-first; it can be partially consumed (e.g. shift 4 bits at a time over 8 OUT instructions for a quad-IO protocol); and it can be moved bit-reversed via MOV.

FIFO joining

By default, an SM has a 4-word TX FIFO and a 4-word RX FIFO. If your SM only transmits (e.g. WS2812), the RX FIFO is wasted — you can join the FIFOs to get an 8-word TX (RX disabled) or 8-word RX (TX disabled). This is critical for protocols that need deeper buffers than 4 words to absorb DMA descriptor latency.

; Example sm_config in C-SDK style
sm_config_set_fifo_join(&c, PIO_FIFO_JOIN_TX);   ; 8-word TX, 0-word RX
sm_config_set_fifo_join(&c, PIO_FIFO_JOIN_RX);   ; 0-word TX, 8-word RX
sm_config_set_fifo_join(&c, PIO_FIFO_JOIN_NONE); ; default 4+4

Pin mapping — IN, OUT, SET, side-set, JMP-pin

Each SM has five independent pin "windows" — five different views of the GPIO array, each configured with its own base pin and width:

WindowUsed byWidthTypical use
IN baseIN pins, N instruction1–32Sample N pins into the ISR. Used by sniffers, slave-mode receivers.
OUT baseOUT pins, N instruction1–32Drive N pins from the OSR. Used by master-mode transmitters.
SET baseSET pins, N instruction1–5Drive a small immediate value to up to 5 pins. Used for short setup sequences (e.g. driving nCS low at start of a SPI transaction).
side-set baseside-set field on every instruction0–5Drive 0–5 pins as a "free" side effect of every instruction. Used for clock generation — see below.
JMP-pinJMP PIN, target instruction1 (single pin)Read one pin and branch on its level. Used for protocol arbitration, error-detect paths, and edge-triggered handshakes.

The five windows can overlap freely. A typical SPI master configuration uses: SET base = nCS pin (so SET 0 / SET 1 toggles nCS); side-set base = SCK pin (so every instruction can clock); OUT base = MOSI pin; IN base = MISO pin. All four windows are different pin numbers, but they're configured into the same SM and used as needed by the instruction stream.

Clock divider — fractional 16.8 timing

Each SM has its own clock divider that turns the system clock (150 MHz on RP2350) into the SM's instruction clock. The divider is a 16-bit integer plus an 8-bit fractional part — fractional dividers, not just integer — so you can hit baud rates exactly on common targets.

SM_clk = sysclk / (int + frac/256)

For example, to run an SM at 9600 Hz (1 instruction per 9600 Hz step) on a 150 MHz sysclk: divider = 150e6 / 9600 = 15625.0, so int=15625, frac=0. To hit 1 MHz: 150 exactly, int=150, frac=0. For an oddball baud like 230400: 150e6/230400 = 651.04..., int=651, frac=0.04*256≈10, so int=651, frac=10. The achieved rate is 150e6/(651+10/256) = 230400 Hz exactly, no jitter.

The fractional divider matters because most protocols specify exact frequencies (UART baud rates, MIDI 31250 baud, I²C 100/400/3400 kHz) that don't divide evenly into 150 MHz. Without fractional dividers, every UART would have ±0.5% baud error.

Per-SM clocks All 12 SMs can have different clock dividers. PIO0 SM0 can run at 25 MHz while PIO0 SM1 runs at 50 kHz simultaneously. The instruction memory they share is read at sysclk; each SM's PC just advances at its own rate.

The four-bit opcode space

PIO instructions are 16 bits wide. The top 3 bits are the opcode; the next 5 bits are the delay-and-side-set field; the bottom 8 bits are the operand. There are exactly nine instructions:

OpcodeMnemonicWhat it does
000JMPConditional or unconditional branch within the program.
001WAITStall until a pin or IRQ flag matches the requested level.
010INShift bits from the IN pin window into the ISR.
011OUTShift bits from the OSR onto the OUT pin window.
100PUSH / PULLMove data between FIFOs and the shift registers (one opcode, distinguished by a flag).
101MOVCopy register-to-register, with optional bit-reverse and inversion.
110IRQSet, clear, or wait on one of the 8 IRQ flags.
111SETWrite a 5-bit immediate to the SET pin window or to scratch X / Y.

That's it. There is no add, no compare, no multiply. Loops are implemented by decrementing a scratch register with the side effect of JMP X-- target (jumps if X != 0, post-decrements). Comparisons are implemented by subtracting via shift-and-XOR. Almost every PIO program is around 8–25 instructions of this very small ISA.

JMP — the only branch

JMP is the only branching instruction. It takes a 5-bit target address (0–31, an absolute index into the block's instruction memory) and an optional condition:

JMP target              ; unconditional
JMP !X target           ; if X == 0
JMP X-- target          ; if X != 0, post-decrement X
JMP !Y target           ; if Y == 0
JMP Y-- target          ; if Y != 0, post-decrement Y
JMP X != Y target       ; if X != Y
JMP PIN target          ; if the configured JMP_PIN is high
JMP !OSRE target        ; if OSR is not yet empty (not all bits shifted out)

The combination of JMP X-- target with no other compare instruction is what most loops use. The fastest possible PIO loop is a 1-instruction body with a 1-instruction back-jump, running at sysclk / 2. With side-set you can effectively make even that loop "free" — the back-jump itself can drive a clock pin.

WAIT — synchronisation primitive

WAIT stalls the SM until a condition becomes true:

WAIT 1 PIN n            ; wait for pin n to be high
WAIT 0 PIN n            ; wait for pin n to be low
WAIT 1 IRQ n            ; wait for IRQ flag n to be set
WAIT 0 IRQ n            ; wait for IRQ flag n to be cleared

WAIT is what lets two SMs (or an SM and a CPU core) coordinate without polling. A common pattern: SM A sets IRQ 4 to signal "I'm done with this transaction"; SM B has been WAITing on IRQ 4; B starts running the moment A finishes. No CPU intervention.

WAIT consumes 1 instruction cycle even when the condition is already true. Critical-path code uses careful WAIT placement to align edges.

IN / OUT — moving bits to and from pins

The bread and butter of any data protocol. IN samples N pins into the ISR; OUT drives N bits from the OSR onto the pins.

OUT pins, 1             ; shift one bit from OSR onto OUT pin 0, advance OSR
OUT pins, 4             ; shift four bits at a time (quad-IO!)
OUT x, 8                ; shift 8 bits from OSR into scratch X (no pin movement)
OUT pindirs, 1          ; shift one bit, but write it to the pin's DIRECTION (1=output)
                        ;   — used to switch a bidirectional bus to drive
IN pins, 1              ; sample 1 pin, MSB-or-LSB-first depending on shift dir
IN x, 8                 ; shift 8 bits from scratch X into ISR

Both instructions can be configured for auto-pull (OSR refills automatically from TX FIFO when its bit count drops below threshold) and auto-push (ISR drains automatically into RX FIFO when it's full). With auto-pull and auto-push enabled, a streaming protocol never explicitly executes PULL or PUSH — the FIFOs feed and drain transparently while the program just keeps OUT-ing and IN-ing.

PULL / PUSH — FIFO interaction

Explicit FIFO ops, used when auto-pull/push isn't appropriate:

PULL block              ; pull a word from TX FIFO into OSR; stall if empty
PULL noblock            ; pull a word into OSR; if empty, copy X into OSR (handy for "idle pattern")
PUSH block              ; push ISR into RX FIFO; stall if full
PUSH noblock            ; push ISR into RX FIFO; if full, drop the data (lossy — use with care)
PULL ifempty            ; pull only if OSR is empty (safe with auto-pull)
PUSH iffull             ; push only if ISR is full (safe with auto-push)

The noblock variants are how a sniffer handles overrun: rather than stalling the SM and missing real-time edges, it drops the overflowed word and increments a host-readable "drop counter". The X copy behaviour of PULL noblock is used by transmit-only state machines to send a configurable idle pattern when the host isn't supplying data.

MOV — the swiss-army instruction

MOV copies a value between any two of {X, Y, OSR, ISR, PINS, STATUS, NULL}, with optional inversion (~) or bit-reverse (::):

MOV x, y                ; X = Y
MOV x, ~y               ; X = ~Y (bitwise NOT)
MOV x, ::y              ; X = bit-reverse(Y)  — useful for endian flips
MOV osr, x              ; OSR = X (refill the output shifter from a scratch reg)
MOV pc, x               ; PC = X (computed branch! used for jump tables)
MOV pins, null          ; drive 0 to all OUT pins
MOV osr, status         ; OSR = "FIFO levels" (used to report state to host)

The MOV pc, x form is how you build a state machine with arbitrary state count: keep the next-state index in X, and at the end of each cycle MOV it into PC.

IRQ — coordination across SMs and cores

IRQ manages 8 (RP2040) or 16 (RP2350) per-block IRQ flags. Each flag can be:

  • Set or cleared by any SM via IRQ set N / IRQ clear N.
  • Waited on by any SM via WAIT 1 IRQ N / WAIT 0 IRQ N.
  • Routed to the M33 cores as a real interrupt (the IRQ gates which flags propagate up).
IRQ set 4               ; raise flag 4
IRQ clear 4             ; lower flag 4
IRQ wait 4              ; raise flag 4 and wait until it's cleared (rendezvous pattern)

The "raise and wait" form (IRQ wait) is the canonical handshake between two SMs: SM A hits "transaction done", raises flag 4, stalls. SM B sees the flag, processes the result, lowers it. SM A's IRQ wait completes; A continues. No CPU involved at all.

SET — small-immediate writes

SET writes a 5-bit immediate to either the SET pin window (up to 5 pins) or scratch X / Y. It's the lightweight version of MOV when the value fits in 5 bits — used to load loop counters and to drive small numbers of static signals (e.g. "drive nCS low").

SET x, 7                ; X = 7 (loop counter for 8-bit shift)
SET pins, 0b00010       ; drive SET pins to that pattern (e.g. nCS=0, IO0=1, IO1=0, ...)
SET pindirs, 0b11111    ; drive all 5 SET pins to OUTPUT

Side-set and delay — the secret weapons

Every PIO instruction has 5 free bits in its encoding for "side-set and delay". Together they make PIO programs 2–4× shorter and let you generate clean clocks for free.

Side-set

If you declare .side_set N at the top of a program, every instruction can drive N pins as a side effect — without spending an instruction cycle. The side-set value is just a 5-bit field that gets latched onto pins at the start of each cycle.

.program spi_clock
.side_set 1                    ; one side-set bit; SCK is driven by it

loop:
    out pins, 1     side 0     ; place data bit, hold SCK low
    nop             side 1     ; rising SCK edge — slave samples
    jmp loop        side 0     ; next bit, SCK falls back

Without side-set, the same program would need explicit SET pins for the SCK pin between every data bit — doubling the instruction count and making the clock period a multiple of 2× the data period rather than 1×. Almost every nontrivial PIO program uses side-set for at least the clock; many use 2 bits (clock + chip-select) or even 4 (multi-channel clocks).

Delay

The remaining bits in the side-set/delay field encode 0–31 cycles of delay — extra instruction cycles to wait after the current instruction completes. This is how you stretch a 1-cycle instruction into a longer wave-form without writing a loop.

nop             side 1  [3]    ; SCK high, hold for 4 total cycles (1 + 3 delay)
nop             side 0  [3]    ; SCK low, hold for 4 cycles
                                ; → SCK toggles every 4 cycles → sysclk / 8 SCK rate

Combining side-set and delay, a 2-instruction loop body produces a square wave at exactly sysclk / (2 × (1 + delay)). For sysclk=150 MHz and delay=1, that's 37.5 MHz. Delay=0 → 75 MHz. Delay=2 → 25 MHz. The fractional clock divider then gives finer control between those.

Why this matters Side-set and delay together are why a PIO program for SPI master is ~5 instructions long instead of ~30. The programs in app/pio/spi.pio, app/pio/qspi.pio, etc. are all small enough that they fit in one block's 32-instruction memory alongside 2–3 other programs. That's the entire reason PIO is interesting: it's not just programmable, it's information-dense.

FIFOs and back-pressure

The TX and RX FIFOs are the only buffer between an SM (running at, say, 50 MHz instruction rate) and the M33 / DMA (running at sysclk and possibly competing with other agents on the bus). At 50 MHz with 4 words of FIFO, the SM has 80 ns of headroom before it starves; if the host can't feed it within that window, the SM stalls.

Every protocol must answer: what does back-pressure mean? Three patterns:

  1. Stall the wire. The SM stops clocking out, holding SCK static. Fine for SPI master / I²C master / UART TX — slows the wire, doesn't lose data. Implementation: default behaviour of OUT with empty OSR and a blocking PULL.
  2. Drop the data. The SM proceeds but skips real data. Required for sniffers — a sniffer that stalls would miss real-time edges and the capture is unreliable. Implementation: PUSH noblock, plus a host-side overrun counter.
  3. Buffer in SRAM via DMA. A DMA channel auto-feeds the TX FIFO from a large RAM buffer (or auto-drains the RX FIFO into one). The buffer can be MB-sized; the SM never stalls until the entire buffer is drained or filled. Implementation: dreq-paced DMA with an 8-word joined FIFO and a ring buffer in SRAM.

edevkit1 uses pattern (1) for masters, pattern (2) for sniffers when the user has explicitly accepted "lossy" capture, and pattern (3) (DMA + SRAM ring buffer) for everything else. The ring buffer for QSPI sniff is sized at 64 KB (one quarter of the 264 KB available SRAM), giving ~13 ms of capture at 50 MHz quad-IO before the host has to drain it.

DMA chaining — back-to-back transactions

RP2350 has 16 DMA channels, each with a "chain to" register that lets one channel's completion auto-trigger another. A common PIO pattern uses two DMA channels in a ping-pong:

DMA ch 0:  src = SRAM ring buffer, dst = PIO0 SM0 TX FIFO,
           dreq = paced by SM0's "TX not full" signal,
           on completion, chain → ch 1
DMA ch 1:  src = (next position in SRAM ring), dst = same TX FIFO,
           dreq = same,
           on completion, chain → ch 0

This lets the SM consume data continuously from a 2 MB ring buffer with zero CPU involvement. The CPU's only job is to top up the ring buffer ahead of where the DMA channels are reading. Effective throughput: limited by SRAM bandwidth (≈600 MB/s) and the DMA channel's peak (≈150 MB/s), not by the M33's interrupt rate.

For receive: same shape, dst = SRAM, src = RX FIFO. CPU drains the ring buffer to USB at its own pace; the SM keeps producing.

IRQ to the M33 cores

PIO IRQ flags can be selectively forwarded to the M33 NVIC. The forwarding mask is per-block: PIO0 has its own IRQ0/IRQ1/IRQ2/IRQ3 lines into the NVIC, with a mask register selecting which of the 8 (16 on RP2350) flags propagate up.

// C-SDK style
pio_set_irq0_source_enabled(pio0, pis_interrupt0, true);  // PIO0 IRQ flag 0 → NVIC PIO0_IRQ_0
pio_set_irq1_source_enabled(pio0, pis_interrupt1, true);  // ...IRQ flag 1 → NVIC PIO0_IRQ_1

This is how a "transaction complete" event in a PIO program reaches the firmware C code: the SM raises IRQ flag N, the flag is mapped to NVIC PIO0_IRQ_n, the M33 NVIC fires the corresponding interrupt handler, the handler reads RX FIFO and signals the requesting thread.

pioasm — the assembler & how Zephyr builds it

pioasm is the assembler. Source files have a .pio extension, the syntax is described in the RP2350 datasheet section 11.4, and the assembler emits either:

  • A C header (.h) with the assembled bytecode as a constant array plus helper structs to configure the SM. This is the format Pico-SDK and Zephyr's raspberrypi,pio driver consume.
  • A raw binary (.bin) — useful for runtime loading.
  • A Python module, for use with the rp2040-pio-emulator simulator.

The Pico-SDK has a CMake helper pico_generate_pio_header() that auto-runs pioasm on every .pio in a target. Zephyr does not. On Zephyr you have to either write your own CMake function or run pioasm manually as a build step. edevkit1 ships its own tools/cmake/PIO.cmake helper:

# tools/cmake/PIO.cmake
function(edevkit_pio_program target pio_file)
  set(out_h "${CMAKE_CURRENT_BINARY_DIR}/${pio_file}.h")
  add_custom_command(
    OUTPUT ${out_h}
    COMMAND ${CMAKE_SOURCE_DIR}/../tools/pioasm/pioasm
            -o c-sdk
            ${CMAKE_CURRENT_SOURCE_DIR}/${pio_file}
            ${out_h}
    DEPENDS ${CMAKE_CURRENT_SOURCE_DIR}/${pio_file}
    COMMENT "Assembling PIO program ${pio_file}"
  )
  target_sources(${target} PRIVATE ${out_h})
  target_include_directories(${target} PRIVATE ${CMAKE_CURRENT_BINARY_DIR})
endfunction()

# usage in app/CMakeLists.txt
edevkit_pio_program(app pio/swd.pio)
edevkit_pio_program(app pio/qspi.pio)
edevkit_pio_program(app pio/i2c.pio)
edevkit_pio_program(app pio/ws2812.pio)

pioasm binaries are vendored under tools/pioasm/ for Linux x86_64, Linux arm64, macOS arm64, macOS x86_64, and Windows x86_64. The build picks the right one based on host triple. See tools/pioasm/README.md for details on rebuilding from upstream sources.

Simulating PIO programs without hardware

The most useful tool for PIO development is the rp2040-pio-emulator (a Python package). It's a cycle-accurate software model of one PIO block, including all 4 SMs, FIFOs, side-set, delay, and IRQ. You feed it a .pio file and a virtual TX FIFO sequence; it tells you what the pins look like cycle-by-cycle.

pip install rp2040-pio-emulator
python -m rp2040_pio_emulator app/pio/swd.pio --feed 0xCAFEBABE

The emulator is what edevkit1's CI uses to verify that PIO programs behave correctly before they're flashed to hardware. Every .pio file under app/pio/ has a sibling test script under tests/pio/test_*.py that runs known input vectors through the emulator and checks the output pin sequence. Catches bugs in side-set placement and delay budgets before they brick a real board.

House rule Every PIO change goes through the emulator first. The flow: edit .pio, run pioasm, run pytest against the emulator, only then flash to hardware. Catches "off by one in side-set" the same hour you wrote it instead of three days into a debug session.

The v1 PIO budget — 12 SMs, 7 protocols

edevkit1's official PIO partition (also documented in CLAUDE.md and edevkit1.html §7):

BlockSM 0SM 1SM 2SM 3
PIO0SWD / JTAG (shared, mode-select)SWO RX (UART decode)WS2812 (status LEDs)free
PIO1I²C (master/slave/sniffer)SPI / QSPI (master/slave)QSPI partner (sample-late or slave-emul)Sniffer (passive)
PIO2reserved for v2reserved for v2reserved for v2reserved for v2

v1 uses 8 of the 12 state machines, with 4 reserved for v2 (logic analyser, additional UARTs, video out, etc.). PIO0 SM3 is the only intra-v1 spare; the others are committed.

Why this partition

  • SWD and JTAG are mutually exclusive at any moment (the user has selected one mode), so they share PIO0 SM0. The PIO program is loaded at mode-switch time. Mode switch costs ~1 ms (re-program SM0's instruction memory and reconfigure pin window).
  • SWO RX is independent — it can run while SWD is active. SWO arrives on its own pin, gets decoded by PIO0 SM1, lands in a SRAM ring buffer via DMA, then back-pressure-aware streams up to USB.
  • Protocol tester (I²C / SPI / QSPI) lives on PIO1, completely separated from the debug stack. Two reasons: (a) protocol tester pins are physically separated on the connector (different voltage domain); (b) I²C and QSPI together can use up most of one block's 32-instruction memory, leaving zero room for SWD code.
  • Sniffer on PIO1 SM3 is the dedicated "passive eavesdropping" SM. Always there, can be aimed at I²C, SPI, or QSPI dynamically. Its program is small (just sample-and-PUSH); the decoding is done host-side.

Worked example — SWD on PIO0 SM0

SWD is a 2-wire ARM debug protocol — see debug_swd.html for the protocol-level explanation. Here's the PIO program that implements it on edevkit1 (simplified):

; app/pio/swd.pio (excerpt)
.program swd_master
.side_set 1                            ; SWCLK is sideset

; Each iteration of the inner loop is 2 PIO cycles (1 falling edge + 1 rising).
; Sysclk = 150 MHz, so each loop step = 13.33 ns; max practical SWCLK ~ 25 MHz with delay 1.

write_loop:
    out  pindirs, 1     side 0  [1]    ; in case we're switching direction
    out  pins, 1        side 0  [1]    ; place data bit, SWCLK low
    nop                 side 1  [1]    ; rising edge: target samples bit
    jmp  !osre, write_loop  side 1     ; loop while OSR has bits left

read_loop:
    in   pins, 1        side 0  [1]    ; sample SWDIO with SWCLK low?
                                       ; (per protocol, master samples on falling edge —
                                       ;  this depends on whether we're in master or slave)
    nop                 side 1  [1]    ; falling-to-rising: target drives next bit
    jmp  read_loop      side 0

The actual program in app/pio/swd.pio handles the full SWD turnaround sequence (master drives → high-Z → target drives → high-Z → master), parity, ack reading, and line-recovery (50+ idle cycles). The edited version above shows only the data-shifting core; about half the actual program is the turnaround dance.

How the C code dispatches it

// firmware/app/src/proto/swd.c (excerpt)
#include <hardware/pio.h>
#include "swd.pio.h"   // generated by pioasm

static void swd_init(uint pio, uint sm, uint pin_swclk, uint pin_swdio)
{
    uint offset = pio_add_program(pio, &swd_master_program);
    pio_sm_config c = swd_master_program_get_default_config(offset);

    sm_config_set_sideset_pins(&c, pin_swclk);
    sm_config_set_out_pins(&c, pin_swdio, 1);
    sm_config_set_in_pins(&c, pin_swdio);
    sm_config_set_clkdiv_int_frac(&c, 6, 0);   // 25 MHz at 150 MHz sysclk

    pio_gpio_init(pio, pin_swclk);
    pio_gpio_init(pio, pin_swdio);

    pio_sm_init(pio, sm, offset, &c);
    pio_sm_set_enabled(pio, sm, true);
}

void swd_write_word(uint pio, uint sm, uint32_t word)
{
    pio_sm_put_blocking(pio, sm, word);   // pushes into TX FIFO; SM auto-pulls
}

uint32_t swd_read_word(uint pio, uint sm)
{
    return pio_sm_get_blocking(pio, sm);  // blocks until RX FIFO non-empty
}

Notice how the C side never touches a single GPIO register, never busy-waits on a clock, and never has to time anything. All the timing-critical work happens inside the PIO state machine; the C side is reduced to "push word in, pull word out".

Pitfalls and bring-up gotchas

SymptomLikely causeFix
Pins don't toggle when you start the SM.Pin is configured for SIO function, not PIO function.Call pio_gpio_init(pio, pin) before enabling the SM. This points the pad mux at PIO0 / PIO1 / PIO2 instead of SIO.
Only the first 4 words push correctly; subsequent pushes block.FIFO is 4 words deep by default. The SM is consuming slower than you're pushing.Enable DMA pacing, or join the FIFOs to 8 words, or reduce the push rate, or speed up the SM.
Side-set pin doesn't move at all.You forgot to pio_gpio_init the side-set pin, or the SM's side-set base/width is wrong.Check sm_config_set_sideset_pins matches your .side_set directive in the .pio.
SM hangs after a few cycles.Auto-pull is on but TX FIFO is empty. The SM stalled at the next OUT instruction.Either feed the FIFO, or use a non-blocking pull pattern with a "default value" via PULL noblock + scratch X.
Edges are jittery / clock period varies.System clock is being divided by an integer that doesn't match the desired rate exactly, and the SM is stretching cycles unpredictably (e.g. PULL stalling on FIFO empty).Use the fractional clock divider with a smaller integer ÷ smaller fractional. Verify the SM never stalls in steady state.
Two SMs that should run synchronously drift.They have the same clock divider but different start times; the divider is per-SM, so they're free-running.Use pio_enable_sm_mask_in_sync(pio, mask) to start them on the same sysclk edge. For more precise sync, gate one with WAIT IRQ from the other.
Instruction memory full when adding a new program.Each block has only 32 instructions, shared by all 4 SMs in that block.Move the new program to a different block, or shrink an existing program (look for repeated patterns and convert to a loop), or use the SM's "EXEC injection" to run one-off instructions without using program memory.
JMP target not where you expect.JMP target is an absolute address into the block's instruction memory, not a relative offset. After pio_add_program() the same source line lives at a different address.Use the assembler's symbolic labels (which expand to absolute addresses post-link). Or use the (your_program_offset + label) idiom that the C-SDK ships.
RP2040 program runs on RP2350 but timing's off.RP2040 default sysclk is 125 MHz; RP2350 default is 150 MHz. If your clock divider is hard-coded for 125, your wave is now 20% faster on RP2350.Express clock dividers as a ratio of the actual sysclk, not as a fixed integer. Use clock_get_hz(clk_sys) / desired_rate.

Where to learn more

  • RP2350 datasheet, chapter 11 — the canonical reference. Read sections 11.1–11.5 end-to-end if you're going to write a PIO program from scratch.
  • RP2350 hardware design guide (Raspberry Pi Foundation) — has worked PIO examples for SPI, UART, WS2812, DPI, with timing diagrams.
  • Pico-SDK pico-examples/pio/ — 30+ reference programs covering UART, I²C, SPI, WS2812, DPI, PWM, audio, even a video output. Reading these is the fastest way to learn PIO idioms.
  • rp2040-pio-emulator (Python) — cycle-accurate simulator, free, runs on a laptop.
  • Hunter Adams's blog (Cornell ECE) — the most accessible third-party tutorial series on RP2040 PIO. Covers oscilloscope-friendly PIO programs and is the source of half the community's PIO knowledge.
  • "Programmable I/O with Raspberry Pi Pico" (Sandy Maguire's textbook) — a full book on PIO, with worked exercises.
  • edevkit1 source under firmware/app/pio/ — the production .pio programs for SWD, JTAG, SWO, I²C, SPI, QSPI, WS2812. Each is heavily commented.
  • debug_swd.html / debug_jtag.html / debug_swo.html / proto_i2c.html / proto_spi.html / proto_qspi.html — protocol-level docs that show how each .pio program implements its protocol.
