SPI — Serial Peripheral Interface, end-to-end

A focused reference on the Serial Peripheral Interface (SPI) — the four-wire, full-duplex, push-pull serial bus that runs almost every flash chip, display panel, and high-throughput sensor in modern embedded systems. This document covers what the wires do, the four CPOL/CPHA clock modes (with ASCII timing diagrams), bit order and word size, speed limits, topologies, slave mode, sniffing, and how edevkit1's PIO-based SPI engine implements all three roles — master, slave, and passive sniffer — on a single connector.

Audience. A firmware engineer who is about to write SPI-driver code, port a flash chip to a new SoC, or use the edevkit1 protocol tester to debug a misbehaving SPI bus on someone else's board. By the end you should be able to read an SPI datasheet timing page without squinting, choose the right CPOL/CPHA from a part's spec, debug the universal "why am I reading 0xFF?" first-time bug, and configure the edevkit1 SPI engine for whichever role your bench setup needs.

Sister docs. For the dual / quad / octal extensions read proto_qspi.html; for the underlying PIO state machines that implement the bus, see platform_pio.html. For the other two protocols on edevkit1's tester, see proto_uart.html and proto_i2c.html. For the connector wiring, voltage-domain rules, and overall position of SPI in the v1 design, see edevkit1.html §6.5.

What SPI is, and why it dominates flash and displays

The Serial Peripheral Interface is a synchronous, full-duplex, four-wire serial bus that Motorola published as part of the 68HC11 microcontroller family in 1979, and which has since become the lingua franca of board-level chip-to-chip communication. It is not a formal standard maintained by a standards body — there is no "SPI specification" the way there's a USB or Ethernet spec — it's a de-facto convention codified, slightly differently, in every datasheet that uses it. The Wikipedia page is currently the closest thing to a normative reference, and it is the one most chip vendors implicitly converge on.

SPI's defining choices, all made in service of being the cheapest possible fast bus:

  • Synchronous. A clock wire (SCK) is shared between master and slave; both shift bits on the agreed-upon clock edges. No baud-rate negotiation, no oversampling, no PLL recovery — the receiver simply latches what's on the data line at clock-edge time. This is what lets SPI scale to 50–133 MHz on a tiny board with no fancy I/O cells.
  • Full-duplex. Two data wires, one in each direction (MOSI, MISO). Every clock edge moves a bit in and a bit out simultaneously, so a write and a read happen in the same clock cycle. Serial-flash command sets exploit this heavily — a 1-byte command shifted out is overlapped with 1 byte of don't-care shifted in, and then status bytes flow back on subsequent clocks while the master shifts dummy bytes.
  • Push-pull (NOT open-drain). Both master and slave drive their data lines actively high or low through CMOS push-pull output stages. No pull-up resistors on data, no wired-AND, no slow rising edges. This is what gets you the speed — and it is also what fundamentally rules out multi-master SPI without external arbitration logic, because two masters trying to drive opposite levels on the same wire produce a short circuit.
  • Master-slave, with selection by chip-select. Exactly one master drives the clock; one or more slaves listen. A per-slave chip-select (nCS, active-low) tells which slave should respond — every other slave keeps its MISO output tristated and ignores SCK.
  • No addressing on the wire. Compare with I²C, where every transaction begins with a 7-bit slave address byte. SPI uses a dedicated nCS pin per slave, so the addressing happens at the GPIO layer — there are zero protocol bytes spent on it. This is one of the reasons SPI is faster in practice than I²C even at the same clock rate: you don't pay an address byte per transaction.
  • No acknowledgment, no flow control, no error detection on the wire. Whatever bits arrive, the receiver believes. If you need integrity, you bake a CRC into the protocol layer above (every modern SPI flash command set does — see W25Q's CRC and JEDEC's SFDP).

Those choices put SPI in a very specific niche. It dominates wherever you need fast, point-to-point, on-board chip-to-chip data transfer with low gate count on the slave side: boot/data flash (every modern MCU boots from SPI flash), TFT/OLED displays, SD-card-mode-1 transfers, high-rate IMUs and ADCs, Ethernet and Wi-Fi MACs as MCU peripherals (W5500, ESP-Hosted, ENC28J60), CAN controllers (MCP2518FD), pixel-driver chains (APA102, anything chained), DAC arrays. Anywhere I²C's 400 kHz / 1 MHz / 3.4 MHz feels limiting, SPI takes over.

Conversely, SPI is a terrible choice for: longer-than-board distances (single-ended push-pull doesn't survive a metre of wire well), buses with many slaves (every slave wants its own nCS GPIO and its own routing channel), or anything needing ack/retry semantics (the protocol gives you nothing to build on).

Mental model SPI is a pair of distributed shift registers clocked by a shared clock. Master and slave each have an internal 8-bit (or 16-bit, or 32-bit) shift register. SCK clocks them both in lock-step; on each edge, the master's MSB falls into the slave's LSB, and the slave's MSB falls into the master's LSB. After 8 clocks, the two registers have swapped contents. That is the whole protocol. Everything else — CPOL, CPHA, nCS, modes — is just convention about which clock edge does the latching and whether the wires idle high or low.

A note on naming — MOSI/MISO vs COPI/CIPO

OSHWA, OSHPark, Adafruit, Sparkfun, and the Linux kernel have all adopted Controller / Peripheral terminology since 2020, mapping MOSI → COPI and MISO → CIPO. The wires are electrically identical; only the labels differ. Datasheets are mixed: most legacy chip datasheets still say MOSI/MISO; many newer reference designs and Linux DTS files say COPI/CIPO. edevkit1 silkscreen uses MOSI/MISO for compatibility with the larger body of existing datasheets, but firmware logs and CLI output accept and emit either spelling. When in doubt, the direction arrow on the schematic disambiguates.

Why nCS is active-low

Active-low select dates back to TTL-era logic where unconnected inputs floated high and you wanted "no slave selected" to be the default unconnected state. Modern CMOS doesn't care, but the convention is universal — every SPI slave datasheet you'll ever read assumes nCS active-low. Don't fight it; if your master GPIO peripheral only outputs active-high, invert it in software or with a discrete inverter.

Why MISO must be tristated

In a star topology with N slaves, all N MISO outputs are wired together to the master's MISO input. Only the selected slave is allowed to drive the line; the other N−1 must hold their MISO outputs in Hi-Z (high-impedance / tristated). If a slave fails to tristate (some cheap parts have a bug here, or worse, a strap-controlled "always-drive" mode), you get bus contention — two outputs fighting, wasted current, and corrupted data. This is the #2 reason a working slave seems to "stop responding" when you add a second slave to the same MISO line. (The #1 reason: forgetting to wire the new slave's nCS to a unique GPIO.)

CPOL & CPHA — the four clock modes

SPI has two clock-related parameters that the master and the slave must agree on, or nothing works:

  • CPOL — clock polarity. What level SCK idles at. CPOL=0 means SCK idles low; CPOL=1 means SCK idles high.
  • CPHA — clock phase. Which clock edge samples the data. CPHA=0 means the receiver samples on the first (leading) edge of each clock cycle and the transmitter shifts on the second (trailing) edge. CPHA=1 swaps that: shift on the leading edge, sample on the trailing edge.

The two bits encode four "modes", numbered 0–3:

ModeCPOLCPHASCK idleSample edgeWhere you'll see it
000lowrisingDefault for ~80% of slaves — almost every SPI flash (W25Q, MX25), most TFT/OLED controllers, ADCs, IMUs (BMI270, LSM6DSO).
101lowfallingLess common. Some audio codecs, occasional Microchip / TI parts.
210highfallingLess common. Some SD cards in SPI mode quirk into this; some legacy parts.
311highrisingThe other common one — SD cards in SPI mode after init, some SPI flash variants advertise both M0 and M3, and many power-management ICs prefer M3 because clock idles high (closer to the "bus inactive" Vdd state).
Practical rule If a datasheet just says "SPI" without specifying a mode, it almost always means Mode 0. If it shows a timing diagram with SCK idling high, it's probably Mode 3. The other two modes are rare enough that when you encounter one you'll know it from context. Many parts (most SPI flash) actually accept both Mode 0 and Mode 3 on the same pins — they observe the SCK level when nCS falls and pick accordingly.

What "wrong CPOL/CPHA" feels like

Getting CPOL/CPHA wrong is the universal first-time SPI bug. It manifests in three distinct ways depending on which bit you got wrong:

SymptomLikely cause
You read all 0xFF from MISO regardless of what you ask for.Either MISO is floating (slave never drives because nCS is wrong), or your slave isn't even seeing the clock (CPOL inverted so the bus appears "always idle").
You read all 0x00.MISO is being held low somewhere — possibly by another slave that didn't tristate, or by a wrong-polarity nCS keeping the actual slave deselected.
You read data that looks almost right but is bit-shifted by exactly one bit. The first bit is junk; the rest is the previous byte's data offset by one position.CPHA off by one. You're sampling on the wrong edge, so each sample picks up the bit the transmitter is about to set, not the one it just finished setting.
Sometimes-right, sometimes-wrong, jittery results.Either CPHA correct but you're at the edge of the timing budget (slow down clock and re-test), or you're hitting an SI issue on a long cable. See speed.
First byte garbage, every subsequent byte fine.The slave's "first bit" isn't ready before the first SCK edge — typical of slaves that need extra setup time on first transaction. Add a dummy byte at the start, or insert a brief delay between asserting nCS and starting SCK.
Debug recipe Hook a logic analyzer to all four wires before you write any driver code. Sigrok's spi decoder takes CPOL/CPHA as parameters and will tell you which mode the slave is operating in when you feed it a known-good capture. Spend the 10 minutes on the bench; it saves the 4 hours of guessing.

Bit order, word size, transaction shape

MSB-first vs LSB-first

MSB-first is the default on probably 95% of all SPI slaves you'll meet — flash, displays, sensors, the lot. The master clocks out the most-significant bit first and the slave returns its data MSB-first. A few protocols (some legacy Microchip parts, a handful of TI ADCs, some MMC-mode-1 transfers) use LSB-first. Both master and slave hardware can usually be configured either way; the constraint is purely "you have to match what the slave expects".

If you receive a byte that looks bit-reversed from what you expect (e.g. you write 0x80 and read back 0x01 when echoed through a loopback), that's the bit-order mismatch tell.

Word size — 8, 16, 32 bits

SPI hardware controllers usually let you pick a "word size" of 8, 16, or 32 bits. The word size determines:

  • How many SCK cycles a single FIFO push generates. A 16-bit word = 16 SCK edges.
  • Whether the controller toggles nCS between consecutive words (some do, some don't — controller-specific).
  • The width of the controller's TX/RX FIFOs in CPU words.

For most flash and display work you'll use 8-bit words; the protocol byte stream is naturally byte-oriented. For high-rate ADCs or DACs that pack a 12-bit or 16-bit sample per transfer, 16-bit words let you push a full sample with one register write — half the FIFO pressure and half the interrupt rate.

A multi-byte transaction

The standard shape of a one-master-one-slave SPI transaction:

1.  Master pulls nCS low.
2.  Master clocks out N bytes on MOSI (and simultaneously receives N bytes on MISO).
3.  Master releases nCS high.

Read-modify-write sequences typically split this into two halves: a "command + address" half where MOSI carries the data and MISO is don't-care, and a "read data" half where MOSI shifts dummy bytes (often 0xFF or 0x00) and MISO carries the slave's response. nCS stays low for the entire transaction. Many slaves — especially flash — interpret a premature nCS rise as "abort the command", and will not commit the operation. This is one of the more interesting gotchas: a buggy SPI controller that toggles nCS between every word will corrupt every multi-byte command on a flash chip.

Speed, setup/hold, the long-cable problem

SPI's clock rate is bounded by three things, in order of which one usually bites first:

  1. Slave's max SCK frequency, per its datasheet. Most generic SPI flash is rated 50–104 MHz in single-IO mode; cheap displays cap at 20–25 MHz; high-end ADCs push 80–100 MHz. The slave's spec is the ceiling.
  2. Setup/hold time at the master's MISO input. The slave drives MISO some delay t_v after the previous SCK edge; the master needs MISO stable for t_su before the sampling edge. The relation is roughly T_SCK ≥ 2 · (t_v + t_su + t_pcb) where t_pcb is the round-trip flight time on the PCB or cable.
  3. Signal integrity — overshoot, ringing, reflections. Push-pull edges with no termination resistor produce ringing on anything more than a couple of inches; faster edges, longer wires, more ringing.

The "long flex cable" problem

A 6-inch flex cable adds roughly 1 ns of one-way flight time. At 50 MHz, the SCK period is 20 ns; at 100 MHz, 10 ns. Once the round-trip time approaches the SCK half-period, the master is sampling MISO before the slave's response from the previous edge has even arrived. You'll see clean SCK and MOSI on a logic analyzer, but MISO will look right at low speeds and turn into garbage past some threshold — typically the same threshold across all transactions, suspiciously close to "round-trip-time = T_SCK/2".

Mitigation, in order of preference:

  • Slow down. SPI is a synchronous bus — half the clock, half the throughput, but it almost always works. 10 MHz on a sloppy cable beats 50 MHz guess-and-pray.
  • Shorten the cable. Below ~3 inches you can run 50 MHz reliably with hand-soldered jumpers.
  • Use the controller's "sample-late" / "RX delay" feature if it exists. Many SoCs let you sample MISO N clock cycles after the nominal sample edge — this directly compensates for round-trip latency. RP2350's PIO programs can be written with the same technique.
  • Use shielded twisted-pair or differential signaling. Overkill for in-box, mandatory if you're crossing a chassis boundary.
edevkit1 cable rule The edevkit1 protocol header is rated for 50 MHz QSPI / 100 MHz single-SPI on cables ≤4 inches. Above 4 inches, the firmware automatically reduces the maximum offered clock to 25 MHz unless overridden by --no-cable-derate in the CLI. You will see this reflected in edevkit spi info.

Topologies — star, daisy-chain, multi-IO

Star topology — one nCS per slave

                    ┌─── nCS_A ───→  Slave A
                    │
   Master ──────────┼─── nCS_B ───→  Slave B
                    │
                    └─── nCS_C ───→  Slave C

   SCK   ───────────────────────────→  (shared, all slaves)
   MOSI  ───────────────────────────→  (shared, all slaves)
   MISO  ←───────────────────────────  (shared; only selected slave drives)

The default and most common SPI topology. SCK, MOSI, MISO are bussed to every slave; each slave gets its own nCS GPIO from the master. Trade-off: the nCS pin count grows linearly with the number of slaves. With more than 4 slaves, this is when you start reaching for a 74HC138-style 3-to-8 decoder driven by 3 GPIOs.

Daisy-chain topology — slaves in series

    Master ─── MOSI ───→ Slave_A.MOSI
                          Slave_A.MISO ───→ Slave_B.MOSI
                                              Slave_B.MISO ───→ Slave_C.MOSI
                                                                  Slave_C.MISO ───→ Master.MISO

    SCK    ──────────────→ shared by all slaves
    nCS    ──────────────→ shared by all slaves (single chip-select for the chain)

The master clocks out N bytes (where N = number of slaves), and the bytes propagate through the chain. After N transactions, every slave has the byte the master originally intended for it. Only specific chips support this — TI's TLC5940 LED drivers, several DAC families, the WS2812-protocol shift-style chains, Maxim's MAX7219 display drivers. Most flash and sensor chips don't, because their MISO is computed from the command, not just shifted from the previous slave.

Daisy-chain saves nCS GPIOs but increases latency proportional to chain length, and a single faulty slave breaks the whole chain. It's a niche topology that you should recognise but won't usually design from scratch.

Multi-IO — DSPI, QSPI, OSPI

Modern SPI flash also defines multi-IO modes that use 2, 4, or 8 data pins simultaneously, all bidirectional, all driven by either the master or the slave depending on the transaction phase. These modes — Dual SPI, Quad SPI, Octal SPI — are out of scope for this document; they get their own treatment in proto_qspi.html. The basic clocking model (CPOL/CPHA, push-pull, nCS-bracketed) is identical; only the data lane width changes.

Slave mode — when the MCU is the peripheral

Most of this doc assumes your MCU is the master. The reverse case — your MCU is the slave on someone else's bus — has its own rules and is the harder direction to get right.

Constraints

  • You don't drive the clock. SCK arrives at whatever rate the master wants, including bursts faster than your CPU can handle byte-by-byte. You must use the SPI controller's hardware FIFO, DMA, or PIO buffer — interrupt-per-byte will not keep up at 10 MHz, let alone 50 MHz.
  • Response time budget is tiny. When the master clocks out a request byte (e.g. "what's your status?"), the slave has typically ≤2 SCK cycles to load the response into its TX FIFO. At 10 MHz that's 200 ns. Most slave-mode designs pre-load the next response based on the previous command.
  • nCS edge marks transaction boundaries. The falling edge says "new transaction starts now"; the rising edge says "abort or commit". Slave firmware almost always wires nCS to a GPIO interrupt or PIO trigger so it can reset its byte-counter and command-decoder state machine on every transaction boundary.
  • FIFO underflow → 0xFF / 0x00. If the slave can't produce the next response byte fast enough, the bus sends whatever the controller defaults to — usually 0xFF on idle MISO. Masters interpret that as "slave returned 0xFF", which can be a valid value, so the failure is silent. Defensive masters CRC every response.
  • Mode is set by the master. The slave has no say in CPOL/CPHA or clock rate; whatever the master is doing, the slave must match. Slave firmware that's auto-detecting CPOL is rare and brittle.

Where slave mode shows up

  • An MCU acting as a co-processor (e.g. a sensor-fusion chip exposing data via SPI to a host SoC).
  • An MCU emulating an SPI flash chip — this is one of edevkit1's headline use-cases: emulate a W25Q128 to a target board for boot-time testing without burning a real chip.
  • An MCU acting as a CAN/BLE/WiFi bridge driven via SPI by a low-end host MCU.
  • Test fixtures that pretend to be a peripheral so the firmware-under-test can run on a bench rig.

Sniffer mode — passively reading a live bus

A sniffer is a third party on the bus that listens to all four wires (SCK, MOSI, MISO, nCS) without driving anything. It's invaluable for debugging two-party SPI failures where the only thing you can change is the observer.

The sampling challenge

To capture every edge of an SPI transaction without missing or doubling bits, the sniffer must sample at at least 2× the SCK frequency, and ideally 4× to pick up SCK edges with margin against jitter. The Nyquist-style minimum is 2×; in practice 3–4× gives you a robust capture.

That bound translates directly into "what clock the sniffer can possibly handle". A sniffer running on a 150 MHz processor or PIO state machine, sampling at 4× SCK, tops out around ~37 MHz SPI. With 2× sampling and aggressive PIO programming you can stretch to ~75 MHz SPI — but you lose timing margin and any clock asymmetry on SCK starts to break the capture.

Where the data goes

Sniffed bytes get buffered into a large RAM ring (edevkit1 reserves up to 256 KiB of SRAM for protocol-tester capture buffers; see edevkit1.html §4.2) and then DMA'd to the host over USB CDC. The producer (PIO) is faster than the consumer (USB FS at ~10 Mbps usable) on any sustained burst, so the buffer doubles as a back-pressure absorber. When it fills, edevkit firmware refuses to silently drop bytes — it raises a BufferFull condition, halts capture, and lets the user choose to slow the master, switch to USB HS (when added), or reduce the SCK rate.

The edevkit1 SPI engine — PIO1 SM1 + SM2

edevkit1's SPI tester is one of the protocol engines defined in §6.5 of the v1 design spec. It's implemented entirely on the RP2350B's PIO1, state machines 1 and 2 — see platform_pio.html for the PIO programming model and instruction-set walkthrough. Using two state machines lets the engine run two roles concurrently: e.g. a master on SM1 plus a sniffer on SM2, or a slave-emulator on SM1 plus a sniffer on SM2.

Pin assignment

The protocol-test connector multiplexes 6 GPIOs (GPIO 6–11 on the RP2350B) as SPI / DSPI / QSPI in a single header. The pin meaning depends on which mode you've selected at config time:

GPIOSPI (1-bit)DSPI (2-bit)QSPI (4-bit)
6SCKSCKSCK
7nCSnCSnCS
8MOSI (IO0)IO0IO0
9MISO (IO1)IO1IO1
10——IO2 (= nWP)
11——IO3 (= nHOLD)

You will not find three separate connectors for SPI, DSPI, and QSPI on the edevkit1 PCB — that was a deliberate decision (see edevkit1.html §15 decision log) to save board area, simplify cabling, and keep the firmware uniform. The downside is that the user has to tell the firmware which mode they want; the upside is one ribbon cable handles all three.

Three roles, configurable per-session

RoleBehaviourTop speed
MasterDrives SCK, MOSI, nCS; reads MISO. Clocks user-supplied byte streams or firmware-built command sequences (e.g. "read JEDEC ID then dump 4 KiB at offset 0").100 MHz single-IO on ≤4" cables; 50 MHz QSPI; 25 MHz on longer cables (auto-derated, see callout above).
SlaveListens for nCS-falling-edge, decodes incoming MOSI bytes against a YAML-defined profile (flash, sensor, modem), produces the matching MISO response in real time. Profiles live under firmware/app/profiles/ — see edevkit1.html §6.5 for the schema.~25 MHz comfortably; up to ~50 MHz with profile-specific tuning (must pre-compute most responses).
SnifferAll output drivers tristated; the PIO program just samples SCK, MOSI, MISO, nCS at the full PIO clock and reports edges over USB. Useful for capturing two-party transactions you don't control.~37 MHz SCK with 4× oversampling; ~75 MHz SCK with 2× oversampling and aggressive PIO timing.

CLI examples

# bring the engine up as an SPI master in mode 0, 8-bit MSB-first, 25 MHz
edevkit spi master --mode 0 --bits 8 --msb --hz 25000000

# probe the slave's JEDEC ID (0x9F + 3 dummy reads)
edevkit spi xfer 0x9F 0xff 0xff 0xff
# → reads back: 0x9F 0xEF 0x40 0x18  (manufacturer + device id)

# emulate a W25Q128 from the bundled profile
edevkit spi slave --profile flash/w25q128

# sniff someone else's bus passively
edevkit spi sniff --hz-hint 10000000 --decode

# concurrent: master on connector A + sniffer on connector B
edevkit spi master --conn A --hz 10000000 &
edevkit spi sniff  --conn B

Code — Zephyr SPI API, master transaction

The Zephyr SPI API is documented at the official site under hardware/peripherals/spi. The modern (post-3.0) idiomatic style uses devicetree-resolved specs (struct spi_dt_spec); the older direct-config style is still supported for cases where the slave isn't in the devicetree.

Devicetree binding for an SPI slave

/* boards/rpi_pico2.overlay — declare a generic spi-device child of spi1 */

&spi1 {
    status = "okay";
    clock-frequency = <25000000>;
    cs-gpios = <&gpio0 7 GPIO_ACTIVE_LOW>;

    flash0: w25q128@0 {
        compatible = "jedec,spi-nor";
        reg = <0>;
        spi-max-frequency = <50000000>;
        size = <0x8000000>;        /* 128 Mbit = 16 MiB */
        jedec-id = [ ef 40 18 ];
    };
};

Master-side transaction in C

/**
 * @brief Read JEDEC ID from an SPI flash chip.
 *
 * Performs a single SPI transaction:
 *   nCS↓  → 0x9F (cmd) → dummy 0xFF × 3 → nCS↑
 * The slave responds on MISO with the 3-byte JEDEC ID after the
 * command byte has been clocked in.
 */
#include <zephyr/device.h>
#include <zephyr/drivers/spi.h>
#include <zephyr/logging/log.h>

LOG_MODULE_REGISTER(spi_demo, LOG_LEVEL_INF);

/* Resolve the spi-device declared in the devicetree overlay above. */
static const struct spi_dt_spec flash =
    SPI_DT_SPEC_GET(DT_NODELABEL(flash0),
                    SPI_OP_MODE_MASTER | SPI_TRANSFER_MSB |
                    SPI_WORD_SET(8) /* 8-bit words */,
                    0 /* delay before/after, ns */);

int spi_read_jedec_id(uint8_t out_id[3])
{
    uint8_t tx_buf[4] = { 0x9F, 0xFF, 0xFF, 0xFF };
    uint8_t rx_buf[4] = { 0 };

    const struct spi_buf tx = { .buf = tx_buf, .len = sizeof(tx_buf) };
    const struct spi_buf rx = { .buf = rx_buf, .len = sizeof(rx_buf) };
    const struct spi_buf_set tx_set = { .buffers = &tx, .count = 1 };
    const struct spi_buf_set rx_set = { .buffers = &rx, .count = 1 };

    if (!spi_is_ready_dt(&flash)) {
        LOG_ERR("SPI bus not ready");
        return -ENODEV;
    }

    int rc = spi_transceive_dt(&flash, &tx_set, &rx_set);
    if (rc) {
        LOG_ERR("spi_transceive failed: %d", rc);
        return rc;
    }

    /* rx_buf[0] is don't-care (clocked while master sent 0x9F);
     * rx_buf[1..3] hold the JEDEC ID. */
    out_id[0] = rx_buf[1];   /* manufacturer */
    out_id[1] = rx_buf[2];   /* device id high */
    out_id[2] = rx_buf[3];   /* device id low / capacity */

    LOG_INF("JEDEC ID = %02x %02x %02x",
            out_id[0], out_id[1], out_id[2]);
    return 0;
}

A few details worth noting in the snippet above. The SPI_OP_MODE_MASTER | SPI_TRANSFER_MSB | SPI_WORD_SET(8) bitfield is how Zephyr encodes mode/order/word-size into a single uint16_t. SPI_DT_SPEC_GET resolves the cs-gpios + max-frequency from the devicetree node, so the C code never has to hard-code GPIO numbers. spi_transceive_dt drives MOSI from tx_set and captures MISO into rx_set in lock-step; passing NULL for either side is allowed if you don't need it.

Direct (non-devicetree) configuration

If your slave isn't in the devicetree (e.g. a bench instrument hooked up at runtime), you can populate struct spi_config directly:

const struct device *spi_dev = DEVICE_DT_GET(DT_NODELABEL(spi1));
struct spi_cs_control cs_ctrl = {
    .gpio = GPIO_DT_SPEC_GET_BY_IDX(DT_NODELABEL(spi1), cs_gpios, 0),
    .delay = 0,
};
struct spi_config cfg = {
    .frequency = 1000000U,
    .operation = SPI_OP_MODE_MASTER | SPI_TRANSFER_MSB
               | SPI_WORD_SET(8) | SPI_MODE_CPOL /* set for mode 2/3 */,
    .cs        = cs_ctrl,
};

spi_transceive(spi_dev, &cfg, &tx_set, &rx_set);

Setting SPI_MODE_CPOL alone gives you Mode 2; combined with SPI_MODE_CPHA you get Mode 3; neither flag → Mode 0; CPHA only → Mode 1. The full encoding is in Zephyr's SPI API documentation.

Verifying with edevkit1 in the loop

If the target is misbehaving, the natural workflow is to put edevkit1 on the bus as a sniffer and capture exactly what the master and slave are doing wire-for-wire:

# on the host, with edevkit1 wired to the target board's SPI bus
edevkit spi sniff --hz-hint 25000000 --decode --output capture.csv

# trigger the transaction on the target
# (e.g. reset-and-boot, or a CLI command on your DUT)

# stop the sniff with Ctrl-C and inspect
column -s, -t capture.csv | head -50
# → sees CMD=0x9F, then ID=ef 40 18 — confirms slave is alive

Common pitfalls & first-time bugs

#SymptomCauseFix
1Reads all 0xFFnCS not asserted, slave never sees a transaction. Or MISO is floating because no slave is driving.Logic-analyze nCS. Confirm GPIO polarity (active-low!). Confirm pin connection.
2Reads all 0x00MISO held low — possibly a second slave on the bus failing to tristate, or wrong-polarity nCS keeping the real slave deselected.Disconnect every other slave. If the issue persists, your nCS is inverted.
3Data shifted by one bitCPHA wrong — sampling on the shift edge instead of the latch edge.Toggle CPHA. Re-test. Half the time the datasheet timing diagram ends up actually being Mode 1, not Mode 0.
4Data bit-reversedBit order mismatch (MSB vs LSB first).Set the controller's transfer order to match the slave's datasheet, or bit-reverse in software at the protocol layer.
5First byte garbage, rest fineSlave hasn't loaded its first byte before the first SCK edge. Sometimes a "wakeup" issue.Insert one dummy byte at the start of the transaction, or add a few microseconds of delay between nCS↓ and SCK starting.
6Multi-byte command corrupts the slaveMaster toggles nCS between bytes (controller bug or a misconfigured "auto-CS-per-word" mode).Force the controller to hold nCS low for the entire transaction — usually a flag like SPI_HOLD_ON_CS or equivalent.
7Works at 1 MHz, fails above 10 MHzCable length is killing your timing budget. Round-trip propagation now exceeds T_SCK/2.Either shorten the cable, slow down, or enable controller-side RX-delay compensation.
8Slave-mode underflow — master sees random 0xFF in the streamYour slave-mode firmware can't keep the TX FIFO fed. Common at >5 MHz SCK with interrupt-per-byte.Switch to DMA, hardware FIFO, or PIO. Pre-compute responses on the previous transaction's command-decode.
9Adding a second slave breaks the first oneSecond slave doesn't tristate MISO when deselected — bus contention.Verify with a logic analyzer that MISO is Hi-Z when neither nCS is asserted. If not, return the second part to the vendor or add an external tristate buffer.
10Random failures only when the radio / motor is onSI / EMI. Push-pull edges plus an unshielded radio nearby.Add ground reference between data lines, route SCK between two ground traces, slow down, or shield the cable.
Triage order When SPI fails for the first time on new hardware, before you read any code: (a) verify with a multimeter that VCC is at the slave's pins; (b) verify with a logic analyzer that nCS goes low and SCK toggles; (c) with the analyzer's SPI decoder, set CPOL/CPHA to all four combinations until one decodes the JEDEC ID / device ID correctly. Then go fix the firmware to match what the analyzer told you. This sequence catches ~80% of "why doesn't my SPI work" before any debugger session.

Where to learn more

Once you've internalised the basics above, the resources below cover deeper or more practical territory.

  1. Sparkfun — Serial Peripheral Interface (SPI) tutorial. The classic introduction. Diagrams of all four modes, simple slave-vs-master sketches, the COPI/CIPO terminology rationale. Good 30-minute read for a beginner; good 5-minute refresher for an experienced engineer.
  2. Wikipedia — Serial Peripheral Interface. The closest thing to a normative reference for SPI. Covers all four modes, daisy-chain, multi-IO, history. Notably better-cited than most tutorials.
  3. Corelis — SPI tutorial. Short and timing-diagram-heavy; good if you want a second source for the CPOL/CPHA edge alignments.
  4. Total Phase — Understanding SPI clock signals (2024). Modern, careful walk-through of the four modes with worked examples. Aligns with this doc's mode-numbering convention.
  5. Analog Devices — Introduction to SPI Interface. ADI's house tutorial, with focus on ADC/DAC integration. Useful if you're hooking SPI to a high-rate analog part — ADI parts dominate that niche.
  6. Raspberry Pi Pico C/C++ SDK — SPI chapter. The canonical reference for the SPI hardware on the RP2040 and RP2350, including the controller register interface. Reading this together with platform_pio.html covers both edevkit1's "real SPI controller" path and its PIO-implemented path.
  7. Zephyr — SPI bus documentation. The Zephyr API used in the code snippets above, including spi_transceive, spi_dt_spec, devicetree bindings, and the operation-flag encoding.
  8. Winbond W25Q128JV datasheet and Macronix MX25R datasheet. The two most-shipped SPI flash families, both excellent reference command sets. If you only ever read one SPI flash datasheet, make it W25Q128JV — JEDEC IDs are EF 40 18 for W25Q128 and C2 28 18 for MX25R128, useful for sniffer-decode validation.
  9. JEDEC JESD216 — Serial Flash Discoverable Parameters (SFDP). The standard for "ask the chip what it can do" via the 0x5A command. Mandatory reading before writing a generic SPI flash driver that has to support unknown parts.
  10. edevkit1.html §6.5 (protocol engines) and §4.2 (capture buffers). The v1 design spec sections that locate SPI in the broader edevkit1 architecture. Always re-read these before changing the SPI engine — they encode constraints the code is expected to respect.
  11. Sister docs in this hub: proto_uart.html, proto_i2c.html, proto_qspi.html, platform_pio.html.
