edevkit · firmware · daily workflow

Firmware development workflow

Everything you do dozens of times a day while iterating on edevkit firmware: build, flash, watch RTT logs, decode crashes, and confirm the device enumerated. Set up once, then it's a three-line cycle.

One-time setup

Required environment variables

Add these to ~/.zshrc if they're not already there. Pin them to your actual paths:

# edevkit — Layout A (wrapper-dir at ~/projects/temp/edevkit/)
export EDEVKIT_REPO="$HOME/projects/temp/edevkit/edevkit"
export EDEV_VENV="$HOME/projects/temp/edevkit/edev"
export ZEPHYR_WORKSPACE="$HOME/projects/temp/zephyrproject"
export ZEPHYR_SDK_INSTALL_DIR="$HOME/zephyr-sdk-1.0.1"

# Override stale launchctl value (some Nordic/SEGGER installers set this
# to literal '${HOME}/zephyrproject/zephyr' which never expands).
export ZEPHYR_BASE="$ZEPHYR_WORKSPACE/zephyr"
macOS launchctl quirk — read this

Several SDK installers (Nordic Connect for VS Code, SEGGER tools) run launchctl setenv ZEPHYR_BASE '${HOME}/zephyrproject/zephyr' with single quotes. macOS stores the literal string without expanding ${HOME}, and inherits it into every shell. west build then can't find your workspace and reports the misleading "unknown command 'build'; do you need to run this inside a workspace?". The export ZEPHYR_BASE line above overrides it. To remove the bad value at its source: launchctl unsetenv ZEPHYR_BASE.

Install probe-rs

Used for both flashing and RTT streaming. One install covers all three OSes:

cargo install probe-rs-tools --locked
# or grab a binary release from https://probe.rs/

The edev_* shell helpers

The edevkit project provides convenience wrappers for west, probe-rs, and other tools. See shell_helpers.html for the full reference (function signatures, defaults, examples). The helpers are sourced automatically when you activate the Python venv — no manual shell-rc edits needed.

HelperWhat it doesNeeds (edev) venv?
edev_buildwest build -b $EDEV_BOARD .Yes (west is Python in venv)
edev_rebuildPristine rebuild (sysbuild + signing)Yes

Stale helper definitions on macOS?

If you see unexpected edev_* behavior (e.g., old function signatures, missing error checks), run type -a edev_flash to find shadow definitions lurking in ~/.zshrc, ~/.bashrc, or the venv's activate script. Remove any stale blocks — they're incompatible with the current canonical version in $EDEVKIT_REPO/tools/env_helpers.sh. This trap bit us on 2026-05-08 when we unified helper sourcing.

Daily cycle

From any sample directory after a code edit:

cd firmware/samples/usb/usb_01      # or whichever sample
edev_build && edev_run              # flash + stream RTT
# Ctrl-C to detach RTT

If you only want to peek at logs without reflashing:

edev_rtt_monit

If you only want to flash without watching logs:

edev_flash

RTT — log monitoring via Pico debug probe

What RTT is

Real-Time Transfer (Segger). The MCU writes log messages into a small ring buffer in SRAM; a host-side tool reads that buffer over SWD via the debug probe. No UART pins burned, no host serial driver needed, no impact on the device under test. Standard Zephyr support — flip three Kconfigs and you have it.

Per-sample prj.conf requirements

CONFIG_LOG=y
CONFIG_USE_SEGGER_RTT=y
CONFIG_RTT_CONSOLE=y
CONFIG_UART_CONSOLE=n
CONFIG_USBD_LOG_LEVEL_INF=y      # if you want USB-stack logs

Wiring

The Pico debug probe (Raspberry Pi Debug Probe, or a second Pico flashed with the debugprobe firmware) needs:

  • Probe GND ↔ Target GND
  • Probe SC ↔ Target SWCLK
  • Probe SD ↔ Target SWDIO

RTT goes over SWD — you do NOT need to wire the probe's UART pins for RTT to work.

Reading the stream

CommandBehavior
probe-rs run --chip RP2350 path/to/zephyr.elfHalt + program flash + reset + stream RTT (= edev_run, which goes through edev_flash for the proper MCUboot+app split rather than probe-rs run's single-ELF flash).
probe-rs attach --chip RP2350 path/to/zephyr.elfConnect to running target, stream RTT only — no reset, no reprogram (= edev_rtt_monit).
probe-rs reset --chip RP2350Reset the target without doing anything else
probe-rs listList detected probes; useful when multiple probes are connected

Coredump — analyzing a Zephyr fault dump

When the firmware faults (bus error, hard fault, undefined instruction, etc.), Zephyr prints a register dump and halts. Example from the usb_01 bring-up:

[00:00:00.001,000] <err> os: ***** BUS FAULT *****
[00:00:00.001,000] <err> os:   Precise data bus error
[00:00:00.001,000] <err> os:   BFAR Address: 0xffffff00
[00:00:00.001,000] <err> os: r0/a1:  0x1000ca75  r1/a2:  0x00000000  r2/a3:  0x1000ca79
[00:00:00.001,000] <err> os: r3/a4:  0xffffff00 r12/ip:  0x00000001 r14/lr:  0x10003a25
[00:00:00.001,000] <err> os:  xpsr:  0x29000000
[00:00:00.001,000] <err> os: Faulting instruction address (r15/pc): 0x10003a3e
[00:00:00.001,000] <err> os: >>> ZEPHYR FATAL ERROR 25: Unknown error on CPU 0
[00:00:00.001,000] <err> os: Current thread: 0x20001248 (main)

What the fields mean

FieldMeaning
BFAR AddressBus Fault Address Register — the address being dereferenced when the fault hit. Often the smoking gun.
PC (r15)Faulting instruction's address. addr2line this to find the source line.
LR (r14)Link register — return address of the function that called the faulting one. addr2line to find the caller.
r0..r3First four function arguments + scratch. Often contain the offending pointer or value.
xpsrProgram status register; mostly useful for confirming Thumb mode (bit 24) and active exception number.
Current threadWhich thread was running. main = boot-time crash; other names = a runtime issue in that thread.
FATAL ERROR 25Zephyr fatal-error code. 25 = "unknown / unhandled" — the actual cause is in the BFAR/PC fields above, not the code.

Recipe — find the source line

Map PC (and LR) to source file:line using the SDK toolchain's addr2line:

TOOL=$ZEPHYR_SDK_INSTALL_DIR/gnu/arm-zephyr-eabi/bin/arm-zephyr-eabi-addr2line
ELF=build-rpi_pico2_rp2350a_m33/<sample-name>/zephyr/zephyr.elf

$TOOL -e $ELF -fpC 0x10003a3e 0x10003a25
# -f: show function name
# -p: pretty single-line format
# -C: demangle C++ symbols (harmless on plain C)

Output:

init_configuration_inst at .../subsys/usb/device_next/usbd_init.c:135
init_configuration_inst at .../subsys/usb/device_next/usbd_init.c:127

The first line is where it crashed; the second is the caller.

Disassembly around the crash

If addr2line doesn't pinpoint enough (inlining, optimization), drop to objdump:

OBJDUMP=$ZEPHYR_SDK_INSTALL_DIR/gnu/arm-zephyr-eabi/bin/arm-zephyr-eabi-objdump
$OBJDUMP -d --start-address=0x10003a30 --stop-address=0x10003a60 $ELF

Common bus-fault patterns

BFAR valueLikely cause
0x00000000NULL pointer dereference — uninitialized struct or function pointer
0xFFFFFFXXNegative errno being treated as a pointer (e.g., -22 = 0xFFFFFFEA)
Some flash addressWriting to a const object that the linker placed in flash (a real cause of usb_01's second crash)
Some RAM address that looks structuredWalking off the end of an array or following a corrupted pointer
Peripheral MMIO addressTouching a clock/peripheral that isn't enabled in devicetree

USB device verification

After the firmware boots, confirm the host actually enumerated it.

Linux 
lsusb -v -d 1209:ede1            # full descriptor dump
sudo dmesg -w | grep -i usb      # live kernel events
sudo wireshark -i usbmon0        # packet capture (loaded modprobe usbmon)
macOS 
# PRIMARY (always reliable, reads kernel I/O Registry live):
ioreg -p IOUSB -l -w 0 | grep -B 2 -A 25 "edevkit"

# system_profiler is more readable but CACHES — sometimes lags by minutes
# on Apple Silicon.  Force refresh and re-query:
sudo killall -HUP USBHostBSDClient
system_profiler SPUSBDataType | grep -B 1 -A 12 "usb_01"

# packet capture: Apple's PacketLogger from "Additional Tools for Xcode"
Windows 
:: Device Manager-style listing (PowerShell)
Get-PnpDevice -PresentOnly |
  Where-Object { $_.InstanceId -like "*VID_1209*" }

:: Graphical descriptor view (Windows SDK)
usbview.exe

:: Packet capture: install USBPcap, then capture via Wireshark
macOS gotchas

On Apple Silicon Macs with Sonoma/Sequoia, system_profiler SPUSBDataType often shows stale or missing devices for several minutes after a re-plug. Always cross-check with ioreg — it reads the kernel I/O Registry live and is the source of truth. system_profiler also reports idProduct as decimal in some output paths, so grepping for 0xede1 can miss devices that ARE enumerated. Grep by manufacturer or product string instead.

Useful alias

alias edev_lsusb='ioreg -p IOUSB -l -w 0 | grep -B 2 -A 25 "edevkit"'

Sanity-check checklist

When something doesn't work, walk this list top-to-bottom:

  1. echo $ZEPHYR_BASE — must print a path that exists; not literal ${HOME}/...
  2. which west — must be inside the (edev) venv
  3. probe-rs list — your probe must show up
  4. edev_build — exits 0
  5. edev_run — flashes; you see the Zephyr boot banner in RTT (the helper surfaces probe-rs reset failures, see shell_helpers.html)
  6. ioreg -p IOUSB ... | grep edevkit — device shows up on host

If step N fails, the bug is between step N-1 and step N — that's where to focus.

Roadmap — desktop GUI for the workflow

Planned · not yet implemented

The shell helpers above are the bare-minimum daily loop. They work, but a real desktop GUI would:

  • Sample browser — list every sample under firmware/samples/usb/ with build status and last-flashed-when timestamp
  • Build button per sample — wraps west build, shows progress + warnings + errors inline (clickable to source)
  • Flash button — wraps probe-rs download, shows progress, validates against connected probe
  • RTT pane — live log streaming with timestamp + level filtering, search, save-to-file, log-color per module
  • Coredump analyzer — when a fault appears in RTT, automatically run addr2line on the PC/LR and link to the source
  • USB device tree — live view of ioreg/lsusb filtered to edevkit VID, with descriptor decode
  • Probe selector — when multiple probes are connected, pick which one is which target

Implementation will likely live in edevkit/host/gui/ as a Tauri app (Rust + web frontend) — same shell as the planned probe/protocol-tester GUI from the v1 design. Until that exists, the edev_* shell helpers above are the canonical workflow.
