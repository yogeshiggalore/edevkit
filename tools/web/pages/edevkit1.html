01Executive Summary

edevkit1 is an open-source embedded development tool that combines on a single board:

Debug probe
CMSIS-DAP v2 — SWD & JTAG for ARM Cortex + RISC-V
SWO trace
Real-time ITM/printf trace from CoreSight
Dual UART
Target log + dev/test, independent voltage domains
I²C tester
Master / slave / sniffer at any address
SPI / DSPI / QSPI
All modes, master/slave/sniffer, up to 50 MHz
Device emulator
Fake flash, sensors, AT modems via YAML profiles
USB-C composite
CMSIS-DAP + 4× CDC + WebUSB on one cable
Tauri GUI + CLI
Cross-platform, scriptable, OTA-updatable

Built around the RP2350B (Cortex-M33 dual-core, 12 PIO state machines, 520 KB SRAM) running Zephyr RTOS 4.x with MCUboot for OTA updates over USB. Firmware delivers all protocols using PIO programs for wire-level timing, with HW UARTs for the two UART channels.

Why build instead of buy? A J-Link Plus + Saleae Logic + Total Phase Aardvark + custom emulator scripts costs ~$1,500 per engineer and still doesn't integrate. edevkit1 hits ~$33/unit BOM, single tool, single GUI — and the device-emulation feature isn't in any off-the-shelf product.

What's locked in for v1

Hardware

  • RP2350B (QFN-80, 48 GPIO)
  • 16 MB boot flash + 128 MB data flash
  • Mixed level shifters (LVC8T245 + TXS0108E + LVC1T45)
  • TPS25940 eFuse on Vtgt (programmable current limit)
  • USB-C (FS) with common-mode choke + ESD
  • WS2812 LED chain, 4 buttons via ADC ladder, buzzer
  • 4-layer FR4 PCB, 60 × 85 mm, ENIG, controlled impedance
  • 3D-printed PETG enclosure

Firmware

  • Zephyr 4.x with USB-Device-Next composite
  • SMP across both Cortex-M33 cores
  • CMSIS-DAP v2 (SWD/JTAG/SWO)
  • MCUboot, ECDSA-P256 signed images, swap & revert
  • SMP server over USB CDC for OTA
  • YAML-based device emulation engine
  • LittleFS for configs & calibration
  • Per-unit serial from RP2350 chip ID
  • Hardware watchdog + panic dump capture

What's deferred to v2

  • v2 8-channel logic analyser (sigrok-compatible via SUMP protocol)
  • v2 CAN, RS-485
  • v2 USB-C PD high-current target power
  • v2 cJTAG & WCH SWIO RISC-V wire protocols
  • v2 Standalone OLED + rotary encoder
  • v2 Manchester-encoded SWO; ETM/parallel trace
  • v2 Multi-target switch (debug 2 boards from one probe)

02Goals & Use Cases

Primary goals

  1. One tool on every engineer's desk. Replace a pile of debugger + scope + logic-analyser + protocol-bridge boxes with a single integrated device.
  2. Unified host experience. Debug, log, sniff, emulate — all in one GUI with shared timeline and correlated events.
  3. Test drivers without real hardware. The device-emulation engine lets developers exercise their I²C / SPI / UART driver code against fake-but-realistic peripherals before the actual silicon arrives.
  4. OTA-updatable. New device profiles, new emulation models, new protocol modes shipped without re-flashing via probe.
  5. Open hardware. KiCad source, Zephyr firmware, Tauri GUI — all in git, all reproducible, all maintainable by the community.

04System Architecture

The host computer (developer's laptop) connects to edevkit1 over a single USB-C cable. The edevkit1 firmware exposes a USB composite device with multiple endpoints, each serving a different logical role. On the target side, edevkit1 has separate physical connectors per protocol, each electrically isolated and voltage-tolerant via dedicated level shifters.

05Hardware Design

5.1 MCU choice — RP2350B

Why RP2350B (not RP2350A)

  • 48 GPIO (vs 30 on RP2350A) — needed for our pin budget
  • QFN-80, 10 × 10 mm (vs 60-QFN, 7 × 7 mm)
  • ~$0.50 cost premium, negligible
  • Same dual Cortex-M33 + Hazard3 RV32, same 12 PIO state machines, same 520 KB SRAM

Why RP2350 family (not STM32 / nRF / etc.)

  • 12 PIO state machines — every protocol gets its own dedicated bit-banger; no fighting for hardware peripherals
  • Dual Cortex-M33 @ 150 MHz with TrustZone — fast, supports SMP cleanly
  • 520 KB SRAM — large ring buffers for sniffer, SWO, SMP
  • Cheap (~$2 in qty 100); strong supply
  • Mature pico-sdk + Zephyr support; large community
  • Open silicon ethos aligns with open-hardware project

5.2 PIO state-machine budget

RP2350 has 3 PIO blocks × 4 state machines = 12 SMs total. Allocation for v1:

PIO0 (debug + low-rate proto) PIO1 (high-rate protocol tester) PIO2 (reserved for v2) SM0 → SWD / JTAG (mode mux) SM0 → I²C (master/slave/sniff) SM0 → free / v2 LA ch0–3 SM1 → SWO RX (UART decode) SM1 → SPI / DSPI / QSPI primary SM1 → free / v2 LA ch4–7 SM2 → WS2812 LED chain SM2 → QSPI partner (CLK + CS) SM2 → free / v2 trigger SM3 → free SM3 → Sniffer (concurrent w/ M/S) SM3 → free / v2 timestamp
Why this is roomy. Earlier drafts crammed everything into 2 PIO blocks and ran out. With LA deferred to v2 (whole PIO2 free) and SPI/QSPI mode-shared (one connector, mode-select), there are 5 free SMs in v1 — plenty of headroom for protocol sniffer concurrency, runtime profile changes, and v2 features without re-spinning hardware.

5.5 Level shifters

Two shifter families used, picked by speed and direction need:

UsePartChannelsDirectionMax speed
SWD/JTAG, SPI/QSPI (high speed, bidirectional)SN74LVC8T2458Explicit DIR pin~150 MHz
I²C (open-drain, bidirectional)TXS0108E8Auto-direction~50 MHz
UART (RX, TX unidirectional)SN74LVC1T451 eachStatic DIR (tied)~150 MHz

Mixed strategy keeps cost down (auto-dir is cheaper to drive but slower; explicit DIR is more PCB-area but faster).

Vtgt vs Vio domains. Debug header references target's voltage (Vtgt) — auto-detected via ADC. Protocol test headers reference edevkit-selected Vio (1.8 / 3.3 V) — set by user in GUI. These are independent domains, intentionally — your target can be 1.8 V while the I²C device under test is 3.3 V, and vice versa.

5.6 Power architecture

USB-C VBUS (5 V, 500 mA host budget) │ ├──► Common-mode choke (Würth 744232090) ──► USB D+/D- to RP2350 │ ├──► ESD protection (USBLC6-2) │ └──► TPS62933 buck (5V → 3.3V, 2A) │ ├──► RP2350 + flash + level shifters (3.3 V system) │ ├──► TLV70218 LDO (3.3V → 1.8V) │ │ │ └──► Vio = 1.8V option │ ├──► Vio mux (3.3V or 1.8V, switched by GPIO28) │ │ │ └──► Vio rail to protocol headers + their level shifters │ └──► TPS25940 eFuse → Vtgt output │ ├──► Programmable current limit (250 mA default) ├──► OCP / OVP / UVLO ├──► Auto-retry on fault └──► INA180 current sense → ADC2 Vtgt is independent from Vio — Vtgt goes to debug header (target's reference V), Vio goes to protocol headers (edevkit-selected reference V).
Power-on sequencing (firmware-managed)
  1. USB connect → 5 V applied
  2. RP2350 boots, runs Stage 2 → MCUboot → app
  3. Firmware enables 3.3 V buck (already on by default — for self-power)
  4. Firmware enables 1.8 V LDO if needed
  5. Firmware sets Vio mux to default 3.3 V (or last-saved value from LittleFS)
  6. Firmware sets all level-shifter DIR pins to safe defaults (input)
  7. Firmware enables level-shifter OE
  8. Firmware enables Vtgt eFuse load switch (only after USB enumerates and host configures)

This sequence avoids glitching target signals during boot, even if a target board is plugged in at power-up.

Raw block device, addressed by slot ID from host:

  • Emulation images: developer uploads firmware images for flash emulation, gets slot handle, references slot in profile (--image-slot 0x05)

  • Stimulation streams: sensor replay CSVs, GPS routes, AT-command scripts uploaded for emulator use
  • Future v2: standalone target firmware images (field-flasher mode without host)

5.9 Protection circuitry

ESD on every external pin

  • USB D+/D− → USBLC6-2 (low-cap, 4-channel)
  • SWD lines → SP0524 ESD array
  • UART, I²C lines → SP0504 / PESD3V3 arrays
  • SPI/QSPI 6-line group → SP0524 + PESD on extras
  • Vtgt input → bidirectional 6 V TVS clamp

Total: ~$0.50 BOM for 4 ESD diode arrays. Cheap insurance.

Vtgt output protection

  • TPS25940 eFuse — programmable current limit (default 250 mA, configurable to 500 mA)
  • OCP, OVP, UVLO
  • Auto-retry on fault clear
  • OPEN-DRAIN fault output → MCU IRQ
  • INA180 high-side current sense → ADC2 → live current shown in GUI

~$1.50 BOM. Saves the board if developer shorts Vtgt to GND.

Reverse-polarity / overvoltage on inputs

  • Bidirectional clamp diodes on each Vtgt-domain input pin
  • Series PTC fuse on Vtgt output (backup to eFuse)
  • Vio rail isolated by load switch with reverse-blocking

USB-C input

  • USBLC6-2 ESD on D+/D−
  • Common-mode choke (Würth 744232090)
  • VBUS clamp Schottky for back-EMF
  • VBUS sense on ADC3 — detect cable yank

5.10 PCB stackup

4-layer, 1.6 mm, 1 oz copper, FR4, ENIG finish, controlled impedance. Standard JLCPCB / PCB Power / Lion Circuits stackup.

┌─────────────────────────────────────────────────────────────┐ │ L1 (top) SIGNAL 1 oz copper, ENIG, soldermask│ ├─────────────────────────────────────────────────────────────┤ 0.2 mm prepreg │ L2 GND PLANE 1 oz, solid pour │ ├─────────────────────────────────────────────────────────────┤ 1.0 mm core (FR4) │ L3 POWER PLANES 1 oz, split: 3.3V / 1.8V / Vio│ ├─────────────────────────────────────────────────────────────┤ 0.2 mm prepreg │ L4 (bottom) SIGNAL + GND 1 oz, secondary, ENIG │ └─────────────────────────────────────────────────────────────┘ Total: 1.6 mm
Routing rules
  • USB D+/D−: differential pair on L1, 90 Ω diff impedance, length-matched ±0.5 mm, reference L2 GND
  • QSPI signals (boot + data flash): on L1, 50 Ω SE impedance, <5 mm trace length, reference L2 GND, length-matched
  • SWD/JTAG: on L1, <25 MHz, length-matched within ±2 mm
  • Crystal: on L1 above L2 GND, stitching vias around
  • Power: L3 split into 3.3 V / 1.8 V / Vio zones; bridged where needed
  • L2 (GND) is sacred — no traces. Continuous pour.
  • Via-in-pad for RP2350 QFN-80 fanout (resin filled + plated)
Board dimensions
  • Outline: 60 × 85 mm
  • Mounting: 4× M2.5 with 4 mm clearance from edge
  • USB-C: top edge, centered
  • Front panel: connectors along one long edge (Cortex Debug + TC2030 + Target UART grouped)
  • Side panel: protocol test connectors (Test UART + I²C + SPI/QSPI) grouped

6.2 Multicore (SMP) thread layout

Both Cortex-M33 cores run a single Zephyr SMP image. Threads are pinned to specific cores to keep USB latency tight.

Core 0 — USB / debug

  • cmsis_dap_dispatcher
  • swo_rx_pump
  • target_uart_pump
  • test_uart_pump
  • shell_thread
  • smp_server_thread

Core 1 — protocol / emulation

  • i2c_engine_thread
  • spi_engine_thread
  • qspi_engine_thread

Cross-core IPC primitives

PrimitiveUseNotes
k_msgqCommand/event passing (host→engine, engine→USB)Default for almost all IPC
ring_buf (in .nocache)SWO trace stream, sniffer events, high-throughput dataSPSC; cache-flushed at producer side
k_spinlock + shared structVoltage state, mode flags, emulation_active bitRead often, written rarely

Cache coherency. Cortex-M33 has 4 KB I-cache + 4 KB D-cache per core. Zephyr kernel objects (k_msgq, k_pipe, ring_buf) handle this internally. Hand-rolled shared structs need sys_cache_data_flush_range() at the producer and sys_cache_data_invd_range() at the consumer, or place them in a .nocache linker region.

6.3 PIO programs

Each protocol has its own .pio file assembled at build time by pioasm (vendored under tools/). The build emits a header per file with pio_program_t + entry-point offsets. Zephyr's pio_rpi_pico driver loads the program into a state machine.

Example — SWD program (simplified) 
; src/pio/swd.pio
.program swd
.side_set 1 opt

public write_cmd:
    out  x, 32           side 0     ; bit count - 1 into X
    out  pindirs, 1      side 0     ; SWDIO = output
write_bit:
    out  pins, 1         side 0 [1] ; drive bit, clock low
    jmp  x--, write_bit  side 1 [1] ; clock high, dec X

public read_cmd:
    out  x, 32           side 0
    out  pindirs, 1      side 0     ; SWDIO = input
read_bit:
    in   pins, 1         side 0 [1] ; sample on clock low
    jmp  x--, read_bit   side 1 [1]
    push                 side 0

This is identical in spirit to raspberrypi/debugprobe's probe.pio. With clkdiv tuned, SWCLK reaches 25 MHz on RP2350.

6.6 Device emulation engine

The emulation engine runs on Cortex-M33 #1 and lets developers test their host-side drivers without the real peripheral hardware. Three target categories are supported in v1:

SPI/QSPI flash

Emulate W25Qxx, MX25Lxx and others. JEDEC commands decoded, backing data served from data flash. Programmable response delays + error injection.

I²C sensor

Emulate LSM6DSOX, BMP280, SHT4x, etc. Register bank in SRAM, optional waveform-replay stream from CSV.

UART AT-command device

Emulate ESP-AT, SIM7000, NMEA GPS. Pattern-matching command table with delay + multi-line responses.

13RISC-V Support

v1 supports RISC-V via standard JTAG out of the box — no hardware changes, no extra firmware.

How it works

Host (probe-rs / OpenOCD with riscv driver) │ │ CMSIS-DAP DAP_JTAG_Sequence (raw JTAG bit shifting) ▼ edevkit firmware (Zephyr) │ │ PIO0 SM0 in JTAG mode ▼ RISC-V target's JTAG TAP

The probe doesn't need to "understand" RISC-V — it's a transparent JTAG transport. The host tool implements the RISC-V Debug spec on top.

Targets that work on v1 today

  • ESP32-C3, ESP32-C6, ESP32-H2 (external JTAG mode)
  • RP2350's own Hazard3 RV32 cores
  • SiFive HiFive, FE310, U540
  • K210, GD32V, BL808, MAX78000
  • Most parts probe-rs or OpenOCD has a config for

Deferred to v2

  • cJTAG (compact JTAG, 2-pin) — used by some Andes-Tech parts; ~1 week firmware work
  • WCH SWIO (1-wire RISC-V debug) — used by CH32V chips; ~1 week firmware + ~1 week host integration

15Decision Log

Record of key architectural decisions made during design discussion, with rationale.

#DecisionRationale
D-1RP2350B over RP2040 / STM32 / nRF12 PIO state machines decisively wins for multi-protocol concurrent operation; dual M33 supports SMP cleanly; mature open-source community.
D-2RP2350B (80-pin) over RP2350A (60-pin)Pin budget exceeds 30 GPIO. RP2350A insufficient even with optimization.
