usb_04 — control transfers in depth

Design + code flow for the fourth sample in the USB Book series. Carries usb_03's descriptor tree forward byte-for-byte, then implements four vendor control requests on EP0 that exercise every shape a control transfer can take: 3-stage IN, 3-stage OUT, 2-stage no-data, and a deliberate STALL. This is the first sample where the device responds to commands — every prior sample only sat still while the host read descriptors out of it.

Authoring status — sample built 2026-05-11, paired with Ch 08

This doc is the locking spec for firmware/samples/usb/usb_04/ and stands in for USB Book Ch 08 ("Control transfers in depth") until that chapter is formally split out. Every API claim has been verified against $ZEPHYR_BASE/include/zephyr/usb/usbd.h and usb_ch9.h on Zephyr v4.4.99-ge2a556ff6417, and every Setup-packet field is traceable to USB 2.0 §9.3.

1. Goal & scope

One sentence: implement four custom vendor requests on EP0 — GET_UPTIME, SET_LOG_MSG, SET_LED_BLINK, STALL_ME — wired through Zephyr's control_to_host and control_to_dev class callbacks, so that the wire-level mechanics of every control-transfer shape (3-stage IN, 3-stage OUT, 2-stage no-data, deliberate STALL) become directly observable from a 20-line Python script on the host.

This is the first sample where the device does work in response to host requests. Up to now (usb_01..03) the device has only existed as a static blob of descriptor bytes that the host enumerates and then ignores. EP0 has been delivering exactly the standard requests GET_DESCRIPTOR and SET_ADDRESS; everything else has gone unanswered. usb_04 keeps the descriptor tree frozen and turns on the request path.

Pairs with USB Book Ch 08 — Control transfers in depth (chapter not yet split out; this design doc is the source-of-truth until it is).

Why this shape was chosen

Four requests is the minimum that hits every transfer shape exactly once. Adding a fifth would teach nothing new; dropping any one would leave a gap. The selection is pedagogical, not realistic — a production protocol would group commands by function and avoid the STALL_ME entirely. What you're learning here is the wire shape, not the API design.

What usb_04 does not cover

  • Bulk I/O. EP1 IN and EP1 OUT still NAK forever, same as usb_02/03. Bulk endpoints have their own dedicated sample later (likely a usb_05_bulk_loopback paired with Ch 13 "Endpoint buffers, MPS, ZLPs"). Mixing control and bulk in one sample would force the reader to debug two new things at once.
  • Standard or class requests. The Zephyr core handles all 11 standard device requests (USB 2.0 §9.4) — GET_STATUS, SET_ADDRESS, GET_DESCRIPTOR, SET_CONFIGURATION, etc. — without our intervention. Class-specific requests would only fire if we declared a recognised class (HID, CDC, MSC); we're vendor-class, so the host never issues them.
  • Asynchronous control transfers. Zephyr's callback model is synchronous: control_to_host must fill buf and return before the function exits. Long-running responses (read a sensor that takes 50 ms) need a different pattern — out of scope.
  • Interface alternate settings & multi-config. Slated for usb_31..35 (Ch 31).

3. The four vendor requests, byte-level

Every USB control transfer starts with an 8-byte Setup packet on EP0. The format (USB 2.0 §9.3, Table 9-2) is fixed:

Offset  Size  Field             Meaning
  0      1    bmRequestType     bit 7    : direction (0 = host→device, 1 = device→host)
                                 bits 5-6 : type      (0 std, 1 class, 2 vendor, 3 reserved)
                                 bits 0-4 : recipient (0 device, 1 interface, 2 endpoint)
  1      1    bRequest          opaque to USB — meaning defined by the class / vendor
  2      2    wValue            request-specific, little-endian
  4      2    wIndex            request-specific, little-endian; for INTERFACE recipient → iface number
  6      2    wLength           number of bytes to follow in the DATA stage (0 means no DATA stage)

For all four usb_04 requests, bmRequestType.type = VENDOR (2) and bmRequestType.recipient = INTERFACE (1). Only the direction bit and bRequest distinguish them. wIndex = 0 always (our one and only interface number).

3a. Setup-packet bytes per request

bRequestNamebmRequestTypewValuewIndexwLengthShape
0x01 GET_UPTIME 0xC1 0x0000 0x0000 0x0008 3-stage IN
0x02 SET_LOG_MSG 0x41 0x0000 0x0000 ≤ 0x0020 3-stage OUT
0x03 SET_LED_BLINK 0x41 count 0x0000 0x0000 2-stage (no DATA)
0x04 STALL_ME 0xC1 or 0x41 0x0000 0x0000 0x0000 SETUP → STALL

Decoding the bmRequestType bytes:

0xC1 = 0b 1 10 00001         → IN  | Vendor | Interface
0x41 = 0b 0 10 00001         → OUT | Vendor | Interface

3b. What each request teaches

RequestWire shapeWhat's new about it
GET_UPTIME SETUP → DATA (in, 8B) → STATUS (out, 0B) Three-stage host-IN. Device fills 8 bytes (a little-endian k_uptime_get() in ms) into the DATA stage. STATUS stage is a zero-length OUT from host → device that ACKs receipt. This is the same shape as GET_DESCRIPTOR, which is how every prior sample's descriptor reads worked — only the bRequest and the data source change.
SET_LOG_MSG SETUP → DATA (out, N≤32B) → STATUS (in, 0B) Three-stage host-OUT. Host sends N bytes of UTF-8 text in the DATA stage; device LOG_HEXDUMPs them. STATUS stage is a zero-length IN from device → host that ACKs successful receipt. The STATUS direction is always opposite to the DATA direction (USB 2.0 §8.5.3); that's how the host knows when the device has finished consuming the payload.
SET_LED_BLINK SETUP → STATUS (in, 0B) Two-stage host-OUT with no DATA stage. wValue carries the "count" payload (up to 16 bits). The entire transfer is just two packets on the wire. This is the most efficient control-transfer shape, used when payload fits in wValue+wIndex's 32 bits combined — common for "set a flag" / "trigger an action" requests in real protocols.
STALL_ME SETUP → STALL Deliberate protocol STALL. Device's control_to_* returns -ENOTSUP; the Zephyr core responds to the host with an EP0 STALL handshake. Host's libusb_control_transfer returns LIBUSB_ERROR_PIPE (-9). EP0 STALLs auto-clear on the next Setup packet per USB 2.0 §8.5.3.4 — unlike bulk-EP STALLs which need a CLEAR_FEATURE(ENDPOINT_HALT) to recover. That's the entire teaching point of this request: STALL on EP0 is cheap and self-healing.

4. The EP0 control-transfer state machine

This is the conceptual heart of Ch 08. Pause here.

USB control transfers are always two or three stages. Every control transfer begins with a SETUP transaction (the host sends an 8-byte Setup packet to EP0 OUT). What happens next depends on wLength and the direction bit in bmRequestType.

4a. Three shapes, side by side

3-stage host-IN  (wLength > 0, direction = 1)
                                                          GET_UPTIME, GET_DESCRIPTOR
   host                                  device
    │  ── SETUP    ─────────────────────►  │   8-byte Setup packet
    │  ◄──── DATA in (1..N packets) ─────  │   device fills, host receives
    │  ── STATUS out (ZLP) ──────────────► │   host ACKs by sending ZLP

3-stage host-OUT (wLength > 0, direction = 0)
                                                          SET_LOG_MSG, SET_CONFIGURATION
   host                                  device
    │  ── SETUP   ──────────────────────►  │   8-byte Setup packet
    │  ── DATA out (1..N packets) ──────►  │   host sends, device consumes
    │  ◄──── STATUS in (ZLP) ────────────  │   device ACKs by sending ZLP

2-stage          (wLength = 0, direction is don't-care for the wire)
                                                          SET_LED_BLINK, SET_ADDRESS
   host                                  device
    │  ── SETUP   ──────────────────────►  │   8-byte Setup packet (payload in wValue/wIndex)
    │  ◄──── STATUS in (ZLP) ────────────  │   device ACKs by sending ZLP

STALL (any shape)
   host                                  device
    │  ── SETUP   ──────────────────────►  │   8-byte Setup packet
    │  ◄──── STALL ─────────────────────   │   device rejects with STALL handshake
                                              (auto-clears on next SETUP — §8.5.3.4)
Why STATUS always flips direction

The STATUS stage exists to give the recipient of the DATA stage a chance to say "done." In a host-IN transfer, the host receives DATA, so the host sends the STATUS ZLP. In a host-OUT transfer, the device receives DATA, so the device sends the STATUS ZLP. In a 2-stage transfer, there's no DATA, so the host arbitrarily sends the STATUS ZLP (it always flows host → device). This isn't a Zephyr convention — it's USB 2.0 §8.5.3.

4b. Where Zephyr's callbacks fit

The Zephyr USB-Device-Next core handles the wire-level state machine for you. Your job is to fill in two callbacks; the core invokes them at exactly one moment in the transfer:

CallbackFires whenWhat you do
control_to_host After SETUP arrives, before the device-IN DATA packets go out Inspect setup->bRequest / wValue; fill response bytes into buf via net_buf_add_*. Return 0 if handled, -ENOTSUP for STALL.
control_to_dev After the host-OUT DATA stage completes (or after SETUP for 2-stage host-OUTs) For 3-stage OUT: read payload from buf->data[0..buf->len]. For 2-stage: payload is in setup->wValue / setup->wIndex. Act on it. Return 0 or -ENOTSUP.

Note what the core does not tell you about: SETUP arrival, individual DATA packets, STATUS completion. Those are the core's job. You only see the request after enough wire-level activity has happened to determine the request's payload — and you're done as soon as you've supplied (or consumed) that payload.

4c. Routing — how the core decides which class gets the request

From $ZEPHYR_BASE/subsys/usb/device_next/usbd_ch9.c:976:

switch (setup->RequestType.recipient) {
    case USB_REQTYPE_RECIPIENT_ENDPOINT:
        c_nd = usbd_class_get_by_ep   (uds_ctx, setup->wIndex);   // wIndex carries EP address
        break;
    case USB_REQTYPE_RECIPIENT_INTERFACE:
        c_nd = usbd_class_get_by_iface(uds_ctx, setup->wIndex);   // wIndex carries iface number
        break;
    case USB_REQTYPE_RECIPIENT_DEVICE:
        c_nd = usbd_class_get_by_req  (uds_ctx, setup->bRequest); // device-level vreq table
        break;
}
if (c_nd != NULL) {
    if (reqtype_is_to_device(setup))
        ret = usbd_class_control_to_dev (c_nd->c_data, setup, dbuf);
    else
        ret = usbd_class_control_to_host(c_nd->c_data, setup, dbuf);
}

Three consequences:

  1. Recipient determines routing, not type. A standard or class request with recipient = INTERFACE still lands in control_to_*. The class is expected to filter on RequestType.type itself. This is G16.
  2. wIndex semantics depend on recipient. For INTERFACE → iface number. For ENDPOINT → endpoint address. Get this wrong and the core routes your request to a different class (or no class).
  3. For recipient = DEVICE with type = VENDOR, the core checks a per-bRequest registration table (the 4th arg to USBD_DEFINE_CLASS, which we pass as NULL). Falls back to a device-level vendor handler if registered. usb_04 uses INTERFACE specifically to avoid this routing complexity.

5. Carry-forward defenses + new (G1–G18)

Gotchas G1–G15 carry forward from usb_01, usb_02, and usb_03. usb_04 introduces three new gotchas specific to control transfers.

Carried over (re-state, re-defend)

#GotchaDefense in usb_04
G1–G14Descriptor-tree and string-handling gotchas — already covered in usb_02 §4 and usb_03 §5.All carry through unchanged — usb_04's descriptor tree is byte-identical to usb_03's.
G15Don't double-add the configuration-string descriptor.main.c does five explicit string adds (lang, mfr, product, serial, iface_str), then lets usbd_add_configuration() auto-register cfg_str. Same five-explicit-plus-one-implicit pattern as fixed usb_03.

New for usb_04 (control-transfer-specific)

#GotchaDefense
G16 Filter on RequestType.type inside control_to_*. The core routes by recipient, not by type. A vendor class with recipient = INTERFACE will get every interface-recipient request — standard (e.g., GET_STATUS for the interface), class-specific (e.g., HID GET_REPORT if we somehow declared HID), and vendor. Without an explicit type check, your class will respond to standard requests with vendor-shaped logic and confuse the host. First line of every control_to_* handler in control_demo.c: 
if (setup->RequestType.type != USB_REQTYPE_TYPE_VENDOR) {
    return -ENOTSUP;
}
The core takes -ENOTSUP as "let the standard machinery handle it." For interface recipient with standard type, that machinery returns sensible defaults (e.g., 16 bits of zero for GET_STATUS).
G17 wLength is the host's upper bound, not the actual transfer size. For host-IN, you can return less data than wLength requested; the core sends a short packet (or a ZLP if your response length is an exact multiple of the EP0 max packet size, 64 for FS) to signal end-of-transfer. Treating wLength as the size to fill leads to two bugs: (a) leaking stack memory if your data is shorter and you pad to wLength, (b) failing to terminate if your data is longer and you stop reading at wLength. For GET_UPTIME, the device unconditionally writes 8 bytes via net_buf_add_le64 and returns. If host requests wLength = 4, the core truncates. If host requests wLength = 16, the core sees 8 bytes < 16, sends a short packet, transfer ends. We LOG_WRN when the truncation case applies — useful during host-side debugging, no functional impact.
G18 For host-OUT data, check buf->len, not setup->wLength. The host declares wLength in the Setup packet, but the DATA stage can finish early — host stops sending after fewer bytes, the core delivers what arrived. buf->len is the actually-received count. Using setup->wLength reads uninitialised tail of buf->data. SET_LOG_MSG uses MIN(buf->len, LOG_MSG_MAX) for the LOG_HEXDUMP_INF length. LOG_MSG_MAX caps the log spam (defence in depth — a malicious or buggy host could lie about wLength too).

7. Boot & runtime flow

BOOT is identical to usb_03 (with class name and PID bumped). RUNTIME is where the new behaviour lives.

BOOT (main thread, in order — order matters; see usb_03 §7 for the why)
  0. serial_init()                            chip ID → serial_buf
  1. usbd_add_descriptor(&ctx, &lang_desc)    idx 0
  2. usbd_add_descriptor(&ctx, &mfr_desc)     idx 1
  3. usbd_add_descriptor(&ctx, &product_desc) idx 2
  4. usbd_add_descriptor(&ctx, &serial_node)  idx 3
  5. usbd_add_descriptor(&ctx, &iface_str)    idx 4
  5a. control_demo_set_iface_idx(usbd_str_desc_get_idx(&iface_str))
  6. usbd_add_configuration(&ctx, USBD_SPEED_FS, &cfg)
                                              stack auto-adds cfg_str → idx 5
  7. usbd_register_class(&ctx, "control_demo", USBD_SPEED_FS, 1)
  8. usbd_init(&ctx)                          validates tree
  9. usbd_enable(&ctx)                        connects FS pull-up
 10. k_sleep(K_FOREVER)                       USB stack runs in its own queue

RUNTIME (USB stack work queue, callback-driven)
  Standard enumeration (unchanged):
  A. host issues GET_DESCRIPTOR(Device)                   served from USBD_DEVICE_DEFINE
  B. host issues GET_DESCRIPTOR(STRING, 0..5, 0x0409)     standard strings
  C. host issues GET_DESCRIPTOR(Configuration)            32-byte cfg+iface+ep+ep
  D. host issues SET_CONFIGURATION(1)                     triggers demo_enable()

  Vendor requests (new — fire when host code runs):
  E. host issues VENDOR | INTERFACE | IN  bRequest=0x01   GET_UPTIME
       → core calls control_to_host(setup, buf)
       → handler writes 8 bytes; returns 0
       → core sends 8 bytes IN, then waits for STATUS OUT ZLP
       → transfer completes; pyusb returns 8 bytes

  F. host issues VENDOR | INTERFACE | OUT bRequest=0x02   SET_LOG_MSG
       → core collects DATA OUT packets into buf
       → core calls control_to_dev(setup, buf) once DATA stage done
       → handler LOG_HEXDUMPs payload; returns 0
       → core sends STATUS IN ZLP; transfer completes

  G. host issues VENDOR | INTERFACE | OUT bRequest=0x03   SET_LED_BLINK
       → wLength=0; no DATA stage
       → core calls control_to_dev(setup, buf) directly after SETUP
       → handler reads setup->wValue; logs count; returns 0
       → core sends STATUS IN ZLP

  H. host issues VENDOR | INTERFACE | (IN|OUT) bRequest=0x04   STALL_ME
       → handler returns -ENOTSUP
       → core sends STALL handshake on EP0
       → STALL auto-clears on next SETUP (USB 2.0 §8.5.3.4)
       → pyusb raises USBError; host code catches it

  I. host attempts bulk read on EP 0x81                   NAK forever (still — carried over)

8. Code: src/main.c

Byte-for-byte adaptation of usb_03's main.c — only the class name, product string, interface-name string, PID, and bcdDevice change. No new functions, no new control flow.

firmware/samples/usb/usb_04/src/main.c
/*
 * usb_04 — control transfers in depth.
 *
 * Boot flow is byte-identical to usb_03 (5 explicit string adds → iface_str
 * idx patch → usbd_add_configuration → register_class → init → enable). The
 * new material is entirely inside control_demo.c, where control_to_host /
 * control_to_dev handle four vendor requests on EP0.
 */

#include <stdio.h>
#include <errno.h>
#include <zephyr/kernel.h>
#include <zephyr/usb/usbd.h>
#include <zephyr/drivers/hwinfo.h>
#include <zephyr/logging/log.h>

LOG_MODULE_REGISTER(usb_04, LOG_LEVEL_INF);

#define DEMO_CLASS_NAME "control_demo"

extern void control_demo_set_iface_idx(uint8_t idx);

USBD_DESC_LANG_DEFINE(lang_desc);
USBD_DESC_MANUFACTURER_DEFINE(mfr_desc,     "edevkit");
USBD_DESC_PRODUCT_DEFINE     (product_desc, "usb_04 - control_demo");
USBD_DESC_CONFIG_DEFINE      (cfg_str,      "Default config");
USBD_DESC_STRING_DEFINE      (iface_str,    "Vendor (control demo, 4 EP0 requests)",
                              USBD_DUT_STRING_INTERFACE);

#define SERIAL_TEMPLATE   "EDV-0000-0000-0000"
#define SERIAL_BUF_SIZE   sizeof(SERIAL_TEMPLATE)

static char serial_buf[SERIAL_BUF_SIZE] = SERIAL_TEMPLATE;

static struct usbd_desc_node serial_node = {
    .str = { .utype = USBD_DUT_STRING_SERIAL_NUMBER, .ascii7 = true },
    .ptr             = serial_buf,
    .bLength         = USB_STRING_DESCRIPTOR_LENGTH(SERIAL_TEMPLATE),
    .bDescriptorType = USB_DESC_STRING,
};

static int serial_init(void)
{
    uint8_t id[8] = {0};
    ssize_t n = hwinfo_get_device_id(id, sizeof(id));
    if (n < 6) {
        LOG_ERR("hwinfo_get_device_id returned %d (need >= 6)", (int)n);
        return -EIO;
    }
    int written = snprintf(serial_buf, sizeof(serial_buf),
                           "EDV-%02X%02X-%02X%02X-%02X%02X",
                           id[0], id[1], id[2], id[3], id[4], id[5]);
    if (written != (int)(sizeof(serial_buf) - 1)) return -ENOSPC;
    LOG_INF("serial: %s", serial_buf);
    return 0;
}

USBD_CONFIGURATION_DEFINE(cfg, USB_SCD_REMOTE_WAKEUP, 50, &cfg_str);

USBD_DEVICE_DEFINE(usbd_ctx, DEVICE_DT_GET(DT_NODELABEL(usbd)),
                   0x1209, 0xede4);

int main(void)
{
    int err;
    (void)serial_init();

    /* 5 explicit string adds — cfg_str stays for usbd_add_configuration (G15). */
    err = usbd_add_descriptor(&usbd_ctx, &lang_desc);    if (err) return -1;
    err = usbd_add_descriptor(&usbd_ctx, &mfr_desc);     if (err) return -1;
    err = usbd_add_descriptor(&usbd_ctx, &product_desc); if (err) return -1;
    err = usbd_add_descriptor(&usbd_ctx, &serial_node);  if (err) return -1;
    err = usbd_add_descriptor(&usbd_ctx, &iface_str);    if (err) return -1;

    uint8_t iface_idx = usbd_str_desc_get_idx(&iface_str);
    LOG_INF("iface_str registered at idx %u", iface_idx);
    control_demo_set_iface_idx(iface_idx);

    err = usbd_add_configuration(&usbd_ctx, USBD_SPEED_FS, &cfg);
    if (err) { LOG_ERR("add cfg: %d", err); return -1; }
    err = usbd_register_class(&usbd_ctx, DEMO_CLASS_NAME, USBD_SPEED_FS, 1);
    if (err) { LOG_ERR("register class: %d", err); return -1; }
    err = usbd_init(&usbd_ctx);
    if (err) { LOG_ERR("usbd_init: %d", err); return -1; }
    err = usbd_enable(&usbd_ctx);
    if (err) { LOG_ERR("usbd_enable: %d", err); return -1; }

    LOG_INF("usb_04 control_demo: enabled, waiting on EP0 vendor requests");
    while (true) { k_sleep(K_FOREVER); }
    return 0;
}

9. Code: src/control_demo.c

This is where every new line of usb_04 lives. The two control callbacks dispatch on setup->bRequest and implement the four request shapes from §3.

firmware/samples/usb/usb_04/src/control_demo.c
/* control_demo — vendor class, 1 interface, 4 vendor control requests on EP0. */

#include <zephyr/kernel.h>
#include <zephyr/usb/usbd.h>
#include <zephyr/usb/usb_ch9.h>
#include <zephyr/sys/byteorder.h>
#include <zephyr/net_buf.h>
#include <zephyr/logging/log.h>
#include <errno.h>

LOG_MODULE_REGISTER(control_demo, LOG_LEVEL_INF);

#define DEMO_CLASS_NAME "control_demo"

/* bRequest values — exposed for the host-side verification scripts in §12. */
#define VREQ_GET_UPTIME       0x01
#define VREQ_SET_LOG_MSG      0x02
#define VREQ_SET_LED_BLINK    0x03
#define VREQ_STALL_ME         0x04

#define LOG_MSG_MAX           32U

/* Descriptor tree — byte-identical to usb_03's strings_demo. */

static struct usb_if_descriptor iface = {
    .bLength            = sizeof(struct usb_if_descriptor),
    .bDescriptorType    = USB_DESC_INTERFACE,
    .bInterfaceNumber   = 0,
    .bAlternateSetting  = 0,
    .bNumEndpoints      = 2,
    .bInterfaceClass    = USB_BCC_VENDOR,
    .bInterfaceSubClass = 0xFF,
    .bInterfaceProtocol = 0xFF,
    .iInterface         = 0,                       /* patched at boot — G13 */
};

static struct usb_ep_descriptor ep_out = {
    .bLength          = sizeof(struct usb_ep_descriptor),
    .bDescriptorType  = USB_DESC_ENDPOINT,
    .bEndpointAddress = USB_EP_DIR_OUT | 0x01,
    .bmAttributes     = USB_EP_TYPE_BULK,
    .wMaxPacketSize   = sys_cpu_to_le16(64U),
    .bInterval        = 0,
};
static struct usb_ep_descriptor ep_in = {
    .bLength          = sizeof(struct usb_ep_descriptor),
    .bDescriptorType  = USB_DESC_ENDPOINT,
    .bEndpointAddress = USB_EP_DIR_IN  | 0x01,
    .bmAttributes     = USB_EP_TYPE_BULK,
    .wMaxPacketSize   = sys_cpu_to_le16(64U),
    .bInterval        = 0,
};

static struct usb_desc_header *fs_desc[] = {
    (struct usb_desc_header *) &iface,
    (struct usb_desc_header *) &ep_out,
    (struct usb_desc_header *) &ep_in,
    NULL,
};

void control_demo_set_iface_idx(uint8_t idx) { iface.iInterface = idx; }

/* ─── Vendor request handlers ──────────────────────────────────────────── */

static int control_to_host(struct usbd_class_data *const c_data,
                           const struct usb_setup_packet *const setup,
                           struct net_buf *const buf)
{
    /* G16 — only vendor requests are ours. */
    if (setup->RequestType.type != USB_REQTYPE_TYPE_VENDOR) {
        return -ENOTSUP;
    }

    switch (setup->bRequest) {
    case VREQ_GET_UPTIME: {
        /* 3-stage IN. G17 — wLength is upper bound; we always write 8 B. */
        if (setup->wLength < sizeof(uint64_t)) {
            LOG_WRN("GET_UPTIME: host wLength=%u < 8 — will truncate",
                    setup->wLength);
        }
        int64_t up = k_uptime_get();
        net_buf_add_le64(buf, (uint64_t)up);
        LOG_INF("GET_UPTIME → %lld ms", (long long)up);
        return 0;
    }

    case VREQ_STALL_ME:
        LOG_INF("STALL_ME (IN) → returning -ENOTSUP");
        return -ENOTSUP;

    default:
        LOG_WRN("control_to_host: unknown vendor bRequest 0x%02x",
                setup->bRequest);
        return -ENOTSUP;
    }
}

static int control_to_dev(struct usbd_class_data *const c_data,
                          const struct usb_setup_packet *const setup,
                          const struct net_buf *const buf)
{
    if (setup->RequestType.type != USB_REQTYPE_TYPE_VENDOR) {
        return -ENOTSUP;
    }

    switch (setup->bRequest) {
    case VREQ_SET_LOG_MSG: {
        /* G18 — trust buf->len, not setup->wLength. */
        size_t n = MIN(buf->len, LOG_MSG_MAX);
        LOG_HEXDUMP_INF(buf->data, n, "SET_LOG_MSG payload:");
        return 0;
    }

    case VREQ_SET_LED_BLINK: {
        /* 2-stage. wValue carries the count; there is no DATA stage. */
        LOG_INF("SET_LED_BLINK count=%u (2-stage; no DATA)", setup->wValue);
        return 0;
    }

    case VREQ_STALL_ME:
        LOG_INF("STALL_ME (OUT) → returning -ENOTSUP");
        return -ENOTSUP;

    default:
        LOG_WRN("control_to_dev: unknown vendor bRequest 0x%02x",
                setup->bRequest);
        return -ENOTSUP;
    }
}

/* ─── Class vtable + registration ──────────────────────────────────────── */

static int  demo_init   (struct usbd_class_data *const c) { return 0; }
static void demo_enable (struct usbd_class_data *const c) { LOG_INF("control_demo: enabled"); }
static void demo_disable(struct usbd_class_data *const c) { LOG_INF("control_demo: disabled"); }

static void *demo_get_desc(struct usbd_class_data *const c, const enum usbd_speed s)
{
    return (s == USBD_SPEED_FS) ? (void *)fs_desc : NULL;
}

    .disable          = demo_disable,
    .get_desc         = demo_get_desc,
    .control_to_host  = control_to_host,    /* ← new in usb_04 */
    .control_to_dev   = control_to_dev,     /* ← new in usb_04 */
};

12b. The verification script

firmware/samples/usb/usb_04/verify.py — runs on Linux · macOS · Windows
#!/usr/bin/env python3
"""Exercise all 4 usb_04 vendor requests."""
import usb.core, usb.util, struct, sys

VID, PID = 0x1209, 0xede4
IFACE    = 0   # recipient = interface, wIndex = 0

# bRequest values mirror src/control_demo.c
VREQ_GET_UPTIME    = 0x01
VREQ_SET_LOG_MSG   = 0x02
VREQ_SET_LED_BLINK = 0x03
VREQ_STALL_ME      = 0x04

# bmRequestType bytes built from constants
RT_IN  = usb.util.build_request_type(usb.util.CTRL_IN,
                                     usb.util.CTRL_TYPE_VENDOR,
                                     usb.util.CTRL_RECIPIENT_INTERFACE)  # 0xC1
RT_OUT = usb.util.build_request_type(usb.util.CTRL_OUT,
                                     usb.util.CTRL_TYPE_VENDOR,
                                     usb.util.CTRL_RECIPIENT_INTERFACE)  # 0x41

dev = usb.core.find(idVendor=VID, idProduct=PID)
if dev is None: sys.exit("usb_04 not found — check VID:PID and udev/Zadig")

dev.set_configuration()

# (1) GET_UPTIME — 3-stage IN
data = dev.ctrl_transfer(RT_IN, VREQ_GET_UPTIME, 0, IFACE, 8)
uptime_ms = struct.unpack("<Q", bytes(data))[0]
print(f"GET_UPTIME      → {uptime_ms} ms ({len(data)} bytes)")

# (2) SET_LOG_MSG — 3-stage OUT
msg = b"hello usb_04"
n = dev.ctrl_transfer(RT_OUT, VREQ_SET_LOG_MSG, 0, IFACE, msg)
print(f"SET_LOG_MSG     → device accepted {n} bytes")

# (3) SET_LED_BLINK — 2-stage (no DATA)
dev.ctrl_transfer(RT_OUT, VREQ_SET_LED_BLINK, wValue=3, wIndex=IFACE, data_or_wLength=0)
print(f"SET_LED_BLINK   → count=3 sent in wValue, no DATA stage")

# (4) STALL_ME — expected to raise
try:
    dev.ctrl_transfer(RT_IN, VREQ_STALL_ME, 0, IFACE, 0)
    print(f"STALL_ME        → UNEXPECTED success; device should have STALLed")
except usb.core.USBError as e:
    print(f"STALL_ME        → expected USBError (\"{e}\")")

Expected output on a working board:

$ python firmware/samples/usb/usb_04/verify.py
GET_UPTIME      → 18342 ms (8 bytes)
SET_LOG_MSG     → device accepted 12 bytes
SET_LED_BLINK   → count=3 sent in wValue, no DATA stage
STALL_ME        → expected USBError ("[Errno 32] Pipe error")
Acceptance — usb_04 is "done" when

(a) The verification script prints all four request lines without crashing; (b) The RTT log shows one matching <inf> control_demo line per request; (c) The uptime value increases monotonically across multiple runs (i.e., is actually k_uptime_get(), not a hard-coded constant); (d) STALL_ME reliably produces a USBError on the host and an <inf> log line on the device, and the next GET_UPTIME still works (proving the STALL auto-cleared per §8.5.3.4); (e) Bulk endpoints still NAK forever — that's still a future sample's problem.

13. Anticipated pitfalls

Symptom → Fix

"pyusb gets [Errno 13] Access denied (insufficient permissions)." Linux udev rule not in place (see §12a), or another driver claimed the interface. Linux: install the udev rule and replug. macOS: rarely happens; if it does, no kernel extension is the cause — check that nothing else (a half-installed driver) is claiming PID 0xede4. Windows: install WinUSB via Zadig.

Symptom → Fix

"GET_UPTIME returns fewer than 8 bytes." Host's wLength was less than 8 (G17). The device wrote 8 bytes but the core truncated. Either request wLength=8 exactly or accept short reads. RTT shows <wrn> control_demo: GET_UPTIME: host wLength=N < 8 — will truncate when this happens.

Symptom → Fix

"SET_LOG_MSG hex-dump shows garbage past the intended message." Code is using setup->wLength instead of buf->len as the consumed-bytes count (G18). The buf was sized to accommodate wLength but only buf->len bytes are valid. The rest is uninitialised pool memory.

Symptom → Fix

"STALL_ME succeeds instead of raising." The callback returned 0 instead of -ENOTSUP (the G4 / G16 hazard in reverse — saying "I handled it" when you didn't). Or the request hit a different class (recipient = INTERFACE with the wrong wIndex). Add a LOG_INF at handler entry to confirm dispatch.

Symptom → Fix

"After STALL_ME, subsequent requests also fail." Shouldn't happen on EP0 — STALL auto-clears on the next SETUP per USB 2.0 §8.5.3.4. If it does happen, suspect (a) the host stack got confused (try unplugging and replugging), or (b) the request returning -ENOTSUP actually panicked the device. Check RTT for fault output.

Symptom → Fix

"All four requests time out with [Errno 110]." The class isn't seeing the requests at all. Likely causes: (a) recipient mismatch — script used CTRL_RECIPIENT_DEVICE but device routes only INTERFACE; (b) wIndex mismatch — script used IFACE=1 but device's interface is 0; (c) bmRequestType.type mismatch — script used CTRL_TYPE_CLASS instead of CTRL_TYPE_VENDOR. Capture with usbmon to see what the host actually sent.

Symptom → Fix

"Build warns 'unused variable c_data'." Harmless. Zephyr passes the class-data pointer to every callback even when the handler doesn't use it. Silence with ARG_UNUSED(c_data) or accept the warning — it has no functional impact.
