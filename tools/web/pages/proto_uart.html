UART — asynchronous serial, the lingua franca of embedded

A practitioner's reference on the Universal Asynchronous Receiver / Transmitter: the oldest, simplest, most universal serial link in embedded systems. Two wires (TX, RX), no clock line, a baud rate agreed up front, and a frame format both sides know about. That's it — and that's why every MCU on Earth has at least one. This document covers the wire-level frame, baud-rate generation, line standards (TTL / RS-232 / RS-422 / RS-485), flow control, modem signals, error reporting, hardware-vs-PIO trade-offs, edevkit1's twin-UART layout, and the tooling and gotchas that consume the most bench time.

Audience. Firmware engineers wiring up a new target's debug console, host-side developers who just want their printfs to land in a terminal, and anyone confronted with a two-wire-three-wire-five-wire serial header on someone else's board and asked to "just talk to it." If you have used a Cortex-M before, this doc fills in the surrounding "but how does the wire actually move bits" part. If you are new to embedded entirely, start here, then read proto_i2c.html and proto_spi.html for the synchronous-bus cousins.

Where edevkit fits. edevkit1 carries two independent UART bridges — one wired to the target's voltage domain (for the target's log output), one wired to the user-selectable Vio domain (for protocol testing against a third device). The two are deliberately not tied together. Both are surfaced to the host as USB CDC-ACM serial ports. See edevkit1.html §6 for the full hardware rationale.

What UART is — and isn't

A UART is the silicon block that converts parallel bytes inside an MCU into a stream of bits on a single wire, and back. The name is an acronym from the dial-up era — Universal Asynchronous Receiver / Transmitter — and the word "Universal" is doing real work: a UART is configurable enough to talk to a 110-baud teletype from 1965, to a 6-Mbaud modern Bluetooth transport, and almost everything in between, with the same handful of registers.

The defining traits, none of which you can change without ceasing to be a UART:

  • Asynchronous. No clock line is shared between the two ends. Each side runs its own clock, and they agree up front on a baud rate — the bit-time on the wire. The receiver re-synchronises on the falling edge of every start bit, then samples each subsequent bit roughly in its middle.
  • Point-to-point. Exactly one transmitter and one receiver per direction. There is no addressing, no arbitration, no multi-drop in the base protocol. (RS-485 bolts multi-drop on the wire side; the bytes still flow to one peer at a time.)
  • Full-duplex by default. TX and RX are separate wires; data can flow both ways simultaneously. Half-duplex variants exist (one-wire UART, RS-485 with the receiver muted while transmitting), but they are downgrades of the base.
  • Byte-oriented frames. A UART transmits one character at a time, where a character is 5–9 bits of payload framed by a start bit and one or two stop bits. There is no concept of a "packet" at the UART level — that is your problem to define on top.

What a UART is not:

  • It is not USB. USB carries serial data, but with framing, error correction, addressing, descriptors, and a 1-ms-tick host master. A USB CDC-ACM endpoint emulates a UART so host software can keep using /dev/ttyACM0, but the wire underneath is utterly unrelated.
  • It is not RS-232. RS-232 is the line-driver standard from 1960 that puts ±3–15 V swing on a connector. Modern MCUs run a UART at 3.3 V or 1.8 V TTL; you bolt on a MAX3232 only when you want to talk to a real RS-232 device.
  • It is not Modbus, SLIP, NMEA, AT, or any other "serial protocol." Those all sit on top of a UART link. UART moves bytes; framing and meaning are layered above.
  • It is not synchronous (despite the U/S in some chips). A "USART" peripheral can also do synchronous serial (clocked SPI-ish modes). When asynchronous, it is just a UART.
why it is everywhere Two wires. One configuration parameter (the baud rate, plus a couple of byte-format bits). Predictable behaviour at any clock speed from kHz to MHz. No external reference, no addressing, no master/slave roles to negotiate. Every MCU has at least one UART, and the bring-up "I see characters!" moment is usually the first sign that an unknown board is alive at all.

Frame format — start, data, parity, stop

The UART line is held idle high (logic 1, also called the mark state, a holdover from teletype current-loop days where current-flowing was the resting condition). To transmit a character the sender drives the line low for one bit-time — the start bit — and then shifts out the data bits, optional parity, and one or more stop bits returning the line to idle.

Frame field-by-field

FieldWidthPolarityPurpose
IdleanyHigh (mark)Default line state. The receiver waits here for a falling edge.

Parity bit0 or 1 bitEven / odd / mark / space / noneOne-bit error detection; modern systems almost always use none and rely on higher layers.
Stop bit(s)1, 1.5, or 2 bitsHigh (mark)Guarantees a high-time gap so the next start bit's falling edge is unambiguous.

The conventional shorthand 8N1 means "8 data bits, No parity, 1 stop bit" — that is the modern default and what 99 % of debug consoles use. 7E1 (7 data + Even parity + 1 stop) shows up in older industrial gear; 8E1, 8O1, and 8N2 occasionally in oddball legacy. Always state the format alongside the baud rate; "115200" alone is ambiguous.

Worked example: sending the ASCII character "A" (0x41) at 8N1

0x41 is binary 0100 0001. UART is LSB-first on the wire, so the bits leave the shift register in the order 1 0 0 0 0 0 1 0, framed by start (low) and stop (high):

      |< 1 bit time >|
      |              |
idle  | s  d0 d1 d2 d3 d4 d5 d6 d7 sp |  idle
______          ____             ___________
      |        |    |           |
       \      /      \_________/
        \____/

bit:    S  1  0  0  0  0  0  1  0  P
        ^                          ^
        start                      stop
        bit  (LSB-first payload)   bit

decoded:    bit0=1 bit1=0 bit2=0 bit3=0 bit4=0 bit5=0 bit6=1 bit7=0
            => 0b01000001  =>  0x41  =>  'A'

At 115200 baud each bit lasts 1 / 115200 s ≈ 8.68 µs, so the entire 10-bit frame for "A" occupies 86.8 µs on the wire. At 9600 baud the same frame takes 1.04 ms. The character throughput is therefore baud / 10 for 8N1 (or baud / 11 with one parity bit).

where the receiver samples A standard UART receiver oversamples the line at 8× or 16× the baud rate. On the falling edge of the start bit it waits half a bit-time, re-checks (rejects if it has gone back to high — the original edge was noise), then samples the centre of each subsequent bit. With a 16× oversample clock, the receiver can absorb up to roughly ±5 % aggregate clock mismatch over a 10-bit frame before it samples the wrong half of the stop bit.

The 9-bit "address" mode

9-bit data plus stop is a legacy convention used by some industrial multi-drop networks (notably 9-bit RS-485 setups, and a handful of automotive buses): the 9th bit is treated as an address-flag, set on the first byte of a packet and clear on subsequent data. Receivers with a hardware multi-processor mode can wake up only on address bytes and ignore data destined for other nodes. You almost never see it on a debug console.

Parity: present, mostly vestigial

Parity adds one bit at the end of the data field. The transmitter computes it so the count of 1s in (data + parity) matches a parity rule, and the receiver checks the count on arrival.

ParityRuleUse
noneNo parity bit transmitted.The default everywhere. Higher layers (CRC, framing) catch errors.
evenTotal 1s in data+parity is even.Detects single-bit errors. Common in older industrial links.
oddTotal 1s in data+parity is odd.Same detection power as even, just a different convention.
markParity bit always 1.Used as a 9th data bit hack — e.g. driver enable on RS-485.
spaceParity bit always 0.Same hack, opposite polarity.

Single-bit parity is a famously weak error detector: it cannot catch double-bit errors and adds one bit of overhead per character. A real link should rely on a CRC or message-level integrity check; parity persists only because changing it would break compatibility with a million field-deployed devices. For a debug console: none.

Baud rates and rate tolerance

The baud rate is the symbol rate on the wire — for plain UART, equivalent to the bit rate, since each symbol is one bit. The two ends of a link must agree on it; mismatch is the single most common cause of "garbage characters" during bring-up.

Why 115200 is the canonical default

It is fast enough that a few KB of boot log doesn't take painful seconds, slow enough that almost any USB-serial bridge supports it, and it falls out of nearly every common MCU clock as a near-perfect divider:

  • From a 12 MHz crystal: 12 000 000 / 115200 = 104.17, error 0.16 %.
  • From a 48 MHz peripheral clock with a 16×-oversampling UART: 48 000 000 / 16 / 115200 = 26.04, the fractional 0.04 trims out via the divider's fractional part.
  • From the RP2350's typical 125 MHz peripheral clock: 125 000 000 / 16 / 115200 = 67.82, well within the UART's fractional divider range.

By the time a board is shipping, "115200 8N1" has become a tribal handshake: stick a TTL-serial cable on the header, open tio /dev/ttyACM0, and you almost certainly see something legible.

Rate tolerance — the ±2.5 % rule

A UART receiver samples each bit near its centre. The cumulative timing error budget over a single character (start + 8 data + stop = 10 bit-times) is roughly ±5 % aggregate before the stop-bit sample lands in the wrong half. By convention each end is allowed up to ±2.5 % of clock error so the worst-case sum stays under the budget. That is why MCUs always reach for a crystal (typically ±20 ppm = ±0.002 %) for the system clock when a UART is involved — the internal RC oscillator on most parts is ±1–3 % over temperature and that eats most of your budget by itself.

At higher baud rates the absolute error tightens: 2.5 % of 6 Mbaud is 150 kHz, and most internal RC oscillators can swing further than that across temperature. So 6 Mbaud requires a crystal-quality reference, properly terminated traces, and short cables. The typical breakdown isn't bit-time error per se — it's jitter from PLL stages or PIO state-machine wakeup that pushes individual edges out of tolerance.

The baud-rate generator

Inside an MCU, the UART block contains a programmable divider that takes the peripheral clock down to 16 × baud (or 8 × baud on lower-cost cores). On older parts the divider was integer-only, and many baud rates were unreachable from common crystals (3.6864 MHz crystals exist precisely because they are an integer multiple of every standard baud rate). Modern UARTs — including the RP2350's PL011-derived block — use a fractional divider: an integer part plus a 6-bit fractional part, so the divider can hit any baud rate within a tiny error margin from any reasonable peripheral clock. See the RP2350 datasheet, UART chapter, for the exact register layout.

Logic levels & line standards

The UART protocol says nothing about voltages on the wire. Several different physical-layer standards sit underneath the same byte-format. They are distinguished by signalling type (single-ended vs differential), voltage swing, drive strength, and how they handle multi-drop.

TTL UART — the MCU-native level

Inside an MCU and on a typical dev-board header, "UART" means single-ended swing between 0 V and the I/O supply (3.3 V on most modern parts, 1.8 V on the latest ULP families). High = idle = mark = 1; low = active = space = 0. This is what every Zephyr / Pi Pico / Arduino tutorial implicitly assumes when it says "connect TX to RX."

TTL UART is happy on a few centimetres of jumper wire at 115200, and on a clean PCB up to maybe 6 Mbaud. It is not happy across a desk on flying leads — cable capacitance rounds the edges off, ground bounce shifts the threshold, and the receiver starts seeing phantom edges. If your "serial" cable is more than about 30 cm and runs more than 230400 baud, expect to add a level shifter or move to a differential standard.

RS-232 — the ancestor

RS-232 dates from 1960 and was specified for connecting a Data Terminal Equipment (DTE, your computer) to Data Communications Equipment (DCE, a modem). The physical layer uses inverted polarity vs TTL and large voltage swings: a "1" (mark) is −3 to −15 V, a "0" (space) is +3 to +15 V, both measured against signal ground. The wide swing buys it noise margin and reach over multi-metre cables.

Modern MCUs do not produce RS-232 levels — they produce TTL. A MAX3232 or similar charge-pump driver chip translates between TTL and RS-232 voltages. This is why the cheap "USB-to-serial" cables in a parts drawer almost always come with RS-232 levels at the connector end (a DB9 or 3.5 mm jack); they expect to talk to a real legacy device. Never plug an RS-232 cable directly into a 3.3 V GPIO — the 12 V swing will likely destroy the pin.

don't mix levels TTL-into-RS-232 input does nothing (the receiver sits in an undefined zone). RS-232-into-TTL input damages the MCU pin (the silicon ESD diodes clamp briefly, then fail open). USB-to-serial dongles labelled "RS-232" need a MAX3232-class chip in series before they touch your board. USB-to-serial dongles labelled "TTL" or "3.3 V" can connect direct — check the dongle's datasheet, every brand handles this slightly differently.

RS-422 — differential, point-to-point, longer reach

RS-422 keeps the UART byte format but moves the wire-side to a differential pair: each direction uses two wires (A and B) and the receiver compares them. Differential signalling rejects common-mode noise, so cables can run hundreds of metres. RS-422 is full-duplex (separate A/B pair per direction = 4 wires plus ground) and supports one driver and up to ten receivers per pair.

Use RS-422 when you need to reach more than 10 m, or you are crossing an industrial environment with motors and contactors throwing transients onto everything, or you need a deterministic point-to-point UART link between two boards in different chassis.

RS-485 — differential, multi-drop

RS-485 is RS-422's multi-drop cousin: same differential signalling, same voltage levels, but the drivers are tri-stateable so multiple devices can share one twisted pair. Half-duplex (one A/B pair total = 2 wires plus ground) is the common case; full-duplex RS-485 (two pairs) is rare. The standard supports up to 32 unit-load drivers per segment, and modern low-load transceivers extend that to 256.

RS-485 is the dominant link in industrial protocol stacks like Modbus RTU, Profibus DP, DMX-512, and most building-automation buses (BACnet MS/TP, KNX). The catch is that someone must coordinate who is transmitting when — usually a master/slave protocol on top, with the master polling each slave by address. The transceiver chip needs a driver-enable (DE) input that the firmware drives high while transmitting and low otherwise; getting the DE timing wrong (releasing too early, holding too long) causes mysterious dropped bytes that Modbus stacks have folklore about.

picking the line standard TTL for anything on the same PCB or stacked board. RS-232 only for talking to legacy gear that demands it (industrial PLCs, old test equipment). RS-422 for a long point-to-point link where you control both ends. RS-485 for a multi-drop industrial bus. The byte format (8N1 etc.) and the baud rate stay the same across all four; only the line driver and receiver chip change.

Flow control — none / RTS-CTS / XON-XOFF

Without flow control, a transmitter that pumps bytes faster than the receiver can drain its FIFO will eventually overrun the RX hardware: bytes arrive while the FIFO is full and the UART silently drops them (and raises the OVERRUN error flag — see below). Flow control is how the receiver tells the transmitter to pause.

Three flavours

RTS/CTS in detail

The names are aggressively unhelpful; they mean different things on a DTE (your computer) versus a DCE (a modem). The original IBM 1960s convention:

  • DTE drives RTS high to say "I'm Ready To Send."
  • DCE responds by driving CTS high to say "Clear To Send, go ahead."

Then, in real-world implementations, RTS got reinterpreted as "Ready To Receive" (asserted while my RX FIFO has space, deasserted when full). Today essentially all hardware flow control uses the second meaning; documentation uses the first. The modern, sane wiring between two MCUs is:

   MCU A                       MCU B
  ┌──────┐                    ┌──────┐
  │  TX  │ ─────────────────→ │  RX  │
  │  RX  │ ←───────────────── │  TX  │
  │ RTS  │ ─────────────────→ │ CTS  │     A asserts RTS when its RX FIFO has room.
  │ CTS  │ ←───────────────── │ RTS  │     B asserts RTS when its RX FIFO has room.
  │ GND  │ ─────────────────  │ GND  │
  └──────┘                    └──────┘

(That is, MCU A's RTS connects to MCU B's CTS and vice versa — a "crossover" or "null modem" wiring.) This is the cabling the canonical "null-modem cable" provides and is what 99 % of MCU-to-MCU UART links assume. If you see "RTS to RTS" in a schematic, it is wrong.

XON/XOFF

Software flow control reserves two ASCII control codes: 0x11 (DC1, XON) means "resume" and 0x13 (DC3, XOFF) means "pause." When the receiver's FIFO fills up it sends an XOFF; when it drains it sends an XON.

This works fine when the link only carries text (terminal sessions, AT modem commands), and falls over the moment you try to send binary data: the byte 0x13 in your firmware blob is indistinguishable from a real XOFF, and it will pause the link until something happens to send 0x11. Some implementations escape 0x11/0x13 in the data stream, doubling the encoding burden. For any binary-data link, prefer hardware flow control or no flow control.

Modem control signals — DTR, DSR, DCD, RI

The full RS-232 connector defines nine signals (the DB9 pinout), only three of which (TX, RX, GND) are needed for a basic UART link. The rest are modem control lines originally specified for DTE-to-modem ("DCE") communication in the 1960s. Most are unused on MCU-to-MCU links, but they survive because USB CDC-ACM emulates them — and host-side serial drivers expose them as TIOCMSET ioctls on Linux, EscapeCommFunction on Windows.

SignalDirectionOriginal meaningModern (mis)use
DTR (Data Terminal Ready)DTE → DCE"My terminal is alive and listening."Often abused as a reset / boot-mode strap on dev boards (Arduino, ESP32). On USB CDC-ACM, the host driver toggles DTR when an application opens the port.
DSR (Data Set Ready)DCE → DTE"My modem is alive and connected."Rarely used on modern hardware. Some USB-serial chips let firmware read it.
DCD (Data Carrier Detect)DCE → DTE"Carrier tone present — the line is up."Vestigial. Some serial-console BMCs use it as a connection indicator.
RI (Ring Indicator)DCE → DTE"The phone line is ringing."Almost never used. Some PC BIOSes still let you "wake on RI."
RTS (Request To Send)DTE → DCE"I want to transmit."Repurposed as "I have RX FIFO space" for hardware flow control. Also abused as a boot-mode strap.
CTS (Clear To Send)DCE → DTE"Go ahead, transmit."Repurposed as "the other side has RX FIFO space" for hardware flow control.

Why these still matter on USB CDC-ACM

The CDC-ACM (Abstract Control Model) USB class was designed to emulate a modem, so it carries DTR, DSR, DCD, RI, RTS, CTS as state bits in its setup-packet protocol. On the host they appear in the standard tty layer; on the device-side firmware (Zephyr USB-Device-Next, TinyUSB, etc.) you get callbacks when the host changes them.

The classic abuse is the Arduino "auto-reset" trick:

  • The bootloader runs on POR for ~1 second listening for an upload command.
  • The PC-side IDE opens the serial port, which drives DTR low.
  • A capacitor on the board converts the DTR falling edge into a brief reset pulse on the MCU.
  • The bootloader hears the upload command before the timeout.

The same pattern, with DTR and RTS together, drives ESP32 boards into the ROM serial-bootloader by simultaneously asserting boot-mode (GPIO0 low) and pulsing reset. On edevkit1 we deliberately do not wire DTR/RTS to anything mechanical — the dev-host can open and close the port freely without rebooting the target, which matters for a debug probe whose job is to keep talking to a target across host disconnects. See edevkit1.html §6.4 for the rationale.

Errors — framing, parity, overrun, break

The UART hardware reports four classes of error condition. Each is a flag bit in the status register that the driver can read after every received byte. Your firmware should at minimum log them; in production you usually count them per-second and surface a metric to higher layers.

ErrorCauseSymptomWhat to do
FramingStop bit was not high when the receiver sampled it.Garbled characters, bursts of bad bytes after a noise event.Almost always a baud-rate mismatch or a bad cable. Re-check both ends' baud + format. Check for ground loops.
ParityReceived parity bit didn't match the data.Single bytes corrupted, flagged.Treat the byte as suspect (drop or escalate to higher-layer CRC). If it happens often: bad cable, EMI, or a parity-mode mismatch (e.g. one side configured for even, the other for odd).
OverrunRX FIFO was full when a new byte arrived; the new byte is dropped.Missing characters, length mismatches, "the device sent N bytes but I only see N−1."Drain the FIFO faster (DMA or higher-priority IRQ) or add hardware flow control. Software overrun — where the FIFO drained but the driver's ring buffer overflowed — is a separate, software-side variant.
BreakThe line was held in the space (low) state for longer than one full frame time.A "break" event in your driver, often used to signal "reset" or "attention."If unintended: usually a connector dropout or a peer that powered off mid-transmit. If intended: someone is sending you a break on purpose, e.g. a Linux BRK from the host.

The "junk first byte after wake-up" gotcha

A very common bring-up bug: the very first byte after the link comes up is garbled, but every subsequent byte is fine.

The cause is the difference between line state and line idle. After power-up or reset the TX pin is floating until the UART block is configured and enabled; an external pull-up may or may not yet be present; the wire might be at any voltage between 0 and Vcc. The first time the transmitter drives a real start bit, the receiver may already be mid-sample on garbage. The fix is to configure the UART, drive TX high, wait at least one bit-time, and only then start sending data. On boards where the TX pin's reset state is high-Z, add an external 10-100 kΩ pull-up to Vcc — cheap insurance.

The matching receive-side fix: if your driver reports a single framing-error on the first byte after enabling RX, swallow it — it's the line settling, not real data.

Hardware UART vs PIO / bit-banged UART

An MCU has hardware UART blocks — dedicated state-machine peripherals that run independently of the CPU and offload all the bit-shifting work. They are configured by writing a few registers, and they raise interrupts when bytes arrive or the TX FIFO drains. The RP2350 has two PL011-derived UART blocks (UART0 and UART1).

What you get from a hardware UART

  • FIFOs. Both directions buffer 32 bytes (RP2350) before the CPU has to intervene, so a 16×-oversampled receiver at 921600 still gives the CPU ~280 µs of slack between interrupts.
  • DMA hooks. Each UART connects to a DMA channel, so the CPU can sleep through a 100-byte transfer entirely.
  • Interrupt sources for every condition — TX empty, RX threshold, RX timeout, error, modem-state change.
  • Predictable timing. The state machine is hardware; baud accuracy is exactly the divider math.
  • Hardware flow control. RTS/CTS is wired through the peripheral; no firmware involvement once configured.

What you give up vs PIO bit-banging

  • Pin choice. Hardware UART pins are fixed by the silicon (with a few mux options). PIO can drive UART on any GPIO.
  • Format flexibility. Hardware UART supports the RS-232 frame zoo (5/6/7/8/9 data, parity, 1/1.5/2 stop). PIO can do those and any other framing — a 12-bit "frame," a custom inter-byte gap, a non-NRZ encoding, whatever you can write in PIO assembly.
  • Custom break behaviour. Want to send a 1.7-bit-time break for a LIN-bus header? PIO. Want to detect a malformed break and recover? PIO.
  • Number of channels. RP2350 has 2 hardware UARTs. Each PIO state machine can be a UART, giving you 12 simultaneous UART channels (3 PIOs × 4 SMs) if you want.

edevkit's mix

edevkit1 uses the two RP2350 hardware UARTs for its two main UART bridges (target log and test) because they need to run reliably at high baud rates with FIFO/DMA support and minimal CPU overhead. We also reserve one PIO state machine for SWO trace decoding (which is itself a UART-like signal at unusual baud rates — see platform_pio.html for the PIO budget). For arbitrary-format serial bus emulation (e.g. a custom three-wire UART variant on a target chip we are pretending to be), PIO is the escape hatch.

when to reach for PIO UART Use the hardware UART unless you have a specific reason not to. Reasons to go PIO: pin assignment is fixed and the hardware mux can't reach it; you need an unusual baud or format; you need more than two simultaneous UARTs; you are sniffing a third-party UART (the hardware UART can transmit and receive but not "passively listen") and need to capture exact edge timing; or you are emulating a peripheral whose UART has bus-specific quirks (LIN, K-Line, ISO 7816 reset semantics).

edevkit1's twin-UART setup

edevkit1 carries two completely independent UART bridges, one per voltage domain. They never share wires, never share ground references except through the chassis, and never share configuration. The host sees them as two separate USB CDC-ACM serial endpoints (typically /dev/ttyACM1 and /dev/ttyACM2; ttyACM0 is the CMSIS-DAP / shell endpoint).

Why two domains, never tied

The Target Log UART references Vtgt — whatever I/O voltage the target board is using (1.8 V on a low-power BLE chip, 3.3 V on a generic ARM dev board, sometimes 5 V on an old AVR). edevkit auto-senses Vtgt with an ADC channel and configures its level shifter accordingly. This way, plugging into a 1.8 V target doesn't require any user action.

The Test UART, by contrast, references Vio — a voltage you picked from the GUI/CLI (3.3 V or 1.8 V) before plugging in a third-party device. Vio is generated locally on edevkit by a switching regulator + LDO chain, and is current-limited by the eFuse so a misconfigured external device can't damage anything.

The two domains must not be tied together: a target running at 1.8 V and an external sensor running at 3.3 V will fight if their grounds drift, and no level shifter can rescue you if their I/O supplies are bonded by accident. edevkit1's silkscreen calls this out next to the connectors. edevkit1.html §6 covers the routing and ESD considerations.

vtgt < 1.65 V The LVC8T245 buffer's data sheet specifies a minimum Vcc of 1.65 V on the low-voltage side. Below that, output drive degrades and the input thresholds shift unpredictably. Firmware detects Vtgt < 1.65 V via the ADC and routes traffic through TXS0108E channels instead, which work down to 1.2 V. The shell prints a one-time warning so the user knows the path changed.

How a target log byte travels

Target MCU (say, an STM32WB55 at 3.3 V)
  │
  │ printk("Hello\n");
  │
  ▼
[STM32 UART TX pin] ── 3.3 V TTL ──► [edevkit GP0]
                                     │
                                     │ RP2350 UART0 RX
                                     │ → ring buffer (4 KB)
                                     │ → CDC-ACM IN endpoint
                                     │
                                     ▼
                        [USB FS, EP3 IN, 64-byte packets]
                                     │
                                     ▼
                          host [/dev/ttyACM1]
                                     │
                                     ▼
                           tio /dev/ttyACM1
                                     │
                                     ▼
                           "Hello\n" appears on stdout

Notably, the 4 KB SRAM ring buffer between the UART and CDC-ACM endpoint exists because USB FS only schedules a CDC-ACM bulk IN once per USB frame (1 ms): if the target dumps a kilobyte burst at 921600 baud, that's ~10 ms of stream, but USB only drains it 64 bytes at a time. Without the ring buffer, the UART would overrun. With it, the ring smooths the burst into the USB pipe with no data loss as long as long-term throughput stays under USB FS's ~10 Mbps usable.

Zephyr UART API — concrete code

Zephyr ships three UART API tiers: polling (simple, blocking), interrupt-driven (the workhorse), and asynchronous (DMA-backed, callback-style). They live behind the same struct device * handle and are selected by which functions you call. The interrupt and async APIs are mutually exclusive for the same device when CONFIG_UART_EXCLUSIVE_API_CALLBACKS is set (default on).

Devicetree binding

edevkit1's overlay (boards/rpi_pico2.overlay) wires up both UARTs:

/* boards/rpi_pico2.overlay (edevkit1) */
&uart0 {                           /* Target Log UART, on Vtgt */

    pinctrl-names  = "default";

&uart1 {                           /* Test UART, on Vio */

Polling API — simplest possible "send + receive a line"

#include <zephyr/device.h>
#include <zephyr/devicetree.h>
#include <zephyr/drivers/uart.h>

static const struct device *const uart_dev =
    DEVICE_DT_GET(DT_ALIAS(test_uart));

void uart_send_line(const char *s)
{
    while (*s) {
        uart_poll_out(uart_dev, *s++);   /* blocks until TX FIFO has room */
    }
}

int uart_read_byte_blocking(uint8_t *out)
{
    while (uart_poll_in(uart_dev, out) != 0) {
        k_yield();                        /* yield so other threads run */
    }
    return 0;
}

Polling APIs are useful for a quick bring-up smoke test or a once-at-boot configuration write to a peripheral. They do not scale — CPU is busy 100 % of a transfer.

Interrupt-driven API — the production default

#include <zephyr/device.h>
#include <zephyr/drivers/uart.h>
#include <zephyr/sys/ring_buffer.h>
#include <zephyr/logging/log.h>
LOG_MODULE_REGISTER(test_uart, LOG_LEVEL_INF);

static const struct device *const uart_dev =
    DEVICE_DT_GET(DT_ALIAS(test_uart));

RING_BUF_DECLARE(rx_rb, 1024);

static void uart_isr(const struct device *dev, void *user_data)
{
    ARG_UNUSED(user_data);

    while (uart_irq_update(dev) && uart_irq_is_pending(dev)) {
        if (uart_irq_rx_ready(dev)) {
            uint8_t buf[32];
            int n = uart_fifo_read(dev, buf, sizeof(buf));
            if (n > 0) {
                uint32_t pushed = ring_buf_put(&rx_rb, buf, n);
                if (pushed != (uint32_t)n) {
                    LOG_WRN("rx ring overflow, dropped %u bytes", n - pushed);
                }
            }
        }
    }
}

int test_uart_init(void)
{
    if (!device_is_ready(uart_dev)) {
        LOG_ERR("uart dev not ready");
        return -ENODEV;
    }

    struct uart_config cfg = {
        .baudrate  = 115200,
        .parity    = UART_CFG_PARITY_NONE,
        .stop_bits = UART_CFG_STOP_BITS_1,
        .data_bits = UART_CFG_DATA_BITS_8,
        .flow_ctrl = UART_CFG_FLOW_CTRL_NONE,
    };
    int err = uart_configure(uart_dev, &cfg);
    if (err) {
        LOG_ERR("uart_configure: %d", err);
        return err;
    }

    uart_irq_callback_set(uart_dev, uart_isr);
    uart_irq_rx_enable(uart_dev);

    LOG_INF("test UART up at 115200 8N1");
    return 0;
}

The pattern: configure the UART, register an ISR, enable RX interrupts, drain the hardware FIFO into a ring buffer in the ISR, consume from the ring in a worker thread. This survives bursts up to (ring-buffer-size / character-time) without back-pressure; beyond that you need either flow control or a bigger ring.

Async API — for DMA-driven bulk transfers

#include <zephyr/drivers/uart.h>

static uint8_t rx_dma_buf[2][512];   /* double-buffered */

static void uart_async_cb(const struct device *dev,
                          struct uart_event *evt, void *user_data)
{
    switch (evt->type) {
    case UART_RX_RDY:
        /* evt->data.rx.buf, .offset, .len describe the chunk */
        process_chunk(&evt->data.rx);
        break;
    case UART_RX_BUF_REQUEST:
        /* the driver wants the next buffer; provide one */
        uart_rx_buf_rsp(dev, rx_dma_buf[1], sizeof(rx_dma_buf[1]));
        break;
    case UART_RX_BUF_RELEASED:
        /* the driver is done with this buffer */
        break;
    case UART_RX_DISABLED:
        /* re-arm if you want continuous RX */
        uart_rx_enable(dev, rx_dma_buf[0], sizeof(rx_dma_buf[0]), 100);
        break;
    default:
        break;
    }
}

The async API is what Zephyr's high-throughput drivers use under the hood — LoRa concentrators, NMEA log capture, etc. It is overkill for a 115200-baud debug console, but the right choice when you are pumping multi-MByte/s through PIO-DMA-backed UART channels. See the Zephyr UART API docs for the full event vocabulary.

Bridging UART ↔ USB CDC-ACM in firmware

edevkit1's bridge is essentially a forever loop in two directions:

/* simplified pseudo-code; real code uses async API for both sides */
static void target_log_bridge_thread(void)
{
    uint8_t buf[256];

    for (;;) {
        /* drain UART -> ring -> CDC */
        size_t n = ring_buf_get(&uart_rx_rb, buf, sizeof(buf));
        if (n) {
            (void)uart_fifo_fill(cdc_dev, buf, n);
        }
        /* drain CDC -> ring -> UART */
        n = ring_buf_get(&cdc_rx_rb, buf, sizeof(buf));
        if (n) {
            (void)uart_fifo_fill(uart_dev, buf, n);
        }
        k_msleep(1);
    }
}

In practice the worker is interrupt-coalesced; the explicit k_msleep(1) is illustrative and would be a k_poll on real ring-buffer events.

The recommended default: tio

For new edevkit users on Linux/macOS we recommend tio. It does everything you want a serial terminal to do, with no surprises:

# open the target log UART at 115200 8N1
tio /dev/ttyACM1

# same, with explicit settings (defaults are 115200 8N1 already)
tio -b 115200 -d 8 -p none -s 1 /dev/ttyACM1

# log the session to a file (timestamped)
tio /dev/ttyACM1 --log session.log --log-strip

# auto-reconnect when the device drops (useful while flashing/resetting)
tio /dev/ttyACM1 --no-reconnect=false

# show inbound bytes as hex
# Ctrl-T, then 'h'  --  toggles hex display

# exit:  Ctrl-T, then 'q'

stty — configuring the line from a shell

stty sets line parameters on a tty without opening a terminal. Useful for scripts and one-shot flashing tools that wrap cat > /dev/ttyUSB0:

# inspect current settings
stty -F /dev/ttyACM1 -a

# set 115200 8N1, no flow control, "raw" (no echo, no line discipline)
stty -F /dev/ttyACM1 115200 cs8 -cstopb -parenb -ixon -ixoff -crtscts raw -echo

# set 921600 baud
stty -F /dev/ttyACM1 921600

# Macs: device path is /dev/cu.usbmodem* and stty looks slightly different
stty -f /dev/cu.usbmodem1101 115200 cs8 -cstopb -parenb raw -echo

# read a line from the shell:
cat /dev/ttyACM1                 # Ctrl-C to stop

# write a string:
echo "AT" > /dev/ttyACM1

stty flags worth memorising: cs8 sets 8 data bits; -cstopb means 1 stop bit (the minus disables 2-stop-bits); -parenb disables parity; raw turns off all line-discipline cooking; -echo stops the kernel echoing typed characters back. Without raw, the kernel will eat your binary data alive.

Capturing a session

# tio: built-in
tio /dev/ttyACM1 --log session.log

# screen: keystroke H, or:
screen -L /dev/ttyACM1 115200

# picocom:
picocom --baud 115200 --logfile session.log /dev/ttyACM1

# raw cat (after stty):
stty -F /dev/ttyACM1 115200 raw -echo
cat /dev/ttyACM1 | tee session.log

The Linux ModemManager problem

Modern Linux desktops ship ModemManager, a daemon that probes new serial devices to see if they're 3G/LTE modems. When you plug in edevkit, ModemManager opens /dev/ttyACM* and sends AT commands to it — which conflicts with your shell session and produces mysterious "device busy" errors or random characters appearing in the buffer.

The fix is a udev rule that tells ModemManager to ignore edevkit's USB VID/PID:

# /etc/udev/rules.d/77-edevkit.rules
# Block ModemManager from probing edevkit's CDC-ACM endpoints.
SUBSYSTEM=="tty", ATTRS{idVendor}=="2e8a", ATTRS{idProduct}=="000a", \
  ENV{ID_MM_DEVICE_IGNORE}="1"

# Make the device readable by anyone in 'plugdev'
SUBSYSTEM=="tty", ATTRS{idVendor}=="2e8a", ATTRS{idProduct}=="000a", \
  GROUP="plugdev", MODE="0660"

(Replace 000a with edevkit's actual product ID once allocated; see edevkit1.html §7 for the registered VID/PID.) Reload with sudo udevadm control --reload-rules && sudo udevadm trigger. macOS and Windows do not have this problem; on macOS the device just appears as /dev/cu.usbmodem*.

Common pitfalls and how to debug them

1. Wrong baud rate

Symptom: Clean garbage on the screen — characters that look almost ASCII but with the high bit set, or repeating patterns of bytes that don't match anything sane. Sometimes only every Nth character is right.

Diagnosis: The two ends are running at different baud rates (or one end mis-configured a fractional divider). The character pattern often hints at the ratio: if the receiver is at half the transmitter's baud, you'll see mostly 0xFE / 0xFF / 0x80 patterns as it samples in the wrong half of every bit.

Fix: Step down to 9600 baud on both sides until you see clean output, then ratchet up to find the highest reliable rate. Verify the transmitter's actual on-wire baud with a logic analyser if you have one — some MCUs have surprisingly inaccurate dividers.

2. TX/RX swapped

Symptom: Total silence in both directions, but the cable is plugged in and the LED is on.

Diagnosis: Cable manufacturers do not agree on what "TX" and "RX" mean on a labelled wire. Some ship through wiring (TX on the cable's TX wire = TX on both ends), others ship crossover wiring (TX on the cable's TX wire = RX on the other end). Both conventions are out there. The labels on dev-board headers also vary — sometimes "TX" means "TX of the board" (the board's transmit pin), sometimes "TX" means "to be wired to your transmit," which is the opposite.

Fix: Swap TX and RX. If both still show silence, swap and try the cable in the other orientation. As a sanity check, scope the TX pin while sending data — a real TX wiggles, a wrongly-connected one stays high.

3. Wrong logic level

Symptom: Either total silence, total noise, or smoke.

Diagnosis: One side is RS-232 (±12 V), the other is TTL (3.3 V). RS-232 into a TTL receiver damages the input; TTL into an RS-232 receiver may silently work, may produce garbage, or sometimes does nothing.

Fix: Check both sides' datasheets. Use a MAX3232 between RS-232 and TTL. Use a level-translator (LSF010x, TXS0108E, LVC8T245) between 5 V and 3.3 V or between 3.3 V and 1.8 V.

4. No common ground

Symptom: Intermittent garbage that worsens when you touch the cable, "ghost" reception when the cable is unplugged, errors that change with hand position.

Diagnosis: Two devices powered from different supplies with no GND wire between them are floating relative to each other. The receiver's logic threshold is meaningless — "high" and "low" are referenced to local ground, and they disagree.

Fix: Always include a GND wire. For longer cables, consider a star-grounded power scheme; for noisy environments, switch to RS-422/485 differential, which is immune to (some) ground offset.

5. ModemManager (Linux)

Symptom: The first character or two from the device is missing, or you see "AT", "ATQ0", or other modem strings appearing in your log right after plug-in.

Fix: Add the udev rule from the tooling section. Or, as a quick test, kill ModemManager: sudo systemctl stop ModemManager.

6. Half-byte garbage on first character after wake-up

Symptom: The first byte after enabling the UART is corrupt or flagged with framing-error; subsequent bytes are clean.

Fix: See the line-state vs line-idle gotcha. Drive TX high before enabling, wait one bit-time, then send. On RX, swallow the first framing error after enabling.

7. RS-485 driver-enable timing

Symptom: Last byte (or two) of every transmit is corrupt; protocol stack times out waiting for response.

Diagnosis: The RS-485 transceiver's DE pin was deasserted before the last byte fully shifted out, cutting off the stop bit and leaving the line in an undefined state for the receiver.

Fix: Hold DE asserted until the TX-complete (TC) flag fires — the actual end of the stop bit, not just "FIFO empty." On Zephyr's UART API that's the UART_TX_DONE async event.

8. Hardware overrun under bursty traffic

Symptom: Mostly works; occasionally drops bytes during high-traffic moments (the boot log dump, a long Modbus frame).

Fix: Move from polling to interrupt-driven, or interrupt-driven to DMA. Increase the ISR's priority above any interfering interrupt. Add hardware flow control if the protocol allows. As a last resort, lower the baud rate.

9. The "endianness" confusion

Symptom: Multi-byte values come out byte-reversed.

Diagnosis: UART itself is byte-by-byte; "endianness" appears at the protocol level. If you call uart_send(&u32_val, 4) on a little-endian MCU, the first byte on the wire is the LSB of the u32; if your peer reassembles it big-endian, the value is reversed.

Fix: Define your wire format up front. Pick big-endian (network byte order) for portable interop, or explicitly serialize byte-by-byte. Don't punning-cast structs onto the wire.

10. The cable is bad

Symptom: Inexplicable, intermittent, hardware-dependent failures.

Fix: When in doubt, swap the cable. Cheap dupont jumper wires fail constantly — the crimp loosens, the wire breaks inside the insulation, the female end loses spring tension. A 30-second cable swap saves an hour of debugging silicon.

debugging order When a serial link doesn't work, check in this order: (1) GND connected, (2) TX/RX not swapped, (3) baud rate matches, (4) levels match, (5) no ModemManager interference, (6) cable is fresh. About 95 % of bring-up failures are in this list.
