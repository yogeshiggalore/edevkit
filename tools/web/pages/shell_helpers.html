edevkit · workflow · shell helpers

One source command. Every dev task on the bench.

A tiny helper script that wraps west, sysbuild, probe-rs, BOOTSEL UF2, mcumgr, and the docs hub launcher behind one-word commands. Source it once from your shell rc; type edev_help any time. Works on macOS, Linux, and (via the PowerShell variant) Windows.

Based on the edevtoolkit/documents/env_script.txt pattern; edevkit-flavored for RP2350B + sysbuild + the umbrella-product layout.

Why a helper script

Without it, every iteration looks like:

cd ~/projects/temp/zephyrproject
west build -p always -b rpi_pico2/rp2350a/m33 --sysbuild \
    -s ~/projects/temp/edevkit/edevkit/firmware/app \
    -d ~/projects/temp/edevkit/edevkit/firmware/build
probe-rs download --chip RP2350 --binary-format bin --base-address 0x10000000 \
    ~/projects/temp/edevkit/edevkit/firmware/build/mcuboot/zephyr/zephyr.bin
probe-rs download --chip RP2350 --binary-format bin --base-address 0x10040000 \
    ~/projects/temp/edevkit/edevkit/firmware/build/app/zephyr/zephyr.signed.bin
…

With it:

edev_activate
cd "$EDEVKIT_REPO/firmware/app"
edev_build
edev_flash

Same effect. Less typing. Layout-vars are the only knobs — no path is hardcoded.

One-time install

Pre-requisite: the four layout env vars from installation.html → Pick a layout must already be exported in your shell rc.

Recommended — Pattern C (loaded with the venv) Run tools/setup_venv.sh (or setup_venv.ps1 on Windows) once. It creates the venv, installs west + docs-hub deps, and appends a one-line source to the venv's activate script. From then on, the helpers are loaded every time you enter the venv — no ~/.bashrc edits required.

Linux / macOS / WSL2 (bash, zsh)

bash "$EDEVKIT_REPO/tools/setup_venv.sh"

# now whenever you activate, helpers come with it:
source "$EDEV_VENV/bin/activate"
edev_help

Windows (native PowerShell)

& "$env:EDEVKIT_REPO\tools\setup_venv.ps1"

# activate brings the helpers along:
. "$env:EDEV_VENV\Scripts\Activate.ps1"
edev_help

What setup_venv does

  1. Creates the venv at $EDEV_VENV if missing (reuses it if present)
  2. Upgrades pip + wheel
  3. Installs west and the docs-hub deps (fastapi, uvicorn, jinja2)
  4. Appends — once, idempotently — this guarded source line to the venv's activate: 
    if [ -n "${EDEVKIT_REPO:-}" ] && [ -f "$EDEVKIT_REPO/tools/env_helpers.sh" ]; then
    source "$EDEVKIT_REPO/tools/env_helpers.sh"
fi
  5. Runs west zephyr-export if your Zephyr workspace is initialised

The actual helpers stay in the repo at $EDEVKIT_REPO/tools/env_helpers.sh. git pull updates them; the venv's activate picks up the new version on the next session. If you ever rm -rf $EDEV_VENV and re-create it, re-run setup_venv.sh to restore the source line.

Alternative — Pattern B (source from shell rc)

If you want the helpers loaded in every shell (not just inside the venv) — for example, you switch in and out of the venv often and don't want to remember edev_activate — append this to your ~/.bashrc / ~/.zshrc instead:

source "$EDEVKIT_REPO/tools/env_helpers.sh"

Trade-off: the helpers (and the $PATH addition for pioasm) are present even in shells unrelated to edevkit. Patterns C and B are mutually exclusive — pick one.

Idempotent Sourcing the helper file twice is a no-op (the PATH addition is guarded; functions just get redefined). Re-running setup_venv.sh doesn't re-append the source line.

Reference

Every helper checks the four layout vars are set before doing anything; if one is missing, you get a friendly pointer back to the install guide instead of a confusing path error.

⚡edev_activate
edev_activate

Source the Python venv at $EDEV_VENV/bin/activate. Brings west, pyocd, probe-rs (if installed), mcumgr, kibot, and the docs-hub deps onto your PATH in one shot.

🔧westz
westz <args…>

Run west from inside $ZEPHYR_WORKSPACE, regardless of your $PWD. West has to resolve modules relative to the workspace top-dir, so this avoids "couldn't find west.yml" errors when you invoke from inside the project repo.

📦edev_create_app
edev_create_app <name> [parent]

Scaffold a new Zephyr application at $PWD/<name>/ (default; override the parent dir with the second arg) with a working CMakeLists.txt, prj.conf (logging + shell enabled), board overlay stub, and a hello-world main.c. Refuses to clobber an existing dir.

🧱edev_build
edev_build [src] [board]

Pristine sysbuild build (MCUboot + signed app). Defaults: src=$PWD, board=$EDEV_BOARD_DEFAULT (rpi_pico2/rp2350a/m33). Output lands at ./build-<board-with-slashes-as-underscores>/ (e.g. ./build-rpi_pico2_rp2350a_m33/).

🧱edev_build_simple
edev_build_simple [src] [board]

Pristine non-sysbuild build. Single zephyr.bin / zephyr.uf2, no MCUboot wrapper. Use for quick samples that don't need OTA.

🚀edev_flash
edev_flash [src] [board]

Flash via probe-rs over SWD. Writes both mcuboot.bin at 0x10000000 and the signed app at 0x10040000. Falls back to a single image if the build is non-sysbuild.

🪄edev_flash_uf2
edev_flash_uf2 [src] [board]

UF2 drag-and-drop flash for BOOTSEL mode. Polls /Volumes/$EDEV_UF2_VOLUME (macOS) or /media/$USER/$EDEV_UF2_VOLUME (Linux) for up to 30 s, then copies the signed UF2 across. Default volume label: RP2350.

📡edev_ota
edev_ota [src] [board]

Push the signed image to a running edevkit via mcumgr using the edevkit connection profile. Run mcumgr -c edevkit image test <hash> + … reset after to swap.

📺edev_console
edev_console [port]

Open a serial console to the kit. Auto-detects /dev/cu.usbmodem* (macOS) or /dev/ttyACM* (Linux). Uses tio if available, falls back to screen at 115200.

📚edev_doc
edev_doc [port]

Open the local docs hub in your default browser. Starts the FastAPI server in the background (port 8765 by default) if it isn't already running.

❓edev_help
edev_help

Print every helper, every default value, and the four layout var values for a quick sanity-check.

Typical workflows

Daily firmware iteration (with SWD probe)

edev_activate
cd "$EDEVKIT_REPO/firmware/app"
edev_build && edev_flash && edev_console

First-flash via BOOTSEL (no probe yet)

edev_activate
cd "$EDEVKIT_REPO/firmware/app"
edev_build
# unplug the kit, hold BOOTSEL, plug back in:
edev_flash_uf2

OTA update to a kit running in the field

edev_activate
cd "$EDEVKIT_REPO/firmware/app"
edev_build
edev_ota
# then swap-and-confirm via mcumgr (see installation.html step 11b)

Scaffold + run a one-off learning sample

edev_activate
edev_create_app blinky_test
cd "$EDEVKIT_REPO/firmware/blinky_test"
edev_build_simple && edev_flash_uf2

Just open the docs

edev_doc                # boots the hub if needed, opens browser

Mapping from edevtoolkit's env_script.txt

If you've used the edevtoolkit pattern before, here's the diff:

edevtoolkitedevkit equivalentWhy
ZEPHYR_WORKSPACE_EDEVZEPHYR_WORKSPACEDrops the project-specific suffix. Now reusable across any Zephyr project on the machine.
(none — venv hardcoded)EDEV_VENVLayout flexibility — three layouts supported (see install guide).
EDEV_BOARD_DEFAULT=rpi_picorpi_pico2/rp2350a/m33RP2350B (Cortex-M33) vs RP2040. Zephyr 3.7+ requires fully-qualified board targets when a board has multiple cores.

Same shape, different defaults The spirit of env_script.txt is unchanged: a thin layer of bash functions over west, with a couple of project-aware env vars. Anyone who learned the edevtoolkit pattern can use this without re-learning anything — just type edev_help.

Hacking on the helpers

The script is plain bash — open it and edit:

$EDITOR "$EDEVKIT_REPO/tools/env_helpers.sh"
exec $SHELL -l         # reload

Add a helper by appending a function. Conventions:

  • Prefix names with edev_ to keep the namespace tidy.
  • Call _edev_check first if the helper needs the layout vars.
  • Default any optional argument with ${1:-fallback} so callers can omit positional args.
  • Print friendly status with the emoji prefix convention (🚀 launching, 🧱 building, 📡 network, ✅ success, ❌ failure).
  • Keep the script idempotent — sourcing twice should be a no-op.
