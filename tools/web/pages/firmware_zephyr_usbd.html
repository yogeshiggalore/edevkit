Zephyr USB-Device-Next reference

The new (post-3.x) Zephyr USB device stack — how it's structured, how to bring up a class, and the seven things that bit us during usb_01_hello_enum. Written for new contributors who'll write samples 02 onward.

Pick the right stack

Zephyr ships two USB device stacks side-by-side. They use different macro families and different headers, and turning the wrong one on gives confusing build errors.

StackKconfigMacro familyStatus
USB-Device-NextCONFIG_USB_DEVICE_STACK_NEXT=yUSBD_* (e.g. USBD_DEVICE_DEFINE)Use this — actively maintained
Legacy USB-DeviceCONFIG_USB_DEVICE_STACK=yusb_* (e.g. usb_dc_*, usb_register_*)Frozen; will be removed
Single most common build error

If USBD_* macros are undefined when you compile, the next-stack Kconfig isn't set. Check prj.conf for CONFIG_USB_DEVICE_STACK_NEXT=y first, before anything else.

Three-layer architecture

┌─────────────────────────────────────────────────────────────┐
│ Application                              src/main.c          │
│  - Builds a usbd_context (VID/PID/strings/configs/classes)   │
│  - Calls usbd_init() then usbd_enable()                      │
├─────────────────────────────────────────────────────────────┤
│ usbd core           subsys/usb/device_next/                 │
│  - Manages enumeration state machine                         │
│  - Dispatches standard requests to class drivers             │
│  - Exposes the USBD_*_DEFINE macro family                    │
├─────────────────────────────────────────────────────────────┤
│ udc driver          drivers/usb/udc/udc_<soc>.c              │
│  - Per-SoC: one driver per USB peripheral                    │
│  - rpi_pico2/rp2350: udc_rpi_pico.c                          │
│  - Bound automatically via devicetree (compatible string)    │
└─────────────────────────────────────────────────────────────┘

You only write at the top layer. The core is generic. The bottom layer is bound by devicetree — you don't touch USB controller registers.

Boot sequence in main()

Every USB-Device-Next sample's main() follows this order. Skipping any step, or running them out of order, breaks enumeration.

1. usbd_add_descriptor(&ctx, &lang)              // language IDs (index 0)
2. usbd_add_descriptor(&ctx, &mfr)               // iManufacturer (index 1)
3. usbd_add_descriptor(&ctx, &product)           // iProduct       (index 2)
4. usbd_add_descriptor(&ctx, &serial)            // iSerialNumber  (index 3)
5. usbd_add_configuration(&ctx, USBD_SPEED_FS, &cfg)
6. usbd_register_class(&ctx, "class_name", USBD_SPEED_FS, 1)  // 1 = config index
7. usbd_init(&ctx)        // freezes descriptor tree, computes wTotalLength
8. usbd_enable(&ctx)      // connects the D+ pull-up — host can now see us
9. k_sleep(K_FOREVER)      // stack runs in its own work queue
Order matters

Configurations must exist before classes can attach to them. Classes must exist before usbd_init validates the tree. usbd_init must succeed before usbd_enable connects the pull-up.

Implementing a class

A class is three things in one C file:

  1. Descriptor blob — packed C struct(s) for the interface(s) and endpoints.
  2. NULL-terminated array of descriptor pointers — what get_desc returns.
  3. Vtable + USBD_DEFINE_CLASS — registers the class instance with the core.

The minimum vtable for a vendor class with no functional behavior:

static int  hello_enum_init   (struct usbd_class_data *const c);  // -> 0
static void hello_enum_enable (struct usbd_class_data *const c);  // log
static void hello_enum_disable(struct usbd_class_data *const c);  // log
static void *hello_enum_get_desc(struct usbd_class_data *const c,
                                 const enum usbd_speed speed);    // returns array

static const struct usbd_class_api hello_enum_api = {
    .init     = hello_enum_init,
    .enable   = hello_enum_enable,
    .disable  = hello_enum_disable,
    .get_desc = hello_enum_get_desc,
    /* feature_halt, update, control_to_dev, control_to_host,
       request, suspended, resumed, sof, shutdown — all NULL */
};

USBD_DEFINE_CLASS(hello_enum, &hello_enum_api, NULL, NULL);

The fourth argument to USBD_DEFINE_CLASS is the optional v_reqs (vendor request handler). Passing NULL is fine when you have no vendor-specific control requests.

The usbd_class_api vtable — actual signatures

Pulled directly from $ZEPHYR_BASE/include/zephyr/usb/usbd.h. These are the spellings; do not guess.

FieldSignatureNotes
initint (struct usbd_class_data *)Class init at usbd_init time. Return 0 on success.
shutdownvoid (struct usbd_class_data *)Symmetric cleanup.
enablevoid (struct usbd_class_data *)Fired on SET_CONFIGURATION(n≥1). The "I'm now operational" hook.
disablevoid (struct usbd_class_data *)Fired on SET_CONFIGURATION(0) or unplug.
control_to_devint (c, setup, const net_buf *buf)Class-specific OUT setup. Return -ENOTSUP for requests you don't handle.
control_to_hostint (c, setup, net_buf *buf)Class-specific IN setup.
requestint (c, net_buf *buf, int err)Endpoint transfer completion. Only fires for non-EP0 endpoints.
feature_haltvoid (c, uint8_t ep, bool halted)Endpoint halt-feature update notification.
updatevoid (c, uint8_t iface, uint8_t alt)Fired on SET_INTERFACE alt-setting change.
suspendedvoid (struct usbd_class_data *)Bus suspend (3 ms idle). Drop power if you can.
resumedvoid (struct usbd_class_data *)Bus resume.
sofvoid (struct usbd_class_data *)Start-of-frame; only fires if you set this. Leave NULL unless you specifically need per-frame ticks.
get_descvoid * (c, enum usbd_speed)Must return a NULL-terminated array of struct usb_desc_header *, NOT a single pointer. See gotcha #4.
NULL is fine

Optional callbacks left as NULL in the vtable initializer are handled by the core. You only need to populate the ones your class actually uses. For a vendor class with zero data endpoints, that's typically just init, enable, disable, and get_desc.

Where to find truth in the tree

Always cross-check against the actual headers in your version. Recall is not authoritative.

# the public API
$ZEPHYR_BASE/include/zephyr/usb/usbd.h

# chapter-9 standard descriptor structs and constants
$ZEPHYR_BASE/include/zephyr/usb/usb_ch9.h

# existing class implementations to copy patterns from
$ZEPHYR_BASE/subsys/usb/device_next/class/usbd_dfu.c       # small, runtime-mode has 0 data endpoints
$ZEPHYR_BASE/subsys/usb/device_next/class/usbd_cdc_acm.c   # composite IAD example
$ZEPHYR_BASE/subsys/usb/device_next/class/usbd_hid.c       # HID-class pattern

One-line greps you'll use often:

# Find the real signature of any usbd_* function
sed -n '/^struct usbd_class_api/,/^};/p' $ZEPHYR_BASE/include/zephyr/usb/usbd.h

# Confirm a macro's argument count before calling it
grep -nE "^#define\s+USBD_DESC_" $ZEPHYR_BASE/include/zephyr/usb/usbd.h

# Find the right UDC node label for your board
grep -i "compatible.*pico-usbd\|usbd:" \
  $ZEPHYR_BASE/boards/raspberrypi/rpi_pico2/*.dts

The seven gotchas (catalogue)

Bugs we hit during usb_01 bring-up, in order of frequency. Read these before writing your next sample.

1. USBD_DESC_SERIAL_NUMBER_DEFINE takes ONE argument

Not (name, "string"). The serial is auto-generated from the chip's hardware ID via hwinfo_get_device_id(). Just USBD_DESC_SERIAL_NUMBER_DEFINE(serial); — done.

2. Devicetree node label must match the board's .dts

Build error: '__device_dts_ord_DT_N_NODELABEL_xxx_ORD' undeclared. The fix is to grep the board file for the actual USB controller node label and use it in both app.overlay (&label { status = "okay"; };) and main.c (DT_NODELABEL(label)).

3. Most callbacks return void, not int

The compiler error is incompatible pointer types. Reference the table above; void for all lifecycle/notification callbacks (enable/disable/feature_halt/update/suspended/resumed/sof/shutdown), int only for init, request, control_to_*.

4. get_desc returns a NULL-terminated array of pointers

Symptom: BUS FAULT, BFAR=0xffffff00 at boot, in init_configuration_inst. Root cause: the core walks the returned value as struct usb_desc_header **, dereferencing each pointer until it finds NULL. Returning (void *)&your_iface makes it interpret the descriptor's bytes as pointers — chaos. The fix:

static struct usb_desc_header *fs_desc[] = {
    (struct usb_desc_header *) &your_iface,
    NULL,    /* terminator */
};

static void *get_desc(c, speed) {
    if (speed == USBD_SPEED_FS) return fs_desc;
    return NULL;
}

5. Don't const interface descriptors

Zephyr writes bInterfaceNumber at init time. If you mark the descriptor const and the linker places it in flash, the write either silently fails or hard-faults. Leave them mutable:

static struct usb_if_descriptor my_iface = {  /* no const */
    ...
};

6. Class name string mismatch is a silent failure

USBD_DEFINE_CLASS(hello_enum, ...) creates a registration keyed on "hello_enum". usbd_register_class(&ctx, "hello_enum", ...) must use the identical string. Any typo → -ENOENT at registration, no enumeration. Defend with a single #define:

#define HELLO_ENUM_NAME "hello_enum"
USBD_DEFINE_CLASS(hello_enum, ...);
...
usbd_register_class(&ctx, HELLO_ENUM_NAME, ...);

7. Returning 0 for unsupported requests hangs the host

If your control_to_host or control_to_dev returns 0 for a request you didn't handle, the core thinks you supplied a response and the host waits forever for data. Always return -ENOTSUP for class-specific requests you don't implement — the core then STALLs the request, which is correct per USB 2.0 §9.2.7.

Optional: byte-perfect device descriptor for Book Ch 05

Zephyr USB-Device-Next sets the device-class triple to 0xEF/0x02/0x01 (IAD) by default, even on single-interface devices. Spec-canonical "see interface descriptors" is 0x00/0x00/0x00. To match Book Ch 05's worked example exactly, override after usbd_init:

err = usbd_init(&ctx);
if (err) goto fail;

usbd_device_set_code_triple(&ctx, USBD_SPEED_FS, 0x00, 0x00, 0x00);

err = usbd_enable(&ctx);

Functionally identical (every host accepts both). Cosmetic only.
