SWO and ITM — single-pin Cortex-M trace

A focused reference on Serial Wire Output (SWO) — the one-pin asynchronous trace stream emitted by every SWD-capable Cortex-M core — and the on-chip Instrumentation Trace Macrocell (ITM), Data Watchpoint and Trace (DWT), and Trace Port Interface Unit (TPIU) blocks that produce, decorate, and serialise the bytes that ride it. SWO is the fastest, lowest-effort observability channel a debugger has into a running Cortex-M — much cheaper than full ETM trace, and far higher bandwidth than printf-over-UART.

What's covered. Why SWO is "free" (one pin you already routed for SWD), the two encoding modes (UART NRZ vs Manchester), the CoreSight chain that produces it (CPU → ITM → DWT → TPIU → SWO pin), TPIU packet framing, ITM stimulus ports and the canonical "printf to ITM" pattern, DWT counters and PC sampling for poor-man's profiling, the exact register sequence to bring SWO up from C, how a debug probe captures the line and demuxes the stream, host tooling (J-Link SWO Viewer, Orbuculum, OpenOCD, pyOCD, probe-rs), and how edevkit1 implements the capture path on a PIO state machine over CMSIS-DAP v2.

Audience. Firmware engineers who already have SWD bring-up working (see debug_swd.html) and want logging, timing, and event tracing without burning a UART. Probe authors who need to understand what bytes to expect on the SWO line. edevkit contributors touching the SWO subsystem — see edevkit1.html for the v1 design context and platform_pio.html for the PIO program that captures the line.

What SWO is, and why it's "free"

Serial Wire Output (SWO) is a single-pin, single-direction, asynchronous serial output that exits the Cortex-M core and carries trace data out of the chip to a debug probe. It's strictly target-to-host — never bidirectional — and rides one GPIO of the silicon, typically a dedicated SWO / TRACESWO ball on the package. On a debug header, SWO sits next to the SWD pair as the third "trace" pin (pin 6 on the standard 10-pin Cortex Debug + ETM connector). See debug_swd.html for the SWD pair (clock + data) it ships alongside.

The reason SWO matters is bandwidth-per-cost. Every other "where is my code right now?" channel costs you something:

ChannelPinsTypical bandwidthCaveats
UART printf2 (TX + GND)~1 MbpsSteals a UART; CPU formatting cost; blocking
SWO (UART mode)12–60 Mbps (silicon-dependent)One-way, target → probe only
SWO (Manchester)1~half of UART modeSelf-clocking; less host support
RTT (Segger)0 — over SWD~1–3 MB/s practicalBurns RAM ring buffer; needs probe to poll memory
ETM (full instruction trace)4–5 (TRACECLK + TRACEDATA[0..3])up to several GbpsRouting nightmare; probe must support trace port

SWO occupies the sweet spot: one pin you've already routed (SWO ships on the same Cortex Debug header as SWCLK/SWDIO), tens of Mbps usable on most modern silicon, and a hardware-multiplexed stream that mixes printf-style instrumentation, exception-trace events, and program-counter sampling into the same byte sequence. It's "free" in the sense that the routing cost is one extra trace and a 22-Ω series resistor; no PCB redesign, no extra connector, no extra IO buffers.

The lineage: SWO is part of CoreSight — Arm's on-chip debug-and-trace IP suite — and was introduced alongside the Serial Wire Debug (SWD) protocol with Cortex-M3 (Armv7-M) circa 2004. Every Cortex-M3, M4, M7, M33, M55, and M85 has it. Cortex-M0 / M0+ (Armv6-M) typically don't ship ITM, so SWO is rare on those parts; some vendors include the trace pin and a stripped-down ITM, most don't. (Memfault's profiling-on-Cortex-M post and the Arm CoreSight TRM both call this out.)

Mental model Think of SWO as the narrow side of the CoreSight trace funnel. ETM is the wide side: a parallel bus carrying every PC the core retires, used for reverse-execution debugging on workstation-grade probes. SWO is the same protocol family, multiplexed down to one pin, carrying the events firmware actually opted in to emit. You won't reverse-execute over SWO, but you will get cycle-accurate timestamps on every printf, every exception entry/exit, and a periodic PC sample at up to ~1 kHz without instrumenting the code.

Encoding modes — UART (NRZ) vs Manchester

The TPIU drives the SWO pin in one of two line-coding modes, selected by the firmware via TPI->SPPR (Selected Pin Protocol Register).

UART / NRZ — the dominant mode

UART mode (sometimes called "asynchronous" or "NRZ" in the Arm docs) sends each byte as a standard 8-N-1 UART frame: one start bit (low), eight data bits LSB-first, no parity, one stop bit (high). The line idles high. There is no clock pin — the receiver recovers timing from the start-bit edge and oversamples each bit at the agreed baud rate. Set SPPR.TXMODE = 0b10 for this mode (the value is 2 in SPPR[1:0]).

This is what 95%+ of host tooling expects, and what every common probe (J-Link, ST-Link v2/v3, CMSIS-DAP v2, MCU-Link, Orbtrace, edevkit) defaults to. The reason: a generic UART block, a PIO state machine, or even a bit-banging GPIO ISR can decode it. There's no specialised "SWO PHY" required — it's a UART that happens to carry framed CoreSight data.

Baud-rate constraint. The TPIU prescaler (TPI->ACPR, "Asynchronous Clock Prescaler Register") divides the trace clock down to the SWO baud:

SWO_baud = TRACECLK / (ACPR + 1)

where TRACECLK is the trace input clock (usually equal to the core clock, but on some silicon you can divide it). This means only baud rates that are integer divisors of the core clock are reachable cleanly. On a 150 MHz part, you can get exactly 150/2 = 75 MHz, 150/3 = 50 MHz, … , 150/25 = 6 MHz, 150/100 = 1.5 MHz, etc. Asking for "115200 baud" usually picks the nearest divisor; the resulting actual baud may be off by a few percent, which a forgiving UART receiver tolerates but a tight one does not. Always pick a baud whose divisor is integer for your TRACECLK; otherwise you'll see occasional framing errors and the host will report scrambled bytes.

Most probes prefer round numbers like 1, 2, 4, 6 MHz. J-Link, OpenOCD's tpiu config, and probe-rs all auto-pick the highest divisor that's within tolerance of the requested baud and report the actual rate back; check the log line.

Manchester — self-clocking, less common

Manchester mode (SPPR.TXMODE = 0b01) line-codes each bit as a transition: a 0 is a high-to-low transition mid-bit-cell, a 1 is low-to-high (the IEEE 802.3 convention). Self-clocking means the receiver doesn't need to know the baud rate ahead of time — every bit cell has a guaranteed edge — so a probe can recover the bit clock from the data itself.

The cost: half the throughput at the same line frequency, more sophisticated decoder (a state machine, not a UART). Almost no host tool uses Manchester anymore — it was useful in the early CoreSight days when probes had only generic UARTs and had to deal with target clocks they couldn't measure. Modern probes measure target clock during SWD enumeration, so they can configure the target and themselves to a known UART baud and skip Manchester. Default to UART mode unless you have a specific reason not to.

Pitfall If host tooling reports "SWO captured but garbage bytes", first suspect a baud mismatch: the firmware computed ACPR against an assumed TRACECLK that's wrong (e.g. you switched from XOSC to PLL after init but didn't reconfigure SWO). Second suspect: TPI->FFCR.EnFCont — see TPIU framing. Third: the probe is in Manchester mode but the target is in UART, or vice versa.

The CoreSight chain — what generates SWO data

SWO is the output pin. The bytes that come out of it are the multiplexed product of three on-chip CoreSight blocks: ITM, DWT, and TPIU. Each generates a different kind of trace packet, the TPIU framer interleaves them, and the SWO pin drains the framer FIFO.

ITM — Instrumentation Trace Macrocell

The Instrumentation Trace Macrocell is a software-driven trace generator. Firmware writes to one of 32 memory-mapped stimulus ports, each at a fixed offset in the ITM's MMIO window (0xE0000000 on Cortex-M3/M4/M7, also 0xE0000000 on M33's secure address space). Each write becomes a TPIU packet emitted on SWO, tagged with the port number. ITM is purely software-driven — it doesn't observe any other state.

ITM is the channel firmware uses for printf-style logging, structured events ("entered state X"), and any application-level instrumentation. Each port is independently enable-able, so you can route different log domains to different ports and have the host filter on port number. See § ITM stimulus ports for the canonical printf pattern.

DWT — Data Watchpoint and Trace

The Data Watchpoint and Trace unit is the hardware-driven trace generator. It observes the bus and the core's internal state, and emits packets for things firmware doesn't have to instrument:

  • Cycle counter (CYCCNT) — a 32-bit free-running counter that increments every core clock cycle. The host uses this to timestamp every other DWT/ITM packet at single-cycle resolution.
  • Event counters — CPI (cycles per instruction overhead), EXC (exception overhead cycles), SLEEP (sleep-mode cycles), LSU (load/store unit cycles), FOLD (folded-instruction count). Each is an 8-bit counter that emits an event packet on overflow.
  • Watchpoints with trace — up to four data-address comparators that emit a packet on read, write, or both, optionally including the address and the data value.
  • PC sampling — periodic emission of the current PC, configurable rate (typically thousands of samples per second).
  • Exception trace — packet on every exception entry, exit, and return, with the exception number.

DWT is the source of the "is my idle hook actually idle?" data. PC sampling specifically is the killer feature: at ~1–10 kHz, accumulated over a few seconds, you get a statistical distribution of where the PC spends its time. Run a flame-graph tool over that and you have a profiler — without instrumenting any code. (Memfault's "Profiling firmware on Cortex-M" post walks through this end-to-end.)

TPIU — Trace Port Interface Unit

The Trace Port Interface Unit sits at the output of ITM and DWT. It does three things:

  1. Framer. Interleaves multiple trace sources (in M-class with SWO, that's just ITM+DWT) into a single byte stream, optionally prefixing each chunk with a source-ID byte so the host can demux. Controlled by TPI->FFCR (Formatter and Flush Control Register).
  2. Async output driver. Drives the SWO pin in either UART NRZ or Manchester mode. Configured by TPI->SPPR + TPI->ACPR.
  3. Buffer / FIFO. A small (typically 4-deep) FIFO between the framer and the line; if it fills, ITM writes to a busy port stall the CPU briefly.

For SWO the framer is usually disabled — there's only one stream going out, so wrapping it in another framing layer just costs bytes and confuses receivers. That's the difference between FFCR.EnFCont = 0 (preferred for SWO) and EnFCont = 1 (continuous-formatted, used for parallel TPIU trace where multiple sources actually need demux). See § Common pitfalls — leaving the formatter on is one of the top three "garbage on SWO" causes.

ITM stimulus ports — printf-to-trace

The ITM exposes 32 stimulus ports as 32-bit MMIO registers:

ITM->PORT[0]   at 0xE0000000
ITM->PORT[1]   at 0xE0000004
ITM->PORT[2]   at 0xE0000008
…
ITM->PORT[31]  at 0xE000007C

A write to ITM->PORT[N] emits an SWIT (Software Instrumentation Trace) packet on SWO with port number N and the written payload. The store size matters: a STRB emits a 1-byte payload, a STRH emits 2 bytes, a STR emits 4 bytes. The header byte tells the host the size.

Before you write, you have to read the port and check the LSB. ITM->PORT[N] read returns a status: bit 0 set means "port FIFO has space, you may write". If it's 0, the port is full — the framer hasn't drained the previous byte yet. Spinning is the conventional approach. Drop-on-full is also reasonable for non-critical logging.

The canonical SWO printf helper

This is the function every ARM-GCC SWO setup ends up writing some variant of:

#include "core_cm33.h"   /* CMSIS-Core for the M33; same idea on M4/M7 */

/**
 * @brief Send one byte to ITM stimulus port `port`.
 * @retval 1 on success, 0 if dropped (port disabled or FIFO full and we don't spin)
 */
static inline int itm_send_char(uint8_t port, uint8_t ch)
{
    /* TRCENA must be set or the whole CoreSight tree is held in reset. */
    if ((CoreDebug->DEMCR & CoreDebug_DEMCR_TRCENA_Msk) == 0)
        return 0;

    /* Port must be enabled in ITM_TER. */
    if ((ITM->TER & (1u << port)) == 0)
        return 0;

    /* Wait until FIFO has room (bit 0 of port read = ready). */
    while (ITM->PORT[port].u32 == 0) { /* spin */ }

    /* 8-bit write → 1-byte SWIT packet. */
    ITM->PORT[port].u8 = ch;
    return 1;
}

/* Hook for newlib's _write so printf goes to ITM port 0. */
int _write(int fd, const char *buf, int len)
{
    for (int i = 0; i < len; i++) itm_send_char(0, buf[i]);
    return len;
}

That's the entirety of "printf over SWO". With TRCENA set, port 0 enabled in ITM->TER, and the TPIU configured (next section), printf("hello\n") from any thread or ISR emits seven SWIT packets on SWO, the probe decodes them, and the host displays them. Total firmware overhead: a few cycles per byte (the spin is rare in practice — TPIU drains faster than printf produces).

Why ports? Port 0 is the printf convention; ports 1–31 are yours. Use them as channels: port 1 = USB events, port 2 = scheduler trace, port 3 = ADC samples (binary, decoded host-side), etc. The host can filter on port number, so you can leave them all enabled and only display what you care about. Don't emit binary data to port 0 — every viewer assumes UTF-8 there.

A subtle compiler problem

The ITM port registers are MMIO with side effects: writing to them transmits data. But if the compiler doesn't know the writes have side effects, it may eliminate them. CMSIS's core_cm33.h declares ITM->PORT[] as volatile, which guarantees the writes are emitted. If you reach for the registers via your own pointer cast — e.g. *(uint32_t *)0xE0000000 = ch; without volatile — link-time optimisation may drop the store. Always go through the CMSIS macros, or define your own with explicit volatile.

TPIU packet framing

The byte stream on SWO is a sequence of packets, each consisting of a 1-byte header followed by 0–4 bytes of payload. The header encodes the source (which CoreSight block emitted it), the port or event number, and the payload size. This is defined in the Armv7-M ARM (DDI 0403) §D4 "ITM packet protocol" and the Arm CoreSight Components TRM (DDI 0314) §11–13.

nnnnn1SS (SS≠0)HW source — DWT-generated1, 2, or 4 bytes; nnnnn = DWT discriminator
0nnnnn00Local timestamp (varint)1–4 follow-bytes; high bit chains
10010100Global timestamp 11–4 follow-bytes
10110100Global timestamp 24 follow-bytes

01110000Overflow0 — host should warn user

The size encoding in the low two bits (SS):

SS = 01  →  1 byte payload
SS = 10  →  2 byte payload
SS = 11  →  4 byte payload
SS = 00  →  not a SWIT/HW header (see other tables)

Worked example — ITM_SendChar('A') on port 0

Firmware does ITM->PORT[0].u8 = 'A';. The ITM emits one SWIT packet:

Header byte: 00000001 = 0x01
             ││││││└┴── SS = 01  (1-byte payload)
             │││││└──── 0       (SWIT, not HW)
             └┴┴┴┴───── port = 00000 (port 0)

Payload byte: 0x41 = 'A'

Wire bytes: 0x01 0x41

As UART NRZ at e.g. 6 MHz, 8-N-1, that's 2 bytes × 10 bit times = ~3.3 µs total.

A 32-bit binary log to port 5 — ITM->PORT[5].u32 = 0xDEADBEEF;:

Header byte: 00101011 = 0x2B
             ││││││└┴── SS = 11  (4-byte payload)
             │││││└──── 0       (SWIT)
             └┴┴┴┴───── port = 00101 (port 5)

Payload bytes (little-endian): 0xEF 0xBE 0xAD 0xDE

Wire bytes: 0x2B 0xEF 0xBE 0xAD 0xDE

Sync packets — how the host re-aligns

If the host starts capture mid-stream, it can't know where one packet ends and the next begins. The TPIU periodically emits a sync sequence: at least five bytes of 0x00 followed by one byte of 0x80. Specifically: 47 consecutive zero bits followed by a single 1 bit, padded out to byte boundaries. Decoders look for this pattern (six bytes of 0x00 0x00 0x00 0x00 0x00 0x80 works as a robust matcher) and use it to byte-align the packet stream.

Sync emission is enabled by DWT->CTRL.SYNCTAP (see Armv7-M ARM §D4.2.5): set it to a non-zero value to have DWT trigger sync packets on cycle-counter rollover. SYNCTAP = 0b01 emits sync every 2^24 cycles (~110 ms at 150 MHz); 0b10 every 2^26; 0b11 every 2^28. Always enable some sync — host tools will refuse to display anything until they see one.

Overflow packets — back-pressure made visible

If the TPIU FIFO fills (target produces faster than line drains), subsequent ITM writes are dropped, and the next free slot in the FIFO emits an overflow packet (0x70). Host tools display this as a "trace overflow" warning. Seeing a few overflows during a verbose burst is normal; seeing thousands means you need a higher SWO baud, fewer enabled ports, or you've enabled DWT PC sampling at too high a rate.

DWT counters, watchpoints & PC sampling

DWT is the hardware-driven half of the trace stream. Most firmware engineers underuse it because the registers look intimidating; they're not, once you've seen them once.

CYCCNT — the cycle counter

DWT->CYCCNT is a 32-bit free-running counter that increments once per core clock when DWT->CTRL.CYCCNTENA = 1. At 150 MHz it wraps every ~28.6 seconds. This is the single most useful register on the chip for performance work:

uint32_t t0 = DWT->CYCCNT;
my_thing();
uint32_t dt_cycles = DWT->CYCCNT - t0;     /* unsigned wrap is fine */

Single-cycle resolution. No interrupt overhead. Works inside ISRs. Costs nothing when not used. Always enable CYCCNT in your debug build. It's the foundation of every other timing measurement — DWT exception trace, ITM timestamp packets, and PC samples are all timestamped against it.

Event counters

DWT->CTRL also enables a battery of 8-bit event counters that emit a DWT-source SWO packet on overflow:

CounterEnable bitCounts
CPICNTCPIEVTENACycles where the CPU stalled because a multi-cycle instruction was in flight (CPI overhead)
EXCCNTEXCEVTENACycles spent in exception entry/exit overhead
SLEEPCNTSLEEPEVTENACycles the core was in sleep (WFI/WFE)
LSUCNTLSUEVTENACycles the LSU stalled the pipeline
FOLDCNTFOLDEVTENAInstructions that were "folded" (executed in zero cycles)

The host accumulates these against CYCCNT to produce a per-second breakdown: "this firmware spent 18% of its cycles in exception overhead, 60% sleeping, 22% running real code". That's the kind of insight you can't get from printf.

Watchpoints with trace generation

The DWT has up to four comparators (DWT->COMP0..COMP3 with mask in MASK0..MASK3 and function in FUNCTION0..FUNCTION3). Each compares against the bus address of every load/store, and on match can trigger a halt, a watchpoint, or emit a DWT-source packet on SWO containing the data value, the PC, or both. The packet shows up as an HW-source SWIT-shaped event the host decodes.

Worked example: track every write to a specific RAM word.

/* Watch every write to my_state.counter */
DWT->COMP0     = (uint32_t)&my_state.counter;
DWT->MASK0     = 0;                               /* exact-match address */
DWT->FUNCTION0 = (1 << 0) |                       /* match read+write */
                 (1 << 11) |                      /* MATCH = data write */
                 (0b10 << 6);                     /* EMITRANGE: data + addr trace */

The exact FUNCTION bit assignments differ slightly between Armv7-M and Armv8-M (the M33 reorganised them — see DDI 0553 §D1.2). Always cross-check the field names against your specific core's TRM.

PC sampling — the killer feature

Set DWT->CTRL.PCSAMPLENA = 1 and POSTINIT + POSTPRESET to a sensible rate (the period is (POSTPRESET + 1) × 2^N cycles where N ∈ {6, 10} via CYCTAP). At ~10 kHz, over 10 seconds you collect 100 000 samples. Group those by function and you have a profiler-grade flame graph of where your firmware actually spends time — and unlike instrumentation profilers, it sees the idle hook, ISRs, and library code equally.

Tools that consume PC samples directly: J-Link's Performance Counter Profiler, Segger Ozone's Code Profile view, and Orbuculum's orbprofile. probe-rs has experimental support too.

Recipe Bring up CYCCNT and PC sampling on day one of every Cortex-M project. Even before you have any printf working, you'll be able to flame-graph the firmware. Once printf-over-ITM is also working, you essentially have all the dynamic-analysis tools a desktop programmer has.

Configuring SWO from C — the register sequence

The bring-up sequence trips up almost everyone the first time, because the registers are scattered across four MMIO blocks (CoreDebug, ITM, DWT, TPI) and the order matters. Here's the canonical sequence for "port 0 enabled, CYCCNT running, 6 MHz UART SWO, no formatter, sync every 2^24 cycles", which is the reasonable default for most projects.

#include "core_cm33.h"   /* or core_cm4.h, core_cm7.h, etc. */

/**
 * @brief Bring up SWO for ITM port 0 + DWT cycle counter.
 *
 * @param core_clock_hz    The clock feeding TPIU's TRACECLKIN (typically core clk).
 * @param swo_baud_hz      Desired SWO baud (e.g. 6'000'000). Must be a clean
 *                         integer divisor of core_clock_hz.
 *
 * Call this after the system clock is stable. Recall it if the system clock changes.
 */
void swo_init(uint32_t core_clock_hz, uint32_t swo_baud_hz)
{
    /* 1. Master enable for the entire CoreSight trace tree.
     *    Without TRCENA, ITM/DWT/TPIU registers read as zero and writes are dropped.
     *    DEMCR is in CoreDebug, not ITM/DWT — common confusion. */
    CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;

    /* 2. Unlock ITM. The Lock Access Register is a CoreSight convention:
     *    most CoreSight blocks ignore writes until you write the magic word
     *    0xC5ACCE55 to LAR ("ACCESS"). ITM, DWT, TPI, and ETM all share it. */
    ITM->LAR = 0xC5ACCE55;

    /* 3. Configure TPIU output mode and baud divider.
     *    TPI is also lockable — unlock first if your part requires it.
     *    Most M-class parts have TPI's lock implicitly cleared when DEMCR.TRCENA = 1.*/
    TPI->SPPR = 0x2;                                   /* 2 = UART NRZ; 1 = Manchester */
    TPI->ACPR = (core_clock_hz / swo_baud_hz) - 1;     /* prescaler = TRACECLK/baud - 1 */

    /* 4. Disable formatter (single-source: no need to wrap in TPIU framing).
     *    Leaving EnFCont = 1 here is the #1 cause of "garbage on SWO". */
    TPI->FFCR = 0x100;                                 /* TrigIn enabled, formatter off */

    /* 5. Configure ITM. */
    ITM->TCR = (1 << ITM_TCR_ITMENA_Pos)    |          /* enable ITM */
               (1 << ITM_TCR_TSENA_Pos)     |          /* local timestamps on */
               (1 << ITM_TCR_SYNCENA_Pos)   |          /* allow sync packets */
               (1 << ITM_TCR_DWTENA_Pos)    |          /* DWT events go to TPIU too */
               (1 << ITM_TCR_SWOENA_Pos)    |          /* SWO output enabled */
               (1 << ITM_TCR_TraceBusID_Pos);          /* bus ID 1 (any non-zero) */

    /* 6. Enable individual stimulus ports. Bit N = port N. */
    ITM->TER = 0x1;                                    /* port 0 only — add bits as needed */

    /* 7. Allow unprivileged code to write to all enabled ports.
     *    Without this, only privileged threads can hit the FIFO. */
    ITM->TPR = 0x0;                                    /* 0 = all ports unprivileged */

    /* 8. Configure DWT — cycle counter on, sync packets on rollover, exception trace. */
    DWT->CTRL = (1 << DWT_CTRL_CYCCNTENA_Pos)   |      /* cycle counter on */
                (1 << DWT_CTRL_EXCTRCENA_Pos)   |      /* exception trace on */
                (1 << DWT_CTRL_SYNCTAP_Pos);           /* sync every 2^24 cycles */
}

Verification. After swo_init() returns, do a printf("hello SWO\n");. If the host viewer (next section) sees "hello SWO", you're done. If it sees nothing, walk the checklist in § Common pitfalls.

Probe-side capture — what the debugger does

From the probe's POV, SWO is just a UART receive line. The work splits into four stages:

  1. Sample the line. A UART block (or a software equivalent — bit-banged GPIO ISR, PIO state machine) recovers bytes from the NRZ stream at the configured baud rate. The probe's UART must be at the same baud as the target's TPIU; the probe is told the baud out-of-band by the host (SWO_Baudrate command — see below).
  2. Buffer. Bytes go into a SRAM ring buffer large enough to absorb USB latency. ~16 KB is typical; edevkit1 uses larger because the SWO baud can outrun USB-FS bursts.
  3. Frame back to host. The probe streams the raw bytes to the host over USB. It does not demux into ITM/DWT packets — that's the host tool's job.
  4. Re-arm on overflow. If the ring buffer fills, the probe records an overflow flag the host can poll. The probe never re-orders or pads; the host gets exactly the wire bytes (sync sequences and overflow markers included) plus an out-of-band overflow indicator.

For CMSIS-DAP v2 the protocol exposes three commands plus a bulk-IN data path. See usb_cmsis_dap.html for the full command set; the SWO subset is:

CommandIDPurpose
DAP_SWO_Transport0x17Pick where SWO bytes ride: command-response (slow) or bulk-IN endpoint (fast). Always pick bulk-IN for production.
DAP_SWO_Mode0x18Pick UART (1) or Manchester (2) decoding.
DAP_SWO_Baudrate0x19Tell the probe what baud to use. Probe replies with the actual baud achieved (might differ from requested).
DAP_SWO_Control0x1AStart / stop capture.
DAP_SWO_Status0x1BQuery trace count, overflow flag.
DAP_SWO_Data0x1CPull SWO bytes from the probe (only when transport = command-response).

For other probes:

  • J-Link exposes SWO over its proprietary USB protocol; J-Link Commander and J-Link RTT Viewer talk it directly. The bytes come back in raw form just like CMSIS-DAP v2.
  • ST-Link/V2 and ST-Link/V3 support SWO via vendor commands; OpenOCD's st-link driver wraps them. ST-Link/V2 caps SWO at ~2 MHz; V3 does ~9 MHz.
  • Black Magic Probe exposes SWO as a USB-CDC ACM serial port — host opens it like any TTY and reads raw bytes. Less integrated but trivially scriptable.

SWO/ITM vs RTT — when to pick which

SWO is not the only logging-over-debug-pin path. Segger's RTT (Real-Time Transfer) achieves the same goal — get bytes from firmware to host without burning a UART — by a totally different mechanism: firmware writes to a known RAM ring buffer, and the probe periodically reads that buffer over the SWD memory-access channel. No SWO pin needed.

SWO / ITMRTT
Pin cost1 (SWO)0 (rides SWD)
RAM cost~0 (FIFO is in TPIU silicon)~1–4 KB ring buffer
Bandwidth2–60 Mbps (silicon)~1–3 MB/s practical (probe-poll-rate-limited)
LatencySub-µs (drains continuously)10s of ms (probe polls every poll-period)
Hardware eventsYes — DWT exception/PC sampleNo — software-only
Cortex-M0/M0+ supportRare (most M0 silicon lacks ITM)Yes (just RAM + SWD)
Available with halted targetNo (CPU stopped, ITM stopped)Yes (probe still polls RAM)
StandardisedYes (Arm CoreSight)No (Segger; reimplemented by others)

Pick SWO when you have ITM-bearing silicon (M3 and up), need hardware events (exception trace, PC sampling), or need the lowest possible logging latency. Pick RTT when you're on Cortex-M0/M0+, you can't route the SWO pin, or you want logging during halted-target inspection.

You can run both simultaneously, with no interference — they share no hardware. probe-rs and J-Link both support that.

edevkit1's SWO implementation

edevkit1 is itself a debug probe — see edevkit1.html §6 — so its job is the capture side of SWO, not the generation side. The implementation is unusual in that the RP2350B has no dedicated SWO peripheral: the capture lives in a PIO state machine, configured as an asynchronous UART receiver.

The PIO program — sketch

The PIO state machine implements a UART receiver. Conceptually it's:

; Pseudo-asm — see app/pio/swo_uart_rx.pio for the real source.
; The full annotated source is in platform_pio.html.

.program swo_uart_rx
.side_set 0 opt

start:
    wait 0 pin 0          ; wait for start bit (line goes low)
    set x, 7        [10]  ; load bit count, delay to mid-bit-cell
bit_loop:
    in pins, 1            ; sample one bit into ISR
    jmp x-- bit_loop [6]  ; loop 8 times with bit-cell delay
    push                  ; push the byte to the RX FIFO
    jmp start             ; back to waiting for next start bit

Concrete details and the actual .pio source live in platform_pio.html. The PIO clock is set to 8 × SWO_baud (oversampling factor 8) to give the receiver tolerance against baud-rate mismatch and start-bit jitter.

Throughput

The RP2350B at 150 MHz can clock a PIO at the full 150 MHz, giving a usable SWO baud of 150/8 = 18.75 Mbps in oversampling-8 mode, or 150/4 ≈ 37 Mbps in oversampling-4. For the edevkit v1 the practical ceiling is ~6–10 Mbps, set by the USB-FS bulk endpoint that has to carry the bytes back to the host (USB-FS is ~10 Mbps usable; see CLAUDE.md gotcha 4 and usb_cmsis_dap.html). v2 with USB-HS will lift that ceiling.

USB transport — bulk-IN, not CDC

The captured SWO bytes ride the CMSIS-DAP v2 bulk-IN endpoint, not a separate USB-CDC ACM port. This is deliberate, and a frequent design question for new contributors:

  • Atomic with debug session. SWO and SWD events on the same probe should be timestamped by the same host-side clock. Splitting SWO onto a CDC means two USB endpoints with independent USB-stack latency, and "did this exception happen before or after that breakpoint?" becomes hard.
  • Standardised. CMSIS-DAP v2 spec already defines SWO commands and a bulk-IN format. Going off-spec with CDC means every host tool has to special-case edevkit.
  • One enumerated USB device. Adding a CDC bumps the composite descriptor count and complicates Windows driver assignment.

The flip side: you can't cat /dev/ttyACMx to see SWO. You need a CMSIS-DAP-aware tool (Orbuculum, probe-rs, pyOCD, J-Link). For developer-friendly raw-text logging that does show up as a TTY, edevkit1 has a separate hardware UART bridge ("target log UART", GPIO 0/1) — see edevkit1.html §7. SWO is for structured trace; the UART bridge is for unstructured printf.

Tooling Workflow on edevkit1: edevkit attach opens a CMSIS-DAP v2 session, then edevkit swo --port=0 demuxes ITM port 0 to stdout, edevkit profile --elf=app.elf runs PC-sampling for a flame graph. The host CLI uses probe-rs as its CMSIS-DAP backend; Orbuculum works as an external alternative.

Common pitfalls and how to debug them

"I see nothing on SWO"

Walk this checklist top-to-bottom — about 95% of "no SWO" reports stop at one of these:

  1. DEMCR.TRCENA not set. Without it, every CoreSight register reads as zero and writes are silently dropped — including ITM_LAR. Symptom: ITM->TER reads as 0 even after you wrote it. Fix: set TRCENA before touching any ITM/DWT/TPI register.
  2. ITM_LAR not unlocked. Without writing 0xC5ACCE55, all ITM register writes go nowhere. Fix: include the unlock in swo_init().
  3. Port not enabled in ITM_TER. Bit N of TER must be 1 for port N to emit. The CMSIS ITM_SendChar defensively checks this and silently drops; if your custom helper doesn't, you'll see no error and no output.
  4. Privileged-only port hit from unprivileged code. ITM_TPR bit N = 1 means port N requires privileged access. Zephyr's user-mode threads or an unprivileged ISR will get a fault or silent drop. Fix: ITM->TPR = 0 for "all unprivileged".
  5. Wrong baud. ACPR is computed from the assumed TRACECLK; if the system clock changed after init, the baud is now wrong. Symptom: probe reports framing errors, host shows scrambled bytes. Fix: re-call swo_init() from your clock-change hook, or pick a TRACECLK source independent of SYSCLK.
  6. Formatter on. TPI->FFCR.EnFCont = 1 wraps every byte in TPIU framing the SWO host doesn't expect. Symptom: structured but consistently-wrong byte sequences. Fix: FFCR &= ~0x102; or just FFCR = 0x100;.
  7. Compiler eliminated the ITM write. If you bypassed CMSIS and wrote your own non-volatile port pointer, LTO may have dropped it. Fix: use ITM->PORT[N] or declare your pointer volatile.
  8. SWO pin not muxed. The GPIO has to be in TRACE/SWO function, not GPIO function. Vendor-specific: STM32 sets it via GPIO_AF_TRACE, NXP via the IOMUX, Nordic via P0.18.MCUSEL = 2. RP2350 doesn't have a dedicated SWO function — edevkit captures, doesn't emit, so this only matters for targets we debug.
  9. Probe is in Manchester mode but target is UART (or vice-versa). Symptom: probe reports "no signal" or "framing errors". Fix: align SPPR on target with DAP_SWO_Mode on probe.
  10. Sync packets not enabled. Without sync, host viewers wait forever for byte alignment and display nothing. Fix: DWT->CTRL.SYNCTAP = 0b01.
  11. Cable too long / SI bad. SWO is single-ended; at 6+ Mbps, ribbon cables longer than ~15 cm degrade. Symptom: occasional bad bytes, worse with target far from probe. Fix: shorten the trace cable; add a 22-Ω series resistor at the source.

"I see overflow markers everywhere"

  • You're producing faster than SWO drains. Either lower the production rate (fewer enabled ports, lower PC-sample rate, less verbose printf), or raise the SWO baud (next divisor up — but your probe must support it).
  • Specific to PC sampling: at 10 kHz on a 150 MHz core, each sample is a 5-byte HW packet — that's 50 KB/s, well within 6 Mbps SWO budget. If you crank to 100 kHz, you're now at 500 KB/s and TPIU FIFO depth becomes the limit, not the line.

"printf over ITM is slow / blocking"

  • The itm_send_char helper above spins waiting for FIFO space. Under heavy printf bursts that's a real stall. Mitigations: drop-on-full instead of spin (lose some bytes, never block), or hand bytes off to a low-priority thread that drains a software ring buffer to ITM in the background.
  • If printf formatting itself is the bottleneck (it usually is on small MCUs — newlib's printf is huge), switch to printf_-shrink alternatives (Marco Paland's printf, picolibc's tinystdio) or skip formatting entirely and emit binary records to a non-zero ITM port for host-side formatting.

M33-specific gotchas

  • Secure / Non-Secure ITM banking. On M33 with TrustZone active, ITM has banked secure and non-secure register copies; the SWO pin carries both, but each bank's TER/TPR/TCR is independent. Without TrustZone enabled (the edevkit1 case — see arm_cortex_m33.html § edevkit notes), only the secure bank exists and behaves like a vanilla M3/M4 ITM.
  • DWT_FUNCTION fields renumbered. Armv8-M reorganised DWT_FUNCTIONn compared to Armv7-M. Code that worked verbatim on M4 needs the bit positions checked against DDI 0553 §D1.2 before it'll work on M33.

Where to learn more

Primary sources, in the order most contributors find useful:

  1. Arm CoreSight Components Technical Reference Manual (DDI 0314) — the authoritative spec for ITM, DWT, TPIU, and the rest of the CoreSight family. Chapters 11–13 cover the ITM packet protocol byte-for-byte. Dense but unambiguous; this is what every probe author and host-tool author reads.
  2. Armv7-M Architecture Reference Manual (DDI 0403) — Part D "Debug" — the same material the CoreSight TRM covers, but in the context of the M-profile architecture. §D4 specifically covers the ITM packet protocol from the core's POV. If you have only one Arm manual on disk, this should be it.
  3. Armv8-M Architecture Reference Manual (DDI 0553) — the M33's ARM. §D1 covers the M33's debug-and-trace, including the renumbered DWT fields and the secure/non-secure banking.
  4. Segger SWO and RTT documentation — practical and well-written. The SWO Viewer manual has worked-example baud calculations and pin-routing notes from the probe vendor's POV. Read the RTT page next to understand the alternative.
  5. Memfault Interrupt — "Profiling firmware on Cortex-M" — end-to-end PC-sampling walkthrough, including host-side flame graph reconstruction. The single best public tutorial on extracting profile data from SWO. Memfault's Cortex-M watchpoints post pairs well for the DWT-comparator side.
  6. Joseph Yiu, The Definitive Guide to Arm Cortex-M3 and Cortex-M4 Processors (and the M33 sequel) — the long-form book treatment. Chapter on debug-and-trace is the most readable single source; covers ITM/DWT/TPIU register-by-register with diagrams.
  7. Orbuculum source — read the decoder (itmDecode.c) for a worked, production-grade ITM/DWT byte-stream parser. ~600 lines, no surprises, well-commented. The fastest way to internalise the packet protocol is to read this once.
  8. CMSIS-DAP v2 specification — particularly the DAP_SWO_* command section. Required reading if you're touching the edevkit firmware's SWO pump or its host-side decoder. Cross-reference usb_cmsis_dap.html for the full command set.
  9.
