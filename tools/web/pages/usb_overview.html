What USB is, and the bits that matter for an MCU

USB — Universal Serial Bus — is a serial bus designed in the mid-1990s to replace the zoo of PC peripheral connectors (PS/2, RS-232, parallel, ADB, MIDI, joystick) with a single hot-pluggable, self-configuring, self-powered cable. Three properties of the design carry through every later layer and are worth keeping in mind when reading anything else:

  1. Host-controller-centric. A USB device never speaks unless spoken to. The host (your PC, phone, embedded host SoC) is the only entity allowed to initiate transfers; devices respond to polls. This makes the protocol cheap on the device end (you're a slave, not a peer), and is why a USB device can never wake itself up faster than the host's poll rate.
  2. Polled, not packet-switched. Even "bulk" transfers happen inside fixed-size frames (1 ms on Full Speed, 125 µs microframes on High Speed and above). Latency is bounded by the frame structure — there is no async push from device to host.
  3. Self-describing. When a device is plugged in, the host asks it a sequence of standard questions, the device answers with descriptors describing what it is, and the host then loads (or has already loaded) a class driver that knows how to talk to it. The driver model is fully data-driven from the device's descriptors.

Speed tiers

The bus has shipped five physical-layer speeds. They are not separate protocols — they share the descriptor and enumeration model — but the wiring, signalling, encoding, and frame timing differ.

TierSpeedYearWiresNotes
Low Speed (LS)1.5 Mbit/s1996D+/D- (twisted pair, no shielding)USB 1.0/1.1. Mice, keyboards, cheap HID. Uses 3 ft max cable. Signalled via NRZI on D+/D-, with idle = D- high, K state, 1.5 K pull-up on D-.
Full Speed (FS)12 Mbit/s1996D+/D-USB 1.0/1.1. Same wires as LS; speed advertised by which line has the 1.5 K pull-up (D+ for FS). Native bus speed of every RP-series silicon (RP2040, RP2350) and most low-end MCUs. This is what edevkit speaks.
High Speed (HS)480 Mbit/s2000 (USB 2.0)D+/D- (with HS chirp negotiation)Adds High-Speed signalling on top of FS wires using a chirp handshake at attach. STM32H7, NXP iMX-RT, Tegra etc. Need a dedicated PHY (often external ULPI, sometimes integrated). MCU support is much narrower than FS.
SuperSpeed (SS)5 Gbit/s2008 (USB 3.0)SSTX±, SSRX± (additional differential pairs alongside D+/D-)USB 3.0/3.1 Gen 1. Different physical lanes from USB 2 — a USB 3 cable carries both 2.0 and 3.0 wires. SuperSpeed+ (SS+)10 / 20 Gbit/s2013+ (USB 3.1 Gen 2 / 3.2)Same SSTX/SSRX pairs, different encoding (128b/132b, optionally dual-lane on USB-C)USB 3.1 Gen 2 = 10 Gbit/s; USB 3.2 Gen 2x2 = 20 Gbit/s on USB-C. Phone/laptop territory. Out of scope for embedded.

The RP2350 USB block is a hard-IP Full Speed (12 Mbit/s) device controller plus a bare-bones host controller. There is no High-Speed PHY on the chip, so edevkit's USB ceiling is 12 Mbit/s gross.

Rule of thumb · 10 Mbps usable Of FS's nominal 12 Mbit/s, you lose ~17% to bit-stuffing (NRZI), framing (1-ms SOF tokens, sync, EOP), per-transaction handshakes (token + data + ACK), and address/endpoint overhead. Plan for ~10 Mbit/s of payload throughput on a healthy bus, and rather less if your endpoints have small packet sizes. This is edevkit's hard bandwidth ceiling — see Bandwidth budgeting.

A bit about NRZI and bit-stuffing (skim if you don't care)

USB encodes bits using NRZI (Non-Return-to-Zero Inverted) on the D+/D- differential pair. A 0 bit causes a transition; a 1 bit is a hold. To prevent long runs of 1s from making the receiver lose its clock, USB inserts a stuffed zero after every six consecutive ones, then strips it on the receive side. This is invisible to higher layers but explains the ~17% framing tax. The receiver's PLL re-locks on every transition.

SE0 (single-ended zero — both D+ and D- driven low) signals reset when held for ≥10 ms, and EOP (end of packet) when held for ~2 bit-times. Speed is communicated by which line has the 1.5 KΩ pull-up: D+ = FS, D- = LS.

Power

USB supplies power on VBUS (5 V on USB 2/USB-A; up to 20 V at higher current via USB-C and PD). Devices that draw > 100 mA must declare a bMaxPower in their Configuration descriptor, and a host can refuse to enumerate them if the upstream port can't supply it. Bus-powered devices negotiate this in the descriptor; self-powered devices set the SELF_POWERED bit in bmAttributes. edevkit is bus-powered: USB-C 5 V → buck → 3.3 V system rail.

Topology — host, hub, device, endpoints

USB is a tiered star: one host root, hubs branching out from it, and leaf devices on hubs. There is exactly one host on a bus. Hubs can cascade up to 5 deep, and a single host can address up to 127 devices (7-bit address space).

│  Root Hub  │

[EDEVKIT]   ← appears to host as one or more interfaces, each with endpoints

Device vs. interface vs. function

A USB device exposes one or more configurations (almost always exactly one in practice — the spec allows several, but Windows for instance won't switch between them). A configuration contains one or more interfaces. An interface contains one or more endpoints. Each endpoint is a unidirectional data pipe between host and device buffer.

Device
└─ Configuration #1
   ├─ Interface #0 (CDC ACM control)        ← claimed by serial driver
   │  └─ Endpoint 0x82 (interrupt IN, 8 B)
   ├─ Interface #1 (CDC ACM data)
   │  ├─ Endpoint 0x01 (bulk OUT, 64 B)
   │  └─ Endpoint 0x81 (bulk IN, 64 B)
   └─ Interface #2 (vendor — CMSIS-DAP)     ← claimed by WinUSB / libusb
      ├─ Endpoint 0x03 (bulk OUT, 64 B)
      └─ Endpoint 0x83 (bulk IN, 64 B)

Compound vs. composite is a constant source of confusion:

  • Compound device — physically, a hub plus one or more devices in the same shell (e.g. a USB-C dock that has a hub, an Ethernet controller, and a card reader internally; the OS sees several distinct USB device addresses).
  • Composite device — logically, one device with multiple interfaces bound to different OS drivers (e.g. edevkit: two CDC ACM interfaces + one vendor interface, all on one address). Composite is the path you almost always want for an MCU.

The control pipe at endpoint 0

Every USB device has a bidirectional control endpoint at address 0 (Endpoint Zero / EP0). This is the only endpoint that exists before enumeration completes — the host uses it to send SETUP packets that ask the device to describe itself, change its address, or activate a configuration. EP0 is bidirectional in the sense that one endpoint number serves both directions (0x00 OUT and 0x80 IN are the same logical pipe). Class drivers also send class-specific control requests through EP0, e.g. CDC_SET_LINE_CODING for serial baud-rate changes.

Transfer types — control, bulk, interrupt, isochronous

Endpoints come in four flavours. The flavour is fixed at descriptor time and determines how the host schedules transactions to that endpoint.

TypeUse caseLatencyThroughputError handling
ControlEnumeration, class-specific small commands~1 frame (1 ms)Reserved 10% of bandwidth on FSHardware retransmits on error; bounded transfer.
BulkBest-effort high-throughput data — file transfer, debug streamsUnbounded (gets leftover bandwidth)Highest sustainable on FS (~10 Mbit/s payload). Up to 64 B max packet on FS, 512 B on HSHardware retries; data integrity guaranteed.
InterruptPeriodic polled small messages — HID input reports, status updatesBounded by polling interval (1–255 ms on FS)Low — 64 B per poll × 1000 polls/s = 64 KB/s max on FSHardware retries.
IsochronousReal-time streams — audio, video, where freshness beats correctnessStrictly periodic (every frame)Reserved bandwidth — guaranteedNo retries; bad packets dropped.

Picking the right type is one of the few real design decisions you make in a USB device:

  • Bulk — anything where you care about throughput and can tolerate jitter. Mass storage, CDC ACM data path, vendor-specific debug streams. CMSIS-DAP v2 uses bulk IN/OUT.
  • Interrupt — anything where bytes are scarce but latency matters. HID keyboards, mouse buttons, status LEDs that the host polls. CDC ACM uses an interrupt endpoint for line-state notifications (DCD/DSR/RING) but a bulk pair for actual data.
  • Isochronous — anything streaming. UAC2 audio, UVC video. Avoid for MCU side-projects: scheduling is finicky, drivers are unforgiving of dropped packets even though the spec says they're allowed, and most MCU stacks don't do iso well.
  • Control — never explicitly chosen for a data path. You get EP0 for free, and you can pass small class-specific commands through it.

Descriptors — the device's self-description

A descriptor is a packed C struct living in the device's flash that the host reads via GET_DESCRIPTOR requests on EP0. The hierarchy is fixed:

Device descriptor                        ← 1 per device
└─ Configuration descriptor              ← 1 (or rarely more) per device
   ├─ (optional) IAD — Interface Association Descriptor
   ├─ Interface descriptor               ← 1+ per configuration
   │  ├─ (optional) class-specific descriptors
   │  └─ Endpoint descriptor             ← 1+ per interface
   ├─ Interface descriptor
   │  └─ Endpoint descriptor(s)
   └─ ...

String descriptors                        ← referenced by index from above
BOS (Binary Object Store) descriptor     ← USB 2.1+; carries platform capabilities

Device descriptor — concrete bytes

The Device descriptor is 18 bytes. Here it is for a hypothetical edevkit unit:

Offset Field                Size  Value     Meaning
─────  ───────────────────  ────  ────────  ──────────────────────────────
 0     bLength               1    0x12      Descriptor length = 18 bytes
 1     bDescriptorType       1    0x01      DEVICE
 2     bcdUSB                2    0x0210    USB 2.10 (BCD; needed for BOS)
 4     bDeviceClass          1    0xEF      Misc — required for IAD-based composite
 5     bDeviceSubClass       1    0x02      Common Class
 6     bDeviceProtocol       1    0x01      Interface Association Descriptor
 7     bMaxPacketSize0       1    0x40      EP0 max packet size = 64 bytes
 8     idVendor              2    0x2E8A    Raspberry Pi (RP-series silicon)
10     idProduct             2    0x10XX    edevkit-assigned PID
12     bcdDevice             2    0x0100    Device version 1.00 (BCD)
14     iManufacturer         1    0x01      String index → "edevkit"
15     iProduct              1    0x02      String index → "edevkit1"
16     iSerialNumber         1    0x03      String index → "EDV-XXXX-YYYY-ZZZZ"
17     bNumConfigurations    1    0x01      One configuration

Key choices to notice:

  • bDeviceClass = 0xEF, SubClass = 0x02, Protocol = 0x01 — this is the magic three-tuple that says "this is a composite device that uses Interface Association Descriptors". Without it, Windows in particular will mis-bind drivers on the second-and-later interfaces of multi-function devices.
  • idVendor = 0x2E8A — Raspberry Pi Trading Ltd's Vendor ID. RP-series silicon ships with this VID and a bank of available PIDs. Open-source projects can use it without licensing fees as long as they pick a PID outside Pi's reserved range.
  • iSerialNumber — non-zero string index telling the host "ask me for a serial". The string itself is constructed at first boot from the RP2350's 64-bit chip ID (see edevkit1.html §11). Hosts use this serial to remember per-device settings (COM port assignments on Windows, /dev/serial/by-id symlinks on Linux).

Configuration descriptor

The Configuration descriptor is read in two stages: first 9 bytes (the header), then the host re-reads with the larger length specified in wTotalLength to grab everything below it (interfaces, endpoints, class-specific descriptors) as one packed blob. Skeleton:

0x09  bLength
0x02  bDescriptorType         (CONFIGURATION)
0xXX  wTotalLength            (LSB)
0xXX  wTotalLength            (MSB)  ← total bytes incl. all sub-descriptors
0xNN  bNumInterfaces          (e.g. 0x05 for edevkit)
0x01  bConfigurationValue
0x00  iConfiguration          (no string)
0xC0  bmAttributes            (0x80 = bus-powered, 0xC0 = self-powered hint)
0x32  bMaxPower               (in 2-mA units; 0x32 = 100 mA)

Interface and Endpoint descriptors

Interface descriptor (9 B):
  bLength=9, type=4, bInterfaceNumber, bAlternateSetting, bNumEndpoints,
  bInterfaceClass, bInterfaceSubClass, bInterfaceProtocol, iInterface

Endpoint descriptor (7 B):
  bLength=7, type=5,
  bEndpointAddress (0x81 = EP1 IN, 0x02 = EP2 OUT, etc.),
  bmAttributes (00=control, 01=iso, 02=bulk, 03=interrupt),
  wMaxPacketSize (8/16/32/64 on FS),
  bInterval (interrupt poll period in ms; ignored for bulk)

String descriptors

Strings are UTF-16LE with no NUL terminator and a 2-byte header:

String descriptor #0 — language IDs:
  0x04 0x03 0x09 0x04
  └─ length, type=STRING, LangID = 0x0409 (US English, little-endian)

String descriptor #1 — manufacturer:
  0x10 0x03 'e'\0 'd'\0 'e'\0 'v'\0 'k'\0 'i'\0 't'\0
  └─ length=16, type=STRING, "edevkit" in UTF-16LE

String index 0x00 is reserved for the language-IDs descriptor. All other strings are referenced by 1-based index from other descriptors (iManufacturer, iProduct, iInterface, etc.). Index 0xEE is special — Windows uses it to detect Microsoft OS Descriptors (see below).

BOS descriptor (USB 2.1+)

The Binary Object Store descriptor was added in USB 2.1 to carry "platform capabilities" — things like USB 3 LPM support, container IDs, and (relevantly for us) Microsoft OS 2.0 descriptors. A device declares bcdUSB = 0x0210 in its Device descriptor to opt in. The BOS descriptor itself is short; the interesting payload is the platform-capability sub-descriptors it contains.

Enumeration — what happens at plug-in

Plugging in a USB device kicks off a fixed handshake. Understanding it is essential when something goes wrong, because the stage at which enumeration fails tells you exactly which descriptor is malformed.

1.  ATTACH         Device pulled D+ high (FS) or D- high (LS).
                   Host sees attach via root-hub status change.

2.  RESET          Host drives SE0 for ≥10 ms.
                   Device clears all state, returns to default address 0.

3.  GET_DESCRIPTOR(Device, 8 B)
                   Host fetches just the first 8 bytes — enough to read
                   bMaxPacketSize0 (so it knows EP0's packet size).

4.  RESET (again)  Some hosts re-reset here to be safe.

5.  SET_ADDRESS(N)
                   Host assigns the device a 7-bit address (1..127).
                   From here on, all subsequent requests use this address.

6.  GET_DESCRIPTOR(Device, 18 B)
                   Full Device descriptor.

7.  GET_DESCRIPTOR(Configuration #0, 9 B)
                   First 9 bytes, just to read wTotalLength.

8.  GET_DESCRIPTOR(Configuration #0, wTotalLength)
                   Full configuration tree — every interface, endpoint,
                   and class-specific descriptor in one buffer.

9.  GET_DESCRIPTOR(String #0)        — language IDs
   GET_DESCRIPTOR(String #iMfr)
   GET_DESCRIPTOR(String #iProduct)
   GET_DESCRIPTOR(String #iSerial)

10. (Windows only) GET_DESCRIPTOR(String #0xEE)
                   If device responds with the magic MS OS String, host
                   issues vendor request 0x?? to fetch MS OS descriptors.

11. SET_CONFIGURATION(1)
                   Activates the chosen configuration. Endpoints become
                   available. Class drivers are bound and start their
                   own initialisation (CDC SET_LINE_CODING, MSC inquiry,
                   etc.).

12. (Class-specific bring-up)
    e.g. CDC ACM:  SET_LINE_CODING, SET_CONTROL_LINE_STATE
         MSC:      INQUIRY, READ_CAPACITY, READ(10) of MBR
         HID:      SET_IDLE, GET_REPORT_DESCRIPTOR

13. Device ready — endpoints exchange data freely.

Standard classes vs. vendor-specific

Once enumeration completes, the host needs a driver. There are two paths:

  1. Standard class — your interface declares one of the well-known class codes (bInterfaceClass in the Interface descriptor), and the OS's built-in driver claims it. No driver install. Constrains your protocol.
  2. Vendor-specific — class code 0xFF, your own protocol on bulk/interrupt endpoints, host-side library does the heavy lifting. Free choice of protocol; need to handle host-driver matching yourself.

When to use a class vs. roll your own

Default to a standard class wherever it fits. The wins are huge: zero driver install on every modern OS, immediate compatibility with shell tools (cat /dev/ttyACM0, screen, file-explorer drag-drop), and a body of tooling you don't have to build.

Go vendor-specific when:

  • Throughput requires bulk endpoints with no per-byte protocol overhead (MSC's SCSI wrapping costs ~30 B/transfer; CDC ACM is fine for raw bytes but its line-coding/control negotiation adds latency at open).
  • You need bidirectional binary message framing that isn't a stream (CMSIS-DAP, Saleae Logic, BMP probe protocols).
  • You want host-side flexibility — libusb gives you full control over packet boundaries, timeouts, and stalls.
  • You're integrating with an existing host library that expects vendor-specific endpoints (probe-rs talks raw to CMSIS-DAP, OpenOCD has its own bulk transports).

Microsoft OS descriptors and WinUSB

Linux and macOS bind libusb/IOUSBLib to any vendor-specific interface they see — you just libusb_open() and start reading. Windows does not. Without help, Windows shows your vendor interface as "Unknown Device" and demands an INF file.

The fix that has been industry-standard since ~2008 is Microsoft OS descriptors: extra non-standard descriptors that tell Windows "use the built-in WinUSB driver for this interface, here's the device interface GUID to publish it under". Two generations exist:

VersionTriggerCarrierStatus
MS OS 1.0String descriptor index 0xEEVendor request returned via EP0Works back to Windows XP. Still widely used.
MS OS 2.0BOS descriptor (USB 2.1+)Platform-capability descriptor inside BOSWin 8.1+. Cleaner. Preferred for new designs.

MS OS 1.0 — the 0xEE String trick

When Windows sees a new device, after fetching the standard descriptors it issues:

GET_DESCRIPTOR(STRING, index=0xEE, langid=0x0000)

If the device returns a 0xEE string descriptor with the magic signature "MSFT100" followed by a vendor-request byte, Windows will then issue that vendor request to fetch:

  • Extended Compat ID descriptor — tells Windows "for interface N, use driver X" (where X is one of the codes in the MS table — "WINUSB" is the one we want; "RNDIS", "PTP", "MTP", "BLUTUTH", "XUSB20" are others). The MS OS 1.0 spec lists the full table.
  • Extended Properties descriptor — additional registry keys, the most important being DeviceInterfaceGUIDs (a 128-bit GUID under which user-mode apps can enumerate this interface via SetupDiGetClassDevs).

MS OS 2.0 — via BOS

Cleaner, no magic string index. The device declares bcdUSB = 0x0210, exposes a BOS descriptor with a Microsoft platform-capability sub-descriptor, and the OS reads compatible-ID and properties from a single binary blob. Zephyr USB-Device-Next supports this directly through USBD_DESC_BOS_VS_DEFINE.

Practical note Most teams ship both MS OS 1.0 and 2.0 descriptors and let the host pick. The 1.0 set keeps Windows 7/8 happy for any ancient PC the field engineer drags out of a closet; the 2.0 set is what modern Windows 10/11 actually consumes. Costs ~200 bytes of flash in total.

WinUSB device interface GUID

edevkit's CMSIS-DAP v2 interface should publish a stable per-product GUID so host tools can find it via standard Windows enumeration. Pick one once with uuidgen, hard-code it in the Microsoft OS descriptor, and never change it across firmware revisions of the same product. The CMSIS-DAP v2 spec recommends a specific GUID ({CDB3B5AD-293B-4663-AA36-1AAE46463776}) so PC tools that hard-coded it can find any v2 probe.

Why edevkit picked Zephyr USB-Device-Next

edevkit firmware runs on Zephyr 4.x, so the realistic choices were Zephyr's own stacks: the legacy USB device stack (deprecated since Zephyr 3.4, and slated for removal) or USB-Device-Next, which was promoted to the default in Zephyr 3.6. The reasons to use the new one:

  • Composite-device support is first-class. The legacy stack required hand-managing endpoint number assignments across classes and broke quietly under multi-class loads. USB-Device-Next allocates endpoints at usbd_init() time across all registered functions and refuses to enumerate if budgets don't fit, with a clear error.
  • Devicetree-driven configuration. Each CDC ACM port, MSC instance, etc. is a node in the board overlay; classes are registered at runtime in code that walks the tree. This matches every other Zephyr subsystem and means board variants don't need code changes.
  • BOS / MS OS 2.0 descriptors built in. USBD_DESC_BOS_VS_DEFINE lets you ship the Microsoft platform-capability blob without fighting the stack.
  • Active development. The legacy stack only gets fixes; new classes (UAC2, UVC, RNDIS) land first on -Next.
  • Same lifecycle as the rest of Zephyr. One Kconfig tree, one logging domain, one sample suite. No second build infrastructure.

The trade-off: USB-Device-Next is younger and less battle-tested than TinyUSB. Zephyr's GitHub has open issues about endpoint allocation collisions in CDC + UAC2 composites (issue #103324 is a recent one). For edevkit's CDC + Vendor + MSC layout these aren't blockers, but it's worth tracking the issue tracker as the stack matures.

Composite-device gotchas (IAD, endpoint numbering)

A "composite" device is one logical USB device with multiple interfaces bound to different drivers. Almost every interesting MCU project ends up composite — even a probe-and-CDC combo is composite. There are five things that bite teams:

1. Interface Association Descriptor

If two of your interfaces logically belong together (CDC's control + data, or audio's control + streaming), they need an IAD immediately before the first interface descriptor in the configuration tree. Without it, Windows treats every interface as an independent function and binds usbccgp.sys as a parent — which is fine for some classes but wrong for CDC ACM (it'll bind usbser.sys only to the control interface and never connect the data interface).

IAD descriptor (8 bytes):
  bLength = 0x08
  bDescriptorType = 0x0B (INTERFACE_ASSOCIATION)
  bFirstInterface              ← e.g. 0
  bInterfaceCount              ← e.g. 2 (control + data)
  bFunctionClass               ← 0x02 (CDC) for an ACM port
  bFunctionSubClass            ← 0x02 (ACM)
  bFunctionProtocol            ← 0x01 (AT commands)
  iFunction                    ← string idx, e.g. "Target log UART"

And remember the trio at the Device descriptor level: bDeviceClass=0xEF, bDeviceSubClass=0x02, bDeviceProtocol=0x01. Without these, Windows ignores the IAD.

2. Endpoint numbering matters

USB endpoints are 4-bit (0..15). On most MCUs each direction is a separate endpoint, so EP1-IN and EP1-OUT are different endpoints sharing a number. Some controllers (RP2350 included) have 16 endpoints per direction; some have only 8 (older STM32). When two classes both want "EP1 IN" you have a collision the stack must resolve — and bad stacks resolve it silently in a way that leaves one interface non-functional.

USB-Device-Next does this allocation correctly at usbd_init(); TinyUSB requires you to set CFG_TUD_*_EP_BUFSIZE macros that influence layout; STM32Cube requires you to assign endpoints by hand in the descriptor template.

3. Endpoint count budget

RP2350's USB device controller has 15 IN + 15 OUT endpoints (plus EP0). Edevkit's worst-case layout uses ~10 endpoints (see below), so we have headroom. Tighter chips (G0xxx, M0+ silicon) have only 4-8 endpoints total and force you to pack hard.

4. Windows class-driver matching is order-sensitive

Windows iterates interfaces in declaration order and binds the first match it finds. If you put the vendor (CMSIS-DAP) interface before the CDC ACM pair, in some Win10 patch levels the CDC pair won't get usbser.sys. Conventional ordering is: CDC ACM(s) first, then HID, then MSC, then vendor-specific last. edevkit follows this.

5. Alternate settings + endpoint zero

Audio and video classes use bAlternateSetting heavily — interface 0 alt-setting 0 has no endpoints (so the device idles and consumes no bandwidth), alt-setting 1+ defines the actual streaming endpoints. Hosts swap alt-settings to start/stop streams. Don't forget your descriptor parser has to handle alt-setting 0 specially — most stacks do this for you, but if you hand-roll it, this is a classic landmine.

#include <zephyr/usb/usbd.h>

USBD_DEVICE_DEFINE(edev_usbd, DEVICE_DT_GET(DT_NODELABEL(usbd)),
                   0x2E8A,    /* VID */
                   0x10D0);   /* PID */

USBD_DESC_LANG_DEFINE(lang);
USBD_DESC_MANUFACTURER_DEFINE(mfr,    "edevkit");
USBD_DESC_PRODUCT_DEFINE(prod,        "edevkit1");
USBD_DESC_SERIAL_NUMBER_DEFINE(sn);            /* derived from chip ID */
USBD_DESC_CONFIG_DEFINE(fs_cfg, "FS Configuration");

USBD_CONFIGURATION_DEFINE(fs_config,
                          USB_SCD_SELF_POWERED,
                          200,                    /* mA */
                          &fs_cfg);

int edev_usb_init(void) {
    int rc;
    rc = usbd_add_descriptor(&edev_usbd, &lang);
    rc |= usbd_add_descriptor(&edev_usbd, &mfr);
    rc |= usbd_add_descriptor(&edev_usbd, &prod);
    rc |= usbd_add_descriptor(&edev_usbd, &sn);
    rc |= usbd_add_configuration(&edev_usbd,
                                  USBD_SPEED_FS,
                                  &fs_config);

    /* Register the two CDC ACM ports. The class instances are auto-created
     * from the matching devicetree nodes; we just bind them to a config. */
    rc |= usbd_register_class(&edev_usbd, "cdc_acm_0", USBD_SPEED_FS, 1);
    rc |= usbd_register_class(&edev_usbd, "cdc_acm_1", USBD_SPEED_FS, 1);

    /* CMSIS-DAP v2 vendor class — defined in our own driver. */
    rc |= usbd_register_class(&edev_usbd, "cmsis_dap_v2", USBD_SPEED_FS, 1);

    if (rc) return rc;
    return usbd_init(&edev_usbd) ?: usbd_enable(&edev_usbd);
}

The exact macro names track Zephyr upstream — check docs.zephyrproject.org/latest/connectivity/usb/device_next/usb_device.html for the current API surface; small renames are common between minor releases.

SWO multiplexed onto CMSIS-DAP bulk IN

One subtlety: SWO trace data does not get its own CDC port. It rides on the CMSIS-DAP v2 bulk IN endpoint as a tagged stream. The CMSIS-DAP v2 protocol defines a DAP_SWO_* command family and reserves a separate "SWO" endpoint pattern that the host (probe-rs, OpenOCD) auto-correlates with the active debug session. This is non-obvious — when you're tempted to "just expose SWO as another /dev/ttyACMx", don't. Host tools won't be able to merge it with the debug session, and you'll lose the timestamping that makes SWO useful. See usb_cmsis_dap.html for the full DAP_SWO command set and ring-buffer design.

Endpoint budget summary

DirectionUsedAvailable on RP2350Headroom
IN (device → host)5 (0x81, 0x82, 0x83, 0x84, 0x85) + EP015 + EP010 free
OUT (host → device)3 (0x01, 0x03, 0x05) + EP015 + EP012 free

Plenty of headroom for v2 expansion (logic-analyzer streams, additional CDC ports for sniffer output).

Why two CDC ports, not one shared Target log UART and Test UART live in different voltage domains on the hardware (Vtgt vs Vio). Multiplexing them onto a single CDC and switching at runtime would need framing in the byte stream and would break cat /dev/ttyACM0. Two separate CDC interfaces gives the host two independent /dev/ttyACM* nodes that user tools can screen, minicom, or tio directly. Cost is two extra interfaces (5 endpoints) — well within budget.

Bandwidth budgeting — the 10 Mbps rule

Full Speed nominal: 12 Mbit/s. Useful payload after framing/overhead: ~10 Mbit/s on a healthy bus, less if you're contending with other traffic. Per-frame (1 ms) budget on FS:

ResourcePer framePer second
Total bus capacity1500 B1.5 MB/s gross
Reserved for control transfers~150 B (~10%)~150 KB/s
Reserved for interrupt/iso (worst case)~1023 Bup to ~1 MB/s
Available for bulk (best case)~1280 B~1.25 MB/s sustained
Bulk packet ceiling on FS64 B/transaction × ~19 trans/frame~1.2 MB/s

edevkit's worst-case load looks like this — and it's a real concern, not a theoretical one:

  • SWD/JTAG debug session via CMSIS-DAP — bursts up to ~600 KB/s during flash programming.
  • SWO trace at 1 Mbit/s instrumented (typical) — ~125 KB/s sustained.
  • Two CDC ACM ports — typically a few KB/s each for log, but can burst much higher.
  • I²C / SPI sniffer — when active, generates the highest data rate on the bus, easily saturating FS.

Total under "everything-on" can exceed 1.2 MB/s. USB FS is the bottleneck. The architectural mitigations are:

  1. Large device-side ring buffers in SRAM. The sniffer and SWO pumps each get a 32–64 KB ring in MCU SRAM. Bursts longer than the buffer cause back-pressure (described in edevkit1.html §6.5), not silent drops.
  2. Back-pressure-aware producers. When the ring fills, the producer (PIO state machine via DMA) gets stalled at the DMA channel. Capture data is never silently overrun.
  3. Configurable host-side polling cadence. The host CLI can request larger bulk-IN packets be queued; with NAK-aware bulk endpoints the device fills the wire as fast as it can.
  4. Active load shedding. If the host's read rate falls below produce rate for > N ms, the firmware emits a marker into the stream and starts dropping the lowest-priority lane (typically SWO). This is a deliberate quality-of-service decision documented in edevkit1.html §9.
Don't be tempted to add HS RP2350 has no HS PHY. Adding one means a separate USB-HS bridge IC (e.g. an FE1.1 USB hub + a USB3300 PHY behind a real HS-capable MCU) — significantly more cost and complexity, and out of v1 scope. The right answer when FS isn't enough is to narrow what's exposed at once, not to widen the bus. See edevkit1.html §15 (Decision Log) for the rationale on staying FS-only for v1.

Where to learn more

USB has 30 years of public documentation. A subset is worth your time.

Primary sources (the spec itself)

  1. usb.org · Documents library — the USB 2.0 spec PDF (the bible — chapter 9 is the device framework, chapter 11 is hubs, chapter 7 is electrical), USB 3.2 spec, and class specs (CDC, HID, MSC, Audio, Video) in separate PDFs. Bookmark it; you'll come back.
  2. Microsoft Learn · MS OS 1.0 descriptors — official Microsoft spec including the compatible-ID table. Pair with the MS OS 2.0 doc on the same site.
  3. Zephyr · USB-Device-Next docs — the API reference for the stack edevkit uses. Read the index page first, then the per-class pages (CDC ACM, MSC, HID).
  4. CMSIS-DAP v2 spec — Arm's official protocol spec for the debug-probe class. See usb_cmsis_dap.html for our annotated version.

Tutorial-style introductions

  1. USB In a NutShell · beyondlogic.org — a free, well-illustrated walk through every USB 2.0 concept. The single best free starting point if you want a friendlier prose version of the spec.
  2. Jan Axelson, USB Complete: The Developer's Guide (5th ed., Lakeview Research, 2015) — the long-form book. If you can read only one thing on USB and you're paid to design devices, read this. The descriptor and enumeration chapters are the clearest treatment in print.
  3. USB Made Simple — a much more compressed tutorial; good for skimming during onboarding. Less depth than NutShell but lower commitment.
  4. TinyUSB source — even if you don't use TinyUSB, reading its descriptor templates is a fast way to absorb idiomatic device-side USB code. examples/device/cdc_msc/ is a tight composite example.
