I²C — the two-wire bus, end to end

A focused reference on I²C (Inter-Integrated Circuit), the open-drain two-wire bus invented at Philips in 1982 and still the dominant on-board chip-to-chip interconnect on modern PCBs. What I²C is

I²C — pronounced "eye-squared-see" or "eye-two-see", written either I²C, I2C, or (in newer NXP docs) I3C's ancestor — is a synchronous half-duplex multi-drop bus invented at Philips Semiconductors (now NXP) in 1982. It was designed to let a TV's microcontroller talk to a handful of audio/video chips on the same PCB without burning a parallel-bus pin-count budget. Forty-plus years later it is still the bus you reach for whenever you need to wire a sensor, EEPROM, RTC, port expander, fuel gauge, OLED, or system controller to an MCU on a circuit board, and the bus you'll encounter on absolutely every embedded platform you ever touch.

The wire-level idea is small enough to fit on a postcard:

  • Two wires. SDA (serial data) and SCL (serial clock). Plus a shared ground. That is the entire bus.
  • Open-drain. Every device on the bus pulls lines low; nobody ever drives them high. The lines float to logic-high through external pull-up resistors (one on SDA, one on SCL) tied to the bus reference voltage.
  • Multi-drop. Many slaves and (optionally) many masters share the same two wires. Each slave has a 7-bit (or 10-bit) address; the master selects which slave to talk to by emitting that address.
  • Master-driven clock. The master generates SCL. Slaves are synchronous to it — though slaves can stretch the clock by holding SCL low, which we'll cover below.
  • Byte-oriented with mandatory ACK. Data flows in 8-bit bytes, MSB first, and each byte is followed by a single ACK or NAK bit driven by the receiver of that byte.
key idea Open-drain is the secret to everything I²C does. Multi-master arbitration, clock stretching, and the ability to hot-add or reset a single device on the bus all fall out of the wired-AND topology that open-drain gives you. There is no scenario in I²C where "both sides drove the wire and one of them won" — instead, the wire goes low if any device pulls it down, and high only if everyone agrees to release it. That single property is the load-bearing assumption behind the rest of the spec.

The protocol is canonically defined in NXP UM10204 (the I²C-bus specification and user manual, currently Rev 7.0, October 2021). That PDF is 64 pages and is the source of truth for everything you'll read here; if you're going to do I²C work seriously, downloading it once and grepping it occasionally pays off forever.

Why open-drain (and why pull-ups are mandatory)

If both lines were push-pull, two devices driving simultaneously — even by accident — would short Vcc to GND through their output stages and one of them would smoke. With open-drain, the worst case is "both pull low at the same time"; the line just stays low and nobody is harmed. From there:

  • Wired-AND. The bus voltage on either line is logically AND of every device's output. If any device drives low, the line is low. The line goes high only when every device has released it. This is what enables arbitration (§7) and clock stretching (§6).
  • Hot-plug-tolerant. Adding or removing a device while the bus is idle does not damage anything — the new device's input transistor is in cutoff until the master asserts SCL/SDA. No bus-contention failure mode exists.
  • Mixed-voltage tolerant (with a translator). A 5 V master and a 3.3 V slave can share the bus through a level translator (PCA9306, PCA9517, NTS0102, etc.), because the lines are never pushed high by anyone — they're let high by the rail's pull-up. That makes voltage translation a passive switch, not an active driver.

The pull-up resistor is not optional. Without it the line never reaches logic-high; data is undefined and the bus appears stuck low. Every I²C bus needs exactly one pair of pull-ups somewhere on the bus — usually next to the master, sometimes built into a sensor breakout (in which case removing the on-board jumper is your responsibility on populated boards).

Pull-up sizing — the rise-time rule

The pull-up value is set by a tug-of-war between two opposing constraints, and there is exactly one variable that controls both: bus capacitance Cb, which is the sum of all devices' input capacitance plus the trace capacitance, typically 50–400 pF on a real board.

ConstraintDirectionWhy
Rise time tr ≈ 0.85 × R × CbWants R smallSpec caps tr at 1000 ns @ 100 kHz, 300 ns @ 400 kHz, 120 ns @ 1 MHz, 80 ns @ 3.4 MHz
Sink current ≤ 3 mA per device (per UM10204 §7.1)Wants R largeRmin = (VDD − VOL,max) / 3 mA. At 3.3 V that's ~1.1 kΩ; at 5 V, ~1.6 kΩ.

Practical rule-of-thumb values that work on most boards:

100 kHz (Standard)4.7 kΩForgiving; fine even with 200 pF of bus capacitance
400 kHz (Fast)2.2 kΩDrop further if bus is heavily loaded
1 MHz (Fast+)1.0–1.5 kΩApproaching the 3 mA sink limit; verify with scope

scope this If the SCL or SDA waveform looks rounded on the way up (RC-charging shape) and sharp on the way down, your pull-up is too weak (resistor too large) or your bus capacitance is too high. The fix is either a smaller pull-up (down to the sink-current limit) or fewer devices/shorter traces. A device that "works at 100 kHz but glitches at 400 kHz" is almost always this exact problem — same hardware, same firmware, different ramp time.

Electrical model — open-drain, pull-ups, capacitance

            │ │  R_pullup   (one of these on SDA, one on SCL)
            │ │
            └┬┘
             │
             ●──────────●──────────●──────────●──────────●  ← bus line (SDA or SCL)
             │          │          │          │          │

           │OD │      │OD │      │OD │      │OD │      │OD │  open-drain output

  Each device's input is a high-Z buffer (also tapped off the line).
  The line voltage = VDD if every OD is OFF.
  The line voltage ≈ V_OL (typically <0.4 V) if any OD is ON.

The behaviour you actually care about, in firmware:

  • To drive a "0", configure the GPIO as open-drain output, value=0. To drive a "1", switch back to input (high-Z) and let the pull-up bring the line up.
  • The line's "real" value is whatever you read back on the input. You wrote a 1, but the line is still 0? Either the pull-up failed (broken hardware) or someone else is asserting the line (clock-stretching, arbitration, or another master).
  • If you write a 0 and read back a 0, that's expected. If you write a 1 and read back a 1, that's expected. The interesting case — and the one that distinguishes I²C from any push-pull bus — is write 1, read 0. That single condition signals stretching, arbitration loss, or hardware fault, and almost every "smart" I²C feature is built on detecting it.

Bit transmission — what one bit looks like

One bit on the wire takes one full SCL cycle. The transmitter (master or slave, depending on which way the byte is going) sets SDA while SCL is low; the receiver samples SDA on SCL's rising edge; SDA must remain stable until SCL falls again.

      ___           ___           ___
SCL _/   \_________/   \_________/   \___
                ↑              ↑
        sample bit N    sample bit N+1
        (rising edge)   (rising edge)

       _________________________
SDA __/   bit N    \    bit N+1     \____
       (set while SCL low)
       (must be stable while SCL high)

Said in spec language: SDA must not change while SCL is high. That single rule has a consequence — any SDA transition while SCL is high is special, and the protocol uses exactly two such transitions as framing markers: START (SDA falls while SCL is high) and STOP (SDA rises while SCL is high). Everything else is data.

Speed grades — Standard, Fast, Fast+, HS, Ultra-Fast

The I²C spec defines five speed modes. In practice you'll only ever encounter the first three on a real board; the latter two require special master IP and are rare outside power-management ICs and laptop battery sub-buses.

note Ultra Fast-mode is barely I²C. It's push-pull, write-only, no ACKs in the conventional sense, and no clock stretching. NXP added it to support display/LED controller use cases that wanted I²C-like framing without I²C's electrical overhead. If you see "UFm I²C" in a datasheet, treat it as a separate bus discipline that happens to share start/stop framing with classical I²C. Don't try to put it on the same wire as a Standard-mode device.

The arbitration window

The bus's actual running speed is set by the slowest participant. If you configure your master for 1 MHz but a slave can only handle 400 kHz, one of three things happens:

  1. If the slave supports clock stretching, it'll hold SCL low until it can keep up — the bus throttles to whatever rate the slave's worst-case stretch allows. (Often this means the slave de-facto sets the clock for that transaction.)
  2. If the slave doesn't support stretching, it will simply fail to set up SDA in time — you'll get bit errors, NAKs, or garbage data. The transaction "works" electrically but the slave isn't actually decoding the transfer.
  3. If the master is hardware-rigid and ignores stretching (some early STM32 controllers were like this), the slave gets no protection. Reads return stale data; writes get accepted partially. This is the most insidious failure mode and the reason "this sensor doesn't work on this MCU" bug reports exist.

Practical guidance: most real-world I²C is run at 100 kHz or 400 kHz. Going to Fast+ requires checking every device datasheet on the bus; going to Hs requires hardware that explicitly supports it (you'll know — there will be a separate set of pads and a master code byte). Don't push a generic sensor bus past 400 kHz without scoping the lines first.

Frame structure — START, address, data, ACK, STOP

An I²C transaction is a sequence of well-defined moves on the bus:

  1. START condition. Master pulls SDA low while SCL is still high. This is the only legal SDA-falling-while-SCL-high transition; it tells everyone "a transaction is starting".
  2. Address byte. 7 address bits (MSB first) followed by 1 R/W bit (0 = master writes, 1 = master reads). Sent on SDA, clocked by SCL.
  3. Address ACK. Master releases SDA on the 9th SCL pulse; the addressed slave pulls SDA low to acknowledge. If no slave at that address, SDA stays high (NAK).
  4. Data bytes. Repeated 8 bits + 1 ACK bit, indefinitely until master decides the transaction is over. Direction is fixed by the R/W bit in the address: in a write, master drives data and slave ACKs; in a read, slave drives data and master ACKs (or NAKs the final byte to signal "I'm done").
  5. STOP condition. Master releases SDA after releasing SCL, so SDA rises while SCL is high. Bus is now free; another master could grab it.

Full frame, on the wire

Concrete example: master writes 0x42 0x55 to slave at 7-bit address 0x50. The on-the-wire bit pattern is:

A few subtleties this diagram makes plain:

  • Each byte is 9 SCL pulses, not 8. The 9th pulse is for the ACK. New developers staring at a logic-analyser capture sometimes miscount and "lose" bits; if you count 9-clock groups instead of 8-clock groups you'll never go wrong.
  • The address byte and the data byte are structurally identical on the wire. They're both 8 bits + 1 ACK. The only way to know which is which is to count from the most recent START. A sniffer that joins mid-transaction can't distinguish address-vs-data without resync — see § Sniffer mode.
  • The R/W bit is part of the address byte. When you read a datasheet and see "I²C address 0xA0/0xA1", that's the 8-bit version: 0xA0 = (0x50 << 1) | 0 (write), 0xA1 = (0x50 << 1) | 1 (read). The 7-bit address is 0x50. Vendor datasheets are inconsistent about which form they print; this is the single most common source of "my driver returns NAK" bugs in junior code. See § Pitfalls.

START and STOP — the only special transitions

Zoomed in:

      START condition                       STOP condition
          ___                                   _________
SCL ____/   \___________               __/   \________________
                                                  ____________
SDA ‾‾‾\______________________         _________/
        ↑                                       ↑
   SDA falls WHILE                          SDA rises WHILE
   SCL is HIGH                              SCL is HIGH

Both are illegal transitions during a normal data byte (recall: SDA must not change while SCL is high). The protocol "steals" the two SDA-changing-while-SCL-high transitions to use as framing markers, and that's why the rule exists in the first place.

7-bit vs 10-bit addressing & reserved addresses

7-bit addressing

The dominant convention. The first byte after START is AAAAAAA · R/W: 7 address bits MSB-first, then the read/write bit. Total address space is 128 (2^7), but with reserved ranges only about 112 addresses are usable for real slaves (range 0x08..0x77 inclusive).

10-bit addressing

Added later for cases where 112 slaves on one bus isn't enough (rare, but happens in laptops and some industrial controllers). 10-bit addressing escapes through a reserved 7-bit prefix:

         first byte                second byte         optional data
       ┌──────────────────────┐  ┌──────────────────┐  ┌──────────────┐
START  │ 1 1 1 1 0 A9 A8 R/W │A │ A7 A6..A0          │A │ data         │A│  ...  STOP
       └──────────────────────┘  └──────────────────┘  └──────────────┘
       ↑       ↑       ↑
       fixed   high 2  R/W of
       prefix  bits of the eventual
       11110xx addr    transaction

10-bit slaves listen on the 0x78..0x7B reserved 7-bit-equivalent range. A 10-bit-address-aware master sends two address bytes; a 7-bit-only master never emits the 11110xx prefix and so can't accidentally collide. 10-bit and 7-bit slaves can therefore co-exist on the same bus, and most real-world buses simply never use 10-bit addressing at all.

Reserved addresses

Per UM10204 §3.1.12, the I²C spec carves out two blocks of reserved 7-bit addresses at either end of the range. Real slaves must avoid these.

"Shifted" vs "unshifted" — the bug every developer hits exactly once

Vendor datasheets are wildly inconsistent about how they print I²C addresses. The same physical chip might be documented as:

  • "7-bit address 0x50" → just the 7 bits. Use this with most modern firmware APIs (Linux i2c-dev, Zephyr i2c driver, Arduino Wire).
  • "8-bit write address 0xA0, 8-bit read address 0xA1" → already shifted left by 1, with the R/W bit baked in. Some legacy datasheets and most NXP/Renesas documentation use this style.
  • "I²C address 0xA0" with no further qualification → ambiguous; usually means the 8-bit form. Verify with a NAK/scan.

Conversion:

7-bit addr  =  (8-bit write addr) >> 1
            =  (8-bit read addr - 1) >> 1     ; (the read addr has bit 0 set)

8-bit write addr  =  (7-bit addr) << 1
8-bit read addr   =  ((7-bit addr) << 1) | 1
classic bug Datasheet says "address 0xA0", you write i2c_write(0xA0, …), you get NAK. The driver shifted your address again, sending out 0x140 — which got truncated to 0x40 — which doesn't match the chip. The fix is i2c_write(0x50, …): pass the 7-bit address, the driver does the shift. If you're using a Zephyr/Linux/Arduino-style API, always pass the 7-bit address. If you're driving a low-level peripheral register directly, you write the 8-bit form and the hardware blasts it onto the wire as-is.

Repeated START

A repeated START (sometimes "Sr") is a START condition issued without a preceding STOP — i.e., the master starts a new transaction without releasing the bus. It looks identical to a regular START on the wire (SDA falls while SCL is high); the difference is purely in where it appears in the transaction sequence.

The canonical use case is the register-read pattern, which most I²C peripherals (sensors, EEPROMs, port expanders) implement:

  S  | addr · W | A | reg_idx | A | Sr | addr · R | A | data byte 0 | A | data byte 1 | N |  P
  ↑                                  ↑                                                       ↑
  START                              repeated START                                          STOP
  (master writes the                 (master reads back from
  register address it                the same chip, no STOP
  wants to read)                     in between)

"Why not just STOP and START again?" — because between the STOP and the new START, any other master on the bus could grab it. If your slave's "current register pointer" is shared mutable state, a competing master could clobber it between your write and your read, and you'd read the wrong register. A repeated-START tells the bus arbitration logic "this is one atomic transaction; nobody else gets to interleave."

On a single-master bus, a STOP-then-START is functionally identical (because there's nobody to race). On a multi-master bus, repeated-START is mandatory for any read-modify-write or read-from-pointer pattern. Most hardware controllers expose this as a flag on the next transfer call (I2C_MSG_RESTART in Zephyr; I2C_M_NOSTART negation in Linux's i2c-dev; the implicit semantics of Wire.endTransmission(false) followed by Wire.requestFrom in Arduino).

Clock stretching

A slave can hold SCL low to ask the master to wait. The master must respect this: after releasing SCL on each clock cycle, it reads back the line; if the line is still low, the slave is stretching, and the master loops, waiting, until the line goes high.

master releases SCL ──┐
                      ↓
                     ___      ___      ___      _________________      ___
SCL  master drives  /   \    /   \    /   \    /   slave HOLDS    \   /   \
     master reads   │   │    │   │    │   │    │   SCL low here   │   │   │
                      ↑                       ↑                      ↑
              normal pulse                  master expected SCL    slave finally
                                            to rise here, but it   releases SCL,
                                            stayed low for ~50 µs  master continues

Common reasons a slave stretches:

  • EEPROM page write. Most I²C EEPROMs (24LCxx, AT24Cxx) take ~5 ms to program a page; they hold SCL low while burning. The master sees a 5 ms stretch on the byte that follows the data; this is the EEPROM saying "wait for me to commit, then we can continue or you can poll."
  • Slow ADC conversion. A high-resolution ADC reading "convert and read result" stretches the clock during the conversion. Convenient for the master — instead of polling for "conversion done", you just issue the read and the slave will release SCL when ready.
  • Soft-I²C-slave implementations. An MCU acting as an I²C slave usually stretches SCL on every byte boundary while it figures out what to do next. This is normal and is the only reasonable way to implement a slave in software without dropping bytes.
  • Sensor data preparation. Some IMUs and pressure sensors stretch briefly while latching a new sample.
historic gotcha Some hardware I²C controllers don't respect clock stretching. The notorious example is the original STM32F1 series I²C peripheral, which had a documented erratum where stretching at certain bit positions could lock the controller. The Raspberry Pi (BCM2835) Broadcom I²C also has a long-standing stretch bug. The symptom is always the same: "this sensor works fine on chip A but fails intermittently on chip B." If you suspect stretch handling, scope SCL during a transaction and look for any cycle where SCL stays low longer than nominal — if the master keeps clocking through it, you've found your bug.

Mitigations

  • Drop bus speed. Lower-speed buses give slaves more wall-clock time per cycle and reduce the practical need for stretching.
  • Use a master with proper hardware support. All modern controllers (RP2350, nRF52, ESP32-S3, STM32F4 and later, etc.) handle stretching correctly. Check the errata sheet before you commit to a master.
  • Use PIO/bit-bang. A software-defined master is the master — it inherently waits on SCL because it's polling the line directly. edevkit1's PIO I²C engine handles stretching naturally; see § edevkit1's I²C engine.
  • Add a bus timeout. SMBus mandates this — if SCL is held low for more than 25–35 ms, the bus is considered hung and the master should issue a STOP and reset. Don't wait forever.

Multi-master arbitration

I²C supports multiple masters on the same bus — they share SCL/SDA and arbitrate for the bus when more than one wants to transmit. The arbitration mechanism is, again, a direct consequence of open-drain wired-AND.

The rule is simple: every master, on every bit it transmits, reads back the SDA line. If it expected to drive a 1 (i.e., released SDA) but reads a 0, somebody else is driving low. That master has lost arbitration and must immediately stop driving and let the other master continue.

Walked through:

  bit:        7   6   5   4   3   2   1   0
  master A:   1   0   1   0   1   1   0   0    ← wanting to address slave 0xAC
  master B:   1   0   1   0   1   0   1   1    ← wanting to address slave 0xAB
  bus (AND):  1   0   1   0   1   0   ?   ?

                                       ↑
                              At bit 2, master A drives 1 (releases SDA)
                              but master B drives 0 (asserts).
                              The bus reads 0.
                              Master A reads back 0 ≠ 1 it intended →
                              master A LOST arbitration.
                              Master A immediately becomes a high-Z observer
                              for the rest of the transaction.

  Master B continues, completes its transfer to 0xAB, normally.
  Master A waits for STOP, then can retry its 0xAC transaction on the next bus-free.

Notes:

  • The "loser" doesn't generate any bus error — it just goes quiet. The "winner" doesn't even notice anything happened. From the wire's POV, exactly one valid transaction occurs.
  • Arbitration can happen during the address byte or during data bytes. The spec is explicit: a master that loses arbitration during a data byte must stop driving SDA and stop driving SCL — it must not interfere with the surviving master's clock.
  • Most application code doesn't see arbitration loss as an error. The driver retries the transaction transparently. Higher layers see only "this took longer than usual."
  • For arbitration to work correctly, masters must be using compatible bus speeds. Mixed-speed multi-master is allowed but rare and finicky.

SMBus and PMBus — I²C with extra rules

The System Management Bus (SMBus), defined by the SBS Implementer's Forum (originally Intel + Duracell, now SBS-IF), is an Intel-led tightening of I²C designed for laptop battery, fan controller, and embedded-controller communication. SMBus 1.0 was published in 1995; SMBus 3.0 in 2014. Electrically and protocol-level it's a strict subset of I²C, with extra rules layered on top:

SMBus ruleVs plain I²C
Bus speed: 10–100 kHzI²C is 0–5 MHz; SMBus locks down to a narrow range
SCL low-timeout: 25–35 msI²C has no timeout; SMBus says "bus is hung if SCL stays low this long"
Mandatory NAK on last read byte from masterI²C strongly recommends; SMBus mandates
Optional Packet Error Code (PEC) — a CRC-8 byte appendedI²C has no integrity check; SMBus PEC catches single-bit errors
SMBus Alert# — a third wire (open-drain interrupt) for slave-to-master notificationI²C has no out-of-band signal
Defined transaction types: Quick, Send Byte, Receive Byte, Write Byte, Block Write, Block Read, Process CallI²C is "any number of bytes"; SMBus formalises
Address resolution protocol (ARP) for dynamic address assignmentI²C addresses are fixed in the slave

PMBus (Power Management Bus, version 1.4 as of 2020) is SMBus + a power-supply command set. It defines about 250 standardised commands (VOUT_COMMAND, READ_VIN, FAN_CONFIG_1_2, etc.) used by digital point-of-load regulators, FPGAs' on-die power management, and modern server PSUs. Wire-level it's SMBus; protocol-level it's a fixed register map.

From edevkit1's POV: an SMBus or PMBus device is just an I²C device, and you can drive it with the regular I²C engine. If you need PEC, the firmware computes the CRC-8 and appends it as a normal data byte; the slave verifies. If you need Alert#, run an extra GPIO line and treat it as an interrupt source. Most edevkit emulation profiles for power-management ICs treat the chip as I²C-with-fixed-register-map plus optional CRC; the SMBus/PMBus flavor is a thin layer in the YAML profile, not a separate engine. See edevkit1.html §11 — emulation profiles for the schema.

Sniffer mode — passive bus capture

A sniffer is a third device on the bus that doesn't drive anything and just listens. Its job is to record SDA and SCL transitions and decode the resulting byte stream — useful for reverse-engineering an unknown peripheral, debugging a flaky multi-master interaction, or capturing the traffic between an MCU and a sensor on someone else's product. Logic analysers (Saleae, sigrok, DSLogic, PulseView) all have I²C decoders that work in software on captured digital traces; edevkit1 implements a real-time sniffer in PIO.

What the sniffer has to do

  1. Detect START. Look for SDA falling while SCL is high. This is the only resync point — if the sniffer joined mid-transaction, every subsequent bit is unaligned until the next START.
  2. Sample SDA on every SCL rising edge. 8 samples = 1 byte. Then the 9th rising edge gives you the ACK bit.
  3. Track byte position. First byte after START is the address (7 address bits + R/W). Subsequent bytes are data. Direction (master-to-slave vs slave-to-master) is determined by the R/W bit in the address byte.
  4. Detect repeated START vs STOP. A START during an ongoing transaction is a repeated-START, not a new transaction. STOP (SDA rising while SCL high) ends the transaction; the next START begins a new one.
  5. Buffer for the host. Push decoded bytes (with timestamps and direction tags) into a ring buffer; let the host drain over USB.
subtle A sniffer that joined the bus mid-transaction has no way to know whether the next byte is an address or data. It must wait for the next START to resync. This is why a sniffer that powers up while the bus is already active will appear to "miss" the first transaction — by design. After the first START it sees, the decode is reliable.

Why a sniffer can't validate R/W mismatches without seeing the address byte

Once mid-transaction, the sniffer sees a byte going by. The data direction is fixed by the address byte's R/W bit, but the sniffer doesn't have that — it only has the ACK bit visible on the 9th SCL pulse, and the ACK is driven by the receiver of the byte, which depends on direction. If you don't know direction, you don't know who drove the ACK. So mid-transaction joins can decode the bytes but not their direction, and that means a "mismatch detection" feature (e.g., "the master tried to write but the slave thought it was a read") is only meaningful when the sniffer caught the address byte.

Practical consequence: edevkit1's sniffer always reports its first bytes-after-power-up as "data, direction unknown" until it sees the first START. After that, every transaction is fully tagged with master/slave direction, ACK/NAK status, and inter-byte / inter-transaction timestamps.

edevkit1's I²C engine (PIO1 SM0)

edevkit1 implements its I²C engine entirely in RP2350B PIO — there is no use of the chip's hardware I²C controllers. Why PIO? Because we need three modes — master, slave, sniffer — that each have different bus-driving semantics, and we need the slave to respond at any address (or any subset of addresses) with arbitrary timing for emulation purposes. Hardware controllers have one personality each; PIO state machines can swap personalities by reloading a different program.

The three modes

Master Drives transactions onto the protocol-tester header to a target peripheral. This is what you use when you have a real sensor / EEPROM / display on the protocol header and you want edevkit1 to talk to it. Supports 7-bit and 10-bit addressing, repeated-START, full clock-stretching tolerance, NAK detection, arbitration loss detection (relevant if the user puts another master on the bus).
Slave Responds to any 7- or 10-bit address (configurable per profile). The slave program decodes the address byte, looks up the matching emulation profile in flash, and either ACKs (and emits canned data on read, or stashes data on write) or NAKs (if no matching profile). This is the foundation of edevkit1's emulation feature: you load a YAML profile for, e.g., a BME280 environmental sensor, and edevkit1 pretends to be that exact chip — answering at the correct 7-bit address with the correct register-map responses.
Sniffer Both SDA and SCL are tristated; the PIO program watches transitions, decodes the bit stream, and pushes timestamped events (START, address, data byte + ACK, repeated-START, STOP) into a SRAM ring buffer. The host pulls the buffer over the CMSIS-DAP v2 vendor endpoint and visualises it in the GUI. No data is dropped silently — if the host is slow, the producer back-pressures and surfaces overflow as a discrete error event so the user knows the capture is incomplete.

The PIO programs themselves live in firmware/app/pio/i2c_master.pio, i2c_slave.pio, and i2c_sniffer.pio; assembled headers are at the same paths with .h. Run tools/pioasm/pioasm to regenerate after edits — Zephyr doesn't auto-build PIO sources. See platform_pio.html for the full PIO walkthrough including the master program annotated instruction-by-instruction.

Driving I²C from Zephyr — worked example

For application code on edevkit1, the easiest way to drive the I²C engine is through Zephyr's i2c.h API — the firmware exposes the PIO-master engine as a regular Zephyr I²C device named i2c_proto. The same API would work on any Zephyr board; the only edevkit1-specific detail is the device-tree node.

        compatible = "edevkit,pio-i2c";

Reading a register from a slave

/* firmware/app/src/proto/i2c_demo.c
 *
 * Read 6 bytes from register 0xF7 of a BME280 (7-bit addr 0x76).
 * Pattern: write reg-addr, repeated-START, read bytes.
 */
#include <zephyr/device.h>
#include <zephyr/drivers/i2c.h>
#include <zephyr/logging/log.h>

LOG_MODULE_REGISTER(i2c_demo, LOG_LEVEL_INF);

#define BME280_ADDR     0x76        /* 7-bit address; never the 0xEC/0xED form */
#define BME280_REG_DATA 0xF7

int bme280_read_data(uint8_t out[6])
{
    const struct device *dev = DEVICE_DT_GET(DT_ALIAS(i2c_proto));
    if (!device_is_ready(dev)) {
        LOG_ERR("i2c_proto device not ready");
        return -ENODEV;
    }

    uint8_t reg = BME280_REG_DATA;
    struct i2c_msg msgs[2] = {
        {
            .buf   = &reg,
            .len   = 1,
            .flags = I2C_MSG_WRITE,
        },
        {
            .buf   = out,
            .len   = 6,
            .flags = I2C_MSG_READ | I2C_MSG_RESTART | I2C_MSG_STOP,
        },
    };

    int rc = i2c_transfer(dev, msgs, ARRAY_SIZE(msgs), BME280_ADDR);
    if (rc) {
        LOG_ERR("i2c_transfer failed: %d", rc);
        return rc;
    }
    return 0;
}

Things worth pointing out in this snippet:

  • I2C_MSG_RESTART on the second message tells the driver to emit a repeated-START between the write and the read — not a STOP-then-START. This is the canonical register-read pattern. Forget it and you'll either lose the register pointer (single-master, mostly works) or race another master (multi-master, fails intermittently).
  • I2C_MSG_STOP on the final message ensures the bus is released; without it the driver could hold the bus, blocking other transactions or other masters.
  • Address is the 7-bit form (0x76), not 0xEC/0xED. Zephyr's API always wants the 7-bit address.
  • Zephyr's i2c_transfer handles arbitration loss, NAK, clock stretching, and bus reset internally; the application sees only success or a clean error code.

Common pitfalls

Every list below is something at least one edevkit contributor has hit on a real board. Read them now, save yourself a debug session later.

  1. Wrong pull-up value — rise-time too slow at high speed. 10 kΩ pull-ups work great at 100 kHz and fail mysteriously at 400 kHz with a moderately-loaded bus. Symptom: occasional bit errors on long messages, fewer errors when you remove devices from the bus, no errors at all if you scope the line and the probe's input capacitance somehow "fixes" it (the probe wasn't fixing anything; you slowed the clock by mistake when you switched the analyser config). Fix: drop pull-ups to 4.7 kΩ for 100 kHz, 2.2 kΩ for 400 kHz, 1 kΩ for 1 MHz, then verify rise time on a scope.
  2. 5 V vs 3.3 V mixed-rail without translation. A 5 V slave driving SDA low pulls the line to 0.4 V — fine for a 3.3 V master, which sees that as a logic 0. But the bus's pull-up on the 3.3 V side only goes to 3.3 V, which a 5 V slave reads as somewhere between "1" and "indeterminate" depending on its VIH,min. Fix: use a level translator (PCA9306 for low-speed, PCA9517 for higher speed, NTS0102 for tiny boards) — these are passive bidirectional FET-based switches that do exactly the right thing on an open-drain bus.
  3. Address written in 8-bit form. "Datasheet says 0xA0, I wrote 0xA0, getting NAK." That's because the driver shifted-and-OR'd, sending 0x140 → truncated to 0x40 → no slave at 0x40 → NAK. Fix: convert to 7-bit (0xA0 >> 1 = 0x50), pass that. (See the "Shifted vs unshifted" warning above. This is the single most common I²C bug in the wild.)
  4. Missing STOP after a NAK leaves the bus stuck. Some drivers, when they get a NAK, return an error to the application and forget to emit STOP. The bus stays in an indeterminate state — slaves think a transaction is still in progress, future transactions misbehave, and a power-cycle is the only recovery. Zephyr's I²C drivers handle this correctly; if you write your own bit-bang master, make sure NAK paths always emit STOP before unwinding.
  5. Sensor needs ≥1 µs between power-on and first I²C access. Many sensors have a "power-on reset to ready" specification of 0.5–10 ms. Issuing your first I²C command immediately after powering the slave can return NAK or garbage data. Fix: k_sleep(K_MSEC(10)) after enabling Vio, before the first transaction. Read the slave's datasheet — there's always a tboot or tPOR spec.
  6. Stuck SCL or SDA — a slave is hung. If a slave was halfway through a byte when the master crashed/reset, it may still be trying to drive SDA. Recovery is the I²C bus-clear procedure: master toggles SCL up to 9 times until SDA goes high, then issues STOP. Zephyr's i2c_recover_bus() implements this; call it after init and after any unexplained NAK to make the bus deterministic.
  7. Mixing pull-up rails. If your pull-ups are tied to 3.3 V but the slave's input only tolerates 2.5 V, you'll exceed VIH,abs and stress the slave's clamps. Match the pull-up rail to the lowest-voltage device on the bus.
  8. Forgetting the ACK on the last byte of a master read. The protocol requires the master to NAK the final byte of a multi-byte read — that's how it tells the slave "I have what I need, release SDA". If the master ACKs every byte forever, well-behaved slaves will keep clocking out data and eventually wrap or send garbage. Most APIs handle this for you (last i2c_msg.len automatically NAKs the last byte) but if you're rolling your own, get the NAK in.
  9. Speed mismatch in multi-device buses. One sensor is 100 kHz max, another is 1 MHz capable, you set the bus to 1 MHz to be fast. The 100 kHz sensor either fails outright or produces corrupted data, and the symptom is "everything works individually, breaks when both are on the bus." Fix: bus speed = min(all devices' max).
  10. Sensor reports "I'm at 0xC0/0xC1", not "0x60". Same as #3, just stated differently. Always interpret datasheet addresses as read = 8-bit-form-with-LSB-set, write = 8-bit-form-with-LSB-clear, and 7-bit-API-form = (write_addr >> 1).
  11. Forgetting that the slave can stretch even on the address byte. A soft-I²C-slave might stretch SCL on the very first ACK pulse while it's looking up whether to ACK at all. Some master drivers assume stretching only happens during data — they bail out with a "no ACK received" error before the slave has even decided. If a slave that "definitely should ACK" is NAK'ing intermittently, scope SCL during the 9th pulse of the address byte and look for an extended low.

Where to learn more

  1. NXP UM10204 — I²C-bus specification and user manual, Rev 7.0 (October 2021) — the canonical spec. 64 pages, very readable. Read sections 3 (the bus), 5 (definitions), and Appendix B (electrical characteristics) once and you'll have the full picture.
  2. TI SLVA704 — Understanding the I²C Bus — short (8-page) practical primer; the rise-time / pull-up sizing math is presented compactly here.
  3. TI SLVA689 — A Basic Guide to I²C — TI's "I²C Made Simple" set of application notes; covers timing margin and bus extender ICs.
  4. SMBus 3.1 specification (SBS Implementer's Forum, 2018) — the full SMBus spec, including PEC, timeouts, ARP, and the standard transaction types.
  5. PMBus 1.4 specification — the power-management command set. Useful if you need to drive or emulate digital regulators.
  6. RP2350 datasheet, §11 (PIO) — the PIO architecture reference. Read this before editing edevkit1's PIO I²C source.
  7. Zephyr I²C subsystem docs — the Zephyr i2c driver API reference. Authoritative for i2c_msg flags, transaction semantics, bus-recover mechanism.
  8. Sister docs in this hub — proto_uart.html (UART), proto_spi.html (SPI/QSPI), platform_pio.html (PIO programming model and edevkit1's specific PIO programs), edevkit1.html (the v1 design spec — see §6 for the I/O subsystem and §11 for the protocol tester).
