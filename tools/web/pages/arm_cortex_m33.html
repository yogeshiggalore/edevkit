Cortex-M33 — the Armv8-M Mainline core, with TrustZone-M

A focused deep dive on the Arm Cortex-M33 — the core sitting in the RP2350B that drives edevkit1, and the dominant choice across modern security-aware MCUs (Nordic nRF5340 / nRF91 / nRF54, NXP LPC55Sxx, ST STM32L5 / U5, Renesas RA, Microchip SAM L1x). This is the document you read after arm_architecture.html when you actually have to write or debug code on this specific core.

What's covered here vs the sister doc. General Cortex-M material — the programmer's model, the exception model, the memory model, CoreSight basics, the M-profile family — lives in arm_architecture.html. This doc is the delta the M33 brings on top: the Armv8-M Mainline ISA, TrustZone-M, the Security Attribution Unit, the new banked registers, stack-limit checking (PSPLIM/MSPLIM), MPU v8, the optional FPv5-SP-D16 FPU, and how all of this is wired up in real silicon — including the RP2350B that ends up on every edevkit1 board.

Audience. A firmware engineer who has used a Cortex-M before (M3/M4/M7) and now has an M33-based SoC on the bench. Or someone bringing up Zephyr/MCUboot on an M33 and wondering what the new cmse_nonsecure_entry, __TZ_set_PSPLIM_NS, and "secure" / "non-secure" build outputs are for. By the end you should understand exactly what TrustZone-M buys you, exactly what it costs, and how to opt in or out per project.

What is the Cortex-M33

The Cortex-M33 is Arm's flagship "mainstream secure microcontroller" core. It was announced in October 2016 alongside the smaller Cortex-M23, and the first commercial silicon shipped in 2017–2018 (NXP LPC55S69, Microchip SAM L11, then everyone else). It implements the Armv8-M Mainline profile — the M-class evolution that finally brought TrustZone to microcontrollers. Where M3/M4 are Armv7-M and M0/M0+ are Armv6-M, the M23/M33 sit on Armv8-M (Baseline / Mainline respectively) and are usually the first M-profile core a vendor ships when they want a "secure-capable" MCU line.

Mentally, place the M33 here:

Armv6-M  Baseline   ── Cortex-M0,  M0+, M1   (cheapest, smallest)
Armv7-M  Mainline   ── Cortex-M3,  M4,  M7   (workhorse 2006–2016)
Armv7E-M Mainline+DSP ── Cortex-M4, M7        (DSP+FPU variants)
Armv8-M  Baseline   ── Cortex-M23             (M0+ replacement, +TrustZone)
Armv8-M  Mainline   ── Cortex-M33, M35P       (M4 replacement, +TrustZone)
Armv8.1-M Mainline  ── Cortex-M52, M55, M85   (Helium / MVE; not M33)
key idea The M33 is the M4 plus eight years of lessons: a near-identical pipeline and DSP feature set, but with TrustZone, stack-limit registers, MPU v8, a refined FPU (FPv5-SP-D16 instead of FPv4-SP-D16), and a slightly tidier reset/exception model. If you've shipped Cortex-M4 firmware before, the M33 will feel familiar — most of your code compiles unchanged with -mcpu=cortex-m33. The big new thing is what you can opt into: TrustZone-M.

"Why M33 instead of M4?"

If you're choosing a vendor SoC today and you have the option, you pick M33 over M4 for one or more of these reasons:

  • Hardware-enforced isolation. TrustZone-M lets you run an untrusted app and a trusted root-of-trust on the same core, with hardware preventing the app from touching secure code, secure RAM, secure peripherals, or secure NVIC entries. This is the only feature you literally cannot synthesize on an M4.
  • Stack overflow as a first-class fault. The PSPLIM/MSPLIM registers turn silent stack-into-heap corruption into an immediate UsageFault — a top-five bug class on M4 simply ceases to exist.
  • Security-certifiable. Many "secure boot", "PSA Certified", or "SESIP Level 3" claims need TrustZone-grade isolation; M33 SoCs are how you check that box.
  • Long support. M3/M4 are now in late-life. New SoC lines are M33 or higher; staying on Armv8-M is the future-proof choice.
  • FreeRTOS / Zephyr / Mbed OS / TF-M now assume v8-M. Trusted Firmware-M (TF-M) only targets Armv8-M (and 8.1-M); if you want a vetted secure partition manager, M33 is the on-ramp.

Reasons you'd still pick M4 in 2026: existing tooling/firmware is already on M4 and rewrites aren't justified, the SoC you need isn't available with an M33 (still common in motor-control parts), or you don't care about TrustZone and want the cheapest possible core. For greenfield designs that need security, M33 is the default.

Microarchitecture — pipeline, interfaces, options

The M33 core is a 3-stage in-order single-issue pipeline: Fetch → Decode → Execute. That is the same depth as the M3 and M4. There is no out-of-order execution, no superscalar issue, and no branch predictor in the classical sense (the core has a small branch-target speculation hint, but it is not user-tunable). Designers picked 3 stages because it minimises the branch-mispredict penalty (one cycle wasted, not eight) and keeps interrupt latency tight — both of which matter more than peak IPC on a microcontroller.

            ┌─────────┐   ┌─────────┐   ┌─────────┐
   PC ────► │  FETCH  ├──►│ DECODE  ├──►│ EXECUTE ├──► result
            └─────────┘   └─────────┘   └─────────┘
                ▲                            │
                │  branch target / EXC_RETURN│
                └────────────────────────────┘

Most "ALU op" instructions retire in 1 cycle. Multiply is 1–2 cycles. Divide is hardware (3–7 cycles). Loads and stores are 2 cycles when both code and data hit different buses; longer when they contend. Single-cycle GPIO is feasible if the GPIO peripheral lives on a dedicated AHB-Lite slave (the RP2350 SIO block, for example).

External interfaces

The M33's bus interface is AHB5 — an evolution of AHB-Lite that adds the security attributes (HNONSEC bit) needed by TrustZone-M. From a vendor's standpoint, AHB5 is mostly "AHB-Lite + a sideband signal," so existing AHB peripherals integrate with minimal changes. Cores can also expose a separate D-AHB for code memory (an "I-AHB" / "D-AHB" split improves memory-bandwidth utilisation when code is in flash and data is in SRAM).

Synthesis-time options (vendor-configurable)

The M33 is delivered to silicon vendors as a configurable RTL macro. Each vendor chooses which features to include, which is why "Cortex-M33" parts have visibly different feature sets across vendors. The major options:

  • FPU — FPv5-SP-D16 (single-precision IEEE-754, 16 D-registers / 32 S-registers). Optional.
  • DSP extensions — the SIMD / saturating-arithmetic instruction set inherited from Armv7E-M (M4). Optional.
  • TrustZone Security Extension — adds Secure / Non-secure states, banked registers, SAU, etc. Optional but increasingly the default.
  • MPU — 0, 4, 8, or 16 regions per security state. Optional but ubiquitous; without TrustZone there's one MPU, with TrustZone there are two (MPU_S and MPU_NS).
  • Coprocessor interface — up to 8 user-defined coprocessors via the Custom Datapath Extension (CDE). Rare.
  • ETM / MTB — instruction trace (ETM) and the cheaper Micro Trace Buffer. Optional.
  • WIC — Wake-up Interrupt Controller for ultra-low-power state-retention sleep. Optional.
  • Bit-banding — removed in Armv8-M. The M33 has no bit-band region. Use the BFI / UBFX instructions or atomic LDREX/STREX sequences instead.
M33 ≠ M55 ≠ M85 The M33 does not have Helium (M-Profile Vector Extensions / MVE). Helium ships only on Armv8.1-M cores: M52, M55, M85. The M33 also has no L1 cache — caches are an M55/M85 feature. If a vendor part numbered as "M33" lists I/D caches, those are system-level caches sitting between the core and external memory (e.g. STM32U5's ICACHE), not core-internal caches.

Source: Arm Cortex-M33 Technical Reference Manual (DDI 0479 / 100230), sections "Configurable options" and "Programmer's model".

What's new in Armv8-M Mainline vs Armv7-M

If you've shipped Cortex-M3/M4 firmware (Armv7-M), most of your code rebuilds for M33 unchanged. The Armv8-M ISA was deliberately a near-superset of Armv7-M so existing toolchains, RTOSes, and CMSIS-Core code keep working. The differences are concentrated in five areas:

Most of the new instructions only have meaning when the Security Extension is implemented (they manage state transitions and address-attribute checks). If your vendor sells a non-TZ M33 variant, those instructions execute as unpredictable / NOP-like.

The other change worth flagging: Armv8-M deprecates the SCB->CCR.UNALIGN_TRP and STKALIGN behaviours from Armv7-M's late-stage refinements. Defaults are now safer (8-byte stack alignment is always on), and a few CCR bits move. Existing CMSIS-Core(M) code handles this, but you'll see a different reset value if you're reading CCR by hand.

Source: Arm Architecture Reference Manual — Armv8-M (DDI 0553); Migrating to Armv8-M (107565).

Stack-limit registers — PSPLIM and MSPLIM

Stack overflow is, empirically, in the top three causes of "weird intermittent crashes" on Cortex-M3/M4 firmware. The stack grows downward; if a function uses too much locals or recursion, the SP walks off the bottom of the stack region and into whatever memory is below — typically .bss or another task's stack. The result is silent corruption that surfaces hours later as a wild branch or a struct field that "spontaneously" went wrong.

Armv8-M fixes this with two new banked special-purpose registers:

  • MSPLIM — the lower bound (limit) of the Main Stack Pointer.
  • PSPLIM — the lower bound of the Process Stack Pointer.

On every push, the core checks SP >= SPLIM. If the proposed new SP would fall below the limit, the write is blocked and a UsageFault is raised with UFSR.STKOF = 1 ("stack overflow"). Crucially, the corrupting write never lands — your stack region's neighbour is intact, and you can debug the fault from a known-good state.

Both registers are banked under TrustZone:

┌──────────────────────────────────────────────────────────────┐
│  Secure state:      MSP_S    PSP_S    MSPLIM_S    PSPLIM_S   │
│  Non-secure state:  MSP_NS   PSP_NS   MSPLIM_NS   PSPLIM_NS  │
└──────────────────────────────────────────────────────────────┘

Setting them up is one MSR per stack you care about. Typical reset-handler sequence on a single-core M33:

/* In Reset_Handler, before main() */
extern uint32_t __StackLimit;        /* defined by linker */
extern uint32_t __StackTop;

__set_MSPLIM((uint32_t)&__StackLimit);   /* main stack lower bound */
__set_MSP   ((uint32_t)&__StackTop);     /* main stack initial top */

/* If you also use a PSP for an RTOS / unprivileged thread mode: */
__set_PSPLIM(task_stack_limit);
__set_PSP   (task_stack_top);

Where __set_MSPLIM is a CMSIS-Core(M) intrinsic that emits MSR MSPLIM, Rn. Your linker script typically defines __StackLimit at the bottom of the stack region and __StackTop at the top.

RTOS interaction

Modern RTOSes set PSPLIM as part of their context switch:

  • Zephyr — when CONFIG_BUILTIN_STACK_GUARD=y on Armv8-M, the kernel writes PSPLIM in arch/arm/core/cortex_m/swap.c on every task switch, pointing to the per-thread stack base. Cheaper and faster than the MPU-based stack guard used on Armv7-M.
  • FreeRTOS — sets PSPLIM in port.c when configENABLE_TRUSTZONE or the M33 port is selected. The task control block carries a pxStackLimit field.
  • ThreadX — same idea; the M33 port writes PSPLIM during thread restore.
tip Stack-limit checks are free — they run on every push in hardware. On Armv7-M the equivalent protection costs you an MPU region per task plus the MPU reprogram cost on every context switch. On M33 with PSPLIM, you get strictly better protection at strictly lower cost. Always enable it.

Source: Armv8-M ARM section B11.5 "Stack pointer limit registers"; Zephyr arch/arm/core/cortex_m; FreeRTOS-Kernel portable/GCC/ARM_CM33.

MPU v8 (PMSAv8)

The Memory Protection Unit on M33 is PMSAv8 — a redesign of the v7 MPU that fixes its biggest pain points: power-of-2 region sizes (you couldn't protect a 12 KB stack without splitting it) and per-region inline attribute encoding (every region carried its full attribute bits, blowing up the number of MPU writes during a context switch).

Region shape — base + limit

v7-M region: base + size (power of 2) + sub-region disable bits. v8-M region: base address + limit address. Any 32-byte-aligned region size is now legal; no more sub-regions, no more "round my 12 KB up to 16 KB and waste 4 KB".

MPU_RBAR (Region Base Address Register)
  31..5   BASE       Region start (32-byte aligned)
  4..3    SH         Shareability (00 non-, 01 reserved, 10 outer, 11 inner)
  2..1    AP         Access permissions (RW/RO × priv/unpriv)
  0       XN         Execute-never

MPU_RLAR (Region Limit Address Register)
  31..5   LIMIT      Region end (inclusive, must be ≥ BASE)
  4..1    AttrIndx   Index into MAIR0/MAIR1 (0..7)
  0       EN         Region enable

Attribute indirection — MAIR0 / MAIR1

The bigger conceptual change: instead of every region carrying a full 8-bit memory-attribute encoding (Normal/Device, cacheability, etc.) inline in its descriptor, v8-M defines 8 attribute slots in two new registers (MAIR0 for slots 0–3, MAIR1 for slots 4–7). Each region's RLAR.AttrIndx is a 3-bit index into those 8 slots. Set up your "this is normal cacheable RAM", "this is device memory", "this is XIP flash" attributes once in MAIR; thereafter every region just references them.

/* Typical M33 attribute setup */
#define ATTR_NORMAL_WT_NA  0x44  /* normal, write-through, no allocate */
#define ATTR_NORMAL_WB_RA  0xFF  /* normal, write-back, read-allocate, write-allocate */
#define ATTR_DEVICE_NGNRE  0x04  /* device, non-Gathering, non-Reordering, Early ack */

MPU->MAIR0 = (ATTR_NORMAL_WB_RA << 0)   /* slot 0: normal RAM   */
           | (ATTR_DEVICE_NGNRE  << 8)   /* slot 1: peripherals  */
           | (ATTR_NORMAL_WT_NA  << 16); /* slot 2: XIP flash    */

/* Region 0: 4..16 KB SRAM, normal RAM, RW, no execute */
MPU->RNR  = 0;
MPU->RBAR = (0x20001000) | (0 << 1) | (1 << 0);  /* AP=RW priv, XN=1 */
MPU->RLAR = (0x20003FFF) | (0 << 1) | 1;          /* AttrIndx=0, EN=1 */

Banked MPU under TrustZone

With the Security Extension, there are two MPUs — MPU_S (at 0xE000ED90) and MPU_NS (at 0xE002ED90). Each has its own region set, MAIRs, and CTRL register. Secure code can program both; Non-secure code can only see MPU_NS. They are completely independent: an address can be readable in one security state and unreachable in the other.

Source: Armv8-M ARM chapter B8 "PMSAv8"; Arm Cortex-M33 Devices Generic User Guide (100166).

FPU — FPv5-SP-D16 and lazy stacking

When configured with the FPU, the M33 implements FPv5-SP-D16: IEEE-754 single-precision (32-bit) floats, with 16 doubleword registers D0..D15 aliased as 32 single-word registers S0..S31. Operations covered: add, sub, mul, mac/msc, divide, sqrt, compare, convert, abs/neg, min/max, fused multiply-add. No double precision.

The "5" in FPv5 vs "4" in FPv4 (M4) is a refinement: FPv5 adds VSEL (conditional select), VRINTx (rounding to integer keeping float type), VMINNM/VMAXNM (with NaN handling per IEEE-754-2008), and improved conversions. From a programmer's standpoint the differences are minor — most C float code compiles identically. The compiler picks them up via -mfpu=fpv5-sp-d16 -mfloat-abi=hard.

Lazy stacking

FP context (S0..S15, FPSCR) is 17 words. If every interrupt entry stacked all of it, ISR latency would balloon by ~17 cycles on a 3-stage core. Armv7-M (M4) introduced lazy stacking: on exception entry, the core only reserves stack space for FP context but doesn't actually save it. If the ISR never touches the FPU, no FP save/restore happens and there's no penalty. If the ISR does execute its first FP instruction, the hardware traps, walks back, lazy-saves the old FP context, then resumes the ISR.

M33 inherits and refines this. Two registers control the behaviour:

  • FPCCR — FP Context Control. Bits: ASPEN (auto-enable FP context preservation), LSPEN (lazy state preservation enable), LSPACT (lazy state pending — set on entry, cleared when actually saved), and several TZ-related bits (S, SFRDY, BFRDY, MMRDY, UFRDY, HFRDY, TS, CLRONRET, CLRONRETS).
  • FPCAR — FP Context Address. Holds the address where lazy stacking will save FP context if needed. The address points 64 bytes into the stack frame (the reserved FP region).

FPU + TrustZone — CLRONRET

An information-leak gotcha: if Secure code uses the FPU and is preempted by a Non-secure interrupt without lazy-saving FP context, the Non-secure handler can read the secure FP register file and exfiltrate secrets. Armv8-M defines FPCCR.TS (treat FP as secure) and CLRONRET (clear FP regs on return from secure to non-secure) to plug this. On M33 with TZ, set both bits at boot in your secure init code. CMSIS-Core(M) headers expose FPU->FPCCR_S |= (FPU_FPCCR_TS_Msk | FPU_FPCCR_CLRONRET_Msk);.

Source: Armv8-M ARM section B11.4 "Floating-point support"; AN298 lazy stacking on Armv7-M (still applicable mechanism).

DSP extensions and the coprocessor interface

The M33 may include the DSP extension — the same SIMD-style instructions originally added to Armv7E-M (M4). When present, they cover:

  • Packed 16-bit / 8-bit arithmetic — SADD16, SADD8, SSUB16, SSUB8, etc. Operate on two 16-bit halves or four 8-bit lanes per 32-bit register.
  • Saturating arithmetic — SSAT, USAT, QADD, QSUB, QDADD, QDSUB. Mandatory in fixed-point DSP code.
  • Multiply-and-accumulate — SMLAD, SMLAL, SMLALD, SMUAD, SMUSD. Single-cycle FIR/IIR taps.
  • Sign extension and packing — SXTB16, UXTB16, PKHBT, PKHTB.

If your part has the DSP option, the CMSIS-DSP library accelerates dramatically (FFT, FIR, IIR, matrix, distance) — typically 4–10× over plain C. The library's source is at github.com/ARM-software/CMSIS-DSP.

Whether the option is present is determined at synthesis time and reported via the SCB feature registers (SCB->ID_PFR1, SCB->ID_DFR0) — see DDI 0553 chapter D1 for the encoding. CMSIS exposes a build-time macro __ARM_FEATURE_DSP that's set by the compiler when invoked with -mcpu=cortex-m33 on a part that has DSP enabled (or with explicit -march=armv8-m.main+dsp).

Custom Datapath Extension (CDE)

An exotic option: M33 supports up to 8 vendor-defined coprocessors via the Custom Datapath Extension. These appear in the ISA as CX1/CX2/CX3 instructions and let an SoC designer add a few cycles of fixed-function logic on the core's pipeline (e.g. a CRC accelerator, a small XOR engine, a packet-header parser). Almost no general-purpose chip ships with CDE; it's mostly used in domain-specific microcontrollers (some ASIL-D motor controllers, niche IoT). Mention it for completeness; you will not encounter it on Nordic / NXP / ST / RP2350.

Source: Armv8-M ARM section A6.3 "DSP extension"; Cortex-M33 TRM section 1.4 "Configurable options".

TrustZone-M — the centerpiece

This is the headline feature of the M33 and the reason most modern security-conscious vendors moved off the M4. TrustZone for Armv8-M (TZ-M) brings the same hardware-isolation idea that Cortex-A has had since 2003 down to a microcontroller-class core, but redesigned to fit a real-time deeply-embedded model: low interrupt latency, no monitor mode, no virtualization layer, no MMU.

Why hardware isolation matters

Imagine your firmware needs to: (a) hold a private key and use it to sign things on behalf of an application, and (b) run a large pile of third-party application code (network stacks, BLE host, ML inference, file system). Without isolation, every line of (b) is in the same address space as the key in (a) — one buffer overflow in the BLE parser, one logic bug in the file system, and the key is exfiltrated. Software-only mitigations (ASLR, stack canaries, MPU-based sandboxing) help but don't fundamentally bound the attack surface.

TrustZone-M draws a hardware line. The core can be in Secure state or Non-secure state at any moment. Memory regions, peripherals, NVIC slots, MPU regions, and stacks are statically attributed as Secure or Non-secure (the SAU/IDAU make that decision per address). When the core is in NS state, any access to an S-attributed address fails — instruction fetch, data load, MMIO write — with a SecureFault delivered to the secure side. The NS code literally cannot read or call the secure code, no matter what bug it has.

Two states, four contexts

Each cell has its own banked SP/MSP/PSP/MSPLIM/PSPLIM/CONTROL/PRIMASK/FAULTMASK/BASEPRI. The CPU also banks SCB (most of it), MPU, SAU access, NVIC bits, SysTick, and the vector table base (VTOR). Crossing between Secure and Non-secure happens only through specific instructions (SG, BXNS, BLXNS) — the next section is dedicated to those.

mental model TrustZone-M is best thought of as two CPUs sharing a pipeline. Each "CPU" has its own MSP, PSP, vector table, NVIC view, SysTick, MPU. They share the actual silicon — the same instruction goes through the same fetch/decode/execute. The "CPU" identity is the security state. Calls cross from one to the other only at vetted gateway addresses; everywhere else, the hardware enforces isolation.

What TrustZone-M is not

  • Not a hypervisor. No second guest OS, no monitor mode, no virtualization. It's two flat address spaces sharing one CPU.
  • Not crypto. TZ-M provides isolation; the actual crypto (AES, SHA, ECDSA) runs in software in the secure partition or on a dedicated peripheral.
  • Not a sandbox for arbitrary mutual-distrust workloads. The security boundary is one-way — secure trusts non-secure to follow the gateway protocol, but secure code is the trusted side. There is no symmetric isolation between two non-secure tasks (you'd use the MPU for that).
  • Not free at design time. Memory partitioning, secure-callable veneers, ABI design across the boundary — these are engineering choices that take real effort to get right. The reward is hardware-enforced isolation; the cost is a new mental model.

Source: Arm — TrustZone for Cortex-M; Building secure software with TrustZone for Armv8-M (100690); Joseph Yiu, Definitive Guide to Cortex-M23/M33, chapters 5–7.

Banked registers under TrustZone

"Banking" means the same logical register has two physical copies, one per security state. The CPU presents whichever copy matches the current state. Most banking is automatic — you reference MSP, you get the copy for whichever state you're in. Some banked registers can be reached cross-state through suffixed access mnemonics (MSP_NS, MRS r0, PSP_S) — but the CPU will only honour those when issued from Secure state. NS state cannot read or write Secure-side banked copies.

SAU and IDAU — memory attribution

How does the CPU know whether a given physical address is Secure, Non-secure, or Non-secure-Callable? Two cooperating units make that decision: the IDAU (Implementation Defined Attribution Unit) and the SAU (Security Attribution Unit). They run in parallel; the result of both is combined to produce the address's attribution.

IDAU — vendor-fixed

The IDAU is hardwired by the silicon vendor and immutable at runtime. It typically encodes coarse, conservative defaults — for example, "the bottom 32 KB of flash is Non-secure-Callable; the upper bits of the address space follow a fixed pattern." On NXP LPC55S69, the IDAU uses address-bit aliasing: address bit [28] selects security — the same physical memory appears at both a secure alias and a non-secure alias 256 MB apart. On RP2350, the IDAU is simpler — it leaves attribution entirely to the SAU.

You don't program the IDAU. Read your silicon vendor's reference manual to learn the fixed map; design your software around it.

SAU — software-defined

The SAU is programmable. It implements 4 or 8 region descriptors (vendor-configured, but always at least 4). Each region is described by:

SAU_RBAR (Region Base Address)         — start (32-byte aligned)
SAU_RLAR (Region Limit Address)        — end + NSC bit + EN bit
  RLAR.LIMIT[31..5]   end address (inclusive)
  RLAR.NSC            1 = Non-secure Callable, 0 = Non-secure
  RLAR.EN             1 = enabled

SAU_CTRL                                — global controls
  ALLNS  1 = treat everything as NS (used during early boot)
  ENABLE 1 = SAU active

An SAU region attributes a memory range as either Non-secure (NSC=0) or Non-secure Callable (NSC=1). Anything not covered by an enabled SAU region is Secure by default (the SAU is "secure unless told otherwise"). This default is what makes secure boot possible — at reset everything is secure, you set up the SAU to carve out NS regions, then you launch NS code.

Combining IDAU and SAU

For each access, both the IDAU and SAU return a "what is this address" answer. The more secure answer wins:

A worked example

Imagine a 1 MB flash + 256 KB SRAM SoC. You want:

  • Flash 0x00000000–0x000FFFFF (1 MB): bottom 256 KB Secure (boot ROM + secure firmware), middle 4 KB Non-secure-Callable (the veneer table), upper Non-secure (the application image).
  • SRAM 0x20000000–0x2003FFFF (256 KB): bottom 64 KB Secure, upper Non-secure.
  • Peripherals 0x40000000–0x60000000: vendor-attributed via IDAU and a secondary firewall; SAU leaves them alone (treated as NS by SAU, hardware firewall does final check).
/* Secure init code, before launching NS */
SAU->RNR = 0;
SAU->RBAR = 0x00040000;                           /* NS flash starts at 256 KB  */
SAU->RLAR = (0x000FFFE0) | (0 << 1) | 1;          /* NS up to ~1 MB, NSC=0, EN  */

SAU->RNR = 1;
SAU->RBAR = 0x0003F000;                           /* NSC veneer region          */
SAU->RLAR = (0x0003FFE0) | (1 << 1) | 1;          /* NSC=1, EN                  */

SAU->RNR = 2;
SAU->RBAR = 0x20010000;                           /* NS SRAM starts at +64 KB   */
SAU->RLAR = (0x2003FFE0) | (0 << 1) | 1;          /* NS rest of SRAM            */

SAU->RNR = 3;
SAU->RBAR = 0x40000000;                           /* NS peripherals             */
SAU->RLAR = (0x5FFFFFE0) | (0 << 1) | 1;

SAU->CTRL = 1;                                    /* enable SAU                 */

After this, secure code runs from 0x00000000, calls into the veneer table at 0x0003F000, and the application running from 0x00040000 is hardware-isolated from secure flash and secure SRAM.

Source: Armv8-M ARM section B8.2 "SAU"; 100690 — Building secure software; Joseph Yiu, chapter 6.

State transitions — SG, BXNS, BLXNS, TT

Crossing the security boundary is the part of TrustZone-M that surprises people. There are exactly four mechanisms, each with a specific direction and protocol.

NS → S — the Secure Gateway dance

Non-secure code wants to call a secure function. It calls a veneer, which lives in NSC memory (Non-secure-Callable: visible to NS as a callable address, but classified as a security boundary). The veneer's first instruction must be SG — the Secure Gateway. Anything else trips a SecureFault.

; Veneer in NSC region, e.g. at 0x0003F010
secure_function_veneer:
    SG                       ; Secure Gateway — switches CPU to Secure state
    B   secure_function      ; tail-call into the real implementation

Mechanically: NS code does BL secure_function_veneer. The CPU fetches at the veneer address, the IDAU/SAU say "this is NSC", and on entry the very first instruction is checked: if it's SG, the state flips to Secure and execution continues. If anything else, SecureFault. The branch immediately after SG goes to the real secure function (which lives in Secure memory, unreachable to NS directly).

S → NS — BXNS / BLXNS

Secure code wants to call into Non-secure. Two instructions, with the NS suffix:

  • BXNS Rn — branch and switch state to NS. Like BX but transitions to Non-secure. Rn's low bit must be 0 (the inversion of the normal Thumb/ARM hint).
  • BLXNS Rn — branch with link, switch state to NS. The hardware pushes a special "FNC_RETURN" frame on the secure stack so when the callee does an ordinary BX LR, the CPU realises the return needs to switch back to Secure and unstacks correctly.

For a one-way "boot the application" handover, you use BXNS. For a callback (S calls NS, NS does work, returns to S), you use BLXNS.

Boot handover sequence

/* Secure bootloader, after SAU setup */
extern uint32_t __ns_vector_table;     /* linker symbol — NS reset vector */

/* Set NS vector table */
SCB_NS->VTOR = (uint32_t)&__ns_vector_table;

/* Set NS MSP from NS vector table[0] (initial MSP) */
__TZ_set_MSP_NS(((uint32_t *)&__ns_vector_table)[0]);

/* Resolve NS reset handler from vector table[1], drop the Thumb bit, BXNS */
uint32_t ns_reset = ((uint32_t *)&__ns_vector_table)[1] & ~1u;
typedef void (*ns_func_t)(void) __attribute__((cmse_nonsecure_call));
((ns_func_t)ns_reset)();

The __attribute__((cmse_nonsecure_call)) tells GCC to emit BLXNS at the call site, scrub general-purpose registers, and clear FP regs if needed.

TT, TTT, TTA, TTAT — query an address's attribution

How does Secure code check, before dereferencing a Non-secure pointer, that the pointer actually points into NS memory and not, say, an NSC veneer being passed as a confused-deputy attack? The TT family answers:

  • TT Rd, Rn — Test Target. Rd ← attribution of the address in Rn as the current security state.
  • TTT Rd, Rn — same, but query MPU privilege bit too.
  • TTA Rd, Rn — Test Target, Alternate. Query as the other security state. Used by Secure code to ask "what would NS see at this address?"
  • TTAT Rd, Rn — alternate + privilege.

Secure entry-point code uses TTA/TTAT (via the GCC builtin cmse_check_address_range()) to validate every NS-supplied pointer at the boundary. Skipping this is the most common TZ-M bug class.

Source: Armv8-M ARM sections C2.5 "SG", C2.6 "BXNS/BLXNS", C2.7 "TT family".

NVIC under TrustZone — ITNS, AIRCR.PRIS

The NVIC keeps the same shape — up to 480 external IRQs, 8 priority bits per IRQ (vendor-implemented, often 3–4) — but each IRQ is independently targeted at Secure or Non-secure state. The targeting register is NVIC->ITNSn (Interrupt Target Non-Secure, one bit per IRQ).

  • ITNS[n] = 0 — IRQ n is targeted at Secure state. Its handler runs in Secure mode, vector resolved from VTOR_S. NS code can't enable, disable, pend, clear, or change priority — those operations are denied.
  • ITNS[n] = 1 — IRQ n is targeted at Non-secure. Handler runs in NS, vector from VTOR_NS. Secure code can still see the bit but typically leaves NS-targeted IRQs to NS to manage.

ITNS itself is Secure-only writable. NS code can read it but not modify. This is how secure boot statically partitions interrupts: a "this peripheral belongs to NS" decision is made once at boot, NS can never re-target one of its IRQs to Secure.

Priority space split — AIRCR.PRIS

Without TZ, an 8-bit priority field gives 256 priority levels. Under TZ, you have a problem: if Secure and Non-secure share the priority space, NS code could set a priority of 0 on its own IRQ and starve the secure side. AIRCR.PRIS prevents this:

  • AIRCR.PRIS = 0 (the default) — single shared priority space. Secure and Non-secure IRQs interleave by their numerical priority. Convenient for development.
  • AIRCR.PRIS = 1 — the priority space splits in half. Secure IRQs occupy 0..127 (always preempt NS), Non-secure IRQs are remapped into 128..255 (logically, NS sees its priorities but the hardware tags them).

For production secure boot you set AIRCR.PRIS = 1 in secure init code. Once set, no NS IRQ can ever preempt a Secure IRQ regardless of how NS configures its own priorities.

NSACR — Non-secure access control

The NSACR register controls which coprocessors NS code may use:

  • NSACR.CP10/CP11 — when set, NS may use the FPU. When clear, FP instructions in NS state trap as a UsageFault.
  • Custom-coprocessor bits if CDE is implemented.

Most projects with FPU set both bits in secure init so the NS application can use floats. Cleared, the FPU becomes a Secure-only resource (rare; mainly for very-tight-isolation profiles).

Source: Armv8-M ARM section B3.10 "NVIC banking"; AIRCR / NSACR encodings in section B3.4.

CMSE toolchain — gcc & armclang attributes

The Cortex-M Security Extensions (CMSE) are the compiler-side conventions that make TrustZone-M usable from C. Both gcc-arm-none-eabi and armclang support CMSE; the surface is identical between them. Enable it with:

# GCC
arm-none-eabi-gcc -mcpu=cortex-m33 -mcmse \
    -mfloat-abi=hard -mfpu=fpv5-sp-d16 \
    secure_app.c -o secure_app.o

# armclang
armclang --target=arm-arm-none-eabi -mcpu=cortex-m33 -mcmse \
    -mfloat-abi=hard -mfpu=fpv5-sp-d16 \
    secure_app.c -c -o secure_app.o

The two attributes you'll write by hand are cmse_nonsecure_entry (on a Secure function callable from NS) and cmse_nonsecure_call (on a function pointer typed as "this points at NS code").

cmse_nonsecure_entry — the secure entry point

/* In a secure source file */
#include <arm_cmse.h>

uint32_t __attribute__((cmse_nonsecure_entry))
sign_message(const uint8_t *msg, size_t len, uint8_t *sig_out)
{
    /* Validate that NS-supplied pointers are actually in NS memory */
    if (!cmse_check_address_range((void*)msg,
                                  len,
                                  CMSE_NONSECURE | CMSE_MPU_READ))
        return -1;
    if (!cmse_check_address_range(sig_out,
                                  64,
                                  CMSE_NONSECURE | CMSE_MPU_READWRITE))
        return -1;

    /* Safe to use msg / sig_out now — we know they're NS, not S leaks */
    return crypto_sign(msg, len, sig_out);
}

The compiler does several things for a cmse_nonsecure_entry function:

  • Emits an SG instruction at the entry, then a normal prologue. The SG sits at the NSC veneer address.
  • On return, scrubs all caller-saved general-purpose registers (R0–R3, R12) of any secret values, clears LR, and uses BXNS LR to return to NS state.
  • If FP is in use, scrubs FP registers and FPSCR via VLLDM (which ties into FPCCR.CLRONRET).
  • Creates a linker symbol for the veneer so the placement script can put it in NSC memory.

cmse_nonsecure_call — calling into NS

/* Function pointer typed as NS */
typedef void (*ns_callback_t)(uint32_t arg) __attribute__((cmse_nonsecure_call));

void secure_dispatch(ns_callback_t cb, uint32_t arg)
{
    /* The compiler emits BLXNS for this call:
       - clears all GPRs the NS callee shouldn't see
       - lazy-stacks FP if needed (CLRONRET)
       - on return, FP-context-clear runs automatically  */
    cb(arg);
}

Without the cmse_nonsecure_call attribute, GCC emits a normal BLX — which from Secure state to an NS address would be detected by the SAU and trap as a SecureFault. The attribute is mandatory whenever you call through a pointer from Secure to Non-secure.

Faults under TrustZone — SecureFault & friends

TrustZone adds one new fault and changes how some old ones are routed.

SecureFault (exception number 7)

Triggered when something violates the security boundary. Always delivered to the Secure side (handler runs in Secure state, vector from VTOR_S[7]). The cause is reported in SFSR (Secure Fault Status Register, at 0xE000EDE4):

SFSR bitMeaning
INVEPInvalid entry point — NS tried to call into S, but the first instruction at the entry wasn't SG.
INVISInvalid integrity signature — NS tried to use a Secure return frame.
INVERInvalid exception return — EXC_RETURN tag inconsistent with security state.
AUVIOLAttribution unit violation — NS access to a Secure address (or vice versa).
INVTRANInvalid transition — branch from S to a Secure address that's not allowed for the current state.
LSPERRLazy state preservation error.
SFARVALIDSFAR (Secure Fault Address Register) holds a valid faulting address.
LSERRLazy state error — relating to FP context security tagging.

HardFault — banked

HardFault is banked: HardFault_S at vector 3 in the Secure VTOR, HardFault_NS at vector 3 in the Non-secure VTOR. By default, all HardFaults go to the Secure handler (so a misbehaving NS can't hide a fault). Bit AIRCR.BFHFNMINS can re-route NS HardFault back to NS — useful for development, dangerous for production.

MemManage, BusFault, UsageFault

Each is banked. The state where the fault originated determines the handler. There are two SHCSRs (one per state); enabling UsageFault in SHCSR_S doesn't affect NS. CFSR (Configurable Fault Status Register) is per-state — when debugging a fault, read the CFSR matching the state in which the fault occurred.

debug tip In a TrustZone-M crash, your debugger's "fault decode" depends on which security state is current. Tools like probe-rs, pyOCD, and JLinkExe can decode CFSR/HFSR/SFSR but you must tell them which state's view to read. Most decoders default to "current state" — halt with monitor halt, then run monitor info securestate (J-Link) before reading status registers.

Source: Armv8-M ARM section B5.5 "Fault model with Security Extension"; Memfault Interrupt blog — Cortex-M fault debugging (Armv7-M version, but the v8-M extension follows the same pattern).

Debug under M33 — DHCSR.S_SDE, secure debug auth

Debug architecture under TZ is the same CoreSight stack you know from M3/M4 — DAP, ITM, DWT, FPB, optional ETM — but with new authentication signals so a debugger can't trivially break secure code's confidentiality.

Authentication signals

Four hardware signals enter the core, each gating a category of debug operations:

SignalWhat it gates
DBGENAll invasive debug — halt, single-step, register access. Required for any debugger to function at all.
NIDENNon-invasive debug — ETM/ITM trace. Allowed even when DBGEN is off (capture without halting).
SPIDENSecure-Privileged Invasive Debug Enable. Required for invasive debug into Secure state.
SPNIDENSecure-Privileged Non-Invasive — secure-side trace.

Vendors typically tie these signals to the silicon's life-cycle state:

  • Open / development — all four asserted. Full debug, including into Secure code. Convenient for development; never ship like this.
  • Closed — DBGEN/NIDEN asserted, SPIDEN/SPNIDEN deasserted. NS code can be debugged; S code cannot. The "production debug" mode for parts that ship with a customer-facing NS application.
  • Locked — all four deasserted. No debug at all. Final-product mode; usually irreversible (a fuse).

DHCSR.S_SDE — what the debugger sees

The Debug Halting Control and Status Register (DHCSR, at 0xE000EDF0) gains a status bit S_SDE ("Secure Debug Enable") which the debugger reads to know whether secure-side halting is permitted. If S_SDE = 0, the debugger must restrict itself to NS-side operations — no halting in secure code, no inspecting secure registers, no setting breakpoints in secure memory. Tooling enforces this in software; the hardware enforces it underneath.

Trace under TZ

ITM and DWT data can be tagged with the security state at the moment of generation. DEMCR.SDME ("Secure DebugMonitor Enable") and the ITM->TER filtering controls let you decide whether NS code can write to secure-tagged trace channels (usually no, to prevent NS from forging "secure" log lines).

Source: Armv8-M ARM section D1 "Debug architecture"; Cortex-M33 TRM chapter 8 "Debug".

Vendor configurations of the M33

The same M33 core looks meaningfully different across silicon vendors because each picks a different combination of synthesis options + adds vendor-specific peripherals (firewalls, crypto blocks, NVM controllers) that interact with TrustZone. Worth knowing the dominant flavours.

Raspberry Pi RP2350 — what edevkit uses

  • Cores: 2× Cortex-M33 + 2× Hazard3 (RISC-V) on the same die. The user picks at boot which pair runs (M33+M33 or RV+RV).
  • Frequency: 150 MHz typical, overclockable.
  • FPU: Yes — FPv5-SP-D16.
  • DSP: Yes.
  • TrustZone: Yes. SAU has 8 regions. IDAU is permissive — defers to SAU.
  • SRAM: 520 KB across 10 banks (8× 64 KB + 2× 4 KB).
  • Flash: XIP from external QSPI (W25Q128 or similar). Boot ROM in mask ROM.
  • Boot security: OTP + signed boot. Signed images verified by ROM before handing off.
  • Distinctive blocks: 12 PIO state machines (programmable I/O — Pi Foundation's signature feature), HSTX (high-speed parallel out), per-bus security gating via the Coprocessor block.

edevkit firmware currently builds for rpi_pico2/rp2350a/m33 and does not enable TrustZone — secure boot lives in the Boot ROM, the application runs entirely in Secure state with TZ effectively unused. This is the Pico SDK / Zephyr default. See edevkit1.html §15 (Decision Log) for the rationale.

Nordic nRF5340

  • Dual M33: Application core (128 MHz, FPU+DSP+TZ, 1 MB flash + 512 KB SRAM) and Network core (64 MHz, no FPU, 256 KB flash + 64 KB SRAM, runs the BLE/802.15.4 stack).
  • Inter-core IPC via 16 IPC channels (mailboxes).
  • SPU (System Protection Unit) — Nordic's bus-level firewall that complements TZ. Enforces S/NS attribution on flash regions, RAM regions, and every peripheral. Programmed alongside SAU in the secure bootloader.
  • CryptoCell-312 — hardware crypto + TRNG + key store. Reachable from Secure side only.
  • TF-M / Zephyr nrf5340_audiodk_app are the canonical reference implementations for "real" TZ usage.

NXP LPC55Sxx (LPC55S69 etc.)

  • Dual M33: both have FPU+DSP+TZ. 150 MHz.
  • IDAU — uses address-bit aliasing. Bit 28 of the address selects security; same physical memory appears at a Secure alias and an NS alias 256 MB apart. Conceptually elegant; in practice surprises new users.
  • PowerQuad — coprocessor for FFT, biquad, matrix, transform. Reachable from CDE-style instructions.
  • CASPER — public-key crypto accelerator (modular arithmetic for ECC/RSA).
  • PRINCE — on-the-fly flash encryption, transparent to the core.
  • ROM-resident "secure boot ROM API" — vendor-supplied entry points NS code can call for things like image authentication.

ST STM32L5 / U5 / H5

  • Single M33; all configurations include FPU+DSP+TZ. Frequencies 110/160/250 MHz across the families.
  • TZSC / GTZC — bus-level TrustZone System / Global TrustZone Controller. Per-peripheral S/NS gating, with the GTZC also providing watermarked memory regions and illegal-access detection.
  • Distinctive: ART Accelerator (instruction prefetch from flash with cache); ICACHE on H5 (system-level cache, not core-internal).
  • STM32CubeMX has a TrustZone wizard that emits the SAU + GTZC + flash watermark code — the lowest-friction onramp to "real" TZ on any vendor part.

Silicon Labs EFR32 Series 2

  • Single M33 + dedicated wireless coprocessor.
  • Series 2 parts (MG24, BG24, FG28) ship with TZ enabled by default.
  • Application Note AN1374 is the canonical "how to use TZ on Silabs" guide and also one of the better cross-vendor TZ-M tutorials.

Migration — from M3/M4 (and from M0+) to M33

Cortex-M4 → Cortex-M33

The friendliest possible migration. Most existing M4 firmware compiles for M33 with two flag changes:

# M4 build:
-mcpu=cortex-m4 -mfpu=fpv4-sp-d16 -mfloat-abi=hard

# M33 build:
-mcpu=cortex-m33 -mfpu=fpv5-sp-d16 -mfloat-abi=hard

What you'll likely change anyway:

  • Add MSPLIM / PSPLIM init. Free stack-overflow protection; no excuse to skip it. ~5 lines in the reset handler.
  • Update CMSIS-Core(M). The CMSIS 6 Cortex-M33 device header has different SCB/MPU/SAU layouts. CMSIS auto-handles this; just regenerate from your vendor's pack.
  • Bit-banding code is gone. If you used BIT_BAND_ALIAS() macros for atomic single-bit access, they've stopped working — use bit-field-insert (BFI/UBFX) or LDREX/STREX. Most vendor HALs already abstracted this.
  • If you want TrustZone — that's the real work. New secure-boot image, NSC veneers, SAU setup, and partitioning peripheral access. Plan a multi-week effort.
  • Update your debugger. probe-rs, pyOCD, OpenOCD all auto-detect M33. Make sure your config selects "cortex-m33" not "cortex-m4" — the SCS layout differs in subtle ways the debugger needs to know about.

Cortex-M3 → Cortex-M33

Same as M4→M33 but you also need to decide whether to enable FPU and DSP. If your M3 code does any float work in software, switching FPU on is a 5–20× speedup on float kernels for free.

Cortex-M0 / M0+ → Cortex-M33

Bigger jump. Things you gain:

  • Full Thumb-2 ISA (32-bit instructions, IT blocks, hardware divide, multi-cycle multiply).
  • 8-bit priority field (256 levels) instead of 2-bit (4 levels).
  • Optional FPU, DSP, TZ.
  • MSPLIM/PSPLIM (no equivalent on M0+).
  • MPU v8 with 8 regions (instead of 0 or 8 v6-M regions).

Things to watch:

  • If your M0+ code did precise nanosecond timing using cycle counts, the M33's longer pipeline + speculative-fetch behaviour means timing must be re-validated. Use DWT->CYCCNT for measurements, never assume.
  • Vector table layout grows — M0+ has up to 32 IRQs, M33 has up to 480. You'll want to autogenerate the vector table or use CMSIS's startup file template.
  • If you used __disable_irq() for naïve mutual exclusion, that still works but you now have BASEPRI for finer-grained masking — adopt it for low-priority RTOS critical sections.

Source: Arm 107565 — Migrating to Armv8-M; Joseph Yiu, Definitive Guide to Cortex-M23/M33 chapter 3.

Concrete code & build snippets

Putting the pieces together. The following snippets are minimal but production-ready patterns.

M33 reset handler with stack limits

/* startup_m33.c (or .S) — the absolute first code that runs */
extern uint32_t __StackTop;       /* top of main stack    */
extern uint32_t __StackLimit;     /* bottom of main stack */
extern uint32_t __etext;          /* end of .text in flash */
extern uint32_t __data_start__;
extern uint32_t __data_end__;
extern uint32_t __bss_start__;
extern uint32_t __bss_end__;

void Reset_Handler(void)
{
    /* 1. Set MSP and MSPLIM as early as possible */
    __set_MSP   ((uint32_t)&__StackTop);
    __set_MSPLIM((uint32_t)&__StackLimit);

    /* 2. Copy .data from flash to RAM */
    uint32_t *src = &__etext;
    for (uint32_t *dst = &__data_start__; dst < &__data_end__; ) *dst++ = *src++;

    /* 3. Zero .bss */
    for (uint32_t *dst = &__bss_start__; dst < &__bss_end__; ) *dst++ = 0;

    /* 4. Vendor system init (PLL, flash wait states, peripheral clocks) */
    SystemInit();

    /* 5. Hand off to libc + main */
    __libc_init_array();
    main();

    /* main returned (it shouldn't on bare metal) */
    while (1) __WFI();
}

Minimal SAU setup for a one-flash, one-RAM partition

/* Called from secure init, before any NS handover */
static void sau_init(void)
{
    /* Region 0: NS flash 0x00040000..0x000FFFFF */
    SAU->RNR  = 0;
    SAU->RBAR = 0x00040000u;
    SAU->RLAR = (0x000FFFE0u) | (0u << 1) | 1u;     /* NSC=0, EN=1 */

    /* Region 1: NSC veneers 0x0003F000..0x0003FFFF */
    SAU->RNR  = 1;
    SAU->RBAR = 0x0003F000u;
    SAU->RLAR = (0x0003FFE0u) | (1u << 1) | 1u;     /* NSC=1, EN=1 */

    /* Region 2: NS RAM 0x20010000..0x2003FFFF */
    SAU->RNR  = 2;
    SAU->RBAR = 0x20010000u;
    SAU->RLAR = (0x2003FFE0u) | (0u << 1) | 1u;

    /* Region 3: NS peripheral window */
    SAU->RNR  = 3;
    SAU->RBAR = 0x40000000u;
    SAU->RLAR = (0x5FFFFFE0u) | (0u << 1) | 1u;

    SAU->CTRL = SAU_CTRL_ENABLE_Msk;                /* enable SAU */
    __DSB(); __ISB();
}

An NSC veneer + secure entry function

/* secure.c, compiled with -mcmse */
#include <arm_cmse.h>

uint32_t __attribute__((cmse_nonsecure_entry))
secure_get_serial(uint8_t *out, size_t out_len)
{
    if (out_len < 16) return 0;
    if (!cmse_check_address_range(out, out_len,
                                  CMSE_NONSECURE | CMSE_MPU_READWRITE))
        return 0;

    /* Pull serial from secure-only OTP */
    extern const uint8_t __otp_serial[16];
    for (size_t i = 0; i < 16; ++i) out[i] = __otp_serial[i];
    return 16;
}

The compiler emits an SG at function entry and a BXNS LR at exit, with all caller-saved registers scrubbed in between. The linker places this function in the NSC region (you tell it via section attribute or a placement file).

Secure-to-Non-secure boot handover

/* End of secure init — launch the NS application */
typedef void (*ns_func_t)(void) __attribute__((cmse_nonsecure_call));

__attribute__((noreturn))
void launch_nonsecure(uint32_t ns_vector_table_addr)
{
    /* 1. Point the NS vector table at the NS app */
    SCB_NS->VTOR = ns_vector_table_addr;

    /* 2. Set NS MSP from vector table[0] */
    uint32_t ns_msp = ((uint32_t *)ns_vector_table_addr)[0];
    __TZ_set_MSP_NS(ns_msp);

    /* 3. Set NS MSPLIM from a linker-defined NS stack limit */
    extern uint32_t __ns_stack_limit;
    __TZ_set_MSPLIM_NS((uint32_t)&__ns_stack_limit);

    /* 4. Resolve NS reset vector, call as cmse_nonsecure_call */
    uint32_t ns_reset = ((uint32_t *)ns_vector_table_addr)[1] & ~1u;
    ns_func_t ns_entry = (ns_func_t)ns_reset;
    ns_entry();      /* compiler emits BLXNS, scrubs regs, transitions */

    /* Should never return */
    while (1) __WFI();
}

Common gotchas & errata-class behaviors

  1. Forgetting SG as the first NSC instruction. Manually-written NSC veneers must begin with SG. Anything else — even a NOP — trips a SecureFault on entry. The CMSE attribute handles this for you; if you write veneers in assembly, get the prologue right.
  2. VTOR_NS in Secure address space. The NS vector table is configured by writing SCB_NS->VTOR, but that register lives in the Secure SCS alias. Only Secure code can program where NS vectors live — NS cannot relocate its own vector table. Forgetting this leads to "I tried to set up an interrupt-driven app but the vector table change has no effect" — you actually need to bounce through a secure-callable veneer that re-programs VTOR_NS.
  3. Lazy stacking + TZ leak window. If FPCCR.TS = 0 and FPCCR.CLRONRET = 0, a Secure FP context can be lazy-stacked and then read by an NS handler. Always set both in secure init when FPU is shared.
  4. NSACR locks you out of FP from NS. Vendor reset values for NSACR.CP10/CP11 are sometimes cleared. NS code's first FP instruction then traps as UsageFault. Set both bits in secure init even if you "don't think" you need to. The fault is unhelpful — its CFSR bit doesn't say "FPU not allowed in NS"; you have to know.
  5. SAU regions overlap is undefined. If two SAU regions overlap, the result is implementation-defined and you'll get inconsistent behaviour across silicon revisions. Tools won't warn you. Audit your SAU map.
  6. Privileged-NS doesn't mean privileged-S. NS Privileged mode and Secure Privileged mode are independent — NS in privileged mode can not escalate into Secure unless it goes through an SG. Newcomers sometimes assume "privileged" is global; it's per-state.
  7. HardFault in NS goes to Secure (by default). If your NS application crashes early in development, the fault handler running is the Secure one. You'll see the trap from the wrong angle. Set AIRCR.BFHFNMINS = 1 during NS development to route NS HardFaults back to NS — but flip it back for production.
  8. SP_LIMIT compares unsigned. Setting MSPLIM = 0 effectively disables the check (SP >= 0 is always true). Some examples online do this on purpose to "disable" the limit; it's a footgun if you forget to set it later.
  9. Bit-banding is gone. Code ported from M3/M4 that uses the 0x22000000 alias region simply reads/writes the wrong addresses on M33. Search-and-replace before you compile, or you'll get faults.
  10. DBGEN tied to lifecycle on Nordic / NXP / ST. On a "closed" device, attaching probe-rs / pyOCD will succeed but you can't halt — the core just keeps running. The fix is restoring the open lifecycle (often requires erasing flash + provisioning); not all parts allow this.
  11. Secure-callable veneers must be in NSC memory. If your linker accidentally places them in plain Secure memory, NS can't see them at all (SAU classifies the address as Secure → SecureFault on call). The build error you'll see is "missing veneer", but tracing it from the error message takes a minute.
  12. Coprocessor instructions (CDE) silently NOP without the option. Most vendors don't include CDE. Code that uses CX1/CX2 intrinsics (rare) compiles fine but executes as NOPs at runtime. If you have a project using CDE, treat the dependency as part of your part-number selection.

edevkit (RP2350) specific notes

The edevkit kit uses the RP2350B — a 48-GPIO version of the same silicon family that powers the Pi Pico 2. From a pure-CPU standpoint, the M33 cores in RP2350 are a vanilla Arm M33 with the FPU, DSP, and TrustZone Security Extension all enabled. From a board-software standpoint, edevkit firmware (Zephyr 4.x via west sysbuild → MCUboot → app) treats those cores as Armv8-M Mainline without the security boundary in active use.

Why this works

The Zephyr board target rpi_pico2/rp2350a/m33 doesn't enable Zephyr's TrustZone-aware build flags (CONFIG_ARM_TRUSTZONE_M=y). The result: secure-only registers (SAU, banked SCB, banked MPU) exist on the silicon but the app runs entirely in Secure state and never sets SAU->CTRL.ENABLE, so SAU defaults apply (everything secure) and there's no NS execution. From the firmware's POV it's "just an M33 without TZ active". This is the recommended Pi/Zephyr default for projects that don't need application-level security partitioning.

When TrustZone becomes relevant for edevkit

Three plausible futures, each motivating TrustZone activation:

  1. Signed firmware + secure boot ROM. The RP2350 boot ROM already implements signed-image verification using the OTP key block. That's "secure boot" in the sense of "the platform won't run an unsigned image" — but it doesn't impose runtime isolation between bootloader and app. Today, MCUboot handles the equivalent at the Zephyr level. No TZ needed.
  2. Untrusted protocol stacks. If a future edevkit unit hosts a USB stack that handles enumeration from arbitrary hosts (e.g. running fuzzing scenarios), partitioning USB/USB-stack into NS while keeping the device's identity keys in S becomes attractive. That's a real TZ use case and would warrant the engineering effort.
  3. Per-customer signing. If edevkits ship with per-unit secrets (signing keys for code-signed firmware artefacts the unit produces), TZ is the right place to keep those. Today the ECC keys for MCUboot sit in flash — one MPU region away from any code that runs.

Until one of those scenarios lands, the design doc's stance is to keep TZ off and treat the M33 as "M4 with stack limits and FPv5". See edevkit1.html §15 — Decision Log for the full rationale and re-opening criteria.

Building for the M33 in this repo

The shell helpers covered in shell_helpers.html wrap the right invocations:

# Default board target on edevkit:
EDEV_BOARD_DEFAULT=rpi_pico2/rp2350a/m33

# Pristine sysbuild build → MCUboot + signed app
edev_build

# Build artifacts land at:
build-rpi_pico2_rp2350a_m33/
├── mcuboot/zephyr/zephyr.bin
└── <app-name>/zephyr/zephyr.signed.bin

The /m33 qualifier tells Zephyr to use the M33 core variant. The other valid RP2350 target is rpi_pico2/rp2350a/hazard3 (the RISC-V cores) — edevkit firmware does not currently target that variant.

Where to go next

You've now got the surface area to read primary sources without drowning. Recommended order:

  1. Armv8-M Architecture Reference Manual (DDI 0553) — the full ISA + memory model + security extension spec. Treat it as a reference, not a tutorial; jump to chapters as questions arise. Sections B5 (faults), B8 (PMSAv8), and B11 (banked registers) are the densest TZ material.
  2. Cortex-M33 Technical Reference Manual — implementation details: pipeline timing, AHB5 interface, error handling, debug. Read the "Configurable Options" chapter first to understand what your specific silicon includes.
  3. Building Secure Software for Armv8-M (100690) — Arm's official "how to actually use TrustZone-M" cookbook. Most other vendor docs reference this one.
  4. Migrating to Armv8-M (107565) — short, practical, focused on M3/M4 → M33 source-level changes.
  5. Joseph Yiu, The Definitive Guide to Arm Cortex-M23 and Cortex-M33 Processors (Newnes, 2021) — the long-form book. If you can read only one thing, read this. Chapters 5–7 cover TrustZone end-to-end with worked examples that go beyond what Arm's official PDFs include.
  6. TF-M (Trusted Firmware-M) source — trustedfirmware.org/projects/tf-m — the reference open-source secure-partition manager for Armv8-M. If you're going to build production TrustZone code, this is the codebase you'll port to your platform. Reading the source teaches more than any tutorial.
  7. Vendor app notes — Silabs AN1374, NXP AN12278 (LPC55), ST AN5347 (STM32L5), Nordic nRF5340 SDK Security chapter. Each picks specific real-world scenarios and shows the SAU + IDAU + peripheral-firewall code in full.
  8. Sister doc — arm_architecture.html — re-read for any general Cortex-M topic this doc deferred (programmer's model, basic exception model, debug architecture overview, software stack, hello-ARM walkthrough).

Also worth bookmarking: the Memfault Interrupt blog — practical writeups on Cortex-M faults, debugging, RTOS, and Armv8-M topics. Search there before searching anywhere else for "why is my M33 crashing in <weird specific way>".
