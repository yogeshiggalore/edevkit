SWD — Serial Wire Debug, the 2-wire protocol every Cortex-M speaks

Serial Wire Debug (SWD) is Arm's compact, 2-wire alternative to JTAG — one clock (SWCLK) and one bidirectional data line (SWDIO). It first appeared with Arm's CoreSight architecture in 2006 and is now the only debug interface most modern Cortex-M parts physically expose. Probe-rs, OpenOCD, J-Link, ST-Link, CMSIS-DAP, Black Magic Probe, and edevkit itself all speak it. This page is the document you read when you need to understand SWD — not just use it — well enough to bring up a custom probe, debug a stuck target, or read OpenOCD source without a glossary.

Audience. A firmware engineer who has used SWD via a probe but never looked at the wires. Or someone bringing up the edevkit CMSIS-DAP probe and wondering why WAIT responses keep happening, what the 0xE79E magic number means, or how to reach core 1 on an RP2040. By the end you should be able to follow an ADIv5 packet trace on a logic analyser, hand-write a register-read sequence, and reason about edge cases like multi-drop, dormant state, and sticky errors.

What's not in this doc. JTAG (see debug_jtag.html), the SWO trace pin and CoreSight ITM/TPIU (see debug_swo.html), CMSIS-DAP USB framing (see usb_cmsis_dap.html), or the RP2350 PIO programming model used by edevkit's bit-banged SWD engine (see platform_pio.html). General Cortex-M debug architecture is in arm_architecture.html and arm_cortex_m33.html. The v1 design context for edevkit is in edevkit1.html.

What SWD is — and why every Cortex-M ships it

SWD is an Arm-defined synchronous serial debug protocol that carries the same Debug Access Port transactions JTAG carries, over two wires instead of four (or five):

  • SWCLK — clock from probe to target. The probe is always the clock master; the target is always the slave. Idle state is low.
  • SWDIO — bidirectional data, sampled on the rising edge of SWCLK. Direction switches mid-transaction and is governed by an explicit turnaround period.

Compare to JTAG, which needs TCK, TMS, TDI, TDO and (optionally) nTRST. On a 48-pin LQFP that's 4–5 dedicated debug pins; on SWD it's 2. Cortex-M parts ship in packages as small as QFN-12 (Cortex-M0+ in some Microchip parts), so SWD is often the only debug interface physically exposed — the chip simply does not have enough pins for JTAG.

That said, SWD is not a separate protocol stack. The Debug Access Port (DAP) — the register interface a debugger pokes to read memory, halt the core, set breakpoints — is identical between JTAG and SWD. SWD is a different physical layer for the same logical transactions. Arm calls the DAP block "SWJ-DP" when the silicon supports both physical layers and lets you switch between them at runtime. Most modern Cortex-M parts (M3 onward) integrate SWJ-DP. M0/M0+ often integrate SW-DP (SWD-only) to save area.

key idea SWD = JTAG's logical transactions, on 2 wires instead of 4. A probe that "speaks SWD" is doing the same DAP register reads/writes as the JTAG-equivalent probe; the difference is purely how those reads/writes are framed on the wire. This is why probe-rs, OpenOCD, and CMSIS-DAP all share most of their code paths between the two transports — only the lowest layer changes.

Why SWD exists at all

Pin economy. The Cortex-M family was designed for cost-sensitive embedded MCUs where every pin you can save is a pin you can sell to the customer as GPIO. Arm's design brief for SWD (originally documented in DDI 0316 for Cortex-M3, then formalised across the family in the Arm Debug Interface Architecture Specification, IHI 0031) was: keep the JTAG register interface (so existing tooling and silicon IP could be reused) but replace the transport with a 2-wire serial bus so MCU vendors could put debug on a 6-pin connector (Vcc, GND, SWCLK, SWDIO, SWO, nRESET) instead of a 20-pin connector.

The original 20-pin Arm debug header is now legacy; the modern standard is the Cortex Debug Connector (Arm's "10-pin Cortex" header) or its even smaller Cortex Debug + ETM variant. edevkit1 supports both (see edevkit1.html §6).

SWD versions and where they appear

VersionFirst introducedSpecHighlights
SW-DP / SWJ-DP v1 ~2006 (Cortex-M3, ADIv5.0) IHI 0031A The original. Single target on the bus. Switch from JTAG via the 0xE79E sequence. ACK codes, parity, turnaround all defined here.
SWDv2 (multi-drop) ~2013 (ADIv5.2) IHI 0031C Adds TARGETSEL register and a connection-handshake sequence so multiple targets can share one SWCLK/SWDIO bus. Used by RP2040, RP2350, multi-die SoCs, and stacked debuggable chiplets.

ADIv6 ~2020 IHI 0074 Modernises addressing (DP/AP both move to a flat 64-bit space), adds new packet types, retains the SWD wire format. Used on Armv8.1-M (M52/M55/M85) and large CoreSight SoCs. Most M-class MCUs in 2026 are still ADIv5.2.

For day-to-day Cortex-M0/M3/M4/M33 work, you're talking ADIv5 over SWDv1 or SWDv2. ADIv6 only matters if you're targeting a Cortex-M55/M85 or a large Cortex-A SoC.

Physical & electrical — pins, voltages, speed, cabling

The two signals

SignalDirectionIdleDriveNotes
SWCLK Probe → Target Low Push-pull Probe is always the clock master. Target sees this as an input. The clock can be stopped indefinitely between transactions — SWD is not free-running. Many probes will gate the clock when no transaction is pending; targets must tolerate that.
SWDIO Bidirectional High (pull-up) Push-pull or open-collector Direction is controlled by the protocol — turnaround periods (1 SWCLK by default, configurable to 1/2/3/4 via the DP WCR register) hand the line off. A pull-up on the target side keeps the line at a defined level during turnaround.

Two more signals are part of every real-world SWD connector but aren't strictly part of the protocol:

  • nRESET — the target's reset pin, driven open-drain by the probe. Used to recover wedged targets and to align the very first SWD transaction with a clean reset. edevkit1 GPIO 16 (see edevkit1.html pinmap).
  • SWO — an asynchronous trace-output pin (UART-encoded ITM frames). Not part of the SWD register protocol; it just happens to be on the same connector. Covered fully in debug_swo.html.

Voltage levels — everything is referenced to Vtgt

SWD has no defined voltage. The signal level is whatever the target's I/O Vcc is. Modern Cortex-M targets run anywhere from 1.62 V to 5.5 V, and the probe must match. A probe that drives 3.3 V into a 1.8 V target will, at minimum, exceed the target's input voltage spec; at worst it back-feeds the target's power rail and damages the chip. Always sense Vtgt and shift levels accordingly.

edevkit1 senses Vtgt on ADC0 (see edevkit1.html §7) and gates SWCLK/SWDIO through an SN74LVC8T245 8-bit dual-supply translator (VccA = 3.3 V from the probe, VccB = Vtgt). The LVC family works down to about 1.65 V; below that, the firmware switches the active level shifters to a TXS0108E auto-direction translator which works down to 1.2 V (at the cost of slower edges and lower max frequency).

gotcha Vtgt sensed below ~1.65 V means LVC8T245 will not drive logic levels reliably at full speed. edevkit1's firmware will detect this and either downshift the SWD frequency, route through TXS0108E channels, or refuse to attach until the user confirms the low-V mode.

SWDIO pull-up — non-negotiable

The Arm spec requires a pull-up on SWDIO (typical 100 kΩ, sometimes 10 kΩ on noisy boards). During turnaround periods both ends are tristate; without a pull-up the line floats and any noise can corrupt the next sample. Most MCUs include an internal pull-up that's enabled out of reset, but you cannot rely on it — some chips disable internal pulls during deep sleep, and your probe board must work even before the target is responsive. Put an explicit pull-up on the probe side. edevkit1 puts a 22 kΩ pull-up on the SWDIO line at the probe-side translator output (see edevkit1.html §6).

rule of thumb Halve the SWCLK frequency every time the cable doubles. If you're seeing intermittent FAULT or unsynced packets, the first thing to try is to drop SWCLK by 2× and see if the problem disappears.

Wire-level transaction structure

Every SWD transaction is exactly the same five-phase shape:

  <----- host drives ------>|<trn>|<-target->|<trn?>|<----- 33 bits ----->|

  1. Request   (8  bits)    | TRN | ACK (3) | TRN | DATA (32) + parity (1)
                                              ^
                                              only on read; not on write

  Read:  HOST → req → TRN → TGT → ack → TGT → data+parity → TRN → HOST idle
  Write: HOST → req → TRN → TGT → ack → TRN → HOST → data+parity → HOST idle

The five phases:

  1. Request — 8 bits, host drives. Selects DP-vs-AP, R-vs-W, the register address, and carries an odd-parity bit. Always ends with the host having driven exactly 8 SWCLK cycles. (Detail: § The 8-bit request packet.)
  2. Turnaround (TRN) — 1 SWCLK cycle (default). Both ends tristate; the SWDIO pull-up holds the line high. This is the moment direction reverses.
  3. ACK — 3 bits, target drives. Encodes OK, WAIT, or FAULT. (Detail: § ACK codes.)
  4. Data + parity — 32 bits LSB-first, plus 1 parity bit. On a read, the target drives this; on a write the host drives it (after another turnaround following the ACK).
  5. Trailing turnaround / idle — on a read, one more TRN cycle hands the line back to the host. On a write, no extra TRN is needed because the host already had control after the post-ACK TRN. The host then drops idle clocks (SWCLK low, SWDIO low) until the next request, or sends the next request immediately back-to-back.

"LSB-first" is the rule for everything except the request packet's parity/stop/park, which are positional. All multi-bit fields go out least-significant-bit first.

tip On a logic analyser, an SWD bus during steady-state debug looks like clusters of ~46-bit bursts (8 req + 1 trn + 3 ack + 1 trn + 32 data + 1 par + 1 trn) separated by SWCLK quiet periods. If you see sustained activity longer than that, you're probably looking at a JTAG-to-SWD switch, a line reset, or a multi-drop handshake.

The 8-bit request packet, field by field

The request is the only host-driven phase that has structured fields beyond raw data. Each request is exactly 8 SWCLK cycles long; the bits go LSB-first on SWDIO:

bit:      0     1      2     3      4     5     6     7
field:  Start  APnDP  RnW   A[2]   A[3]   Par  Stop  Park
value:   1      X      X     X      X      X    0     1

   Start  = 1   (always)
   APnDP  = 1 selects an AP register
            0 selects a DP register
   RnW    = 1 read,  0 write
   A[2:3] = the 2-bit field that picks one of 4 register slots
            within the bank selected by SELECT
   Par    = odd parity over {APnDP, RnW, A[2], A[3]}
   Stop   = 0   (always)
   Park   = 1   (always)

Two oddities worth flagging:

  • Only 2 address bits. SWD lets you address only 4 registers per bank. Bigger register sets are reached by writing the SELECT DP register first to choose which AP and which register bank within that AP. We come back to this in § DP and AP.
  • Address bits are A[2] and A[3], not A[0] and A[1]. Why: the underlying register space is 32-bit-word addressed, so bits [1:0] are always zero, and the spec just numbers the on-the-wire bits using the byte-address convention. Annoying but harmless once you internalise it.

Example — "read DP register 0xC (RDBUFF)"

Read DP register at byte offset 0xC (the RDBUFF register; we'll come back to what it does):

APnDP = 0   (DP)
RnW   = 1   (read)
A[2]  = 1   ; 0xC = bits [3:2] = 11
A[3]  = 1
Parity over {0, 1, 1, 1} = 1 (odd parity = sum of 1s mod 2)
Start = 1, Stop = 0, Park = 1

Bits LSB-first: 1 0 1 1 1 1 0 1   (start, APnDP, RnW, A2, A3, par, stop, park)

As a byte (MSB on left, conventional notation): 0b10111101 = 0xBD

So a "read RDBUFF" request is the literal byte 0xBD shifted out LSB-first. Many SWD implementations precompute the eight legal request bytes (4 reg × 2 R/W × AP/DP) into a small lookup table:

// DP requests              // AP requests
DP_RD_IDCODE  = 0xA5         AP_RD_CSW   = 0x87
DP_WR_ABORT   = 0x81         AP_WR_CSW   = 0xA3
DP_RD_CTRLSTAT= 0x8D         AP_RD_TAR   = 0x8F
DP_WR_CTRLSTAT= 0xA9         AP_WR_TAR   = 0xAB
DP_WR_SELECT  = 0xB1         AP_RD_DRW   = 0x9F
DP_RD_RDBUFF  = 0xBD         AP_WR_DRW   = 0xBB

(Verify the exact values for your project — off-by-one parity bugs are a classic source of "my probe doesn't work" issues. The CMSIS-DAP firmware source is the canonical reference for these constants.)

Parity: odd over four bits

Parity in the request packet is odd parity over {APnDP, RnW, A[2], A[3]}:

par = APnDP ^ RnW ^ A[2] ^ A[3]

Result: the five-bit field {APnDP, RnW, A[2], A[3], Par} always contains an odd number of ones. The target will compute the same XOR over the bits it received and compare; if they differ, the target sets a sticky error and the next ACK will be FAULT (or, on some implementations, the target ignores the request entirely, leading to a host-side timeout).

ACK codes — OK / WAIT / FAULT

After the host's request and one turnaround cycle, the target drives 3 ACK bits LSB-first:

CodeBits (LSB-first on the wire)Binary valueMeaning
OK1 0 00b001Request accepted. Continue with the data phase. Important caveat below.
WAIT0 1 00b010Target is busy — usually because the previous AP transaction hasn't completed (e.g. an in-flight bus access on a slow memory). The host should retry the same request after a short delay, or issue a WRITE ABORT if the wait persists.
FAULT0 0 10b100A sticky error bit is set in CTRL/STAT — either parity error, AP error, or overrun. The host must clear the relevant sticky bit via a WRITE ABORT before any more AP transactions will succeed.
NoResponse / Protocol Error0 0 0 or 1 1 1 (etc.)0b000 / 0b111 / otherNot a defined ACK; usually means the target is unreachable, the line was glitched, or you got out of sync with a half-eaten request. Recover with a line reset.
important An OK on a read does not mean the requested value has reached the host yet. AP reads are posted — the data field that follows an AP-read request is the previous read's result. To get the value you actually asked for, you have to either issue another AP read (and discard or use the now-arrived earlier value) or read the DP RDBUFF register, which holds "the most recent AP read result". This is the single biggest pitfall when hand-writing SWD.

The WAIT loop pattern

repeat:
    drive request
    read ack
    if ack == OK:    proceed with data phase
    if ack == WAIT:  small delay; loop (with a max-retry count)
    if ack == FAULT: read CTRL/STAT, write ABORT, propagate error
    if ack == 0b000: line reset, re-attach

OpenOCD, probe-rs, and CMSIS-DAP all follow this pattern. The retry count is typically ~64 retries with exponentially increasing delays, totalling on the order of milliseconds before giving up.

Data phase, parity, turnaround

If ACK == OK, the request and target are now committed to a 32-bit data exchange:

  • Read: target continues to drive SWDIO immediately after the ACK — no extra turnaround — and shifts out 32 data bits LSB-first plus 1 parity bit. Parity is the XOR of all 32 data bits (even parity over data: a 1 if the data has an odd number of 1s). After the parity bit there's one more TRN cycle handing the line back to the host.
  • Write: after the ACK, the target tristates and a second turnaround happens (1 SWCLK cycle by default). Then the host drives 32 data bits LSB-first plus 1 parity bit. No final turnaround needed — the host already owns the line.

The default turnaround is 1 cycle. The host can change it for the whole link by writing the WCR register in the DP (1, 2, 3, or 4 cycles — useful on long cables where the wire-direction reversal needs more settling time). The target reports its supported value in the WCR read; the actual in-use value is the smaller of host and target capability. Most probes leave it at the default 1.

ASCII timing — one full read transaction

SWCLK : __|¯|_|¯|_|¯|_|¯|_|¯|_|¯|_|¯|_|¯|_   (8 cycles, request)
SWDIO : ...d0  d1  d2  d3  d4  d5  d6  d7....    host drives, LSB-first

         |¯|_                                       turnaround = 1 cycle
SWDIO :   z..    line floats, pull-up holds high

         |¯|_|¯|_|¯|_                              ack = 3 cycles
SWDIO :   a0 a1 a2     target drives

         |¯|_|¯|_|¯|_ ... |¯|_|¯|_       data + parity = 33 cycles
SWDIO :   d0 d1 d2 ... d31 par      target drives, LSB-first

         |¯|_                                       trailing turnaround = 1 cycle
SWDIO :   z..    line floats, host takes over for next request

Total: 8 + 1 + 3 + 33 + 1 = 46 SWCLK cycles per read, 8 + 1 + 3 + 1 + 33 = 46 cycles per write (with the post-ACK TRN replacing the read's trailing TRN). At 25 MHz that's ~1.84 μs per transaction — ~540 k transactions/second peak, before any host-side overhead. A "read 1 KB of memory" is roughly 256 word transactions ≈ 0.5 ms.

DP and AP — the two register banks

Logically, the DAP is a tree:

          SWD wire (SWCLK / SWDIO)
                    |
         +----------+----------+
         |  Debug Port (DP)    |    one per chip (or one per core in multi-drop)
         +----------+----------+
                    |
         +----------+----------+
         |    AP bus (APBUS)   |    addressed by SELECT.APSEL (8 bits = up to 256 APs)
         +-----+----+-----+----+
               |    |     |
             AP0   AP1   AP2 ...
              |     |     |
         MEM-AP  JTAG-AP  custom-AP
              |
         MCU memory bus — this is where you read 0x20000000

Two banks of registers visible to the SWD layer:

  • Debug Port (DP) — a small set of registers controlling the wire itself: identity, power-up, sticky errors, AP bus select. Always reachable directly via APnDP=0 in the request packet.
  • Access Port (AP) — one of up to 256 sub-blocks behind the DP, addressed via SELECT.APSEL. The most common AP is the MEM-AP (a memory-mapped bridge to AHB or APB on the chip), but you may also see JTAG-AP (carries JTAG-style scan chains over the DAP — useful when the chip has a JTAG-only legacy block), or vendor-specific APs.

Banked register access via SELECT

SWD's request packet only carries 2 address bits, so each bank exposes 4 registers at a time. The SELECT DP register chooses:

SELECT @ DP 0x8 (write only)
+-------+--------+----------+----------+
| 31:24 | 23:8   | 7:4      | 3:0      |
| APSEL | (rsvd) | APBANKSEL| DPBANKSEL|
+-------+--------+----------+----------+

  APSEL     = which AP we're talking to        (0..255)
  APBANKSEL = which bank of 4 registers within that AP   (0..15)
  DPBANKSEL = which bank within the DP itself  (0..15; mostly 0)

So to read MEM-AP register at offset 0x10 (TAR), you'd write SELECT with APSEL=0, APBANKSEL=1, then issue an AP-read with A[3:2]=00 — the first slot of bank 1. Most debuggers maintain a software shadow of SELECT and only re-write it when the bank actually changes, to save round-trips.

Posted reads — why RDBUFF exists

AP reads are pipelined. When you issue an AP-read request, the target starts fetching the value (which may take a bus cycle on the chip's internal AHB), but the data field that comes back on the SWD wire after the ACK is the result of the previous AP read. So:

1. host: AP-read CSW              -> data phase carries OLD value (garbage on first read)
2. host: AP-read CSW              -> data phase carries CSW value from step 1
3. host: DP-read RDBUFF           -> data phase carries CSW value from step 2
                                     (RDBUFF "drains" the pipeline; no new AP read)

The standard idiom is: issue the AP read you want, then read RDBUFF to retrieve the result. The total cost of one logical AP read is therefore two SWD transactions (one AP-read, one RDBUFF read).

For burst reads, debuggers chain AP-reads back-to-back and only read RDBUFF once at the end — each AP-read returns the previous one's value, so a sequence of N AP-reads followed by 1 RDBUFF gives you N usable values with N+1 transactions. This is the basis of how a memcpy from target memory becomes ~3.5 SWCLK cycles per byte at peak (much faster than the naive 46-cycles-per-32-bits would suggest).

IDCODE

The first thing any SWD attach does is read IDCODE. The format is:

    +-------------+---------------+-------------+--+

    +-------------+---------------+-------------+--+

  bit 0 = always 1 (legacy from JTAG IDCODE convention)
  DESIGNER = JEDEC continuation code + JEP106 ID (Arm = 0x23B, ST = 0x020, etc.)

CTRL/STAT — the workhorse

CTRL/STAT @ DP 0x4
+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+
| 31  | 30  | 29  | 28  | 27:12 | 11 | 10  |  6  |  5  |  4  |  1  | 0
| CSY | CSY | CDB | CDB |       | WD | RD  | STK | STK | STK | TRN | OR
| SPW | SPW | GPW | GPW |       | ER | OK  | YOR | YEC | YERR| MOD | UNDETECT
| UPACK| UPREQ| UPACK| UPREQ|   | RR |     | UN  | MP  |     |     |
+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+

The bits you actually care about:

  • CDBGPWRUPREQ / CDBGPWRUPACK — "please power up the debug domain". You write 1 to REQ and wait for 1 in ACK. Without this, the debug logic stays off and AP reads will return errors.
  • CSYSPWRUPREQ / CSYSPWRUPACK — "please power up the system domain". Same handshake. The MCU will not respond on the AP bus until you complete both power-up requests. Every SWD attach starts with these two writes.
  • STICKYERR — set when an AP transaction returned an error. While this bit is set, every subsequent AP request answers FAULT. Clear by writing 1 to it via ABORT.
  • STICKYORUN — the host pushed a request faster than the target could process it (typical when the previous AP transaction is still in flight and you didn't honour the WAIT response). Clear via ABORT.
  • STICKYCMP — sticky compare; used by some advanced trace features. Most debuggers ignore it.
  • WDATAERR — a parity error on a write data phase. Indicates SI problems on the cable. Clear via ABORT.

ABORT

Write-only at DP 0x0. The bit fields:

BitNameEffect of writing 1
0DAPABORTCancel the in-flight AP transaction. Use only after extended WAIT.
1STKCMPCLRClear STICKYCMP.
2STKERRCLRClear STICKYERR. The most-used abort bit.
3WDERRCLRClear WDATAERR.
4ORUNERRCLRClear STICKYORUN.

You'll typically write 0x1E to ABORT after a FAULT to clear all sticky bits at once.

RDBUFF

Read-only at DP 0xC. Returns the result of the previous AP read without initiating a new AP transaction. You read this once at the end of any AP-read sequence to drain the pipeline. Reading RDBUFF does not consume an AP-bus cycle, so it cannot cause a new WAIT.

MEM-AP registers — CSW, TAR, DRW, BD0..BD3

The MEM-AP is the AP variant that exposes the chip's internal memory bus (AHB on Cortex-M3/M4/M33, AHB5 on M33+TZ, APB on some peripherals). To "read 0x20000000" really means "configure CSW + TAR, then read DRW".

0x000CSW (Control/Status Word)Access size (8/16/32-bit), auto-increment mode, privilege bits, in-flight bit. Configured once at attach and rarely changed.
0x040TAR (Transfer Address)The memory address. Auto-increments after each DRW access if CSW.AddrInc=01 (saves you from re-writing TAR for sequential reads).
0x0C0DRW (Data Read/Write)Reading DRW reads memory at TAR; writing DRW writes memory at TAR. The "actual" data port.
0x10..0x1C1BD0..BD3"Banked Data" — a 4-word window into memory at TAR & ~0xF. Lets you read/write 4 sequential words without re-writing TAR. Useful for register dumps.
0xF8FBASEPointer to the ROM table for this AP — a table-of-contents of debug components (DWT, FPB, ITM, etc.). The probe walks this at attach to discover what debug features are present.
0xFCFIDR (AP IDentification)Identifies what kind of AP this is. IDR == 0 means "no AP at this slot"; non-zero = MEM-AP / JTAG-AP / vendor-AP, decoded by class field.

  AddrInc: 00=off, 01=single (TAR auto-inc), 10=packed (size-aware)

The two fields you'll touch:

  • Size — almost always 010 (word) for Cortex-M memory access.
  • AddrInc — 01 lets the debugger chain reads through sequential addresses without re-writing TAR. Burst memcpy uses this.

Worked example — reading 0x20000000

Concrete sequence to read the first word of SRAM at 0x20000000 on a Cortex-M target. Assume SWD is already up (line reset done, JTAG-to-SWD switch done). All transactions are SWD; "DP/AP" refers to APnDP; "R/W" is RnW.

tx#  type      reg      value           ack    notes
---  --------  -------  --------------  -----  -----------------------------
 1   DP read   IDCODE   ----            OK     verify chip is alive
 2   DP write  ABORT    0x0000001E      OK     clear all sticky bits (just in case)
 3   DP write  CTRL/    0x50000000      OK     CDBGPWRUPREQ | CSYSPWRUPREQ
                STAT
 4   DP read   CTRL/    ----            OK     wait for ...PWRUPACK = 1 in bits 31, 29
                STAT                                  (loop until both ACKs are set)
 5   DP write  SELECT   0x00000000      OK     APSEL=0 (MEM-AP), APBANKSEL=0, DPBANKSEL=0
 6   AP write  CSW      0x23000052      OK     word size, AddrInc=single, debug-priv
 7   AP write  TAR      0x20000000      OK     target address
 8   AP read   DRW      ----            OK     posted: starts the read of 0x20000000
 9   DP read   RDBUFF   ----            OK     ... and now we get the value
                                               -> data field carries the word
                                                  contents at 0x20000000

bonus  AP read  DRW    ----             OK     because TAR auto-incremented to
                                                  0x20000004, this posts a read of it
       DP read  RDBUFF ----             OK     -> gets the value at 0x20000004
       ... and so on for sequential reads

To write 0xDEADBEEF to 0x20000000 instead, replace tx 8–9 with:

 8   AP write  DRW      0xDEADBEEF      OK     write goes through; no RDBUFF needed
 9   DP read   CTRL/    ----            OK     check STICKYERR cleared; if set, the
                STAT                                  write hit a fault (e.g. unmapped addr)

This is the canonical pattern. Every probe in existence eventually boils down to sequences of CSW/TAR/DRW writes and DRW/RDBUFF reads.

JTAG-to-SWD switch — 0xE79E and dormant state

Multi-protocol parts (i.e., parts with SWJ-DP) power up in some default state — usually JTAG, occasionally SWD, occasionally dormant. To get into SWD reliably regardless of the power-up default, the host transmits a known protocol-switch sequence on SWCLK/SWDIO. The probe must do this at every attach.

Classic JTAG-to-SWD switch (ADIv5)

The legacy sequence works on every Cortex-M3/M4 part with SWJ-DP:

1. Send >= 50 SWCLK cycles with SWDIO = 1                  ; line reset (puts both
                                                              JTAG and SWD into a known
                                                              "ready to listen" state)

2. Send the magic 16-bit JTAG-to-SWD select code:
       0x79E7 LSB-first (which is the byte sequence 0x9E, 0xE7
       on the wire, MSB-first that's 0xE79E)

3. Send another >= 50 SWCLK cycles with SWDIO = 1          ; line reset to
                                                              re-sync SWD

4. Send 1+ idle cycles with SWDIO = 0                      ; SWD now ready

5. Read DP IDCODE                                          ; verify reception

The "0xE79E vs 0x79E7" confusion is real: the bits go on the wire LSB-first. Many people quote the byte order they see on a logic analyser (which is MSB on the left, so they read 0xE79E); others quote the value as it appears in shift-register notation (0x79E7). Both refer to the same physical bit sequence: 0,1,1,1,1,0,0,1,1,1,1,0,0,1,1,1 on the wire. Most CMSIS-DAP firmware uses 0xE79E in the source.

Dormant state and selection alert (Armv8-M)

From Armv8-M onward, Arm added a "dormant" power-up state where neither JTAG nor SWD responds to anything until the probe transmits a 128-bit selection alert followed by an 8-bit activation code. This was needed because Armv8-M parts often share the debug pins with other functions (e.g. boot ROM modes, vendor-proprietary serial protocols), and Arm wanted a sequence so distinctive that random toggling of the pins won't accidentally enter SWD.

1. >= 8 SWCLK cycles with SWDIO = 1                       ; abort any in-flight selection
                                                              alert from a previous attempt

2. 128-bit Selection Alert (constants from ADIv5.2 / ADIv6):
       bits   0..63 : 0x86852D956209F392 (LSB-first on wire)
       bits  64..127: 0x19BC0EA2E3DDAFE9 (LSB-first on wire)

3. 4 SWCLK cycles with SWDIO = 0                          ; gap

4. 8-bit activation code:
       For SWD: 0x1A   (LSB-first on wire)
       For JTAG-DP: 0x0A
       For JTAG-Serial: 0x?? (vendor)

5. Standard SWD line reset (>= 50 SWCLK with SWDIO = 1)
6. Read DP IDCODE

If the chip only supports dormant state (e.g. RP2040 and RP2350 on power-up), the classic 0xE79E switch will not work and you must issue the dormant-to-SWD activation. Most modern debuggers issue both sequences in succession at attach — the first works on legacy parts, the second on Armv8-M parts, and either is harmless on the wrong chip.

edevkit firmware behaviour edevkit1's CMSIS-DAP implementation issues, in order: line reset → 0xE79E classic switch → line reset → dormant deactivate (0xB3) → selection alert → SWD activation 0x1A → line reset → IDCODE read. Each step is harmless on a chip that doesn't need it. This is the same pattern used by OpenOCD's swd driver and CMSIS-DAP firmware v2.

Line reset, idle, and sticky-error recovery

SWD has two distinct "reset" concepts and they are easy to confuse:

ConceptWhat it doesHow to do itWhen to use
Line resetResets the SW-DP state machine. Does not clear sticky errors. Does not reset the chip.Drive SWDIO high for ≥ 50 SWCLK cycles, then issue an idle SWDIO-low cycle.After protocol-switch, after a malformed transaction, after a long idle when you're not sure what state the SW-DP is in.
WRITE ABORT (DP @ 0x0)Clears sticky-error bits in CTRL/STAT and optionally cancels an in-flight AP transaction.DP-write to register 0 with bits set per the ABORT table.After receiving a FAULT ACK; periodically as a safety measure during long sessions.
nRESET pulseResets the entire chip including the CPU. Does not reset SW-DP unless the chip's reset domain includes the debug logic (rare on modern parts).Drive nRESET low for ≥ 100 μs.Recovering a wedged target; aligning attach with a fresh boot.
Power cycleEverything.Toggle Vtgt.Last resort. Loses any debug state including breakpoints.

Standard recovery algorithm

after FAULT:
    1. read CTRL/STAT to inspect which sticky bit fired
    2. write ABORT with appropriate clear-bit set (usually 0x1E to clear all)
    3. retry the original transaction; if FAULT persists, escalate

after WAIT (repeated > N times, default ~64):
    1. write ABORT with DAPABORT=1 to cancel the in-flight AP transaction
    2. read CTRL/STAT to clear sticky bits
    3. retry

after no-ACK (0b000 or 0b111 received):
    1. line reset (50+ SWDIO-high cycles)
    2. read IDCODE
    3. if IDCODE looks right, re-do power-up sequence and continue
       otherwise, drop into JTAG-to-SWD switch retry
       (probe may have desynced from target's bit position)

after total silence (target offline):
    1. nRESET pulse
    2. wait for target reset to complete
    3. line reset
    4. JTAG-to-SWD switch
    5. read IDCODE
       if still nothing, the target is broken or unpowered

Multi-drop SWD (SWDv2) and TARGETSEL

SWDv2 lets multiple targets share a single SWCLK/SWDIO bus. The debugger picks which target it's currently talking to by writing a special TARGETSEL DP register at attach time; thereafter, only the selected target answers, and the others sit silent.

The use cases:

  • Multi-core chips with one DP per core — the canonical example is the RP2040, which has three DPs on one SWD bus: core 0, core 1, and a "rescue DP" for force-erasing flash. Each DP has a unique 32-bit instance ID; the debugger writes TARGETSEL with the desired ID before reading IDCODE for the first time.
  • Multiple chips sharing one connector — e.g. a board with several MCUs all sharing a single 4-wire SWD trace to save layout area. Each chip's DP gets a unique ID.
  • Stacked debuggable chiplets — modern packaging where several die share a debug bus.

TARGETSEL handshake

The handshake is delicate because multiple targets may try to drive ACK at the same time. The protocol works around this by defining the TARGETSEL write as a special, no-ACK transaction:

1. Standard SWD line reset                 (or dormant->SWD activation)
2. DP write to TARGETSEL register          (write-only, DP @ 0xC, DPBANK=2)
   - request bits as normal: APnDP=0, RnW=0, A[3:2]=11, parity, etc.
   - turnaround
   - target ACK is **ignored / not driven** by spec - all targets stay silent
   - turnaround
   - host drives the 32-bit target ID + parity
3. After this, only the target whose ID matches will respond to subsequent
   transactions. Other targets put themselves into a "lurking" state.
4. Read DP IDCODE to confirm.

Worked example: RP2040 TARGETSEL

RP2040 instance IDs (from the RP2040 datasheet §2.3.2):

Core 0    : 0x01002927
Core 1    : 0x11002927
Rescue DP : 0xF1002927

To attach to core 0 of an RP2040:

1. Line reset (50+ SWDIO-high cycles)
2. Dormant->SWD selection alert + activation 0x1A
3. Line reset
4. DP write TARGETSEL = 0x01002927  (no ACK; target IDs check against this)
5. DP read IDCODE -> should return 0x0BC12477 (Cortex-M0+ DP)
6. Standard CTRL/STAT power-up sequence
7. AP transactions to core 0's MEM-AP work normally

To switch to core 1 mid-session, line-reset and TARGETSEL=0x11002927; the previously-attached core 0 stops responding.

Since RP2350 has a single DP that fronts both M33 cores, multi-drop is optional on RP2350; you can attach without TARGETSEL. The DP supports it for setups with multiple RP2350s on one SWD bus, but for a single-chip edevkit setup, classic SWDv1 attach works.

edevkit1's SWD implementation — PIO0 SM0

The implementation lives in firmware/app/pio/swd.pio and the C glue in firmware/app/src/cmsis_dap/swd.c. The half-duplex challenge

SWDIO is bidirectional and switches direction mid-transaction. PIO state machines can only "side-set" pin directions deterministically as part of an instruction's opcode — so the PIO program has to flip OEOVER exactly when the protocol says to. The swd.pio program uses side-set to control SWDIO's OE:

; sketch — actual program is more involved
.program swd
.side_set 1 opt    ; side-set OE for SWDIO

; -- request phase: drive SWDIO --
shift_request:
    pull block               side 1     ; OE=1, host drives
    set y, 7                            ; 8 bits
req_loop:
    out pins, 1     [1]      side 1     ; bit out, hold OE=1
    jmp y--, req_loop

; -- turnaround: tristate --
trn1:
    nop             [1]      side 0     ; OE=0, line floats

; -- ack phase: read 3 bits --
    set y, 2                            ; 3 bits
ack_loop:
    in pins, 1      [1]      side 0     ; sample SWDIO, OE stays 0
    jmp y--, ack_loop
    push block

; ... and so on for data phase, with another OE flip on writes

The full program assembles to ~24 PIO instructions (the entire SWD wire protocol fits in one of PIO's 32-instruction-per-block memory). At the configured PIO clock divider (typically sys_clk / 5 at 150 MHz sysclk → 30 MHz PIO clock), each SWCLK cycle is 1 PIO instruction, so peak SWCLK = 30 MHz. In practice the firmware caps SWCLK at 25 MHz to leave margin for cable SI.

DMA-driven bursts

For burst memory reads, the firmware queues up sequences of pre-encoded request bytes in DMA memory, kicks PIO, and reads back the result words via a second DMA channel. End-to-end this lets edevkit1 sustain ~95% of the theoretical peak (~510 k transactions/sec at 25 MHz) without involving the CPU per-transaction. CMSIS-DAP USB framing then carries those results back to the host at the rate the bulk endpoint can drain them — typically the bottleneck on USB FS at ~10 Mbit/s usable. (See usb_cmsis_dap.html for the USB-side details.)

Vtgt auto-sense and level shifting

edevkit1's level-shifter logic flow on a fresh attach:

  1. Read ADC0 (Vtgt sense) once. If < 0.5 V, declare "no target" and bail.
  2. If 1.65 V ≤ Vtgt ≤ 5.5 V, configure the LVC8T245 translators with VccB matched and OE asserted.
  3. If 1.2 V ≤ Vtgt < 1.65 V, switch the muxed channels through the TXS0108E auto-direction translators instead, and lower SWCLK ceiling to 10 MHz.
  4. If Vtgt > 5.5 V, halt and warn — this should never happen on a USB-powered probe.
  5. Issue line reset, JTAG-to-SWD switch, and dormant-deactivate sequences.
  6. Read IDCODE; if non-zero and non-0xFFFFFFFF, attach is good.

Common pitfalls when bit-banging SWD

  1. Wrong protocol on power-up. Multi-protocol parts power up in JTAG (or dormant) and silently ignore SWD requests. Always issue the JTAG-to-SWD switch and the dormant-deactivate sequences before the first IDCODE read. If you see IDCODE = 0 or 0xFFFFFFFF, this is your problem 80% of the time.
  2. Vtgt below 1.65 V breaking LVC. Many Cortex-M parts run at 1.8 V; some at 1.2–1.5 V. LVC8T245 needs ≥ 1.65 V on VccB to drive valid logic levels. Below that, SWDIO won't reach the target's VIH threshold and you'll see intermittent ACKs — the most maddening symptom because it depends on noise. Detect Vtgt and switch to TXS0108E or refuse to attach.
  3. Missing pull-up on SWDIO. During turnaround, both ends are tristate. Without a pull-up the line drifts and any ambient noise can corrupt the next sample. Symptom: random parity errors at high SWCLK rates. Fix: 22–100 kΩ pull-up to Vtgt on the probe side.
  4. SWCLK clock-stretching when target is asleep. If the target enters deep sleep (e.g. WFI in low-power mode), the debug-domain clock may stop, and SWD transactions will return WAIT indefinitely. Standard pattern: write CDBGPWRUPREQ in CTRL/STAT to keep the debug clock alive, or wake the target with a dummy AP-read to a peripheral that triggers the wakeup logic. On STM32 parts this is a real headache — some sleep modes turn off the SWD interface entirely and require nRESET to recover.
  5. Forgetting RDBUFF on AP-reads. The ACK after an AP-read says "transaction was accepted" — but the data field carries the previous AP-read's value. If you forget to chase with an RDBUFF read (or another AP-read), you'll get values that look right except they're one transaction stale.
  6. Parity bit on writes vs reads. On reads, the target computes parity over the data it sends and the host checks it. On writes, the host computes parity over the data it sends and the target checks it — setting WDATAERR in CTRL/STAT on a mismatch. Forgetting to send the write-parity bit (an off-by-one in the host's bit count) is one of the most common bugs in hand-rolled SWD code. Symptom: writes silently fail; reads work.
  7. Confusing turnaround count. The default is 1 cycle but some probes negotiate 2 or 3 via WCR. If your code hard-codes 1, it'll desync against any probe that's negotiated otherwise. Keep it at the default unless you have a measured reason.
  8. Sticky errors persisting across "fixes". Once STICKYERR is set, every subsequent AP read returns FAULT — including the AP read you'd think would clear the underlying issue. You must first WRITE ABORT with the appropriate clear bit, then retry. Forgetting this leads to "I fixed the bug but probe still says FAULT" cycles that waste hours.
  9. nRESET reset domain confusion. Pulsing nRESET on most modern parts does not reset the debug logic — SW-DP, AP, and the chip's debug peripherals stay alive across an nRESET event. This is intentional (it's how you single-step out of reset). But it means clearing your in-software notion of debug state requires either a line reset or a power cycle, not just nRESET.
  10. Cable-length-induced ringing on the rising edge of SWCLK. Cables longer than ~6" without proper series termination ring on transitions. The target may sample SWDIO during the ringing window and see the wrong value. Symptoms are speed-dependent: works at 1 MHz, fails at 10 MHz. Fix: shorten cable, add 22–33 Ω series termination at the probe-side translator output, or lower SWCLK.
  11. Multi-drop without TARGETSEL. On RP2040, attaching without writing TARGETSEL gets you "all DPs answer simultaneously" — usually a corrupted IDCODE that looks like a legitimate value but isn't (typically OR-ed bits from both cores). Fix: always issue TARGETSEL on RP2040; it's harmless on parts that don't need it (the spec defines TARGETSEL writes as no-ACK, so a non-multi-drop chip will simply not respond, and the line-reset that follows resyncs you).
  12. Mixing line reset and protocol switch at high speeds. The 50+ SWDIO-high cycles required for a line reset must be sent at a SWCLK rate the target can sample reliably, even at the slowest expected target clock. Sending them at 25 MHz on a chip that's currently running its core at 1 MHz can fail. Drop SWCLK to ≤ 1 MHz for the reset sequence, then ramp back up. CMSIS-DAP firmware handles this automatically.

Where to learn more

Reading order if you want to go deep:

  1. Arm Debug Interface Architecture Specification (IHI 0031, ADIv5.2) — the canonical reference. Chapters 4 (SWD wire protocol), 6 (DP), 8 (MEM-AP) are the core. Treat it as a reference, not a tutorial. The latest revision (D, 2022) covers SWDv2 and dormant state.
  2. Arm ADIv6 Architecture Specification (IHI 0074) — the next-generation DAP architecture, used on Cortex-M55/M85 and large CoreSight SoCs. Most M-class MCUs in 2026 are still on ADIv5.2; come back here when you hit a part that says "ADIv6".
  3. CMSIS-DAP firmware source — Arm's reference implementation of an SWD probe, written in C, running on Cortex-M itself. The DAP_SWD.c and DAP_vendor.c files are the cleanest source for "how do I actually emit these bits". Read this alongside the spec and SWD will click.
  4. OpenOCD SWD drivers — particularly cmsis_dap.c and the swd_*.c files. Open-source, decades of bug-fixes, exhaustive comments. The reference for "how do real probes handle every edge case".
  5. probe-rs source — the modern Rust implementation. The probe-rs/src/probe tree is well-organised and the SWD layer (arm/communication_interface.rs, arm/sequences.rs) is more readable than OpenOCD for someone coming in fresh.
  6. Black Magic Probe firmware — a tiny, focused SWD/JTAG probe firmware. Fits the entire DAP layer in < 32 KB. Excellent for understanding what's essential vs what CMSIS-DAP layers on top.
  7. Joseph Yiu, The Definitive Guide to Arm Cortex-M0/M0+ Processors (2nd ed., Newnes 2015) — chapter 14 ("Debug System") is the best long-form treatment of SWD aimed at engineers, not spec-writers. The M3/M4 and M23/M33 books (same author) have parallel chapters.
  8. "Messing with SWD" by KernelPicnic — an excellent informal walk-through of bit-banging SWD on an STM32. Reading it after the Arm spec reinforces everything.
  9. RP2040 datasheet, §2.3 — Raspberry Pi's own write-up of multi-drop SWD with TARGETSEL, including the exact magic numbers. The clearest publicly-available description of SWDv2.
  10. RP2350 datasheet, §3.5 — the equivalent for the chip on edevkit1, including dormant state, the optional TARGETSEL, and the M33-side Debug Authentication interface.
  11. Memfault Interrupt blog — Cortex-M debug interfaces — bookmark this. Practical, concrete, focused on real-world debugging issues.
