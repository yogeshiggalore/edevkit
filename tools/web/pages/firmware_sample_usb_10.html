usb_10 — CDC ACM echo (your first real I/O)

Design + code-flow spec for the first edevkit USB sample that does real bidirectional data transfer. usb_02–04 only ever moved bytes on EP0 (descriptor reads, vendor commands). usb_10 brings the bulk pair to life via Zephyr's bundled CDC ACM class — the host's OS auto-loads cdc_acm / AppleUSBCDCACMData / usbser.sys, a ttyACMx / cu.usbmodemX / COMn node appears, and any terminal program can talk to the device. This is the canonical first-byte-of-data sample.

Authoring status — sample not yet built, doc is the locking spec

This document is the design-locking spec for firmware/samples/usb/usb_10/. Every API claim has been verified against Zephyr 4.4 source on 2026-05-13 (subsys/usb/device_next/class/usbd_cdc_acm.c, include/zephyr/usb/class/usb_cdc.h, include/zephyr/usb/usbd_msg.h, dts/bindings/serial/zephyr,cdc-acm-uart.yaml, samples/subsys/usb/cdc_acm/).

1. Goal & scope

One sentence: stand up a single CDC ACM virtual serial port on edevkit such that, on any host OS, a stock terminal program can connect to the auto-created tty / COM node and every character typed is echoed back by the device — gated behind a "wait for host to open the port" DTR semaphore so no bytes are lost before userspace is listening.

This is usb_05's slot in the matrix sense (the first real I/O sample) but the matrix-numbering convention reserves usb_05..usb_09 for Tier-0 / Part II additions if any are added later, so this sample takes the Tier-1 number usb_10 — pairing with Ch 15 (CDC ACM), which is the first chapter where the device "does something useful" rather than "exists statically." Sister sample usb_11 (pending, same chapter) will lift to two CDC ACM instances on the same device, demonstrating composite multi-IAD layouts.

Pairs with USB Book Ch 15 — CDC ACM. Read Ch 15 first — this document assumes you've internalised the IAD-plus-two-interfaces shape and the four class control requests, and focuses on the Zephyr-side mechanics of getting it running on the RP2350B.

Why "echo" and not "greeter / streamer / interactive shell"

An echo loop is the simplest sample that proves both directions of the bulk pair are wired: every keystroke must arrive on bulk OUT, get into a FIFO, come back out on bulk IN, and land back in the user's terminal. A one-way streamer would prove only IN; a greeter (send-on-connect) would prove neither direction's steady-state behaviour. With echo, you can also use cat > /dev/ttyACM0 and cat /dev/ttyACM0 in two windows to bulk-test the throughput. The literal echo implementation is one line — the value is in everything that has to be right around it.

What usb_10 does not cover

  • Multiple CDC ACM instances. Splitting into two virtual ports is the job of usb_11 — that's where two IADs in the same configuration get exercised. usb_10 stays at one instance to keep the descriptor count manageable.
  • Bridging to a hardware UART. usb_10 is purely virtual — bytes go USB-in → ring-buffer → USB-out. There is no uart_dev_for_usart0 involvement. A "USB-to-real-UART bridge" (the bring-up tool every project ends up writing) is its own sample later.
  • Custom line-coding handling. The firmware accepts whatever baud / parity / stop bits the host requests, stores them via Zephyr's CDC ACM class, and ignores their semantics. We don't reconfigure anything based on them. Ch 15 §4a explains why this is correct for a pure-virtual port.
  • SerialState notifications. The interrupt-IN endpoint exists and NAKs forever; we never proactively notify the host of anything. usb_11 may exercise it; usb_10 does not.
  • Flow control of any kind. RTS/CTS isn't honoured; XON/XOFF isn't filtered. Ring-buffer back-pressure (RX FIFO full → NAK the bulk OUT) is implicit through Zephyr's class, not something the application code touches.
  • OTA / bootloader interactions. CDC ACM coexisting with MCUboot+SMP is a real concern for edevkit1 v1, but it's out of scope here. usb_10 builds without MCUboot.

The structural shift is from "we implement the class" to "we configure the bundled class." Most of the sample's lines of code now live in the boot-time wiring (enable USB, wait for DTR, hook up the UART IRQ) and the IRQ handler that copies bytes between two ring-buffers. The descriptor tree we own zero bytes of.

3. The descriptor tree Zephyr will emit (byte-by-byte)

You won't write these bytes — Zephyr's CDC ACM class generates them from the DT node — but you need to recognise them in an lsusb -v capture for verification. The byte layout below is what usbd_cdc_acm.c's struct usbd_cdc_acm_desc serialises (from the class source we read at subsys/usb/device_next/class/usbd_cdc_acm.c:56):

Configuration Descriptor (9 bytes — auto, not from class)
  09 02 4B 00 02 01 00 80 32         len, type, wTotalLength=0x004B (75), 2 ifaces, ...

  Interface Association Descriptor (8 bytes) ───── G20
    08 0B 00 02 02 02 01 00          len=8, type=IAD, bFirstIf=0, bIfCount=2,
                                      bFunctionClass=0x02 CDC, subclass=0x02 ACM,
                                      protocol=0x01 AT_CMD_V250 (Zephyr default),
                                      iFunction=0

  Interface 0 — Communications, ACM (9 bytes)
    09 04 00 00 01 02 02 01 iI       bInterfaceNumber=0, bAlternateSetting=0,
                                      bNumEndpoints=1, class=0x02, subclass=0x02,
                                      protocol=0x01, iInterface=<DT label>

    CDC Header functional descriptor (5 bytes)
      05 24 00 10 01                   subtype=0x00 HEADER, bcdCDC=0x0110

    CDC Call Management func desc (5 bytes)
      05 24 01 00 01                   subtype=0x01, bmCapabilities=0x00,
                                       bDataInterface=0x01

    CDC ACM functional descriptor (4 bytes)
      04 24 02 02                      subtype=0x02 ACM, bmCapabilities=0x02
                                       (SET/GET_LINE_CODING + SET_CONTROL_LINE_STATE)

    CDC Union functional descriptor (5 bytes)
      05 24 06 00 01                   subtype=0x06, bControlInterface=0,
                                       bSubordinateInterface0=1

    Endpoint Descriptor (7 bytes)
      07 05 82 03 10 00 0A             addr=0x82 IN, attr=0x03 interrupt,
                                       MPS=16, bInterval=10ms

  Interface 1 — Data (9 bytes)
    09 04 01 00 02 0A 00 00 00       bInterfaceNumber=1, bNumEndpoints=2,
                                      class=0x0A CDC_DATA, subclass=0, protocol=0

    Endpoint Descriptor (7 bytes)
      07 05 01 02 40 00 00             addr=0x01 OUT, attr=0x02 bulk, MPS=64

    Endpoint Descriptor (7 bytes)
      07 05 81 02 40 00 00             addr=0x81 IN, attr=0x02 bulk, MPS=64

Total: 9 + 8 + 9 + 5 + 5 + 4 + 5 + 7 + 9 + 7 + 7 = 75 bytes (matches wTotalLength)

That's 11 descriptors, four of them CDC-functional-class-specific, for one virtual serial port. Compare to usb_03/usb_04 where the configuration descriptor totalled 32 bytes (1 cfg + 1 iface + 2 EP). CDC ACM is ~2.3× the byte count and ~3× the descriptor count of a bare vendor class. That's the price of "OS auto-loads a driver."

How to verify these bytes match what your build actually emits

After first successful flash, run lsusb -v -d 1209:ede5 on Linux (or ioreg -p IOUSB -l on macOS, or USBView on Windows) and diff against the layout above. Pay particular attention to: (a) the device descriptor must be 0xEF / 0x02 / 0x01 — if it's 0x00 / 0x00 / 0x00 the host won't parse the IAD; (b) the IAD's bFirstInterface + bInterfaceCount must straddle exactly the Communications + Data pair; (c) the Union descriptor's bSubordinateInterface0 must point at the Data interface (1, not 0).

4. EP0 traffic during enumeration

From host plug-in to "device is fully usable", the host issues approximately the following control transfers. Zephyr handles all of them — your code never sees these unless you log at USBD_LOG_LEVEL_DBG. Listed in capture-order so you can match a Wireshark trace.

#RequestFromNotes
1GET_DESCRIPTOR(Device, len=64)StandardInitial probe. Host doesn't know MPS yet, asks for 64 bytes, gets 18.
2SET_ADDRESS(addr)Standard2-stage.
3GET_DESCRIPTOR(Device, len=18)StandardFull device descriptor at the new address.
4GET_DESCRIPTOR(Configuration, len=9)StandardJust the cfg header — needs wTotalLength.
5GET_DESCRIPTOR(Configuration, len=wTotalLength)StandardFull 75-byte tree.
6GET_DESCRIPTOR(String, lang)StandardThen product / mfr / serial / interface strings, all by index.
7SET_CONFIGURATION(1)StandardThis is the moment the bulk pipes come alive.
8GET_LINE_CODING (bRequest=0x21)ClassHost asks: what's the current baud / data / parity / stop?
9SET_LINE_CODING (bRequest=0x20)ClassIssued when userspace opens the tty and applies its termios.
10SET_CONTROL_LINE_STATE (bRequest=0x22, wValue=0x0003)ClassThis is the DTR edge we wait for. wValue bit 0 = DTR, bit 1 = RTS.

Steps 8-10 fire after every open() of the tty / COM, not just after enumeration. If userspace closes and reopens the port, you'll see another SET_CONTROL_LINE_STATE(0x0003) preceded by SET_CONTROL_LINE_STATE(0x0000) on the close. That's why the DTR gate uses a semaphore (one-shot) rather than a flag — we only care about the first open. If you do want to handle reconnects, watch USBD_MSG_CDC_ACM_CONTROL_LINE_STATE messages from the USBD callback and reset state on every DTR transition.

5. The Zephyr API — CDC ACM is a UART

This is the conceptual shift from usb_04. There's no struct usbd_class_api for you to fill in. The class exists, it's already attached to the USBD context the moment your DT node is present and the Kconfig is on. What you get is a Zephyr UART device handle (const struct device *) that happens to be backed by USB instead of a physical UART, and you talk to it with the standard Zephyr UART API (include/zephyr/drivers/uart.h).

5a. Getting the device handle

// At file scope:
const struct device *const cdc_dev = DEVICE_DT_GET_ONE(zephyr_cdc_acm_uart);
//                                                     ^^^^^^^^^^^^^^^^^^^^
//                            DT compat with hyphens (zephyr,cdc-acm-uart) becomes
//                            underscore-form in C (zephyr_cdc_acm_uart). G19.

// In main, gate on it being ready:
if (!device_is_ready(cdc_dev)) {
    LOG_ERR("CDC ACM device not ready");
    return -ENODEV;
}

DEVICE_DT_GET_ONE errors at compile time if there isn't exactly one DT node with that compatible. (Use DEVICE_DT_GET with DT_NODELABEL(cdc_acm_uart0) if you want to be explicit, or once you go to usb_11's two-instance variant — then GET_ONE won't compile.)

5b. USBD setup and the message callback

The boot order is the same one usb_04 used (and that reference_zephyr_usbd_api documents), with one CDC-specific addition: we register a message callback on the USBD context so the class can wake us up on DTR transitions. The callback receives struct usbd_msg events of type USBD_MSG_CDC_ACM_CONTROL_LINE_STATE and USBD_MSG_CDC_ACM_LINE_CODING (defined in include/zephyr/usb/usbd_msg.h). In the handler, call uart_line_ctrl_get(msg->dev, UART_LINE_CTRL_DTR, &dtr) to learn the new state. If DTR went 0→1, k_sem_give a "host attached" semaphore.

// Conceptual — function bodies in §9
static void on_usbd_msg(struct usbd_context *ctx, const struct usbd_msg *msg);
   // dispatch on msg->type:
   //   USBD_MSG_CDC_ACM_CONTROL_LINE_STATE → check DTR, k_sem_give if 0→1
   //   USBD_MSG_CDC_ACM_LINE_CODING        → optional: log new baud
   //   USBD_MSG_VBUS_*                     → optional: usbd_enable / disable
   //   USBD_MSG_RESET / USBD_MSG_CONFIGURATION → log only

5c. The RX/TX path — interrupt-driven UART API

Once DTR is asserted, install an interrupt callback on the UART device and enable RX interrupts. Zephyr's CDC ACM class then drives the same UART IRQ semantics as a hardware UART: on incoming USB-OUT data, uart_irq_rx_ready goes true; you drain via uart_fifo_read. To send, enable TX-ready interrupts, then fill via uart_fifo_fill. The intermediate buffer is a Zephyr ring_buf you allocate.

RX path (host → device):
   USB bulk OUT URB arrives
     → Zephyr CDC ACM class reassembles into RX FIFO (1024 B, DT-configured)
       → UART IRQ fires with uart_irq_rx_ready
         → Your IRQ handler calls uart_fifo_read into a stack buffer
           → Stack buffer copied into ring_buf
             → If ring_buf was empty, enable TX IRQ to drain

TX path (device → host):
   UART IRQ fires with uart_irq_tx_ready
     → Your IRQ handler calls ring_buf_get into a stack buffer
       → Stack buffer flushed via uart_fifo_fill
         → Zephyr CDC ACM class wraps into bulk IN URBs
           → ZLP added by class when length is exact multiple of MPS — G22

Back-pressure:
   If ring_buf is full (host writes faster than host reads back the echo):
     → uart_irq_rx_disable, set rx_throttled = true
     → Zephyr class lets its USB-OUT URB queue fill
       → Eventually the class NAKs the next bulk OUT until we re-enable RX
         → Host's write() blocks transparently.
   When ring_buf drains:
     → uart_irq_rx_enable, rx_throttled = false
       → Class resumes accepting URBs.

This is exactly the pattern in Zephyr's reference sample samples/subsys/usb/cdc_acm/src/main.c. You can read that file once for reference, but writing it from scratch by reading just §5c is the more instructive path — it's about 60 lines of C and every line earns its place.

5d. The "Wait for DTR" gate, in detail

From plug-in to "host userspace has opened the tty," the sequence is:

1. Device boots; main() runs.
2. main() initialises the USBD context, registers the message callback,
   registers the bundled CDC ACM class, calls usbd_enable() — USB pull-up
   asserts, host begins enumeration.
3. Enumeration completes (Table in §4 steps 1-7). Bulk pipes are now active
   but no host program has open()'d the tty yet → no SET_CONTROL_LINE_STATE
   with DTR set has been seen.
4. main() calls k_sem_take(&dtr_sem, K_FOREVER) — blocks here.
5. User runs `screen /dev/ttyACM0 115200` → kernel CDC ACM driver calls
   usbd_cdc_acm_open() internally → issues SET_CONTROL_LINE_STATE(0x0003).
6. Class delivers USBD_MSG_CDC_ACM_CONTROL_LINE_STATE → our callback fires
   → uart_line_ctrl_get returns DTR=1 → k_sem_give(&dtr_sem).
7. main()'s k_sem_take returns. We log "DTR set". Now we hook up the IRQ
   handler and enable RX. From here on the echo loop runs.

Without the gate, step 4 is skipped, and the device starts sending its first byte (a greeting or anything else) the moment enumeration completes — but the host's userspace has nowhere to put it. On Linux it ends up in cdc_acm's small pre-open buffer and is later returned to read(); on macOS it gets dropped silently. This is the single most common "works on Linux but not on macOS" pattern for USB-serial firmware, and the fix is to gate every send on the semaphore.

Don't gate the IRQ handler installation itself

It's tempting to call uart_irq_callback_set() only after DTR. Don't — the call is cheap and idempotent. The thing to gate is any transmit that would race with the host's first read(). RX from the host can be queued; the host won't send until userspace has opened the port either. The asymmetric loss is on the device-to-host direction. (Zephyr's stock sample gates the IRQ install too; that's fine but a hair over-conservative.)

6. Carry-forward defenses + new (G19–G22)

Carried over (re-state, re-defend)

  • G1. CONFIG_USB_DEVICE_STACK_NEXT=y still mandatory.
  • G3. Class registration by name string — does not apply here. The bundled CDC ACM class registers itself via its devicetree node; you don't call usbd_register_class. (You do get the class data via USBD_CDC_ACM_DT_DEVICE_DEFINE macros — but for usb_10 the DT instantiation does this implicitly and we don't reference it from main.)
  • G4 / G16. Returning -ENOTSUP from unsupported control callbacks — does not apply here, since the bundled class owns all CDC class requests. No callbacks for you to write.
  • G14. CONFIG_HWINFO=y if you're generating a per-unit serial via hwinfo_get_device_id as in usb_03 — yes, do this. The shared sample_usbd.{c,h} from earlier samples handles it.

New for usb_10 (CDC-ACM-specific)

IDSymptom if violatedDefense
G19 Build fails with "no such DT node" / device handle is NULL at runtime / OS doesn't enumerate. The DT compatible string is "zephyr,cdc-acm-uart" (lowercase, with hyphens and a comma). In C code it shows up as zephyr_cdc_acm_uart — hyphens and commas become underscores. DEVICE_DT_GET_ONE(zephyr_cdc_acm_uart) is correct; DEVICE_DT_GET_ONE(zephyr,cdc-acm-uart) won't compile. Also: do not call usbd_device_set_code_triple() here — the bundled class needs the device-level triple to be the IAD sentinel 0xEF/0x02/0x01, which Zephyr sets for you. Overriding it to 0x00/0x00/0x00 breaks host driver loading.
G20 USB stack comes up before main() runs; the first DTR edge is missed; the "Wait for DTR" semaphore never releases on first connect. Set CONFIG_CDC_ACM_SERIAL_INITIALIZE_AT_BOOT=n in prj.conf. This defers usbd_enable() until your main() calls it explicitly, ensuring the message callback is registered before any control transfers can fire. Zephyr's reference sample sets =n for exactly this reason; the default in some Zephyr versions is =y which would silently break the gate.
G21 Greeting / boot banner is lost on macOS, works on Linux. (Or worse: works locally but breaks in CI where macOS opens the port slightly later.) Gate every device-to-host transmit on the DTR semaphore. Don't send anything before k_sem_take(&dtr_sem, K_FOREVER) returns. The stock symptom is "first byte is dropped" — most users assume their TX path is broken; it's not, the host wasn't listening yet. See §5d for the full sequence.
G22 Host's read() blocks forever after the device sends an exact-multiple-of-64-bytes response, even though all 64 bytes arrived. This is the ZLP rule (Ch 15 §6a). For CDC ACM via Zephyr's bundled class, the class handles this for you — there's a zlp_needed flag in struct cdc_acm_uart_data that tracks "last IN was 64 bytes, no more came in the callback window, send a ZLP." But the rule only fires if you let Zephyr's class drive the TX from its own ring buffer. If you reach below the API and submit raw bulk URBs yourself (you shouldn't, but you might in a custom variant), you'd have to emit ZLPs manually. For usb_10 as designed: trust the class. Just be aware that's what's happening so the symptom doesn't surprise you.

│       put = ring_buf_put(&echo_rb, buf, n)                              │
│       if (put < n) LOG_ERR("dropped %d bytes", n-put)                   │
│       if (put > 0) uart_irq_tx_enable(cdc_dev)                          │
│       if (ring_buf is full) uart_irq_rx_disable; throttled = true       │
│     if (tx_ready):                                                      │
│       n = ring_buf_get(&echo_rb, buf, 64)                               │
│       if (n == 0) uart_irq_tx_disable; continue                         │
│       if (throttled) uart_irq_rx_enable; throttled = false              │
│       send = uart_fifo_fill(cdc_dev, buf, n)                            │
│       if (send < n) LOG_ERR("short fill %d/%d", send, n)                │
└─────────────────────────────────────────────────────────────────────────┘

9. Spec: src/main.c — function-by-function

You're writing main.c from scratch. Here's the exact spec for each function. Glue code (logging, helpers) you can add to taste.

9a. Top of file — includes, log module, statics

Required headers and what you need from each:

  • <zephyr/kernel.h> — k_sem, K_SEM_DEFINE, K_FOREVER.
  • <zephyr/device.h> — const struct device *, DEVICE_DT_GET_ONE, device_is_ready.
  • <zephyr/drivers/uart.h> — uart_irq_* + uart_fifo_* + uart_line_ctrl_get + UART_LINE_CTRL_DTR.
  • <zephyr/sys/ring_buffer.h> — ring_buf, ring_buf_init, ring_buf_put, ring_buf_get.
  • <zephyr/usb/usbd.h> — usbd_context, usbd_enable, usbd_msg, usbd_msg_type_string.
  • <zephyr/usb/usbd_msg.h> — USBD_MSG_* enum (CDC_ACM_LINE_CODING, CDC_ACM_CONTROL_LINE_STATE, etc.).
  • <zephyr/logging/log.h> — LOG_*.
  • "sample_usbd.h" — your boilerplate from usb_04.

File-scope statics (note the names — they're referenced in §8's flow):

  • cdc_dev — const struct device *const, initialised with DEVICE_DT_GET_ONE(zephyr_cdc_acm_uart).
  • echo_buf[1024] — backing memory for the ring buffer.
  • echo_rb — struct ring_buf, init'd at start of main.
  • rx_throttled — bool, defaults to false.
  • dtr_sem — K_SEM_DEFINE(dtr_sem, 0, 1) — one-shot, host-attached signal.

9b. on_usbd_msg(struct usbd_context *ctx, const struct usbd_msg *msg)

When called: by the USBD core, from a worker thread, on every state-change event. What it does:

  1. Log the message type name via usbd_msg_type_string(msg->type) at LOG_DBG. (Optional; useful during bring-up, noisy in production.)
  2. If msg->type == USBD_MSG_CDC_ACM_CONTROL_LINE_STATE:
    • Read DTR via uart_line_ctrl_get(msg->dev, UART_LINE_CTRL_DTR, &dtr).
    • If dtr != 0, k_sem_give(&dtr_sem).
  3. If msg->type == USBD_MSG_CDC_ACM_LINE_CODING (optional):
    • Read baud via uart_line_ctrl_get(msg->dev, UART_LINE_CTRL_BAUD_RATE, &baud); log it. Don't act on it.
  4. Other message types — log only.

Note that msg->dev for CDC-ACM messages is the CDC ACM UART device, not the USBD device. That's the same handle you use for uart_* calls.

9c. on_uart_irq(const struct device *dev, void *user_data)

When called: by the UART subsystem whenever an RX or TX event is pending. What it does: loops on uart_irq_update + uart_irq_is_pending while there's still pending work. Inside the loop, two branches:

RX branch (if !rx_throttled && uart_irq_rx_ready(dev)):

  1. Determine read budget: n = MIN(ring_buf_space_get(&echo_rb), sizeof(stack_buf)) where stack_buf is local, e.g. uint8_t stack_buf[64].
  2. If n == 0: ring buffer is full — call uart_irq_rx_disable(dev), set rx_throttled = true, continue.
  3. Otherwise: recv_len = uart_fifo_read(dev, stack_buf, n).
  4. rb_len = ring_buf_put(&echo_rb, stack_buf, recv_len).
  5. If rb_len < recv_len: log a drop count (shouldn't happen given step 1's budget check, but log defensively).
  6. If rb_len > 0: uart_irq_tx_enable(dev) — kick the TX path so the echo happens.

TX branch (if uart_irq_tx_ready(dev)):

  1. rb_len = ring_buf_get(&echo_rb, stack_buf, sizeof(stack_buf)).
  2. If rb_len == 0: ring is empty — uart_irq_tx_disable(dev), continue.
  3. If rx_throttled: re-enable RX, clear the flag (we just made space).
  4. send_len = uart_fifo_fill(dev, stack_buf, rb_len).
  5. If send_len < rb_len: log a short-fill warning (the UART TX FIFO was tighter than we expected; the next IRQ will drain more — but log it).

9d. main(void)

Sequence:

  1. Init the ring buffer: ring_buf_init(&echo_rb, sizeof(echo_buf), echo_buf).
  2. Assert device_is_ready(cdc_dev) — return -ENODEV on failure.
  3. Call sample_usbd_init_device(on_usbd_msg) — stash the returned context (may not need to keep it).
  4. If usbd_can_detect_vbus(ctx) is false (RP2350 case — VBUS detection is not surfaced through the UDC driver currently): call usbd_enable(ctx) explicitly. Log on failure.
  5. LOG_INF("Waiting for DTR (host to open the port)...").
  6. k_sem_take(&dtr_sem, K_FOREVER).
  7. LOG_INF("DTR set, host attached.").
  8. (Optional) uart_line_ctrl_set(cdc_dev, UART_LINE_CTRL_DCD, 1) and UART_LINE_CTRL_DSR, 1 — pushes SerialState notifications upstream. Stock samples do this; we don't need to.
  9. uart_irq_callback_set(cdc_dev, on_uart_irq).
  10. uart_irq_rx_enable(cdc_dev).
  11. Return 0. The Zephyr scheduler keeps the system alive; the echo happens entirely from IRQ context.

10. Spec: prj.conf · CMakeLists.txt · boards/*.overlay

10a. prj.conf

Required Kconfigs (each annotated with why):

  • CONFIG_USB_DEVICE_STACK_NEXT=y — G1, mandatory for USBD_*.
  • CONFIG_USBD_CDC_ACM_CLASS=y — pulls in the bundled CDC ACM class and instantiates one device per matching DT node.
  • CONFIG_SERIAL=y — UART subsystem. CDC ACM class registers itself as a UART driver and needs this.
  • CONFIG_UART_LINE_CTRL=y — exposes uart_line_ctrl_get / _set. Without it, the DTR check in on_usbd_msg won't link.
  • CONFIG_UART_INTERRUPT_DRIVEN=y — explicit, even though CONFIG_USBD_CDC_ACM_CLASS selects it transitively. Belt and braces.
  • CONFIG_CDC_ACM_SERIAL_INITIALIZE_AT_BOOT=n — G20, defer USB enable to main().
  • CONFIG_HWINFO=y — G14, for per-unit serial in sample_usbd.c.
  • CONFIG_LOG=y, CONFIG_USE_SEGGER_RTT=y, CONFIG_RTT_CONSOLE=y, CONFIG_UART_CONSOLE=n — same RTT-only logging path as usb_04. Critical: we use RTT for logging because the UART console would collide with the CDC ACM virtual UART. If you wanted logs over CDC ACM itself, you'd set zephyr,console = &cdc_acm_uart0 in the chosen node and toggle these accordingly — but that's a separate sample.
  • CONFIG_USBD_LOG_LEVEL_INF=y, CONFIG_USBD_CDC_ACM_LOG_LEVEL_OFF=y — keep stack logs informative, silence the CDC ACM class's own per-byte debug spam.
  • CONFIG_HEAP_MEM_POOL_SIZE=2048 — class uses some heap.
  • CONFIG_THREAD_NAME=y, CONFIG_BOOT_BANNER=y — debugging affordances.
Do NOT enable
  • CONFIG_USBD_MSG_DEFERRED_MODE — message callback would fire from system workqueue with subtle reentrancy on shared state. The default (callback from USBD thread, synchronous) is what we want.
  • CONFIG_UART_LINE_CTRL_PROPAGATE — would auto-forward line-coding to a downstream UART. We have no downstream UART.

12c. Automated round-trip — verify.py

Use pyserial for the host-side test:

# Pseudocode — write this yourself
import serial, sys, time, os
from glob import glob

# Discover the right path per OS
def find_port():
    if sys.platform == 'linux':   return '/dev/ttyACM0'
    if sys.platform == 'darwin':  return sorted(glob('/dev/cu.usbmodem*'))[-1]
    # Windows: caller passes 'COMn' as argv[1]
    return sys.argv[1]

s = serial.Serial(find_port(), 115200, timeout=2)
# Slight pause: opening the port releases the firmware DTR semaphore;
# give it a beat to install the IRQ handler before sending.
time.sleep(0.1)

payload = b'Hello, edevkit!\n'
s.write(payload)
got = s.read(len(payload))
assert got == payload, f"mismatch: sent {payload!r}, got {got!r}"
print("OK — round-trip verified")

Run it on every supported OS. Both this script and the screen-based manual test should pass; if either fails on one OS only, the issue is almost always G21 (DTR gate) or G19 (DT compatible string).

Acceptance — usb_10 is "done" when
  • Build succeeds on rpi_pico2/rp2350a/m33 with no warnings.
  • Device enumerates on Linux / macOS / Windows with class triples per §3 and §12b.
  • verify.py passes on all three OSes from a cold boot — no manual retries, no replug.
  • RTT log shows "Waiting for DTR" before any terminal opens, "DTR set" exactly when the terminal opens.
  • Holding down a key for 5 seconds (~150 chars at typematic) round-trips without drops.
  • Closing the terminal and reopening it succeeds — no replug required.

13. Anticipated pitfalls

Symptom → Fix

"Build fails with __device_dts_ord_0 referenced but not defined." The DT node didn't materialise. Confirm the overlay path is boards/<board_name>.overlay exactly (sysbuild looks at the board name with slashes replaced by underscores: rpi_pico2_rp2350a_m33.overlay). Run west build -t menuconfig after a build and inspect the generated build/zephyr/.config for CONFIG_USBD_CDC_ACM=y + the DT instance numbers — if zero, the node didn't reach the binding.

Symptom → Fix

"Device enumerates as "Unknown Device" on Windows." Class triple isn't IAD-sentinel. See §1's "Locked decisions" — Zephyr should set this automatically, but if you've added a usbd_device_set_code_triple() call somewhere (carry-over from a vendor-class sample), remove it. Or you've forgotten CONFIG_USBD_CDC_ACM_CLASS=y, in which case the IAD doesn't get registered and the device shows up as a single-interface bare device.

Symptom → Fix

"Linux works, macOS shows no /dev/cu.usbmodem* node." Almost always missing or duplicate iSerial. macOS's path generator falls back to nonsense and may not register the device under /dev/ at all. Check that sample_usbd.c's USBD_DESC_SERIAL_NUMBER_DEFINE macro is in place and that CONFIG_HWINFO=y. Plug into a freshly-rebooted Mac to bypass any cached negative-result.

Symptom → Fix

"DTR semaphore never releases — main blocks on k_sem_take forever even after opening screen." Either (a) the message callback isn't registered (sample_usbd_init_device is called without your callback, or you're using the older 3-arg form), (b) the host's terminal program isn't asserting DTR (some terminal programs allow disabling DTR — check settings), or (c) you set CONFIG_CDC_ACM_SERIAL_INITIALIZE_AT_BOOT=y (G20) and missed the edge during init. Verify by adding LOG_INF("USBD msg: %s", usbd_msg_type_string(msg->type)) at the top of on_usbd_msg — you should see USBD_MSG_CDC_ACM_CONTROL_LINE_STATE in RTT exactly when you open the terminal.

Symptom → Fix

"Echo drops every other character when typing fast." Ring buffer's filling up and you've removed the throttle path by mistake. Confirm: (i) n = MIN(ring_buf_space_get(...), sizeof(stack_buf)) in the RX branch — if you read sizeof(stack_buf) unconditionally, you'll overflow ring_buf_put; (ii) you re-enable RX in the TX branch when the throttle flag clears; (iii) your stack buffer isn't smaller than 1 (basic sanity).

Symptom → Fix

"Send fails after a 64-byte exact-multiple write() — host's read() hangs." Surprising but possible. Zephyr's class handles the ZLP for the IN direction, but only if the TX FIFO is genuinely drained at IRQ time. If your IRQ handler exits with the FIFO half-empty (you returned without filling more bytes even though ring_buf still has data), the class may not see the exact-multiple end-of-transfer and may delay the ZLP. Workaround: always drain ring_buf fully in the TX branch before exiting. The stock pattern in §9c does this correctly because of the while (uart_irq_update && ...) outer loop.

Symptom → Fix

"Reboot device while terminal is open — terminal hangs, then says 'lost'." Expected and correct. The terminal saw the USB device disappear (no more URBs completing) and reports the loss. Reconnecting requires reopening the terminal — Zephyr's CDC ACM doesn't fake "persistent device" semantics across firmware reboots. If you need that for development, run a wrapper script like while true; do screen /dev/ttyACM0 115200; sleep 1; done.

Symptom → Fix

"Logs go missing — RTT is silent." If you set zephyr,console in the DT to &cdc_acm_uart0 (or the source code does), Zephyr's LOG_* backend now writes to CDC ACM, not RTT. Either revert the chosen node to UART0 or accept that logs ride on the same virtual port you're echoing — in which case you'll see them interleaved with your echo. For usb_10 as specified, the chosen node should stay on RTT (or the board's default).
