edevkit · firmware · USB sample walkthrough

How to write a USB sample, end-to-end

A step-by-step walkthrough that takes you from "empty directory" to "device enumerates on the host". Worked example throughout is usb_01_hello_enum; the same eight phases apply to every USB sample in the series.

Eight phases
  1. Plan the sample — what capability, which book chapter, what descriptor footprint
  2. Read the right Zephyr code first — verify API surface before writing any line
  3. Scaffold the directory — five files in the sample folder
  4. Write the class — descriptors, vtable, registration
  5. Wire main() — boot sequence in eight calls
  6. First build — expect failures; they're informative
  7. Flash + RTT — boot banner, fault dump if any
  8. Verify on the host — ioreg / lsusb / Get-PnpDevice
  9. Document + commit — README, memory, ship
Estimated total time, from cold start

First sample (you've never done this before): 4–6 hours, of which 1–2 hours is reading Zephyr internals and the rest is debug cycles. From usb_02 onward, with the workflow internalized: 1–2 hours per sample.

0Plan the sample

Time: 15 min Output: a one-page plan in your head (or a sticky note) Tools: book chapter, this doc

Before opening an editor, answer these four questions out loud:

QuestionFor usb_01
What ONE USB capability does this sample teach?Bare enumeration — the device shows up in the OS USB tree and does nothing else.
Which book chapter does it pair with?Ch 05 — Enumeration walkthrough.
What's the minimum descriptor footprint?1 config + 1 interface + 0 endpoints → wTotalLength = 18 bytes.
What VID/PID?VID 0x1209 (pid.codes umbrella), PID 0xede0 + sample_number → 0xede1.

Pin those answers. Every later decision flows from them. If you can't answer them yet, the sample isn't designed yet — go back to the USB Book and find the matching chapter.

Phase 0 done when

You can say in one sentence what the sample does and why it exists, and you know which PID it'll claim.

1Read the right Zephyr code first

Time: 30 min Output: verified knowledge of the actual API spellings Tools: grep, your editor

This is the step that, when skipped, costs you hours later. Verify three things before writing a single line of sample code.

1a. The class API surface

sed -n '/^struct usbd_class_api/,/^};/p' \
  $ZEPHYR_BASE/include/zephyr/usb/usbd.h

Read every callback's signature. The signatures evolve between Zephyr versions and rarely match what training-data recall says. Note which ones return void (most) vs int (only init, request, control_to_*). The cheat sheet from the bring-up is in the Zephyr USBD reference.

1b. The closest existing class — your template

# For a 0-endpoint vendor class like usb_01, copy from DFU's runtime mode:
ls $ZEPHYR_BASE/subsys/usb/device_next/class/

# Open and skim — do NOT copy verbatim, just learn the structure:
$ZEPHYR_BASE/subsys/usb/device_next/class/usbd_dfu.c

Look for: how they declare USBD_DEFINE_CLASS, how their vtable is shaped, how get_desc assembles its return value, what they do in init/enable/disable.

1c. The right UDC devicetree node label

grep -i "compatible.*pico-usbd\|usbd:" \
  $ZEPHYR_BASE/boards/raspberrypi/rpi_pico2/*.dts

Find the node label (the identifier before the : in label: name@addr). For RP2350 it's typically usbd. You'll need the exact spelling for both app.overlay and main.c.

Phase 1 done when

You have three things written down: (a) the actual usbd_class_api field signatures, (b) the path to the existing class you'll copy patterns from, (c) the exact UDC node label.

2Scaffold the directory

Time: 10 min Output: five empty/skeleton files Tools: shell, editor 
cd $EDEVKIT_REPO/firmware/samples/usb
mkdir -p usb_01/src
cd usb_01

# create the five files (empty for now; we'll fill them next)
touch CMakeLists.txt prj.conf app.overlay README.md
touch src/main.c src/hello_enum.c

What each file is for:

FilePurposeApprox. size when done

prj.confKconfig switches: USB-Device-Next stack, RTT console, log levels~15 lines

2a. CMakeLists.txt

Six lines of substance. Copy from any sample under $ZEPHYR_BASE/samples/subsys/usb/:

cmake_minimum_required(VERSION 3.20.0)
find_package(Zephyr REQUIRED HINTS $ENV{ZEPHYR_BASE})
project(usb_01_hello_enum)

target_sources(app PRIVATE
  src/main.c
  src/hello_enum.c
)

2b. prj.conf

The Kconfigs that matter, with comments explaining why:

# --- USB stack ---
# Use the new device stack, NOT the legacy one.  Forgetting this leaves
# every USBD_* macro undefined.  This is the most common build failure.
CONFIG_USB_DEVICE_STACK_NEXT=y

# --- Console via RTT (no UART pins burned) ---
CONFIG_LOG=y
CONFIG_USE_SEGGER_RTT=y
CONFIG_RTT_CONSOLE=y
CONFIG_UART_CONSOLE=n
CONFIG_USBD_LOG_LEVEL_INF=y

# --- USB stack uses some heap ---
CONFIG_HEAP_MEM_POOL_SIZE=2048

2c. app.overlay

Enables the USB controller node you found in Phase 1c:

&usbd {           /* the actual node label from your board's .dts */
    status = "okay";
};
Phase 2 done when

The directory exists with all five files; CMakeLists/prj.conf/app.overlay are filled in. main.c and the class file are still empty stubs — that's the next phase.

3Write the class implementation

Time: 60 min (first time) Output: src/<class_name>.c compiles standalone Tools: editor, the existing class you cited in 1b

Three pieces, in this order:

3a. The descriptor blob

For usb_01 — one interface descriptor, vendor-specific, zero endpoints:

static struct usb_if_descriptor hello_enum_iface = {
    .bLength            = sizeof(struct usb_if_descriptor),
    .bDescriptorType    = USB_DESC_INTERFACE,    /* 0x04 */
    .bInterfaceNumber   = 0,                     /* assigned at init */
    .bAlternateSetting  = 0,
    .bNumEndpoints      = 0,                     /* the whole point */
    .bInterfaceClass    = USB_BCC_VENDOR,        /* 0xFF */
    .bInterfaceSubClass = 0xFF,
    .bInterfaceProtocol = 0xFF,
    .iInterface         = 0,
};
Don't const this

Zephyr writes bInterfaceNumber at usbd_init time. If you mark the descriptor const, the linker may place it in flash and the write either silently fails or hard-faults. Leave it mutable.

3b. The NULL-terminated descriptor pointer array

This is the gotcha that costs people the most time. get_desc returns an array, not a single pointer:

static struct usb_desc_header *hello_enum_fs_desc[] = {
    (struct usb_desc_header *) &hello_enum_iface,
    NULL,    /* terminator — REQUIRED */
};

3c. The vtable + class registration

Only four real callbacks for a 0-endpoint vendor class. Everything else is NULL and the core handles that.

static int hello_enum_init(struct usbd_class_data *const c)
{
    LOG_DBG("hello_enum: init");
    return 0;
}

static void hello_enum_enable(struct usbd_class_data *const c)
{
    /* Fired on host SET_CONFIGURATION(1) — the milestone log line. */
    LOG_INF("hello_enum: configured (enabled)");
}

static void hello_enum_disable(struct usbd_class_data *const c)
{
    LOG_INF("hello_enum: deconfigured (disabled)");
}

static void *hello_enum_get_desc(struct usbd_class_data *const c,
                                 const enum usbd_speed speed)
{
    if (speed == USBD_SPEED_FS) {
        return hello_enum_fs_desc;   /* decays to struct usb_desc_header ** */
    }
    return NULL;
}

static const struct usbd_class_api hello_enum_api = {
    .init     = hello_enum_init,
    .enable   = hello_enum_enable,    /* void(c) — NOT int */
    .disable  = hello_enum_disable,   /* void(c) */
    .get_desc = hello_enum_get_desc,
    /* feature_halt, update, control_to_dev, control_to_host, request,
       suspended, resumed, sof, shutdown — all NULL */
};

USBD_DEFINE_CLASS(hello_enum, &hello_enum_api, NULL, NULL);
Two silent killers

(a) The string in USBD_DEFINE_CLASS(hello_enum, ...) MUST exactly match the string passed to usbd_register_class(&ctx, "hello_enum", ...) in main.c. Mismatch → silent -ENOENT at registration, no enumeration, no error log. Defend with a single #define.
(b) Returning 0 from control_to_* for unsupported requests claims you handled it, and the host waits forever for data. Always return -ENOTSUP for unsupported control requests.

Phase 3 done when

The class file compiles standalone (you can arm-zephyr-eabi-gcc -c it with appropriate include paths and get an object file). Doesn't matter that nothing calls it yet.

4Wire main()

Time: 30 min Output: src/main.c with the eight-call boot sequence

Two parts: file-scope descriptor declarations, then main() wires them up in a fixed order.

4a. File-scope declarations

USBD_DESC_LANG_DEFINE(lang);                           /* US English */
USBD_DESC_MANUFACTURER_DEFINE(mfr,    "edevkit");
USBD_DESC_PRODUCT_DEFINE(product,     "usb_01");
USBD_DESC_SERIAL_NUMBER_DEFINE(serial);                /* one arg only! */

USBD_CONFIGURATION_DEFINE(cfg, USB_SCD_REMOTE_WAKEUP, 50, NULL);

USBD_DEVICE_DEFINE(usbd_ctx,
                   DEVICE_DT_GET(DT_NODELABEL(usbd)),  /* from Phase 1c */
                   0x1209, 0xede1);
The serial macro takes ONE argument

USBD_DESC_SERIAL_NUMBER_DEFINE(name) auto-generates the serial from hwinfo_get_device_id(). Passing a string literal as the second argument is a build error. (See gotcha #1 in the Zephyr USBD reference.)

4b. The eight-call boot sequence

int main(void)
{
    int err;

    /* (1-4) Add string descriptors in language order: lang first, then
     *       manufacturer (index 1), product (2), serial (3). */
    err = usbd_add_descriptor(&usbd_ctx, &lang);     if (err) goto fail;
    err = usbd_add_descriptor(&usbd_ctx, &mfr);      if (err) goto fail;
    err = usbd_add_descriptor(&usbd_ctx, &product);  if (err) goto fail;
    err = usbd_add_descriptor(&usbd_ctx, &serial);   if (err) goto fail;

    /* (5) Add the configuration scoped to FS speed. */
    err = usbd_add_configuration(&usbd_ctx, USBD_SPEED_FS, &cfg);
    if (err) goto fail;

    /* (6) Register the class instance.  String must match
     *     USBD_DEFINE_CLASS in hello_enum.c.  "1" = config index. */
    err = usbd_register_class(&usbd_ctx, "hello_enum",
                              USBD_SPEED_FS, 1);
    if (err) goto fail;

    /* (7) Finalize the descriptor tree (validates, computes wTotalLength). */
    err = usbd_init(&usbd_ctx);
    if (err) goto fail;

    /* (8) Connect the D+ pull-up — host can now see us. */
    err = usbd_enable(&usbd_ctx);
    if (err) goto fail;

    LOG_INF("usb_01 hello_enum: USB enabled, idle");
    while (true) k_sleep(K_FOREVER);

fail:
    LOG_ERR("usb_01 init failed at step: %d", err);
    return err;
}

The goto fail is for clean error reporting — when something fails you'll see the exact errno in RTT instead of a silent halt.

Phase 4 done when

Both source files exist and (you think) compile. Time to find out for real.

5First build — expect failures, they're informative

Time: 20–60 min Output: build-rpi_pico2_rp2350a_m33/.../zephyr.elf Tools: edev_build 
cd $EDEVKIT_REPO/firmware/samples/usb/usb_01
edev_build      # = west build -b rpi_pico2/rp2350a/m33 .

First build almost never works. The errors you're most likely to see, in order:

ErrorCauseFix
USBD_* undefined Wrong USB stack Kconfig Add CONFIG_USB_DEVICE_STACK_NEXT=y to prj.conf
'__device_dts_ord_DT_N_NODELABEL_xxx_ORD' undeclared Wrong UDC node label Re-grep board .dts, fix app.overlay + main.c
macro takes N args but passed M Wrong macro arity (e.g. USBD_DESC_SERIAL_NUMBER_DEFINE with 2 args) grep the macro definition in usbd.h, fix call site
incompatible pointer types on a vtable field Callback signature mismatch (typically int vs void) Check the actual struct usbd_class_api field; align your function signature

Fix one error at a time. Don't try to "improve" code you didn't write yet — make it compile clean first, then iterate.

Phase 5 done when

You see FW: built [size info] and the ELF exists at build-*/usb_01/zephyr/zephyr.elf. Compiler warnings are OK; errors are not.

6Flash + RTT — boot banner or fault dump

Time: 30–90 min depending on bugs Output: RTT log showing successful enumeration Tools: edev_run, addr2line if it crashes 
edev_run        # = edev_flash + probe-rs attach (RTT). Ctrl-C to detach.
                # If the boot banner doesn't appear, cold-boot (unplug USB or pulse RUN→GND) — see the RP2350 SYSRESETREQ quirk.

Outcome A — clean boot + enumeration

*** Booting Zephyr OS build v4.4.0-... ***
[00:00:00.001,000] <inf> usbd_init: bNumInterfaces 1 wTotalLength 18
[00:00:00.001,000] <inf> usb_01: usb_01 hello_enum: USB enabled, idle
[00:00:00.323,000] <inf> usbd_core: Actual device speed 1
[00:00:00.405,000] <inf> hello_enum: configured (enabled)

If you see this — congratulations. The wTotalLength = 18 line is the byte-level milestone (matches Book Ch 05). The configured (enabled) line means SET_CONFIGURATION(1) arrived from the host. Skip to Phase 7.

Outcome B — fault dump

***** BUS FAULT *****
  Precise data bus error
  BFAR Address: 0xffffff00
  PC: 0x10003a3e
  LR: 0x10003a25
>>> ZEPHYR FATAL ERROR 25

Don't panic. The PC and LR addresses are your map. Decode them:

TOOL=$ZEPHYR_SDK_INSTALL_DIR/gnu/arm-zephyr-eabi/bin/arm-zephyr-eabi-addr2line
ELF=build-rpi_pico2_rp2350a_m33/usb_01/zephyr/zephyr.elf

$TOOL -e $ELF -fpC 0x10003a3e 0x10003a25

Output looks like:

init_configuration_inst at .../usbd_init.c:135
init_configuration_inst at .../usbd_init.c:127

Open that file at that line, read the surrounding code, figure out what your code is feeding it that it doesn't like. The BFAR address is the value the kernel tried to dereference — read its byte pattern for clues:

BFAR patternLikely cause
0x00000000NULL pointer — uninitialized struct or function pointer
0xFFFFFFXXNegative errno being treated as a pointer
Some flash addressWriting to a const object placed in flash (e.g., constified interface descriptor)
Looks like part of your descriptor bytesget_desc returned a single descriptor instead of a NULL-terminated array (gotcha #4)

Full coredump catalogue + addr2line recipe is in the firmware workflow doc.

Outcome C — silence

RTT shows no output, not even the boot banner. Most likely:

  • CONFIG_USE_SEGGER_RTT=y + CONFIG_RTT_CONSOLE=y not set in prj.conf → no RTT to read
  • Probe wiring wrong → SWD never connects → probe-rs would say "no probe" though
  • Target hard-faulted on early boot before logs flushed → probe-rs run would print exception info first
Phase 6 done when

RTT shows configured (enabled). The host completed enumeration. Time to confirm from the host side.

7Verify on the host

Time: 5 min Output: device descriptor matches design from Phase 0

Use the OS-native tool. On macOS, prefer ioreg; system_profiler caches and lags.

# macOS — primary
ioreg -p IOUSB -l -w 0 | grep -B 2 -A 25 "edevkit"

# Linux
lsusb -v -d 1209:ede1

# Windows (PowerShell)
Get-PnpDevice -PresentOnly |
  Where-Object { $_.InstanceId -like "*VID_1209&PID_EDE1*" }

Cross-check what you see against your Phase 0 plan:

idVendor0x1209 (= 4617 decimal)
idProduct0xede1 (= 60897 decimal)

USB Product Nameusb_01
USB Serial Numbersome hex string from chip ID
bcdUSB512 (= 0x0200, USB 2.0)

Full per-OS verification recipes (including PacketLogger / Wireshark / USBPcap) live in the firmware workflow doc.

Phase 7 done when

The host's USB tool reports your device with the VID/PID/strings you designed in Phase 0. The sample is functionally complete.

8Document and commit

Time: 30 min Output: README.md, optional memory update, commit

8a. Sample-local README

Mirror the structure of usb_01/README.md:

  • One-paragraph intro: what the sample teaches, which book chapter it pairs with
  • "What you'll see when it works" — the expected RTT log + ioreg snippet
  • Build/flash/monitor commands
  • File layout
  • Descriptor design choices (VID/PID/strings/bcdDevice) — why each value
  • Bugs you hit, in order, with fixes (helps the next person who clones this sample)
  • Cross-references to the book chapter and the workflow docs

8b. Memory update (only if curriculum changed)

If the sample taught us something that revises the plan — like "this sample's purpose changed" or "this Zephyr API gotcha turned out different than expected" — update memory at:

  • ~/.claude/.../memory/project_usb_book_plan.md for status changes
  • ~/.claude/.../memory/reference_zephyr_usbd_api.md for new API gotchas

Phase 8 done when

The sample is committed, README is in place, memory reflects current state. The next person can clone the directory as a starting point for their sample with confidence.
