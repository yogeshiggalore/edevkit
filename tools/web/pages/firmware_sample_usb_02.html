usb_02 — full descriptor tree, declared not implemented

Design + code flow for the second sample in the USB Book series. Extends usb_01_hello_enum from a single-interface zero-endpoint enumeration to a complete device → configuration → interface → endpoint descriptor tree — while explicitly defending against every mistake we burned during usb_01 bring-up.

1. Goal & scope

One sentence: declare a complete USB descriptor tree (device → configuration → interface → 2 bulk endpoints) so that lsusb -v on the host shows all four descriptor levels — without implementing any data flow on those endpoints.

Pairs with USB Book Ch 06 — Device, configuration, interface, endpoint (chapter not yet written; this design doc is the source-of-truth until it is).

What "declared not implemented" means here

The two bulk endpoints are present in the descriptor blob the host reads, so the OS will allocate kernel resources for them (a libusb session can libusb_claim_interface() the interface fine). But the firmware never calls usbd_ep_enqueue() on either endpoint, so any actual read/write from the host will time out. That's the intentional teaching point: a descriptor announces capability, the implementation backs it up — and they are two separate concerns. usb_04 (Ch 08) will fill in the data-flow side.

What usb_02 does not cover

  • Bulk/interrupt data flow — covered in later samples; usb_02's bulk endpoints exist on the wire only as descriptor entries.
  • Custom string descriptors per interface — iInterface = 0 for now; usb_03 redefines the serial-number string format.
  • Class- or vendor-specific control requests — usb_04 (Ch 08) is "Control transfers in depth".
  • HS / SuperSpeed — RP2350 is FS-only. We define fs_desc[]; get_desc() returns NULL for any other speed.

3. The descriptor tree, byte-level

What the host reads back over EP0 in response to GET_DESCRIPTOR(Configuration), in order. The 9-byte configuration descriptor at the head is generated by USBD_CONFIGURATION_DEFINE in main.c; the rest comes from our class's fs_desc[] array.

OffsetBytesFieldValueMeaning
0x009Configuration descriptor09 02 20 00 01 01 00 A0 32bLength=9, bDescriptorType=CONFIG, wTotalLength=0x0020 (32), bNumInterfaces=1, bConfigValue=1, iConfig=0, bmAttributes=0xA0 (bus-powered + remote-wakeup), bMaxPower=0x32 (50 → 100 mA).
0x099Interface descriptor09 04 00 00 02 FF FF FF 00bLength=9, bDescriptorType=IFACE, bInterfaceNumber=0, bAlternateSetting=0, bNumEndpoints=2, vendor class 0xFF/0xFF/0xFF, iInterface=0.
0x127Endpoint descriptor (OUT)07 05 01 02 40 00 00bLength=7, bDescriptorType=ENDPOINT, bEndpointAddress=0x01 (OUT, EP1), bmAttributes=0x02 (Bulk), wMaxPacketSize=0x0040 (64), bInterval=0.
0x197Endpoint descriptor (IN)07 05 81 02 40 00 00bLength=7, bDescriptorType=ENDPOINT, bEndpointAddress=0x81 (IN, EP1), bmAttributes=0x02 (Bulk), wMaxPacketSize=0x0040 (64), bInterval=0.

Total: 0x20 = 32 bytes. The first 9 bytes are the configuration header that contains the rest (which is why wTotalLength=32, not 23). All multi-byte fields are little-endian — so wTotalLength=32 appears on the wire as 20 00.

4. Carry-forward defenses (the 7+3 gotchas)

The seven mistakes we hit during usb_01 bring-up are catalogued in firmware_zephyr_usbd.html. usb_02 introduces three more risks specific to endpoint-bearing descriptors. Every one of these is defended-against explicitly by the code in §7-§8.

Carried over from usb_01 (re-state, re-defend)

#GotchaDefense in usb_02
G1get_desc must return a NULL-terminated array of struct usb_desc_header *, never a single descriptor pointer. Bare-pointer return → BUS FAULT, BFAR=0xffffff00 at boot.fs_desc[] in descriptors_demo.c ends with an explicit NULL entry. The cast is per-entry to (struct usb_desc_header *).
G2Don't const interface or endpoint descriptors. The stack writes bInterfaceNumber and bEndpointAddress at usbd_init time. Flash-placed descriptors cause silent failure or hard-fault.All four descriptor structs are non-const. The fs_desc[] array itself can stay non-const too for symmetry.
G3Class-name string in USBD_DEFINE_CLASS(name, ...) must exactly match the string passed to usbd_register_class(&ctx, "name", ...). Mismatch → silent -ENOENT.A single #define DESC_CLASS_NAME "descriptors_demo" at the top of descriptors_demo.c; main.c includes the same header (or duplicates the literal — see code) so they cannot drift.
G4Return -ENOTSUP from unsupported control_to_* callbacks, never 0. Returning 0 claims the request was handled; host then waits forever. -ENOTSUP makes the core STALL, which is correct per USB 2.0 §9.2.7.Both control callbacks are simply left as NULL in the vtable — the core checks for NULL before calling, and the no-handler path STALLs by default. We add no half-implemented stubs.
G5CONFIG_USB_DEVICE_STACK_NEXT=y is mandatory; without it every USBD_* macro is undefined.Carried over verbatim in prj.conf.
G6The UDC node label must exist in the active board DTS. rpi_pico2 uses usbd, NOT usb_dev0.DEVICE_DT_GET(DT_NODELABEL(usbd)) in main.c; &usbd { status = "okay"; }; in the overlay.
G7Boot sequence order matters: descriptors → configuration → register class → init → enable. Out-of-order calls fail validation silently.main.c follows the documented eight-step order with one error-checked block per step.

New for usb_02 (endpoint-specific)

#GotchaDefense
G8wMaxPacketSize is little-endian on the wire, but stored as uint16_t in struct usb_ep_descriptor. Writing .wMaxPacketSize = 64 on a little-endian MCU happens to work; on a big-endian one it would silently advertise 0x4000 = 16384 bytes — over the FS bulk limit, fail validation. Use sys_cpu_to_le16() always.Both endpoint descriptors use .wMaxPacketSize = sys_cpu_to_le16(64U). RP2350 is little-endian so this is cosmetic correctness, but the macro documents intent and matches Zephyr's own loopback.c.
G9Bulk endpoint bInterval is reserved and must be zero (USB 2.0 §9.6.6 Table 9-13). Non-zero values are accepted by some hosts and rejected by others — undefined behaviour.Both bulk endpoints set .bInterval = 0 explicitly. We document bInterval=1 for interrupt endpoints in the comment, in case usb_02 is later extended.
G10Endpoint addresses 0x01 and 0x81 are the same EP number (1) in opposite directions — fine, but a common mistake is to use 0x01 and 0x82 (different EPs), which works but wastes hardware EPs that RP2350 has limited supply of (15 IN + 15 OUT, but DPRAM is the real constraint).We pair the IN and OUT on EP1 deliberately. Comment in the code explains the convention.

6. Boot & runtime flow

Identical structurally to usb_01 — the only difference is what's in the class file. Steps 1–9 happen in main(); steps A–D are runtime callbacks fired by the USB stack thread.

BOOT (main thread, in order — order matters)
  1. usbd_add_descriptor(&ctx, &lang)         language IDs first
  2. usbd_add_descriptor(&ctx, &mfr)          string descriptors next
  3. usbd_add_descriptor(&ctx, &product)
  4. usbd_add_descriptor(&ctx, &serial)
  5. usbd_add_configuration(&ctx, USBD_SPEED_FS, &cfg)
  6. usbd_register_class(&ctx, "descriptors_demo", USBD_SPEED_FS, 1)
                                                 ^^^^^ exact string match!
  7. usbd_init(&ctx)                          validates the descriptor tree;
                                                 writes bInterfaceNumber +
                                                 bEndpointAddress back into
                                                 the (non-const!) descriptors
  8. usbd_enable(&ctx)                        connects the FS pull-up;
                                                 host now sees the device
  9. k_sleep(K_FOREVER)                       USB stack runs in its own
                                                 work queue; main idles

RUNTIME (USB stack work queue, callback-driven)
  A. host sends bus reset → core resets internal state
  B. host issues GET_DESCRIPTOR(Device)        served from USBD_DEVICE_DEFINE
  C. host issues SET_ADDRESS(N)                core handles, no callback
  D. host issues GET_DESCRIPTOR(Configuration) core walks our fs_desc[]
                                               and concatenates the bytes
  E. host issues SET_CONFIGURATION(1)          → triggers our enable() cb
  F. host issues GET_INTERFACE / SET_INTERFACE  → core handles defaults
  G. host attempts bulk read on EP 0x81        → no buffer enqueued →
                                                  NAK forever (intentional)

7. Code: src/main.c

Almost identical to usb_01/src/main.c. The three changed lines are flagged inline. Read the comments — every numbered step corresponds to a row of the boot-sequence flow above.

firmware/samples/usb/usb_02/src/main.c
/*
 * usb_02 — descriptor-tree expansion of usb_01_hello_enum.
 *
 * Builds the same boot sequence as usb_01 but registers a class
 * ("descriptors_demo") that declares a full device → cfg → iface → 2 ep tree.
 * No data flow on the bulk endpoints — that's intentional, and is the
 * teaching point this sample exists for.
 */

#include <zephyr/kernel.h>
#include <zephyr/usb/usbd.h>
#include <zephyr/logging/log.h>

LOG_MODULE_REGISTER(usb_02, LOG_LEVEL_INF);

/* G3 defense: single source of truth for the class-registration string. */
#define DESC_CLASS_NAME "descriptors_demo"

/* String descriptors — same shape as usb_01 (lang, mfr, product, auto-serial). */
USBD_DESC_LANG_DEFINE(lang);
USBD_DESC_MANUFACTURER_DEFINE(mfr,     "edevkit");
USBD_DESC_PRODUCT_DEFINE     (product, "usb_02");           /* changed from usb_01 */
USBD_DESC_SERIAL_NUMBER_DEFINE(serial);                      /* auto from chip ID */

/*
 * Configuration descriptor:
 *   bmAttributes = 0xA0  (bus-powered + remote-wakeup)
 *   bMaxPower    = 50    (in 2-mA units → 100 mA)
 *   iConfig      = NULL  (no per-config string)
 *
 * The wTotalLength field is filled in by the stack at usbd_init() time
 * after walking every registered class's fs_desc[] for this configuration.
 */
USBD_CONFIGURATION_DEFINE(cfg, USB_SCD_REMOTE_WAKEUP, 50, NULL);

/*
 * Device-level identity.
 *   VID 0x1209 — pid.codes umbrella
 *   PID 0xede2 — sample 2 of the edevkit USB Book series  (changed from usb_01)
 *   The board DTS exposes the UDC node as &usbd; do not guess "usb_dev0".
 */
USBD_DEVICE_DEFINE(usbd_ctx, DEVICE_DT_GET(DT_NODELABEL(usbd)),
                   0x1209, 0xede2);

int main(void)
{
    int err;

    /* (1)–(4) Add string descriptors. Order matters — index slots in the
     * device descriptor are assigned in the order we add them.            */
    err = usbd_add_descriptor(&usbd_ctx, &lang);
    if (err) { LOG_ERR("add lang: %d", err);    return -1; }
    err = usbd_add_descriptor(&usbd_ctx, &mfr);
    if (err) { LOG_ERR("add mfr: %d", err);     return -1; }
    err = usbd_add_descriptor(&usbd_ctx, &product);
    if (err) { LOG_ERR("add product: %d", err); return -1; }
    err = usbd_add_descriptor(&usbd_ctx, &serial);
    if (err) { LOG_ERR("add serial: %d", err);  return -1; }

    /* (5) Add the configuration. We're FS-only on RP2350. */
    err = usbd_add_configuration(&usbd_ctx, USBD_SPEED_FS, &cfg);
    if (err) { LOG_ERR("add cfg: %d", err); return -1; }

    /* (6) Attach the descriptors_demo class instance to configuration 1.
     *     Class-name string MUST match USBD_DEFINE_CLASS in the class file. */
    err = usbd_register_class(&usbd_ctx, DESC_CLASS_NAME,
                              USBD_SPEED_FS, 1 /* config index */);
    if (err) { LOG_ERR("register class: %d", err); return -1; }

    /* (7) Validate + finalize. The stack walks every class's fs_desc[],
     *     computes wTotalLength, and writes bInterfaceNumber +
     *     bEndpointAddress back into our (non-const!) descriptor structs. */
    err = usbd_init(&usbd_ctx);
    if (err) { LOG_ERR("usbd_init: %d", err); return -1; }

    /* (8) Connect the pull-up. From this moment the host can see us. */
    err = usbd_enable(&usbd_ctx);
    if (err) { LOG_ERR("usbd_enable: %d", err); return -1; }

    LOG_INF("usb_02 descriptors_demo: enabled, idle on EP0");

    /* (9) Stack runs in its own work queue. Nothing for main thread to do. */
    while (true) {
        k_sleep(K_FOREVER);
    }
    return 0;
}

8. Code: src/descriptors_demo.c

This is where usb_02 actually differs from usb_01. Three descriptor structs (interface + 2 endpoints) instead of one, all wired together via a 4-entry NULL-terminated fs_desc[] array.

firmware/samples/usb/usb_02/src/descriptors_demo.c
/*
 * descriptors_demo — vendor class, 1 interface, 2 bulk endpoints.
 *
 * Declares a complete descriptor tree (device → cfg → iface → 2 ep)
 * for teaching purposes. Endpoints are advertised but no buffers are
 * ever enqueued — host bulk reads/writes will NAK until usb_04 wires
 * up the data path.
 *
 * Pattern derived from Zephyr's own subsys/usb/device_next/class/loopback.c
 * (Apache-2.0). See attribution at the end of firmware_sample_usb_02.html.
 */

#include <zephyr/kernel.h>
#include <zephyr/usb/usbd.h>
#include <zephyr/usb/usb_ch9.h>
#include <zephyr/sys/byteorder.h>       /* sys_cpu_to_le16  */
#include <zephyr/logging/log.h>

LOG_MODULE_REGISTER(descriptors_demo, LOG_LEVEL_INF);

/* G3 defense: same string as in main.c. If the two ever drift, registration
 * silently returns -ENOENT and the device never enumerates. Keep both
 * usages anchored to a single literal. */
#define DESC_CLASS_NAME "descriptors_demo"

/* ===========================================================================
 * Descriptor structs — non-const because the stack writes into them at init.
 * (G2 defense: don't const these. Don't even const the array.)
 * =========================================================================*/

static struct usb_if_descriptor iface = {
    .bLength            = sizeof(struct usb_if_descriptor),
    .bDescriptorType    = USB_DESC_INTERFACE,        /* 0x04             */
    .bInterfaceNumber   = 0,                         /* written by stack */
    .bAlternateSetting  = 0,
    .bNumEndpoints      = 2,                         /* CHANGED vs usb_01 */
    .bInterfaceClass    = USB_BCC_VENDOR,            /* 0xFF             */
    .bInterfaceSubClass = 0xFF,
    .bInterfaceProtocol = 0xFF,
    .iInterface         = 0,                         /* no string        */
};

/*
 * Endpoint addresses follow the convention IN and OUT share the same EP
 * number, only the direction bit differs. EP1 OUT = 0x01, EP1 IN = 0x81.
 * (G10 defense — pair on the same EP number to conserve UDC resources.)
 *
 * bInterval = 0 is mandatory for bulk per USB 2.0 §9.6.6 Table 9-13. (G9)
 *
 * wMaxPacketSize wrapped in sys_cpu_to_le16() to be byte-order-correct on
 * any architecture. (G8) On RP2350 (little-endian) this is a no-op, but
 * documents intent and matches Zephyr's loopback.c reference pattern.
 */

static struct usb_ep_descriptor ep_out = {
    .bLength          = sizeof(struct usb_ep_descriptor),
    .bDescriptorType  = USB_DESC_ENDPOINT,           /* 0x05             */
    .bEndpointAddress = USB_EP_DIR_OUT | 0x01,       /* 0x01 — bulk OUT  */
    .bmAttributes     = USB_EP_TYPE_BULK,            /* 0x02             */
    .wMaxPacketSize   = sys_cpu_to_le16(64U),        /* FS bulk max      */
    .bInterval        = 0,
};

static struct usb_ep_descriptor ep_in = {
    .bLength          = sizeof(struct usb_ep_descriptor),
    .bDescriptorType  = USB_DESC_ENDPOINT,
    .bEndpointAddress = USB_EP_DIR_IN  | 0x01,       /* 0x81 — bulk IN   */
    .bmAttributes     = USB_EP_TYPE_BULK,
    .wMaxPacketSize   = sys_cpu_to_le16(64U),
    .bInterval        = 0,
};

/* ===========================================================================
 * The NULL-terminated descriptor pointer array. (G1 defense.)
 *
 * Order on the wire mirrors order in this array, AFTER the configuration
 * descriptor that the stack prepends:
 *
 *     [ cfg ]  [ iface ]  [ ep_out ]  [ ep_in ]
 *      9         9          7           7        =  32 bytes (wTotalLength)
 *
 * Each entry is cast to the base header type so the core can walk them
 * uniformly via bLength.
 * =========================================================================*/

static struct usb_desc_header *fs_desc[] = {
    (struct usb_desc_header *) &iface,
    (struct usb_desc_header *) &ep_out,
    (struct usb_desc_header *) &ep_in,
    NULL,                                            /* terminator */
};

/* ===========================================================================
 * Class API callbacks — only the four we need; the rest stay NULL and the
 * core checks for NULL before calling. (G4 defense: we add no half-stubs.)
 * =========================================================================*/

static int demo_init(struct usbd_class_data *const c_data)
{
    LOG_DBG("descriptors_demo: init");
    return 0;
}

static void demo_enable(struct usbd_class_data *const c_data)
{
    /* Fired on SET_CONFIGURATION(1). Host considers us "configured" now. */
    LOG_INF("descriptors_demo: configured (enabled)");
}

static void demo_disable(struct usbd_class_data *const c_data)
{
    /* SET_CONFIGURATION(0) or unplug. */
    LOG_INF("descriptors_demo: deconfigured (disabled)");
}

static void *demo_get_desc(struct usbd_class_data *const c_data,
                           const enum usbd_speed speed)
{
    if (speed == USBD_SPEED_FS) {
        return fs_desc;
    }
    /* HS / SS not supported on RP2350. Returning NULL is correct. */
    return NULL;
}

/* ===========================================================================
 * Vtable. Optional callbacks deliberately left as NULL:
 *   feature_halt, update, control_to_dev, control_to_host, request,
 *   suspended, resumed, sof, shutdown
 * The core treats NULL as "not implemented" and STALLs / no-ops as
 * appropriate — which is what we want for a descriptor-only sample.
 * =========================================================================*/
static const struct usbd_class_api demo_api = {
    .init     = demo_init,
    .enable   = demo_enable,
    .disable  = demo_disable,
    .get_desc = demo_get_desc,
};

/* G3 defense: literal string MUST match DESC_CLASS_NAME in main.c. */
USBD_DEFINE_CLASS(descriptors_demo, &demo_api, NULL, NULL);

Acceptance — usb_02 is "done" when

(a) All three OS verification commands above show the device with VID:PID 1209:ede2, (b) lsusb -v reports wTotalLength=0x0020 and lists both endpoint descriptors with addresses 0x01 and 0x81, and (c) RTT shows the "configured (enabled)" line within ~50 ms of plug-in. Bulk reads/writes on the host side timing out is expected and correct — that's usb_04's problem.

12. Anticipated pitfalls

Symptom → Fix

"Device enumerates as VID 1209 PID ede2 but configuration descriptor still shows wTotalLength=18." Stale cached descriptor in the OS — bump bcdDevice from 0x0100 to 0x0101, rebuild, replug. macOS in particular caches descriptors keyed on {VID, PID, bcdDevice}; bumping the version forces a re-read.

Symptom → Fix

"usbd_init: -22 (-EINVAL) at boot." One of the descriptors failed validation. Most likely cause: bNumEndpoints in the interface descriptor doesn't match the count of USB_DESC_ENDPOINT entries the stack found in fs_desc[]. We have bNumEndpoints = 2 and 2 endpoint structs in the array — counts must match exactly.

Symptom → Fix

"Hard-fault at boot, BFAR points into Flash." An endpoint or interface descriptor was inadvertently const (G2) and the stack's write-back to bEndpointAddress hit read-only memory. Check that none of iface, ep_out, ep_in are declared const.

Symptom → Fix

"usbd_register_class: -2 (-ENOENT)." Class-name string mismatch (G3). Compare the literal in USBD_DEFINE_CLASS(descriptors_demo, ...) against DESC_CLASS_NAME in main.c — they must be byte-identical.

Symptom → Fix

"Host shows the interface but bulk reads time out (expected)." Not a bug — see §1. usb_02 declares the bulk endpoints in descriptors but never submits net_bufs to back them. Use libusb / pyusb / Wireshark to confirm the host is issuing IN tokens and seeing NAKs.

Symptom → Fix

"usbd_init: -12 (-ENOMEM) on a board that worked for usb_01." The UDC ran out of internal DPRAM allocating the new bulk endpoints. RP2350's UDC has a fixed DPRAM budget; usb_01 used only EP0 (64 B IN + 64 B OUT). usb_02 adds 64 + 64 for the bulk endpoints. If the build ever grows past four bulk endpoints, watch for this.
