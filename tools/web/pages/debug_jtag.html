What JTAG is

JTAG stands for Joint Test Action Group, the working group that drafted what became the IEEE 1149.1 standard, "Standard Test Access Port and Boundary-Scan Architecture", first ratified in 1990. The original problem it solved had nothing to do with debugging firmware. In the late 1980s, surface-mount packages (PLCC, then BGA) were making bed-of-nails PCB testers impractical: with no exposed pin to land a probe on, you couldn't drive a logic level into a node and observe it on the next chip. JTAG's answer was to put the test-probe inside every chip: a small "boundary-scan" register sits between each pin and the core logic, and a four-wire serial port lets you shift values into and out of every pin, on every chip on the board, simultaneously. Connect TDI on chip 1 to TDO on chip 2, daisy-chain through the whole board, and a single 4-wire connector lets a test rig drive and observe every soldered pad without touching any of them mechanically.

That was the 1990 use case. Within a few years vendors realised the same TAP that drove the boundary-scan register could also drive any register internal to the chip — including registers that control the CPU's debug logic. By the mid-1990s, every serious microcontroller used JTAG as the debug-and-flash-programming interface. The boundary-scan use case still exists (PCB test houses use it heavily), but for embedded developers, "JTAG" today means "the ARM/MIPS/PowerPC/RISC-V/Xilinx debug interface that happens to also support boundary scan if you want it". Nearly every CPU that ships a debug interface ships JTAG; many modern parts also ship a stripped-down 2-wire variant (SWD on ARM, cJTAG / IEEE 1149.7 elsewhere) but the 4-wire port is still the lingua franca.

The IEEE 1149.1 standard has been revised over the decades — 1990 (original), 1993, 2001 (the version most commonly cited), and 2013 (which folded in optional procedural-language extensions). The core four-wire interface and the 16-state TAP controller have been stable since 1990; everything else has been additive.

key idea JTAG is, fundamentally, a serial path that can address any register inside any chip on the chain. Picking which register is the Instruction Register step; reading or writing that register's data is the Data Register step. The 16-state state machine exists purely to multiplex those two operations onto the same TDI/TDO wires without ambiguity. Once the state machine "clicks" mentally, every JTAG operation reduces to "go to state X, shift N bits, repeat".

Terminology — TAP, TDR, BSR, IDCODE, BYPASS

The standard uses very specific words that the rest of this document leans on:

TermExpansionWhat it is
TAPTest Access PortThe four-wire physical interface plus the 16-state controller behind it. Every JTAG-compliant device has exactly one TAP.
TAP controller—The 16-state finite-state machine inside the chip that interprets TMS values on TCK rising edges. See below.
IRInstruction RegisterA shift register, vendor-defined width (4–10 bits typical), that selects which Data Register is currently exposed between TDI and TDO.
DRData RegisterWhichever shift register the current IR value names. Could be the BSR, the IDCODE register, BYPASS, or any vendor-specific TDR.
TDRTest Data RegisterThe general term for any DR. Vendors invent dozens — IDCODE, BSR, BYPASS, USERCODE, plus debug-specific ones (e.g. ARM's 35-bit DPACC/APACC).
BSRBoundary-Scan RegisterThe classic IEEE-1149.1 DR — one cell per pin, lets you observe (sample) or drive (preload) the chip's pads without the core logic running. The name "boundary scan" comes from this register specifically.
IDCODE—A mandatory 32-bit DR that returns the chip's identity: {version[31:28], part_number[27:12], JEDEC_manufacturer[11:1], 1'b1}. The trailing 1 bit lets an enumeration tool walk a chain by bit-counting.
BYPASS—A 1-bit DR that connects TDI to TDO with one cycle of latency. Selected by the all-1s IR encoding (always). Lets you "skip" a chip in a chain so a longer DR shift only touches the device of interest.

The four (or five) signals

An IEEE-1149.1 TAP has four mandatory signals plus one optional reset:

PinDirection (chip POV)Purpose
TCKinputTest Clock. The TAP samples TMS and TDI on the rising edge and shifts out TDO on the falling edge. Typical maxima: 10–50 MHz on modern parts; some FPGAs accept 100+ MHz.
TMSinputTest Mode Select. The bit driven on TMS at each TCK rising edge selects the next state of the TAP FSM. TMS is the only control input — every JTAG operation is encoded as a sequence of TMS values.
TDIinputTest Data In. Serial input to whichever register (IR or DR) is currently selected for shifting.
TDOoutputTest Data Out. Serial output from the same register. Tri-stated except when actively shifting, so multiple chips' TDOs can chain through TDI of the next.
TRSTinput (optional)Test Reset. Asynchronous active-low reset that forces the TAP FSM to Test-Logic-Reset regardless of TMS/TCK. Often omitted on cost-sensitive boards — you can always reach Test-Logic-Reset by holding TMS=1 for 5 TCK cycles.
RTCKoutput (optional)Return-TCK, on some ARM implementations. The chip echoes TCK back so the probe can run "adaptive" clocking — useful when the target's clock is much slower than the probe and the probe needs to wait for each edge to propagate. Not part of IEEE 1149.1; defined by ARM's RealView reference designs.

Voltages, slew rates, and pull-up requirements are not specified by IEEE 1149.1 — that's left to the silicon vendor. In practice, Cortex-M parts use the same I/O pads as their GPIOs (1.8 V or 3.3 V LVCMOS), and probe vendors put a level-shifter at the connector. JTAG is also strictly synchronous: there is no asynchronous data path, no clock recovery, and no autobaud. The probe owns TCK and clocks every transaction at whatever rate it likes (within the target's max).

why TMS is the only control input IEEE 1149.1 was designed to be cheap to implement. Adding more control wires would have meant more pins on every chip in 1990 — at a time when pin count was the dominant cost. The committee's solution: encode all control as a sequence of TMS values, sampled on a single shared clock. The cost is that every JTAG operation requires walking the 16-state FSM through a specific path. The benefit is that the chip-side hardware needed to implement the standard is a four-bit state register, two shift registers, and a comparator — small enough to drop into any silicon.

The TAP state machine

The TAP controller is a 16-state Mealy-style finite-state machine clocked by TCK. The next state is determined entirely by the current state and the value of TMS on the most recent TCK rising edge. Every JTAG operation is a walk through this graph.

The states divide into two parallel "columns" — one for the Data Register, one for the Instruction Register — joined at the top by the Select-DR-Scan / Select-IR-Scan branch. This symmetry is intentional: the same sequence of TMS values that shifts the DR also shifts the IR, just in the other column.

State categories

The 16 states fall into five logical groups:

GroupStatesWhat happens
ResetTest-Logic-ResetIR is forced to its reset value (selects IDCODE if implemented, BYPASS otherwise). All TDRs return to defined defaults. Held by TMS=1; entered automatically by 5+ cycles of TMS=1 from any state.
IdleRun-Test/IdleThe "do nothing" state. Used as a parking spot between operations. Some vendor instructions (RUNBIST, etc.) use Run-Test/Idle as the active state — TCK pulses here drive built-in self-test logic.
CaptureCapture-DR, Capture-IRThe selected register is loaded in parallel from its source: Capture-IR loads the standard fixed pattern ...01 (low bits = 01, upper bits vendor-defined); Capture-DR loads whatever data the chip wants you to read this cycle (e.g. ACK bits, peripheral status, BSR-sample of the pins).
ShiftShift-DR, Shift-IREach TCK rising edge with TMS=0 shifts one bit out on TDO and one bit in from TDI. This is where actual data movement happens. Stay here for N cycles to shift N bits.
UpdateUpdate-DR, Update-IRThe shifted value is committed in parallel to the register's "live" side. Until Update fires, the chip's behaviour reflects the previous register value, not what's in the shift chain.
TransitSelect-*, Exit1-*, Exit2-*, Pause-*"Decision" states — used to navigate. Pause exists so a slow probe can park mid-shift without losing position.

The Capture / Shift / Update split is the critical insight. Shifting does not change the chip's behaviour; only Update does. This is what lets the BSR drive PCB-level test patterns: you preload the desired pin states by shifting through Capture-DR → Shift-DR → Exit1-DR, then a single Update-DR pulse simultaneously drives all the pins on all the chips in the chain. No glitching during the shift.

Worked example — TMS sequence to "shift 32 bits into DR"

You are in Run-Test/Idle and want to shift a 32-bit value into the currently-selected DR, then return to idle. The TMS sequence (one bit per TCK cycle):

cycle  TMS  state-after-rising-edge      action / data on TDI
─────  ───  ───────────────────────────  ─────────────────────────────
 0      1   Select-DR-Scan
 1      0   Capture-DR                   (chip parallel-loads DR)
 2      0   Shift-DR                     bit  0 → DR[0]
 3      0   Shift-DR                     bit  1 → DR[1]
 4      0   Shift-DR                     bit  2 → DR[2]
 5..32  0   Shift-DR  (×31 more)         bits 3..30 → DR[3..30]
33      1   Exit1-DR                     bit 31 → DR[31]   ← LAST bit shifts on the
                                                              same edge that exits
34      1   Update-DR                    (chip parallel-commits DR)
35      0   Run-Test/Idle

Two things to note. First, the bit count is "shift cycles + 1" — the final bit goes in on the TMS=1 edge that exits Shift-DR, not on a separate cycle. Probe libraries pre-bake this off-by-one so application code can just say "shift 32 bits". Second, the order is LSB first: bit 0 of your value becomes the TDI value on cycle 2, bit 31 on cycle 33. TDO follows the same convention — the first TDO bit you read corresponds to the previous value's bit 0.

IR and DR — the two registers behind every TAP

Every TAP exposes exactly one IR and at any given moment exactly one DR. The IR's value names which DR is "selected"; the DR is then the only data shift register reachable through the TDI→TDO path.

The Instruction Register

The IR is a vendor-defined-width shift register (4 bits is the minimum; 4, 5, 8, and 10 are common; some FPGAs go to 16+). IEEE 1149.1 reserves four IR encodings:

EncodingMandatory?Selects DR
BYPASS = all 1sMandatory1-bit BYPASS register (TDI→FF→TDO).
EXTEST = all 0s (per 1990 spec) or vendor-defined (per 2001 spec)MandatoryBoundary-Scan Register, drive mode (preloaded values appear on pins).
SAMPLE/PRELOADMandatoryBSR, sample mode (snapshot of current pin states, no drive).
IDCODEOptional but near-universal32-bit IDCODE register. Auto-selected at Test-Logic-Reset if implemented.

Everything else is vendor-defined. ARM's CoreSight DAP, for example, defines ABORT, DPACC, APACC, IDCODE, BYPASS at fixed 4-bit IR encodings — see below.

The Capture-IR state loads a fixed pattern into the IR shift chain: the standard mandates that the lowest two bits load as 01 regardless of vendor. This gives test tools a way to verify the IR width: shift in all 1s while reading TDO, count how many cycles pass before the trailing 1 from the load pattern appears, and you've measured the IR length in bits. A surprising number of bring-up bugs reduce to "the tool guessed the IR width wrong".

The Data Register

The DR is whichever shift register the current IR value points at. A single chip will typically have many DRs implemented — IDCODE (32 bits), BYPASS (1 bit), BSR (one cell per pin, often 100+ bits on a big SoC), USERCODE (32 bits, FPGA bitstream identifier), and as many vendor-specific TDRs as the silicon designer chose. Only one is on the TDI→TDO path at a time, named by IR.

The DR's width is implementation-defined per instruction; the same TMS walk through Capture-DR / Shift-DR / Update-DR works for all of them. The number of bits you shift has to match the selected DR's actual width — shifting too few bits leaves the chip in an undefined state at Update; shifting too many means data wraps around and you overwrite earlier bits. Tools rely on the BSDL file (or a vendor-supplied driver) to know the right shift length.

Worked example — reading IDCODE end to end

The most universal "is JTAG even alive" check: enter Test-Logic-Reset, walk to Shift-DR, shift out 32 bits. At Test-Logic-Reset the IR auto-loads to IDCODE (if implemented), so the first 32 bits of TDO are the chip's IDCODE. No prior IR write needed.

The trailing 1 at bit 0 is mandatory for IDCODE. It's how chain-discovery tools tell IDCODE apart from BYPASS: when a chip is in Test-Logic-Reset, if the first TDO bit you see is 1, the chip implements IDCODE (and the next 31 bits are the rest of the code); if it's 0, the chip dropped to BYPASS at reset and you've already read its only DR bit.

Multi-target chains

Multiple JTAG-compliant chips on one PCB share TCK and TMS but daisy-chain TDI→TDO: chip 1's TDI is driven by the connector, chip 1's TDO connects to chip 2's TDI, and so on, with the last chip's TDO going back to the connector. From the probe's POV, the chain looks like one giant shift register whose total length is the sum of all the chips' currently-selected register widths.

       probe                 chip A           chip B           chip C
       ┌────┐                ┌────┐          ┌────┐          ┌────┐
TDI ──►│ TDI├──TDI───TDO────►│ TDI├──TDO────►│ TDI├──TDO────►│ TDI│──TDO──► back to probe
       └────┘                └────┘          └────┘          └────┘
                                ▲               ▲               ▲
                              TCK,TMS broadcast to all simultaneously

Two consequences:

  1. Every register operation hits every chip. A TMS walk to Shift-IR shifts all the chips' IRs at once: the total shift length is IR_A + IR_B + IR_C. To address one chip, you put the others in BYPASS (IR = all 1s) and put your target's "real" instruction in just that chip's IR slot.
  2. Chain order matters. The chip nearest the probe's TDI sees the most-significant bits of the IR stream (because shift-DR/IR is LSB-first at each chip, but the chips are concatenated). When you assemble an IR shift for a 3-chip chain (4-bit IR each, target = chip B with instruction 0xC), you send: 1111_1100_1111 → chip C gets 1111 (BYPASS), chip B gets 1100 (your instruction), chip A gets 1111 (BYPASS).

Discovering an unknown chain

OpenOCD, UrJTAG, and similar tools all walk the same algorithm to discover a chain blindly:

1. Force Test-Logic-Reset (TMS=1 × 5+).
   → All TAPs auto-select IDCODE-or-BYPASS.

2. Walk to Shift-DR. Drive TDI=0 for many cycles, watch TDO.

3. Decode the TDO stream:
      - Read 32 bits. If they end with bit-0 = 1: it's an IDCODE.
                                  Record the IDCODE value, move on.
      - If bit-0 = 0: that chip is in BYPASS (no IDCODE).
                      Record "1 BYPASS bit", move on by 1 bit.

4. After the last chip's TDO byte, you'll start seeing the TDI=0
   value you've been driving — that's the "end of chain" marker.
   Count: that's how many devices total.

5. To measure each chip's IR width, force TLR again, walk to
   Shift-IR, drive TDI=1 forever, count cycles until you see the
   Capture-IR pattern (..01) emerge on TDO. Total IR width = chain
   IR width; deduce per-chip width from BSDL or vendor knowledge.

Most ARM dev boards have exactly one TAP on the JTAG chain (the Cortex-M's debug port), so the discovery algorithm is overkill in practice. Where you do see longer chains: FPGA + MCU on the same board (Xilinx + STM32, Lattice + nRF, etc.), or multi-die SoCs that expose a separate TAP per die.

BSDL — Boundary-Scan Description Language

BSDL is a VHDL-subset DSL, defined as part of IEEE 1149.1, that vendors use to describe a chip's TAP to test tools. Every BGA/QFN package you can buy from a name-brand vendor ships with a corresponding .bsd file, available from the vendor's website. A BSDL file declares:

  • The chip's package and pin-to-ball mapping (pin 1 = signal X, pin 2 = signal Y, …).
  • The IR width and the meaning of every IR encoding the chip supports — including all the boundary-scan instructions (EXTEST, SAMPLE, INTEST, RUNBIST, CLAMP, IDCODE, USERCODE, ...).
  • The Boundary-Scan Register cell list — which pin each cell drives or observes, and in which order they appear in the BSR shift chain.
  • The IDCODE value(s) supported by the part.
  • Any pin-level disable / safe-state behaviour for EXTEST.

A snippet (heavily abbreviated, real files are 500–5000 lines):

entity STM32F407VG is
   generic (PHYSICAL_PIN_MAP : string := "LQFP100");
   port (PA0 : inout bit;
         PA1 : inout bit;
         ... );

   use STD_1149_1_2001.all;
   attribute COMPONENT_CONFORMANCE of STM32F407VG : entity is "STD_1149_1_2001";

   attribute INSTRUCTION_LENGTH of STM32F407VG : entity is 5;
   attribute INSTRUCTION_OPCODE of STM32F407VG : entity is
       "BYPASS  (11111),"     &
       "EXTEST  (00000),"     &
       "SAMPLE  (00010),"     &
       "PRELOAD (00010),"     &
       "IDCODE  (00001)";

   attribute IDCODE_REGISTER of STM32F407VG : entity is
       "0001"        &        -- version
       "0100000110000111" &    -- part number
       "00000100000"  &        -- JEDEC mfr (ST = 0x20)
       "1";                    -- mandatory LSB

   attribute BOUNDARY_LENGTH of STM32F407VG : entity is 384;
   attribute BOUNDARY_REGISTER of STM32F407VG : entity is
       "0   (BC_1, *,        controlr, 1),"  &
       "1   (BC_1, PA0,      output3,  X, 0, 0, Z),"  &
       "2   (BC_1, PA0,      input,    X)," &
       ...

Tools that consume BSDL: UrJTAG (open-source PC-side boundary-scan controller), topJTAG (Windows commercial), XJTAG (industry-standard for production test), and the boundary-scan modules of OpenOCD. They use BSDL to map "set pin PA5 high, observe pin PB3" requests onto the right shift-DR bits.

when you'll touch BSDL Mainstream ARM firmware development almost never needs BSDL: ARM debug works through CoreSight's JTAG-DP instructions, not through the boundary-scan register. You'll need BSDL for: PCB factory test (using XJTAG/UrJTAG), board bring-up of a brand-new board where you suspect a manufacturing solder defect (boundary-scan can find shorts/opens without having any firmware), or working with FPGAs (where BSDL is the only way the synthesis flow knows how to talk to the device's TAP for pin mapping).

JTAG-DP — the ARM Debug Port over JTAG

ARM's CoreSight Debug Access Port (DAP), defined in the ARM Debug Interface Architecture Specification (ADIv5 and ADIv6), is the standard way every Cortex-M, Cortex-A, and Cortex-R part exposes its debug logic. The DAP is a small piece of always-on logic that sits in the chip's debug power domain and arbitrates access to Access Ports (APs), each of which is a bus master onto some part of the SoC. The most common AP is the MEM-AP, which exposes the AHB system bus and is how a probe reads/writes Cortex-M registers, SRAM, and peripherals.

The DAP comes in two physical flavours:

  • JTAG-DP — DAP commands ride on a standard IEEE-1149.1 TAP. The chip looks like any other JTAG device on the chain.
  • SW-DP — DAP commands ride on the 2-wire SWD protocol. See debug_swd.html.

A SoC almost always implements both, multiplexed onto the same pads (TCK/SWCLK and TMS/SWDIO are the same pins). Which one is "active" at reset is vendor-defined; ARM provides a magic switching sequence to flip between them at runtime — see below.

The JTAG-DP instruction set

The JTAG-DP IR is 4 bits wide. Its instructions:

IR encodingInstructionSelected DR (bits)Purpose
1000ABORT35Force-abort an in-flight transaction; clear sticky errors.
1010DPACC35Read/write a Debug Port register (CTRL/STAT, SELECT, RDBUFF, ...).
1011APACC35Read/write the currently-selected Access Port register (TAR, DRW, IDR, ...). This is the workhorse — every memory read/write the probe issues is an APACC.
1110IDCODE32Standard IEEE 1149.1 IDCODE — identifies the DAP itself (e.g. 0x4BA00477 for a Cortex-M4 generic DAP).
1111BYPASS1Standard IEEE 1149.1 BYPASS.

The 35-bit DR for AP/DP transfers

Both DPACC and APACC select the same 35-bit shift register. The bit layout, on the DR shift in (from probe to chip):

shift-in (TDI side):
   ┌────────────────────────┬───────────────────┬───┐
   │      WDATA (32 bits)   │    A[3:2] (2)     │RnW│
   │  data being written    │ register-select   │  │
   └────────────────────────┴───────────────────┴───┘
        bits 34..3              bits 2..1        bit 0
                                                  0=Write
                                                  1=Read

shift-out (TDO side, from previous transaction):
   ┌────────────────────────┬─────────────────────┐
   │      RDATA (32 bits)   │    ACK (3 bits)     │
   │  data read previously  │ result of last txn  │
   └────────────────────────┴─────────────────────┘
        bits 34..3                bits 2..0
                                   010 = OK/FAULT
                                   001 = WAIT (retry)

Note the pipelining: each 35-bit shift kicks off a transaction and reads the result of the previous transaction. This means a probe doing a memory read has to issue a dummy RDBUFF read at the end to drain the last result. ARM's ADIv5 specification §3.3 describes this in detail; if you're writing your own probe firmware, this single fact will trip you up at least once.

Compare to SWD: the same DPACC/APACC operations encode in 46 bits on SWD (8-bit header + 32-bit data + ACK + parity + turnaround) but with no pipelining — each line transaction is self-contained. JTAG is more verbose per round-trip but pipelines beautifully; SWD is more compact per line but stricter about turnarounds. Read the SWD doc for the full comparison.

Reading Cortex-M memory through JTAG-DP — the full path

Reading 0x20000000 (start of SRAM on a generic Cortex-M) through JTAG-DP requires:

1. IR ← APACC                    (4-bit IR shift)
2. DR ← write(SELECT, AP=0)      (35-bit shift, configures which AP)
3. DR ← write(MEM-AP TAR, 0x20000000)   (35-bit, target address)
4. DR ← read(MEM-AP DRW)         (35-bit, kicks off the read)
5. DR ← read(RDBUFF)             (35-bit, drains the pipelined result)
   → RDATA is the 32-bit value at 0x20000000

Total: ~145 TCK cycles for one 32-bit read on a single-device chain.

OpenOCD, probe-rs, pyOCD all hide this behind a clean API; if you peek under the hood you'll see exactly this sequence (or its SWD equivalent) for every memory read your debugger issues.

JTAG vs SWD — when each wins

For ARM Cortex-M parts, JTAG and SWD are functionally equivalent — they are two physical-layer encodings of the same DAP commands. ARM ships both because they serve different audiences.

Pick JTAG when: you're working with a multi-chip board (FPGA + MCU is the canonical case), you need boundary-scan PCB diagnostics, you're working with a non-ARM target, you're already running a logic analyser that grokked TDI/TDO and can decode JTAG traces.

Pick SWD when: the target is ARM, you have pin-count pressure on the chip or connector (SWD frees 2 pins for application use), or you want SWO trace bundled with debug. SWD is the default on most modern ARM dev boards for exactly these reasons.

Most ARM Cortex-M boards expose only SWD on their debug header — the manufacturer ships a 10-pin Cortex Debug Connector wired for SWD, not JTAG. The chip still supports JTAG, but TDI and TDO aren't broken out. If you need JTAG on those parts (e.g. for boundary scan), you'll have to find pads on the PCB and probe directly.

The JTAG-to-SWD switch sequence

An ARM Cortex-M comes out of reset with the DAP in either JTAG mode or SWD mode, depending on what the vendor wrote into the boot ROM and OTP. To force the part into SWD from a known-but-arbitrary state, ARM defines a magic 16-bit sequence shifted on SWDIO (which is the same physical pin as TMS).

The classic (ADIv5) sequence

The full "JTAG → SWD" handshake is:

1. Send at least 50 cycles of SWDIO=1 with SWCLK toggling.
   → Forces the DAP out of any state. (SWD line reset; also resets a JTAG TAP
     because TMS=1 ×5+ → Test-Logic-Reset.)

2. Shift the 16-bit magic value 0xE79E (LSB first) on SWDIO,
   one bit per SWCLK rising edge.

   binary, transmit order (LSB first):
      0  1  1  1  1  0  0  1  1  1  1  0  0  1  1  1
      └────── 0x9E ──────┘ └────── 0xE7 ──────┘

   on the wire (LSB first per byte, low byte first):
      cycle 0: 0       cycle 8:  1
      cycle 1: 1       cycle 9:  1
      cycle 2: 1       cycle 10: 1
      cycle 3: 1       cycle 11: 0
      cycle 4: 1       cycle 12: 0
      cycle 5: 0       cycle 13: 1
      cycle 6: 0       cycle 14: 1
      cycle 7: 1       cycle 15: 1

3. Send another 50+ cycles of SWDIO=1 with SWCLK toggling.
   → SWD line reset, now the DAP is firmly in SWD mode.

4. Read the IDCODE on SWD by issuing a DPACC read of register 0.
   → Confirms the part responded; you're on SWD.

The reverse sequence (SWD → JTAG) uses the magic value 0xE73C. Both magic values were chosen because their bit patterns don't appear in any legal JTAG TMS sequence walking through the FSM — so a chip in JTAG mode will see them as innocuous TMS noise and stay in JTAG mode unless it specifically recognises the SWD-mode-request pattern.

The dormant-state sequence (ADIv5.2 / ADIv6)

Newer ARM parts (Armv8-M, including the RP2350's M33) implement a more general "dormant state" — a fourth DP state, distinct from JTAG-active and SWD-active, that the part enters in response to a different magic. From dormant you can select JTAG or SWD with a follow-up sequence. The dormant-state mechanism lets future protocols (cJTAG, multi-drop SWD, vendor-specific) coexist on the same pads.

The full dormant-state sequences are documented in ADIv6 §B5 — they are longer (up to 188 bits including the activation code) and tool libraries (probe-rs, OpenOCD, CMSIS-DAP firmware) bake them in. From the application engineer's POV, you ask the probe library for "SWD mode" and the right sequence runs under the hood. From the firmware POV, if you're writing a probe (as edevkit1 does), you copy the sequences verbatim from the ADIv6 spec; they're not memorisable.

RP2350 specifically The RP2350 boot ROM defaults to SWD mode on the SWCLK/SWDIO pads. To use JTAG on an RP2350 (which it supports, exposing both Cortex-M33 cores and the Hazard3 RISC-V cores via the same TAP) you need to issue the SWD-to-JTAG dormant-state sequence first. Most users never do — SWD is faster and uses fewer pins, and the RP2350's CMSIS-DAP firmware (used by Pi's official debug probe) only exposes SWD. edevkit1 firmware will support both modes; the JTAG path exists primarily for boundary-scan and non-ARM targets, not for ARM.

level shifting All ARM debug connectors are VTref-referenced: the probe samples VTref (pin 1) and switches its line drivers to that voltage. Never tie VTref to the probe's own supply — the whole point is that targets at 1.8 V, 3.3 V, or 5 V can all share one connector. edevkit1 routes its probe pins through an LVC8T245 buffer with VTref-tied OE and direction control; see the v1 design spec for the schematic.

edevkit1's JTAG implementation

edevkit1 implements JTAG entirely in software, using one of the RP2350's PIO state machines as a programmable bit-banger. The PIO blocks (12 state machines across 3 PIO blocks on RP2350B) are tiny dedicated processors with deterministic single-cycle GPIO access, designed exactly for protocol implementations like this. See platform_pio.html for the PIO programming model.

The single PIO state machine drives TCK, TMS, and TDI from one side-set; reads TDO via the SM's input mapping; and uses an autopull/autopush FIFO to receive instruction bytes from the host. The same SM is loaded with one of two PIO programs at runtime: swd.pio for SWD mode, jtag.pio for JTAG mode. Switching between modes at runtime swaps the program (a few hundred microseconds).

The JTAG PIO program — sketch

The full source lives at firmware/app/pio/jtag.pio (forthcoming). The skeleton:

; jtag.pio — bit-bang IEEE 1149.1 on PIO0 SM0
; side-set: TCK   (1 bit)
; out pins: TMS, TDI  (2 bits, but only 1 is driven per cycle)
; in pin:   TDO
;
; The host pushes 8-bit "command" bytes:
;   [7:6] = mode    (0=TMS clock, 1=TDI/TDO data, 2=set OE, 3=reserved)
;   [5]   = read    (1=push TDO bit back to RX FIFO)
;   [4:0] = count   (number of bits, 0..31)
;
; followed by data bytes if mode == 1 (TDI/TDO data, packed 8 bits per byte).

.program jtag
    pull block            ; fetch the next command from TX FIFO
    out x, 5              ; X = bit count (decremented per cycle)
    out y, 1              ; Y = read flag
    out pins, 2     side 0 ; load TMS/TDI initial; TCK low

bit_loop:
    nop             side 1 ; TCK rising edge — chip samples TMS/TDI here
    in pins, 1            ; read TDO into ISR (1 bit at a time)
    jmp y--, push_bit
    jmp pop_data
push_bit:
    push iffull           ; when ISR fills, send 32-bit word back to host
pop_data:
    out pins, 2     side 0 ; next bit on TMS/TDI; TCK falling edge
    jmp x--, bit_loop
    jmp 0                  ; → fetch next command

The PIO state machine runs at sys_clk / divider. With sys_clk = 150 MHz on RP2350B and divider = 3, each loop iteration takes 4 cycles, giving a TCK rate of ~12.5 MHz. That's fast enough for any Cortex-M target and most FPGAs; faster TCK is achievable by tightening the loop (combining out and side on the same cycle), at the cost of program size. The v1 design spec targets 25 MHz SWD and 10 MHz JTAG — JTAG is intentionally rate-capped because cable SI degrades at 4-wire, 6"-cable distances faster than at 2-wire SWD.

CMSIS-DAP layer on top

The PIO program is the bit-pump; on the host-USB side, edevkit1 implements CMSIS-DAP v2 over a USB bulk endpoint. CMSIS-DAP defines an HID/bulk command set ("DAP_JTAG_Sequence", "DAP_JTAG_IDCODE", "DAP_TransferBlock", etc.) that probe-rs / OpenOCD / pyOCD all already speak. The firmware translates each CMSIS-DAP command into a sequence of PIO commands and pushes them down. See usb_cmsis_dap.html for the host-side framing in full.

Common pitfalls

The bugs you'll hit, sorted by frequency in real-world bring-up:

  1. Confusing TRST with Test-Logic-Reset. These are not the same thing. TRST is an asynchronous physical pin that, when low, holds the TAP FSM in Test-Logic-Reset regardless of TCK. Many cost-sensitive boards omit TRST entirely (it's optional in IEEE 1149.1). On those boards, the only way to reach Test-Logic-Reset is to drive TMS=1 for at least 5 TCK cycles — that path through the FSM is guaranteed to land you in Test-Logic-Reset from any starting state. If your probe firmware assumes TRST exists and skips the 5-cycle TMS reset, the chain initialises incorrectly and the first IDCODE read returns garbage. Always start every session with the TMS-only reset.
  2. Wrong TCK speed for a chip that needs RTCK. Adaptive-clocking parts (some legacy ARM946/ARM11 SoCs, some Xilinx FPGAs in low-power modes) require the probe to wait for an RTCK echo before issuing the next TCK edge. If your probe runs free-clocking at, say, 10 MHz but the target's internal clock is 1 MHz, the TAP samples late or skips edges entirely. Symptom: IDCODE works (because the chip is fast enough to read its own pre-loaded value), but APACC reads return WAIT forever. Fix: drop TCK below the target's slowest expected clock, or implement RTCK following.
  3. BSDL file mismatch — wrong IR width. Boundary-scan tools rely on the BSDL file declaring the right INSTRUCTION_LENGTH. If the file is for a similar but not identical part (e.g. STM32F407 vs STM32F417), the IR shift length is wrong and every operation becomes silent garbage — no error, just wrong data. Always verify the IDCODE matches the BSDL's declared IDCODE_REGISTER attribute before trusting any other operation.
  4. Forgetting BYPASS for non-target chips on a chain. A 3-chip chain where you set IR for chip B but leave A and C in their default IR (which might be EXTEST!) can spike random pins on A and C. Always explicitly load BYPASS into IRs you don't care about — the all-1s pattern is mandatory and safe on every IEEE-1149.1 chip.
  5. Off-by-one on the last shift bit. The last bit of a Shift-DR or Shift-IR is clocked in simultaneously with the TMS=1 transition that exits the Shift state. New probe-firmware authors often add an extra cycle, or omit one — both fail silently. Double-check on a logic analyser.
  6. Driving TDI through a chip that's in EXTEST. EXTEST disconnects the chip's core from its pins and drives the pins from the BSR. If a TDI line on chip B is shared with a sensor or oscillator, EXTEST will drive that line — you can corrupt your own clock or external SPI. Use SAMPLE/PRELOAD for non-destructive observation, EXTEST only when you're actually driving pins for test.
  7. Cable too long. JTAG is fully synchronous and TDI/TDO are unterminated — at 25 MHz over 12" of ribbon, the round-trip skew between TCK and TDO eats your timing budget. Symptom: works at 1 MHz, fails at 10 MHz. Fix: shorten the cable (≤6" recommended) or drop TCK. SWD is more forgiving here because there's no TDI/TDO loop — only one bidirectional line.
  8. Power sequencing with the probe. If the probe is powered (5 V from USB) and the target is powered down, the probe's debug pins can backfeed the target through ESD diodes. Always verify the probe respects VTref — i.e. the probe's level shifters are disabled when VTref reads zero. edevkit1 does this with the LVC8T245 OE pin tied to a Schmitt-buffered VTref-detect circuit.
  9. Mixing SWD-only headers with JTAG cables. Many modern dev boards expose the 10-pin Cortex Debug Connector but only wire SWD — pins 6 (TDO/SWO) and 8 (TDI) are no-connects. Plugging a JTAG-only probe in does nothing useful. Check the schematic before assuming your cable will work.

Where to learn more

Recommended reading, ordered roughly easiest → most authoritative:

  1. The Wikipedia JTAG article — surprisingly thorough, with diagrams of the TAP FSM, IDCODE format, and a survey of vendor-specific extensions. Start here for context, especially the "history" section.
  2. The OpenOCD User's Guide, especially the chapters on "JTAG Hardware Dongles" and "Reset Configuration". OpenOCD is the de-facto open-source JTAG probe orchestrator; reading its docs teaches you what knobs every probe exposes.
  3. OpenOCD source — src/jtag/core.c and src/jtag/drivers/*.c. The jtag_add_* primitives (jtag_add_dr_scan, jtag_add_ir_scan, jtag_add_tlr) are direct expressions of the IEEE 1149.1 state-machine walks. Reading 200 lines of core.c teaches more than 20 pages of any PDF.
  4. IEEE 1149.1-2013 (or -2001) — the standard itself. Buying a copy ($200+) is rarely worth it for software-side work; older 1990 versions are floating around in PDF if you need to verify a corner case. The 2013 revision adds the procedural-language extension (1149.1-PDL) which most embedded developers can ignore.
  5. ARM ADIv5 Architecture Specification (IHI0031) — the canonical spec for JTAG-DP and SW-DP. §3.3 on the 35-bit DPACC/APACC encoding is essential if you're writing probe firmware. ADIv6 (IHI0074) supersedes for Armv8/8.1-M parts but ADIv5 is still the right entry point.
  6. ARM CoreSight Technology System Design Guide — explains how DAPs, APs, and the trace funnel fit together. Read after ADIv5 when the system-level picture stops making sense.
  7. Cortex Debug Connector specification — the connector PDFs from ARM's developer site. Authoritative for the 10-pin and 20-pin headers.
  8. Stuart Hughes's "How JTAG works" series on YouTube and his book chapters — practical, with worked examples on Xilinx FPGAs. Useful if your background is FPGA rather than MCU.
  9. BSDL files at vendor sites — every major vendor publishes BSDL for every package. ST: www.st.com → "BSDL"; NXP: www.nxp.com → "Boundary Scan Description Language"; Xilinx/AMD: per-device download. UrJTAG ships an extensive BSDL collection too.
  10. Joe Grand's JTAGulator — a hardware tool for finding JTAG pinouts on unknown boards. The Hackaday writeups around it are an excellent practical introduction to the protocol.
  11. Sister docs in this hub: debug_swd.html (the 2-wire reformulation, including the full SWD line protocol), debug_swo.html (single-wire trace and ITM), usb_cmsis_dap.html (the host-facing CMSIS-DAP framing), platform_pio.html (the RP2350 PIO programming model), arm_architecture.html (general M-profile programmer's model), arm_cortex_m33.html (M33 specifics, debug authentication, secure-debug), and the edevkit1 v1 design spec for how the debug subsystem is wired on the actual board.
