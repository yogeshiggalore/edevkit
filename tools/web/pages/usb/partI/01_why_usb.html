Why USB is the way it is

A short history, the one design choice that shapes everything else, and why the API looks the way it does once you know.

A 90-second history

USB was published in January 1996 by a consortium of seven companies — Compaq, DEC, IBM, Intel, Microsoft, NEC, and Nortel — to solve one problem: the back of a PC was a mess. Mice, keyboards, modems, scanners, joysticks, and printers each had their own connector, their own electrical spec, and frequently their own interrupt request line. Adding a new device often required cracking the case open and setting jumpers. Plug-and-play was a slogan, not a fact.

USB has been revised six times since 1996, almost always to add bandwidth. The descriptor model, transfer types, and control/bulk/interrupt/isochronous categories have barely changed. Code you wrote for a USB 2.0 CDC device in 2002 still enumerates on a USB-C port on a 2025 laptop. Each new revision layered on top of the previous one rather than replacing it: a USB4 host still negotiates down to USB 2.0 for legacy devices, and a USB 3.0 device still has a USB 2.0 fallback path on the same connector.

Naming has been renamed twice — same connector, three names

USB 3.0 was retroactively renamed "USB 3.1 Gen 1" when 3.1 was published in 2013, then renamed again to "USB 3.2 Gen 1×1" when 3.2 came out in 2017. All three names refer to the same 5 Gbps single-lane SuperSpeed link. Since the 2022 branding refresh the official user-facing name is just USB 5Gbps and the Gen-numbers are deprecated. When vendor docs say "USB 3.x", mentally substitute "any version of SuperSpeed" — it almost never matters which sub-revision.

What changed in SuperSpeed (USB 3.x), and what didn't

SuperSpeed is the biggest architectural break in USB's history. We don't implement any of it on edevkit1 (the RP2350 has no SS PHY), but it's worth knowing what's different because it changes what "USB" means on a modern host:

  • A second physical lane, in parallel. SuperSpeed adds a TX/RX differential-pair set alongside D+/D-, not in place of them. A USB 3.0 cable carries VBUS, GND, D+, D-, plus SS_TX± and SS_RX±. A USB 2.0 device plugged into a SS port simply uses the original four wires; a USB 3.0 device uses both sets simultaneously and the host links to it at the higher speed.
  • Full duplex. USB 2.0 is half-duplex on a single shared pair — host and device take turns. SuperSpeed has dedicated TX and RX pairs, so host and device can transmit at the same time. Effective bidirectional throughput is roughly doubled even before counting the raw speed bump.
  • Polling goes away. The biggest departure from the model in this chapter. In USB 2.0 the host polls each endpoint every frame and an idle device replies NAK over and over (Wireshark traces are full of them). In SuperSpeed the device sends an Endpoint Ready notification when it has data, and the host services it on demand. No NAK storms, far less idle bus traffic, and devices can sleep through long idle periods. The "host is a polite tyrant" mental model softens here — still host-led, but conversational rather than interrogative.
  • Explicit link power states (U0–U3). Instead of USB 2.0's binary "active vs suspend", SuperSpeed has four negotiated link states: U0 (active), U1 (standby, fast wake), U2 (sleep, slower wake), U3 (suspend). Hosts and devices step between them automatically — drives can spin down between transfers without requiring an OS-level suspend.
  • Different line coding. 8b/10b at 5 Gbps on USB 3.0 (so ~4 Gbps of payload after encoding overhead). 128b/132b at 10 Gbps on USB 3.1 Gen 2 (~9.7 Gbps payload). Encoding efficiency improves at each generation while keeping enough transitions for clock recovery.
  • Separate PHY required on each end. A SuperSpeed host or device needs an additional SerDes — you can't bit-bang it out of a regular MCU peripheral or even a PIO state machine. RP2350 has no SS PHY, which is why edevkit1 is firmly USB 2.0 full-speed and our highest expectation is 12 Mbps on the wire.
  • What stayed the same. Descriptors, the four transfer types, class codes, VID/PID matching, the enumeration sequence — all unchanged. A SuperSpeed-capable device has the same descriptor tree as its USB 2.0 ancestor, with one extra SuperSpeed Endpoint Companion descriptor per endpoint that adds fields like bMaxBurstSize. This is why the rest of this book applies regardless of which USB speed your device ends up negotiating.
Scope of this book

The edevkit1 hardware is RP2350B, which exposes a USB 2.0 full-speed peripheral (12 Mbps). Everything in Parts I–VII applies to FS and HS identically. SuperSpeed-specific topics (the second lane, U-states, Endpoint Ready notifications, the SS Endpoint Companion descriptor) we cover only to the depth of the bullets above — relevant for understanding modern hosts, not for implementing on RP2350.

The one design choice that shapes everything else

USB is host-master polled. There is exactly one host on a bus, and the host initiates every single transaction. Devices never speak unless asked. Even an interrupt-style transfer — say, a HID keyboard reporting a key-press — works by the host periodically asking "do you have data for me?" and the device answering yes/no.

This is the opposite of how USB feels when you write firmware for it. You think you're writing "send a key code." You're really filling a buffer that the next time the host polls, will be drained. The host's polling cadence determines your latency, not your code.

Compare this to the protocols you already know:

That last column is the one that drove almost every USB design decision. If you want a user to plug a device in mid-session and have it work without reboots, drivers, or address conflicts, you need:

  • A way to detect a device appeared (electrical signaling on D+/D-).
  • A way to assign it an address dynamically, without conflict (host hands one out).
  • A way for the device to describe itself (descriptors).
  • A way for the OS to figure out which driver to load (class codes + VID/PID).
  • A way for the host to police the bus so a misbehaving device cannot wedge it (polled — devices can't initiate).

Every USB descriptor, every standard control request, every "why is this so verbose?" moment in your firmware traces back to that list.

The host polling loop

The host doesn't poll continuously — that would saturate the bus. Instead, time on the wire is sliced into frames:

  • Full-speed (12 Mbps): 1 ms per frame. Frame number increments. Each frame begins with a Start-of-Frame (SOF) packet.
  • High-speed (480 Mbps): still 1 ms frames, but each is divided into 8 microframes of 125 µs.
  • SuperSpeed (5 Gbps+): different structure — bus intervals replace frames. Out of scope here.

Within a frame, the host schedules transactions in priority order: control transfers and isochronous endpoints get reserved bandwidth first, interrupt endpoints get scheduled at their requested polling interval, then bulk transfers fight for whatever's left. This is why a misbehaving device can't starve a HID keyboard — the keyboard's interrupt slot is reserved.

Three things to notice in that picture:

  1. Every transaction is host-led. Even when the device "sends" data on an IN endpoint, the host sent the IN token first; the device merely answered.
  2. NAK is normal, not an error. "I don't have data for you yet" is a perfectly valid response and the host will simply ask again next frame. (Hard errors are different: a USB host retries a failed transmission up to 3 times before reporting the failure to client software — §4.5.2.)
  3. Multiple endpoints share one frame. A single device can have a dozen logical "channels" (endpoints) interleaved across frames. We'll cover endpoint addressing in Chapter 06.

Why this design pays off

Hot-plug works

Because the host owns the bus, it owns address assignment. New device shows up → host detects it (D+ or D- pull-up appearing), runs through enumeration, hands it an address from a pool. No driver-side coordination, no conflict, no reboot. We'll walk byte-by-byte through enumeration in Chapter 05.

Power management works

Devices declare their power needs (bMaxPower in the configuration descriptor). The host can refuse to enable a configuration that exceeds the budget. The host can also tell a device to suspend (cut to ≤ 2.5 mA / 500 µA depending on era), and the device has to obey. The user pulling out a phone charger from a hub doesn't crash the rest of the bus.

Driver matching works

Class codes (e.g. 0x03 = HID, 0x09 = Hub, 0x0A = CDC-Data) are part of the descriptor tree. The OS reads the descriptors during enumeration and matches against the class-driver registry. A vendor-specific device can additionally pin itself to a specific driver via VID/PID in the device descriptor or, on Windows, via Microsoft OS 2.0 descriptors (Chapter 09). Either way, the OS doesn't need to ask the device "what are you?" — the device already told it.

One bus, 127 devices, seven tiers

Address field is 7 bits, so 127 devices can be on a single bus tree (address 0 is reserved for "newly attached, not yet addressed"). Hubs are themselves USB devices that proxy traffic — the host sees through them. From a firmware author's perspective, the hub topology is invisible: you write code as if you're the only device.

Two architectural limits are worth knowing (USB 2.0 §4.1.1):

  • Maximum 7 tiers including the root — i.e. up to 5 chained hubs between the host and the deepest device. Beyond that, propagation delays violate the protocol's timing budget.
  • Compound devices occupy two tiers (an embedded hub plus its function on one cable), so they cannot be enabled if attached at tier 7. Only standalone functions can live at the bottom of the tree.

Mixed-speed devices don't slow the bus

A USB 2.0 high-speed hub contains a Transaction Translator (TT). When a low- or full-speed device is plugged into an HS hub, the host talks to the hub at 480 Mbps and the hub talks to the slow device at its native rate. Slow traffic stays "behind" the hub instead of monopolizing the bus all the way up to the host. From the device-firmware perspective this is invisible — you don't know whether you're connected directly or through a TT — but knowing the mechanism explains why a USB 1.1 mouse plugged into a modern laptop doesn't kill HS throughput on the same root port. (For a real-world failure mode TTs cause, see appendix → troubleshooting.)

What this design makes hard

Latency floor

If you want a device to react to an external event in < 1 ms, USB FS is the wrong protocol. The host won't poll your interrupt endpoint more often than once per frame. HS gets you down to 125 µs, but it's still floored by the host's microframe scheduler. For sub-microsecond reactions, you need an out-of-band signal (GPIO, SWO trace, dedicated wire). edevkit1's CMSIS-DAP probe is fine — debug-step latencies are dominated by SWD timing, not USB. A real-time control loop running over USB CDC, however, will disappoint you.

Peer-to-peer is impossible

Two USB devices cannot talk to each other directly. They must talk through a host. USB On-The-Go (OTG) and USB Dual-Role-Device (DRD) work by letting one party become the host temporarily, but at any given moment there is still exactly one host. The RP2350 used here is device-only; it cannot host. (USB hosts are doable on RP2040/RP2350 with PIO — see pico-pio-usb — but we're not building one.)

Verbosity

A bare-minimum USB device that does nothing useful still has to: respond to enumeration, expose a device descriptor (18 bytes), a configuration descriptor with at least one interface (≥ 18 bytes more), at least three string descriptors (manufacturer, product, serial), and answer standard control requests. That's a few hundred bytes of read-only structure before your first byte of payload. SPI, by contrast, is "wiggle these pins."

Driver requirements

An I²C sensor works the moment its data sheet's MCU-side driver is implemented. A USB device may need a host-side driver to be loaded — and on Windows, that historically meant signed .inf files. Microsoft OS 2.0 descriptors (Chapter 23) let modern devices skip the .inf by binding to WinUSB at enumeration time, but the friction is real.

How to think about USB going forward

The mental model that helps most:

The host is a polite tyrant. It owns the bus, runs the schedule, hands out addresses, polices misbehavior. Your device is a well-described mailbox the host visits on a fixed schedule. Most of "USB programming" is making sure your descriptors describe yourself accurately enough that the host knows when and how to visit, and that your endpoint buffers are ready when it arrives.

Once that frame is in your head, the API stops looking weird. Of course you have to declare your maximum packet size — the host needs to know how much to ask for. Of course you can't just push data out — the host has to ask first. Of course descriptors are static — the host caches them.

Concretely, every chapter from here on builds on this foundation:

  • Chapter 02 goes one layer down: the wire, the encoding, what D+/D- are doing.
  • Chapter 03 goes one layer up: the packets that ride that wire, and the framing rules.
  • Chapter 04 covers the four transfer types and which problem each is designed for.
  • Chapter 05 walks the full enumeration sequence byte by byte — the moment your device first announces itself to the world. By the time you finish Chapter 05, you'll have usb_01_hello_enum running and recognize every descriptor byte the host is reading from your firmware.

Common pitfalls

Symptom → Fix

"My device sends data but the host never sees it." You can't send. You can only have data ready when polled. If your code is conceptually usb_send(buf) and you expect the bytes to leave immediately, you'll be confused by latency that depends on poll interval and bus load. Re-frame: usb_buffer_for_next_poll(buf).

Symptom → Fix

"I see a flood of NAKs in Wireshark and assume it's broken." NAK is the normal "no data this poll, try again later" response on IN endpoints. Wireshark just shows them more visibly than other captures. Look at the rate, not the presence — sustained 100% NAK rate on an endpoint that should have data is a bug; intermittent NAKs are correct behavior.

Symptom → Fix

"I expected 1 ms latency on a CDC port and got 8 ms." Bulk endpoints don't have guaranteed scheduling. The host may aggregate transfers and only get to your bulk-IN once it has a free slot. If you need bounded latency, use an interrupt endpoint and accept its lower throughput. We'll quantify this in Chapter 04.
