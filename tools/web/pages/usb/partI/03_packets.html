The packet — the smallest unit USB cares about

Every USB transmission, from "hi I'm here" to "here are 64 bytes of CDC data", is a packet. Every packet has the same outer shape:

┌──────┬──────┬─────────────────────────┬──────────┬─────┐
│ SYNC │ PID  │  payload (varies)       │   CRC    │ EOP │
└──────┴──────┴─────────────────────────┴──────────┴─────┘
   8b      8b           0..N bytes           5b/16b
  • SYNC — a fixed bit pattern (KJKJKJKK at FS, longer at HS) the receiver locks onto for clock recovery.
  • PID — Packet Identifier, the most important byte in USB. Tells the receiver what kind of packet this is. We dissect it in a moment.
  • Payload — empty for handshakes, addresses for tokens, actual data for DATA packets.
  • CRC — 5 bits for tokens, 16 bits for DATA. Handshakes have no CRC.
  • EOP — End-of-Packet. The SE0-then-J pattern from Chapter 02.

SYNC, CRC, and EOP are handled by your USB peripheral hardware. From firmware, you see PID and payload. We focus on those.

Bit/byte ordering on the wire

One small fact that bites everyone the first time they read a USB capture: USB transmits least-significant bit first within each byte (USB 2.0 §8.1). Multi-byte fields are little-endian. CRCs are transmitted MSb-first after inversion. Your peripheral hardware and your USB analyzer both handle this for you, so byte-level firmware sees normal LE bytes; but if you ever scope D+/D- and try to decode bits by hand, the bit order is reversed from how the byte prints.

The packet payload is also bit-stuffed on the wire — after 6 consecutive 1-bits the transmitter inserts a 0, which the receiver removes. This affects only the wire timing (the packet is slightly longer than the payload would suggest); it never reaches firmware. Chapter 02 covers the encoding details.

The PID byte

The PID is 8 bits but encodes only 4 bits of information. The lower 4 bits are the type code; the upper 4 bits are the bitwise complement of the lower 4. This redundancy lets the receiver detect a corrupted PID without consulting the CRC — handy because the rest of the packet's interpretation depends on the PID.

Anatomy of each packet family

Token packets — fixed 3-byte payload

After the SYNC + PID byte, a token carries 11 bits of address fields plus 5 bits of CRC:

SYNC | PID(IN/OUT/SETUP) | ADDR (7) | ENDP (4) | CRC5 (5) | EOP
                          └─────────11 bits──────┘

ADDR is the device address (1..127, with 0 for "newly attached"). ENDP is the endpoint number within that device (0..15). Together they uniquely target one buffer pair on one device. The 5-bit CRC covers ADDR + ENDP.

SOF tokens replace ADDR+ENDP with an 11-bit frame number that increments each frame. The frame number rolls over at 2048. Devices that care (audio class, mostly) use it to align their buffers.

Data packets — variable payload

SYNC | PID(DATA0/1/2/M) | data (0..N bytes) | CRC16 | EOP

Payload size is bounded by the endpoint's wMaxPacketSize (8/16/32/64 for FS, up to 512 for HS bulk, 1024 for HS iso). The CRC16 covers the data bytes only.

The DATA0/DATA1 toggle is the part most worth understanding because it explains a real failure mode. When the host sends OUT data to a device:

  1. Host sends OUT token, then DATA0 with payload.
  2. Device buffers it, replies ACK.
  3. Host sees ACK, flips its toggle to DATA1 for the next packet.
  4. Device sees DATA1, accepts it (its expected toggle is also DATA1 now), replies ACK.

If the ACK is corrupted in transit, the host doesn't know whether the device accepted the data. So the host re-sends the same DATA0 packet. The device sees DATA0 when it was expecting DATA1 — it knows this is a duplicate, drops it, and ACKs again. Without the toggle, the device would deliver duplicate bytes upward whenever an ACK got lost.

When toggles bite

If your firmware ever drops out of sync with the host on the toggle bit (e.g., an endpoint reset that doesn't reset the toggle), the host and device disagree on what's a duplicate. Symptoms: every other packet appears lost. The fix is the standard control request CLEAR_FEATURE(ENDPOINT_HALT), which both clears any STALL and resets the toggle to DATA0. We meet this in Chapter 08.

Handshake packets — PID-only

A handshake is just a SYNC + PID + EOP — no payload, no CRC.

SYNC | PID(ACK/NAK/STALL/NYET) | EOP

Three of the four are routine:

  • ACK — the receiver got a valid CRC-passing packet. Issued by host on OUT, by device on IN. Either side may issue it.
  • NAK — issued by device only. "I can't accept/send right now, ask again." The host treats this as not-an-error and reschedules.
  • STALL — issued by device. "This endpoint is in error state; I won't transfer until you clear me." The host stops scheduling that endpoint. Recovery: CLEAR_FEATURE(ENDPOINT_HALT) on endpoint 0. Control endpoint 0 stalling is special — it's "I don't support that request" and self-clears with the next SETUP packet.
  • NYET — HS only. PING/NYET protocol lets a host pre-poll to avoid wasting bandwidth on a NAK.

PING scope. The ping protocol applies only to HS bulk and control endpoints during the data and status stages — never during the SETUP stage of a control transfer. The host pings the device with a special PING token; the device replies ACK ("ready") or NAK ("not yet"). Knowing this scope prevents misreading captures where SETUP transactions appear to skip the ping (USB 2.0 §8.5.1).

Transactions — packets in pairs and triples

A USB transaction is the smallest meaningful host-device exchange. There are three transaction shapes, one for each direction:

OUT transaction (host → device)

sequenceDiagram
  participant H as Host
  participant D as Device
  H->>D: OUT token (addr=N, ep=2)
  H->>D: DATA0/1 (payload, CRC16)
  alt receiver OK
    D-->>H: ACK
  else device buffer full
    D-->>H: NAK
  else endpoint halted
    D-->>H: STALL
  end

IN transaction (device → host)

sequenceDiagram
  participant H as Host
  participant D as Device
  H->>D: IN token (addr=N, ep=3)
  alt device has data
    D-->>H: DATA0/1 (payload, CRC16)
    H->>D: ACK
  else device empty
    D-->>H: NAK
  else endpoint halted
    D-->>H: STALL
  end

SETUP transaction (the only one that's special)

sequenceDiagram
  participant H as Host
  participant D as Device
  H->>D: SETUP token (addr=N, ep=0)
  H->>D: DATA0 (8-byte SETUP packet, CRC16)
  D-->>H: ACK (mandatory — devices may NOT NAK or STALL a SETUP)

That last sequence diagram contains the single most important rule in USB protocol design: a device must always ACK a SETUP packet. It cannot NAK (would let a confused device wedge enumeration) or STALL (would prevent the host from clearing the stall via control endpoint 0). The device may then NAK or STALL subsequent data/status stages of the control transfer, but the SETUP itself must always land.

One exception worth knowing: if the SETUP token itself is corrupted (PID-check fails or CRC fails), the device sends no response at all — neither ACK nor STALL. The host sees the timeout and retries (USB 2.0 §8.4.6.4 / §8.7). The "always-ACK" rule applies only to well-formed SETUP packets that the device successfully decoded.

The 8-byte SETUP packet inside that DATA0 is the structure that drives every standard USB request:

Frames and microframes

Tokens, data, and handshakes are all individual packets. Frames group them on a 1 ms (FS) or 125 µs (HS microframe) cadence. The host emits an SOF token at the start of every frame, then schedules transactions in priority order (control + iso first, interrupt next, bulk last).

┌────────────── 1 ms (FS frame) ──────────────────────────────┐
│ SOF │ ctrl xact │ iso xact │ int xact │ bulk xact │ idle  ...│
└─────┴───────────┴──────────┴──────────┴───────────┴──────────┘
                                                  ↑
                                           SOF (next frame)

Two consequences for firmware:

  • Your interrupt endpoint with bInterval = 4 means "poll me at most every 4 frames" — your data has up to a 4 ms freshness window.
  • Bulk transfers fight for leftover bandwidth. On a quiet bus, a bulk OUT packet may be delivered within microseconds of the host's queue. On a busy bus, it can wait many frames.

Microframes (HS only) divide each 1 ms frame into 8 slices of 125 µs. A high-speed interrupt endpoint can be polled up to once per microframe; high-speed isochronous can pack three packets per microframe (the DATA2/MDATA PIDs are part of that mechanism).

Bus turn-around timing

Every transaction has a bounded "turn-around" window — the time the receiver gets to respond before the sender treats the transaction as failed. USB 2.0 §7.1.18 / §8.7.2 set the limits:

  • Full-speed bus turn-around must complete within ~7.5 bit-times after the EOP of the previous packet; the receiver may take longer to start driving but the response must finish within the host's timeout.
  • High-speed turn-around is much tighter, ~736 ns from EOP to the start of the response.

Practically: if your firmware can't have the IN-endpoint buffer ready by the time the host's IN token arrives, the right answer is to NAK promptly — not to delay the bus. Slow ACKs/responses past the turn-around window are treated as failed transactions and trigger the 3-retry recovery from Chapter 01.

CRCs — what they protect, what they don't

Two CRCs at the protocol layer:

  • CRC5 on tokens — covers the 11 bits of ADDR/ENDP (or frame number for SOF). Polynomial 0x05 = x⁵ + x² + 1. A failing CRC5 means the host's address field corrupted; the device must drop the packet silently. The host will retry up to 3 times.
  • CRC16 on DATA payloads — covers the data bytes only (not PID). Polynomial 0x8005 = x¹⁶ + x¹⁵ + x² + 1. A failing CRC16 means the receiver does not ACK; the sender retransmits.

The protocol layer does not prevent logical errors — wrong descriptor, wrong endpoint, wrong wLength. Those are the application's problem, surfaced through the standard request layer (Chapter 08).

Common pitfalls

Symptom → Fix

"Wireshark shows lots of NAKs and I think the device is broken." NAK is correct behavior whenever a device hasn't filled its IN buffer or can't accept an OUT yet. Look at the rate over time, not the presence. Sustained 100% NAK on an endpoint that should have data means your firmware isn't writing to that endpoint's buffer.

Symptom → Fix

"Every other bulk transfer disappears." Data toggle is desynced. After certain endpoint resets, software may forget to re-zero the toggle. Check your USB stack's reset path; in Zephyr USB-Device-Next, this is normally handled but custom class drivers can break it.

Symptom → Fix

"Endpoint stalled and I can't recover." Stalls on endpoint 0 are self-clearing on the next SETUP. Stalls on other endpoints need an explicit CLEAR_FEATURE(ENDPOINT_HALT) from the host. If your device is stuck STALL'ing endpoint 0 specifically, the bug is in the SETUP handler — log every SETUP packet that arrives.

Symptom → Fix

"My firmware sees a SETUP it didn't expect to handle and the host hangs." Devices must always ACK SETUP. If you can't service the request (unsupported descriptor, e.g.), you stall the data stage by writing a STALL response on endpoint 0's IN/OUT — not by dropping the SETUP. The next SETUP will clear the stall automatically.
