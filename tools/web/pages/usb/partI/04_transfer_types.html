Four transfer types — overview

USB defines exactly four transfer types. Every endpoint on every device must be exactly one of them. Each was designed to solve a specific class of data-flow problem, and the host's bandwidth allocator treats them differently.

Three of the four (control, bulk, interrupt) are functionally identical in their error handling — they CRC, ACK, and retransmit. They differ in scheduling: how much bandwidth the host reserves for them, and how often it polls. Isochronous is the odd one out — no retries, no ACKs, no NAKs.

Control transfers

Endpoint 0 is always a control endpoint. It's the only one guaranteed to exist on every USB device. Almost every standard interaction — enumeration, descriptor reads, configuration changes — happens here.

A control transfer is structured, not freeform:

sequenceDiagram
  participant H as Host
  participant D as Device
  Note over H,D: Setup stage (always)
  H->>D: SETUP token + 8-byte SETUP packet
  D-->>H: ACK
  Note over H,D: Data stage (optional, IN or OUT)
  H->>D: IN/OUT tokens + DATA0/1 packets
  D-->>H: ACK (per packet)
  Note over H,D: Status stage (always, opposite direction of data)
  H->>D: OUT/IN token + zero-length DATA1
  D-->>H: ACK

Three stages — setup, optional data, status — make up one logical operation. The data stage carries the descriptor bytes or the request payload. The status stage is a zero-length packet in the opposite direction; its purpose is for the receiver to confirm "yes I'm done with this transfer." If the data stage was OUT, status is IN, and vice versa.

The spec caps periodic transfers (iso + interrupt) at 90% of each FS frame and 80% of each HS microframe (USB 2.0 §5.6.4 / §5.7.4). The remaining ≥10% (FS) / ≥20% (HS) is shared between control and bulk, with control prioritized — so control transfers always make progress even when bulk traffic is saturating the leftover. This is what makes control the right choice for "configure the device": it's never fully starved.

Limitations: max packet size on EP0 is 8 (LS), 8/16/32/64 (FS), 64 (HS). For practical purposes a control transfer caps at one descriptor's worth of bytes — a few hundred. They're not for moving data.

Bulk transfers

Bulk endpoints carry the largest payloads USB supports and are error-checked, but the host gives them no scheduling guarantees. They occupy whatever bandwidth is left after control, interrupt, and isochronous transfers have had their turn.

Properties:

  • Max packet size: 8/16/32/64 for FS, 512 for HS, 1024 for SS.
  • Error-checked (CRC + retransmit).
  • NAK on no-data; the host reschedules.
  • Half-duplex per endpoint (one direction per endpoint number).
  • Latency is whatever the host decides, which depends on bus contention.

Bulk is the workhorse of high-throughput classes: CDC ACM data pipes, mass storage, debug probes (CMSIS-DAP), printer class, and any vendor-specific high-volume transport. It is what edevkit1's CMSIS-DAP and SWO trace endpoints will use.

The bandwidth math

On full-speed (12 Mbps), the theoretical maximum bulk throughput is around 1216 KB/s per endpoint, per direction — that is, 19 packets × 64 bytes per 1 ms frame (USB 2.0 §5.8.4 table 5-9). IN and OUT count as separate endpoints; each one gets its own 1216 KB/s budget independently. In practice you see 800–1200 KB/s in any one direction depending on the host's scheduler, the other endpoints sharing the bus, and your USB stack's per-transfer overhead. ~10 Mbps usable total is the rule-of-thumb most edevkit firmware should plan against.

HS (480 Mbps) bulk routinely hits 35–40 MB/s, but RP2350's PHY is FS-only.

RP2350 MPS limits — silicon-level

The RP2350 datasheet §12.7.1.1.1 bounds these per-endpoint: control 8 / 16 / 32 / 64 bytes, bulk and interrupt up to 64 bytes, isochronous up to 1023 bytes per packet. Each non-iso endpoint occupies a 64-byte buffer in the 4 KB DPRAM; iso endpoints can be 128 / 256 / 512 / 1024 bytes via the buffer-control register's offset field. Ch 14 §3c opens up the DPRAM layout.

Interrupt transfers

"Interrupt" here is misleading — there's no actual interrupt mechanism. It's just a polled endpoint where the host promises to poll at least every N frames. That's what makes it good for low-latency event delivery: a button press, a HID mouse move, a CDC ring-state notification.

Properties:

  • Max packet size: any value 1..8 (LS), 1..64 (FS), 1..1024 (HS) — interrupt endpoints can choose any size in range, not just powers of two (USB 2.0 §5.7.3).
  • Error-checked (CRC + retransmit).
  • Polled at bInterval frames (FS/LS) or 2^(bInterval-1) microframes (HS).
  • Reserved bandwidth — the host won't let bulk traffic crowd it out.
  • Half-duplex per endpoint.
  • HS high-bandwidth mode (§5.9.1): an HS interrupt endpoint can declare 2 or 3 transactions per microframe, raising its peak to 24 Mbps. Rare; only used by very-high-rate input devices.

bInterval semantics differ by speed and transfer type. For FS interrupt endpoints, bInterval is in frames (1 ms units) and may be 1..255. For HS interrupt endpoints, it's 2^(bInterval-1) microframes, so bInterval=1 means 125 µs, bInterval=4 means 1 ms, etc. This trips up everyone the first time.

Interrupt endpoints aren't for high throughput — at FS with bInterval=1 and 64-byte MPS, you're capped at ~64 KB/s. They're for "this short message must arrive within Δt of being ready."

Isochronous transfers

Isochronous is the only USB transfer type with guaranteed bandwidth and no error correction. It's designed for real-time streaming where missing a deadline is worse than missing a byte: audio, video, USB-Audio (UAC), USB-Video (UVC).

Properties:

  • Max packet size: up to 1023 bytes (FS), 1024 (HS), 3 × 1024 with high-bandwidth (HS).
  • No CRC verification at the application layer — packets arrive even if corrupted. (CRC16 is still on the wire; the host signals corruption but doesn't retransmit.)
  • No ACK/NAK — packets are fire-and-forget.
  • Reserved bandwidth — slot is held even if you don't fill it.
  • One iso packet per (micro)frame typically, though HS supports up to 3.

Iso reservations bite: an iso endpoint with a 1024-byte MPS reserves 1024 bytes of bandwidth every microframe whether or not you have data. Don't open one unless you're going to keep it fed. We touch on iso briefly in usb_26_iso_audio_lite for completeness; it's not part of the CMSIS-DAP / OTA path.

Endpoint zero — a special case

Endpoint 0 is bidirectional (both IN and OUT) and always control. Every device has exactly one EP0. It's where descriptors come from, where standard requests arrive, and where stalls clear automatically. Other endpoint numbers are unidirectional — endpoint 1 IN and endpoint 1 OUT are two separate logical endpoints.

The maximum packet size of EP0 is in the device descriptor (bMaxPacketSize0), and you must answer GET_DESCRIPTOR(device) with the first 8 bytes of that descriptor before the host has any other context — making EP0's MPS field one of the few that actually matters at the byte level. We meet that wrinkle in Chapter 05.

Pipes — the host-side abstraction

One piece of vocabulary the spec uses heavily: a pipe is a logical connection between host software and one device endpoint. Each device endpoint, once configured, exposes a pipe. There are two flavors (USB 2.0 §5.3.2):

  • Message pipe — bidirectional, structured, used only for control transfers. The default control pipe on EP0 is the one every device has.
  • Stream pipe — unidirectional, unstructured. Bulk, interrupt, and isochronous endpoints expose stream pipes. Bytes flow through them with no protocol-level framing.

Endpoint = the device-side resource (buffer + handler). Pipe = the host-side handle that talks to it. They're often used interchangeably, but the spec is precise; if a tutorial says "stream pipe" you now know it means "any non-control endpoint."

Bandwidth budgeting

The host's scheduler allocates bandwidth in this priority order, every (micro)frame:

  1. Isochronous + interrupt (the "periodic" group) — up to 90% per FS frame or 80% per HS microframe. Each iso/int endpoint reserves its required bytes per its bInterval.
  2. Control transfers — share the remaining ≥10% (FS) / ≥20% (HS) with bulk; control is prioritized so it never starves.
  3. Bulk transfers — whatever's left after #1 and #2.

If you stuff a device with a bunch of iso/interrupt endpoints with aggressive intervals, bulk can starve. Conversely, if you only have bulk endpoints, you'll get the full 80–90% of bandwidth. edevkit1's plan — two CDC bulk pairs + two vendor bulk endpoints — is bandwidth-friendly precisely because nothing reserves bandwidth.

Split transactions — when LS/FS lives behind an HS hub

USB 2.0 §5.10 defines split transactions: when an HS host wants to talk to a low- or full-speed device that is plugged into an HS hub, the HS host sends a SSPLIT token at HS to the hub, the hub does the real LS/FS transaction with the device at slow speed, and the host later issues CSPLIT at HS to collect the result. The hub's Transaction Translator (TT) hides the speed mismatch.

From the device's perspective, this is invisible — the device just sees normal LS/FS traffic. From the firmware-author's perspective, it matters in two cases: (a) Wireshark captures behind a TT include extra SSPLIT/CSPLIT packets that look unfamiliar; (b) some hubs have buggy TTs that mishandle iso splits — if your iso endpoint works through one host but not another, suspect the hub. edevkit1 (FS-only) will encounter this whenever it's plugged into a USB-C hub with HS uplink.

For 90% of devices, the answer is "control on EP0 for setup, bulk for data." That's CDC, MSC, CMSIS-DAP, and most vendor classes. Reach for interrupt only when you have a specific latency requirement (HID input, CDC notification endpoint). Reach for isochronous only when you're doing audio or video.

NAK semantics by transfer type

How the host reacts to a NAK varies, and it's worth knowing because it explains observed throughput:

TypeNAK reaction
ControlHost retries the same transaction within the current transfer. Many NAKs ⇒ stall the transfer eventually.
BulkHost reschedules the transaction in a future frame. Sustained NAK = no progress, but no error.
InterruptHost waits until the next polling interval, then asks again.
IsochronousN/A — iso has no ACK/NAK.

NAK does not consume bandwidth wastefully on bulk endpoints — the host learns quickly to back off. This is why a NAK-y bulk endpoint mostly hurts itself, not the rest of the device.

Worked example: edevkit1's planned topology

Looking ahead to Tier 3 (sample usb_30_dap_skeleton in spirit, before we descope), edevkit1's USB shape will be:

  • EP 0 IN/OUT — control, MPS 64. Standard.
  • EP 1 OUT, EP 1 IN — vendor bulk, MPS 64. CMSIS-DAP command/response (later, not in the USB book).
  • EP 2 IN — vendor bulk, MPS 64. SWO trace stream.
  • EP 3 OUT, EP 3 IN, EP 3 INT — CDC ACM. Target log UART.
  • EP 4 OUT, EP 4 IN, EP 4 INT — CDC ACM. Test UART.

Five bulk endpoints, two interrupt endpoints (CDC notify), no iso. Worst-case interrupt reservation: 2 × 64 bytes per polling interval. With bInterval = 16 (typical for CDC notify), that's 8 bytes/ms reserved — negligible. Everything else competes for bulk bandwidth, and we'll engineer back-pressure so SWO can't starve CMSIS-DAP commands. That's a Part VI / Tier 3 problem.

Common pitfalls

Symptom → Fix

"My HS interrupt endpoint polls at 1 ms but I asked for bInterval=1." HS interpretation is 2^(bInterval-1) × 125 µs, so bInterval=1 means 125 µs. Use bInterval=4 for 1 ms on HS.

Symptom → Fix

"I declared an iso endpoint but my bulk throughput tanked." Iso reserves bandwidth even when idle. The host preallocates the slot you asked for. Drop unused iso endpoints from the configuration.

Symptom → Fix

"Used interrupt for high-throughput because I needed low latency, but my throughput is terrible." Interrupt is throughput-poor by design — it's there for periodic small messages, not bulk pipes with low latency. If you need both, the answer is to use bulk and make sure your firmware writes to the IN buffer eagerly so the host's first poll succeeds.

Symptom → Fix

"Two bulk endpoints, one is hot, one is starved." Linux's xHCI driver in particular interleaves polls by endpoint, but some hosts walk endpoints in numeric order. If endpoint 1 always has data, endpoint 2 may rarely be polled. Solution: don't oversubscribe; make your firmware NAK quickly when EP1 has nothing, so the host moves on.
