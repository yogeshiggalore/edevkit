The full picture

From the user's perspective, plugging a USB device in is instantaneous. From the wire's perspective it's about 20 packets and ~50 ms of activity, in this order:

sequenceDiagram
  participant U as User
  participant H as Host
  participant D as Device

  U->>D: Plugs in cable
  D->>H: Pulls D+ high (FS speed signaling)
  H->>D: Bus reset (SE0 ≥ 10 ms)

  Note over H,D: Phase 1 — discover EP0 packet size
  H->>D: GET_DESCRIPTOR(device, 8 bytes) on addr=0
  D-->>H: First 8 bytes of device descriptor

  H->>D: Bus reset (some hosts)

  Note over H,D: Phase 2 — assign address
  H->>D: SET_ADDRESS(N)
  D-->>H: ACK (still on addr=0)
  Note over D: Switch to addr=N for next request

  Note over H,D: Phase 3 — read full descriptor tree
  H->>D: GET_DESCRIPTOR(device, full 18 bytes) on addr=N
  D-->>H: Full device descriptor
  H->>D: GET_DESCRIPTOR(config, 9 bytes)
  D-->>H: Config descriptor header (gets wTotalLength)
  H->>D: GET_DESCRIPTOR(config, wTotalLength)
  D-->>H: Full config tree (config + interfaces + endpoints)
  H->>D: GET_DESCRIPTOR(string, 0)
  D-->>H: Supported language IDs
  H->>D: GET_DESCRIPTOR(string, iManufacturer/iProduct/iSerial)
  D-->>H: UTF-16LE strings

  Note over H,D: Phase 4 — activate
  H->>D: SET_CONFIGURATION(1)
  D-->>H: ACK

  Note over H,D: Class drivers take over from here

Four phases: discover EP0, assign address, read descriptors, activate configuration. We walk each in sequence.

Six device states (the lifecycle behind enumeration)

USB 2.0 §9.1.1 defines six visible device states. Enumeration is the journey through the first five; suspended is orthogonal and reachable from any of them.

StateTriggered byDevice behavior
AttachedCable plugged in, no VBUSPull-up not yet asserted
PoweredVBUS appliedDevice asserts speed pull-up; host detects attachment
DefaultBus reset completeDevice responds on EP0 at address 0
AddressSET_ADDRESS status stage ACK'dDevice responds only at its new address; not yet configured
ConfiguredSET_CONFIGURATION(n≥1)Endpoints active; class drivers can talk to the device
SuspendedBus idle > 3 ms (orthogonal)Drop to suspend current; resume signal returns to prior state

The four phases below correspond to the transitions Powered → Default → Address → Configured. SET_CONFIGURATION(0) deconfigures, sending the device back to Address state without losing its address.

Phase 0 — speed signaling and bus reset

Before any USB packet is exchanged, the device announces its speed by which D wire it pulls up (Chapter 02 recap):

  • D+ pulled to 3.3 V via 1.5 kΩ → device is full-speed.
  • D- pulled to 3.3 V via 1.5 kΩ → device is low-speed.
  • D+ pulled, then chirp during reset → device is high-speed.

The host sees the appropriate D-line rise and starts a bus reset: drive SE0 (both lines low) for ≥ 10 ms. During this time:

  • The device's USB controller resets itself.
  • Any previous device address is forgotten — the device returns to address 0.
  • Any previous configuration is forgotten — the device is "addressed but not configured" once reset clears.
  • Endpoint 0 is now the only active endpoint, MPS unknown to the host yet.

When SE0 ends, the device must be ready to respond to control requests on EP0 within 100 ms.

Phase 1 — discover EP0's packet size

The host's first move is a control read of the device descriptor — but with a surprising twist: it reads only the first 8 bytes on the first try, even though the full descriptor is 18.

Why? Because the host doesn't yet know the device's bMaxPacketSize0 (the EP0 maximum packet size). This field lives at offset 7 of the device descriptor. The host needs that value to know how big subsequent control transactions can be — but to read it, it has to make a control request, which itself needs a packet size assumption. The protocol resolves this circular dependency by mandating that every device must respond to a partial descriptor read of at least 8 bytes. After receiving those 8 bytes, the host knows the actual MPS and can do larger reads.

The host now knows EP0 is 64-byte. It also knows the full descriptor is 18 bytes (bLength = 0x12). Most hosts at this point issue a second bus reset — Linux usbcore in particular does this almost always. The spec doesn't strictly require it, but it's universal in practice.

Phase 2 — assign address

The device is still at address 0. The host issues SET_ADDRESS with the new address it picked from its pool:

SETUP byteHexFieldMeaning
000bmRequestTypeH2D, Standard, Device
105bRequestSET_ADDRESS
2-307 00wValuenew address = 7 (varies)
4-500 00wIndex—
6-700 00wLengthno data stage

This is a control transfer with no data stage — straight from setup to status. The device must answer the status stage on its old address (0), then switch to the new one for any subsequent transaction. Get this wrong and the device disappears from the host's view immediately after assignment.

In Zephyr USB-Device-Next, this is handled by the stack — your application code never sees SET_ADDRESS. But knowing it's there helps you debug a particular failure mode: if your USB controller doesn't switch addresses at the right moment (a hardware bug or driver bug), enumeration dies right here. Wireshark will show the host then trying to talk to address N and the device not responding.

On RP2350 specifically

The RP2350 USB controller does not auto-parse SET_ADDRESS — firmware (Zephyr's udc_rpi_pico.c) reads the SETUP packet from DPRAM at 0x50100000 + 0x00, extracts the new address from byte 2 of the SETUP, and writes it to the ADDR_ENDP register at USBCTRL_REGS_BASE + 0x000 after the status stage completes. Datasheet §12.7.3.8.1. The "controller driver bug" failure mode is concretely: the driver reads the wrong byte, masks bits 6:0 incorrectly, or writes ADDR_ENDP too early. See Ch 14 §4d.

Three timing rules the spec pins down (USB 2.0 §9.2.6):

  • Status stage of SET_ADDRESS must complete within 50 ms of receiving the request.
  • The address change must be in effect within 2 ms after the status stage. Earlier or later breaks enumeration.
  • After a bus reset ends, the host gives the device a ≥ 10 ms recovery interval before sending the next request — your firmware doesn't have to be ready instantly, but it does have to be ready by then.

Phase 3 — read the full descriptor tree

Full device descriptor

Same SETUP as Phase 1 but now wLength = 18 (or sometimes 64 — many hosts ask for more than they need). The device replies with all 18 bytes:

VID/PID note

0x1209 is the pid.codes umbrella VID for open-source / hobbyist projects, and 0xede7 is a placeholder we'll replace before release. Chapters in Part II go deeper on VID/PID assignment for open-hardware projects.

Configuration descriptor — partial then full

Just like the device descriptor, the config descriptor is read in two passes. The first pass asks for 9 bytes (the size of the config descriptor header). That header contains wTotalLength, the size of the entire config tree (config + interfaces + endpoints + class-specific descriptors). The host then issues a second read for that exact length.

The reason for the two-pass dance: descriptor trees can vary from a few dozen bytes (a simple HID device) to several hundred (a complex composite). The host doesn't want to over-read or under-read.

For usb_01_hello_enum the config tree is minimal:

  • 1 × Configuration descriptor (9 bytes)
  • 1 × Interface descriptor (9 bytes)
  • 0 endpoints (the device declares an interface but doesn't use any data endpoints — it's just for visibility)

So wTotalLength = 18. We dissect every byte of the config tree in Chapter 06.

String descriptors

The host then reads three string descriptors (or four — many hosts ask for index 0 first to learn supported language IDs). UTF-16LE encoded, prefixed with a 2-byte header (length + type). For our sample:

  • Index 0 (language IDs) — typically 04 03 09 04 = 4-byte descriptor, type STRING, language 0x0409 (US English).
  • Index 1 (manufacturer) — "edevkit" → 18 bytes (2 header + 14 UTF-16LE chars).
  • Index 2 (product) — "usb_01 hello-enum" → varies.
  • Index 3 (serial) — derived from RP2350 chip ID, e.g. "EDV-1A2B-3C4D-5E6F".

Strings are optional. If a descriptor's iManufacturer / iProduct / iSerialNumber field is 0, that string is "not provided" and the host doesn't ask. usb_01 provides all three because system_profiler on macOS surfaces them in the GUI and seeing your text appear there is a satisfying first-victory milestone.

Phase 4 — activate the configuration

The final step is SET_CONFIGURATION:

SETUP byteHexFieldMeaning
000bmRequestTypeH2D, Standard, Device
109bRequestSET_CONFIGURATION
2-301 00wValueconfiguration index 1
4-700 00 00 00wIndex/wLength—

This is the moment the device transitions from addressed to configured. Endpoints declared in the configuration become active. Class drivers on the host side begin probing them. From the device firmware's perspective, this is when you start seeing class-specific traffic on EP1, EP2, etc.

For the trivial usb_01 sample, nothing class-specific happens after this — the device is simply visible in the OS's USB tree. Subsequent samples (CDC, HID, MSC, vendor) hook into this moment with their respective class drivers.

The eleven standard device requests

Enumeration uses just three: GET_DESCRIPTOR, SET_ADDRESS, SET_CONFIGURATION. The full set defined by USB 2.0 §9.4 is eleven, summarized here so you know what else might appear in a Wireshark capture once class drivers attach. Every device must implement these (or STALL them with a clear "I don't support that").

bRequestNameDirectionWhat it does
0x00GET_STATUSD2H2-byte status of device, interface, or endpoint (e.g., halt bit, remote-wakeup bit)
0x01CLEAR_FEATUREH2DClears a feature flag (typically ENDPOINT_HALT to recover from STALL, or DEVICE_REMOTE_WAKEUP)
0x03SET_FEATUREH2DSets a feature flag (e.g., enable remote wakeup, force endpoint halt)
0x05SET_ADDRESSH2DAssigns the bus address (Phase 2)
0x06GET_DESCRIPTORD2HReads a descriptor; high byte of wValue = type, low byte = index
0x07SET_DESCRIPTORH2DOptional; lets host overwrite a descriptor. Most devices STALL this.
0x08GET_CONFIGURATIOND2HReturns current bConfigurationValue (0 = unconfigured)
0x09SET_CONFIGURATIONH2DActivates a configuration (Phase 4); 0 deconfigures
0x0AGET_INTERFACED2HReturns the active alt-setting for an interface
0x0BSET_INTERFACEH2DSwitches an interface to a different alt-setting (covered in Ch 25)
0x0CSYNCH_FRAMED2HIso-only; reports frame number for clock alignment

Speed-dependent descriptors (HS only)

High-speed-capable devices must additionally support two descriptors that low- and full-speed devices don't have: DEVICE_QUALIFIER (0x06) and OTHER_SPEED_CONFIGURATION (0x07). They tell the host "if I were running at the other speed, here's what I'd look like." A full-speed-only device — like edevkit1 — must STALL these requests when the host asks (USB 2.0 §9.6.2, §9.2.7). Don't return zero bytes or empty descriptors; the spec is explicit that STALL is the correct response.

Request error — what STALL means at this layer

Per §9.2.7, "Request Error" is the catch-all device response for "I don't support that request, or its parameters are invalid." The device signals it by stalling the data or status stage of the control transfer (not by NAKing the SETUP, which is forbidden — see Ch 03). Hosts treat this as a soft failure and move on. So: unknown bRequest? STALL the data stage. Unsupported descriptor type? STALL. Unsupported feature selector? STALL.

Common pitfalls

Symptom → Fix

"Device enumerates as 'Unknown Device' on Windows but works on Linux." Almost always a malformed string descriptor (missing language IDs, wrong type byte, length mismatch). Linux is forgiving; Windows is strict. Check that string index 0 returns the language-ID descriptor (04 03 09 04), not a UTF-16LE string.

Symptom → Fix

"GET_DESCRIPTOR(device) works but GET_DESCRIPTOR(config) times out." Your wTotalLength field is wrong — the host asked for a longer descriptor than your firmware actually returns, so the IN endpoint never finishes. Recount: wTotalLength = sum of all bytes from the config descriptor through every interface and endpoint descriptor that follows.

Symptom → Fix

"Device disappears immediately after SET_ADDRESS." Address transition timing bug. The status stage of the SET_ADDRESS transfer must be ACK'd on address 0; the move to the new address happens after that ACK. Most stacks get this right; if you're writing a controller driver, this is the most common bug.

Symptom → Fix

"macOS doesn't show updated strings even after I rebuilt the firmware." macOS aggressively caches USB string descriptors keyed on (VID, PID, bcdDevice). Bump bcdDevice by 1 in your descriptor when iterating, or run sudo killall -HUP USBHostBSDClient.

Symptom → Fix

"Linux says 'device not accepting address' followed by repeated retries." Either your device didn't ACK the SET_ADDRESS status, or it didn't switch addresses correctly. Capture with usbmon and check whether the device is still responding on address 0 (failed transition) or on the new address (host bug, less likely).
