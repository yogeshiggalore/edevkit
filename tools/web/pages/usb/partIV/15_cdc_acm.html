CDC ACM — your first real I/O

Up to this point the device has only existed as a static descriptor tree (Ch 06–07) and a request-response talker on EP0 (Ch 08). CDC ACM is where USB finally moves bytes. It is also the first standard class we touch — the host doesn't need any custom driver, any .inf, any Zadig dance: it sees the device, looks up the class triple, loads cdc_acm / AppleUSBCDCACMData / usbser.sys, and presents the device to userspace as /dev/ttyACMx, /dev/cu.usbmodemXXXX, or COMx. From the firmware side it looks like a UART. From the host side it looks like a serial port. From the wire side it's a six-descriptor, two-interface, four-endpoint composite. This chapter is about that gap.

You'll learn
  • Why CDC has to be a composite class — and why every CDC ACM device begins with an Interface Association Descriptor (IAD).
  • The five CDC functional descriptors that sit between the control interface and its endpoint — Header, Call Management, ACM, Union — and what each one tells the host.
  • The four class-specific control requests that ride EP0: SET_LINE_CODING, GET_LINE_CODING, SET_CONTROL_LINE_STATE, SEND_BREAK — when they fire, what their byte layouts look like, and which ones a virtual serial port can safely ignore.
  • Why baud rate, parity, and stop bits are fiction on a CDC ACM device (and what to do anyway).
  • The interrupt-IN endpoint that nobody uses — SerialState notifications — and what they're for when somebody finally needs them.
  • The bulk pair that does the actual work, plus the ZLP rule that determines whether your transfers terminate cleanly.
  • Why CDC ACM is "full-duplex over a half-duplex wire" — three layers, three duplex states — and how the bulk pair plus the host scheduler rebuild the duplex abstraction the application sees.
  • The Full-Speed bandwidth budget: ~18 bulk transactions per frame, ~576 KB/s per direction simultaneous theoretical, and what UART baud rates that actually translates to on real stacks.
  • How each major OS finds, loads, and exposes a CDC ACM device — and the one Windows-7-era detail (usbser.inf) that's only just stopped mattering.

1. What CDC is — and why we even have a class for serial

By the late 1990s USB was clearly going to win, and the question facing the USB-IF was: how do you replace RS-232? Not the wires — those obviously went away — but the idea of a stream-of-bytes-with-some-control-lines that every operating system, programming language, and terminal program already knew how to talk to. The answer was the Communications Device Class (CDC), published as a USB-IF spec in 1999 and refined across multiple subclasses. The headline subclass — the one that maps "serial port" to "two USB interfaces and four endpoints" — is the Abstract Control Model (ACM). Everything in this chapter is CDC and ACM; "CDC ACM" is the noun.

The pitch is exactly what you'd expect from a 1999 spec: pretend the device is a modem. CDC has lots of subclasses — 0x02 ACM, 0x06 ECM (Ethernet), 0x0C EEM, 0x0D NCM, plus a half-dozen telephony models nobody implements any more — and the "modem-like things" lineage shows in every byte. There are functional descriptors for Call Management. There are notifications about ring detect, carrier detect, and a break condition. There's a control request to send an AT command. CDC ACM in 2026 is RS-232 in modem cosplay — useful precisely because the cosplay is universally recognised.

Three things to keep in mind about CDC ACM
  • It's two interfaces, not one. The spec separates "the control plane" (Communications interface, class 0x02) from "the data plane" (Data interface, class 0x0A). Both are mandatory.
  • Most of the modem-era complexity is optional. A CDC ACM virtual serial port in 2026 implements 5% of the spec and gets away with it because the OS only checks for the 5% it cares about.
  • The OS does almost all the work. You declare the right class triples, the host finds them, the OS-bundled driver wraps the bulk endpoints in a tty / COM port, and userspace talks to that. The firmware never sees the words "tty", "COM", or "serial port" — it sees a UART API.

1a. Where CDC ACM lives in the descriptor universe

Recall from Ch 06 the descriptor hierarchy: Device → Configuration → Interface → Endpoint. Standard classes (CDC, HID, MSC) extend that tree with class-specific descriptors that hang off the interface. CDC ACM extends it more than most:

Device  (one — same as every sample)
 └─ Configuration
     ├─ Interface Association Descriptor       ← new — Ch 19 expands on this
     │
     ├─ Interface 0  (Communications, class 0x02, subclass 0x02 ACM)
     │   ├─ CDC Header functional descriptor              ← class-specific
     │   ├─ CDC Call Management functional descriptor     ← class-specific
     │   ├─ CDC Abstract Control Management func. desc.   ← class-specific
     │   ├─ CDC Union functional descriptor               ← class-specific
     │   └─ Endpoint 0x82 — interrupt IN, MPS=16, bInterval=10ms
     │
     └─ Interface 1  (Data, class 0x0A)
         ├─ Endpoint 0x01 — bulk OUT, MPS=64
         └─ Endpoint 0x81 — bulk IN,  MPS=64

If you printed every descriptor byte-by-byte from a stock Zephyr CDC ACM instance and laid them end-to-end, you'd see about 75 bytes of descriptors for a single virtual serial port — vs ~32 bytes for usb_02's bare two-bulk-endpoint vendor class. Every byte in those extra 43 is either an IAD or a functional descriptor. Sections 2–5 walk through them in order.

2. The composite shape — and why CDC needs an IAD

A USB interface is the smallest unit of "thing the OS loads a driver against." Pre-2005, the host's heuristic for finding a driver was: read the device's class triple from the device descriptor; if it's 0x00/0x00/0x00 (the "see interface descriptors" sentinel), iterate over each interface and look up that interface's own class triple. One driver per interface. One interface per device, almost always. Easy.

CDC broke this. CDC ACM has two interfaces — Communications and Data — but they belong to the same logical function (the virtual serial port). A naïve host would happily load one driver for interface 0 (CDC class) and a different driver (or none) for interface 1 (CDC-Data class) and end up unable to wire them together. CDC's original answer in 1999 was the Union functional descriptor (§3d below): it tells the host "interface 0 is the master; interface 1 is its subordinate." That worked, sort of — Linux still uses the Union descriptor today — but it required every host stack to learn CDC-specific parsing rules.

The clean fix arrived in 2003 with an ECN to USB 2.0 introducing the Interface Association Descriptor (IAD). The IAD is a generic, class-agnostic way to say "the next N interfaces are one function — load one driver for all of them." It sits in the descriptor stream before the first interface it groups:

Offset  Size  Field              Value (typical CDC ACM)
  0      1   bLength             0x08   (IAD is exactly 8 bytes)
  1      1   bDescriptorType     0x0B   (USB_DESC_INTERFACE_ASSOC)
  2      1   bFirstInterface     0x00   (Communications iface number)
  3      1   bInterfaceCount     0x02   (this + next; both belong together)
  4      1   bFunctionClass      0x02   (CDC)
  5      1   bFunctionSubClass   0x02   (ACM)
  6      1   bFunctionProtocol   0x01   (AT commands V.250 — or 0x00 "none")
  7      1   iFunction           0x00   (optional string descriptor index)

For a host to honour the IAD, the device must also set its device-level class triple to 0xEF / 0x02 / 0x01 — the "Miscellaneous / Common Class / IAD" sentinel that tells modern hosts "this device uses IADs; parse them." Zephyr USB-Device-Next does this for you automatically: as soon as one registered class advertises an IAD, the core sets the device descriptor's bDeviceClass triple to 0xEF / 0x02 / 0x01. This is the same triple from the gotcha note in reference_zephyr_usbd_api — Zephyr defaults to it on every device, even single-interface vendor ones, which is harmless but not byte-canonical to the spec.

A common misconception

"The IAD is part of the Communications interface." It isn't — it sits in the configuration descriptor's flat byte stream between the configuration header and the first interface, with no parent-child relationship to either. Hosts that pre-date the 2003 ECN ignore it (it's just an unknown descriptor type to them) and fall through to the Union functional descriptor instead. That's why CDC ACM keeps the Union descriptor even though the IAD has been mandatory in practice for ~20 years.

3. Functional descriptors — CDC's class-specific extension

Standard descriptors (Device / Configuration / Interface / Endpoint) all use bDescriptorType values < 0x10. The USB-IF reserved 0x21..0x2F for class-specific descriptors, and CDC takes 0x24 for functional descriptors attached to a Communications interface, plus 0x25 for class-specific endpoint descriptors (unused by ACM). Every CDC functional descriptor follows the same 3-byte header:

Offset  Size  Field                  Notes
  0      1   bFunctionLength         total length of this descriptor incl. header
  1      1   bDescriptorType         always 0x24 — CS_INTERFACE
  2      1   bDescriptorSubtype      what kind of functional descriptor this is
  3..N      (subtype-specific)        body

For CDC ACM the four subtypes that matter are 0x00 Header, 0x01 Call Management, 0x02 Abstract Control Management, and 0x06 Union. They appear in that order, between the Communications interface descriptor and its single interrupt-IN endpoint:

3a. Header functional descriptor (subtype 0x00)

Offset  Size  Field            Value
  0      1   bFunctionLength   0x05
  1      1   bDescriptorType   0x24   (CS_INTERFACE)
  2      1   bDescriptorSubtype 0x00  (HEADER)
  3      2   bcdCDC            0x0110 (CDC spec 1.10)  little-endian

The Header descriptor announces "this interface is CDC, and we're claiming spec version 1.10." The version field lets a host with old quirks key on a specific CDC release. Every CDC interface starts with one of these and no host has cared about the version field in 20 years — but it's required.

3b. Call Management functional descriptor (subtype 0x01)

Offset  Size  Field             Value (typical)
  0      1   bFunctionLength    0x05
  1      1   bDescriptorType    0x24
  2      1   bDescriptorSubtype 0x01   (CALL_MANAGEMENT)
  3      1   bmCapabilities     0x00   (no call mgmt — virtual serial)
                                       bit 0: device handles call mgmt itself
                                       bit 1: call mgmt over data class iface
  4      1   bDataInterface     0x01   (interface number of the Data iface)

Call Management is a modem-era construct: it describes how the device dials, answers, and hangs up. For a virtual serial port the answer is "it doesn't" — bmCapabilities = 0x00. Some hosts (notably older Linux) want this descriptor to exist with capabilities zero, rather than be absent. So you include it, you zero it, and you move on.

3c. Abstract Control Management functional descriptor (subtype 0x02)

Offset  Size  Field             Value (typical)
  0      1   bFunctionLength    0x04
  1      1   bDescriptorType    0x24
  2      1   bDescriptorSubtype 0x02   (ABSTRACT_CONTROL_MANAGEMENT)
  3      1   bmCapabilities     0x02   (supports SET/GET_LINE_CODING +
                                        SET_CONTROL_LINE_STATE)
                                       bit 0: supports CDC_SEND_ENCAPSULATED_*
                                       bit 1: supports SET/GET_LINE_CODING,
                                              SET_CONTROL_LINE_STATE, SerialState
                                       bit 2: supports SEND_BREAK
                                       bit 3: supports NETWORK_CONNECTION notif.

The ACM functional descriptor is where the device declares which slice of the ACM-subclass API it implements. 0x02 — "I do line coding and line state" — is the universal lowest common denominator. Zephyr's CDC ACM ships 0x02 exactly. If you wanted SEND_BREAK you'd OR in 0x04; few devices bother.

3d. Union functional descriptor (subtype 0x06)

Offset  Size  Field             Value (typical)
  0      1   bFunctionLength    0x05
  1      1   bDescriptorType    0x24
  2      1   bDescriptorSubtype 0x06   (UNION)
  3      1   bControlInterface  0x00   (master — the Communications iface)
  4      1   bSubordinateInterface0  0x01   (Data iface)

The Union descriptor is the spec-canonical way to link the Communications interface to its Data interface. It pre-dates the IAD by four years and is the fallback for hosts that don't honour IADs. Always include it. The body can list multiple subordinates (some CDC subclasses use that — NCM, for example) but ACM always has exactly one subordinate. A Union descriptor with one subordinate is 5 bytes.

Together the four functional descriptors are 5 + 5 + 4 + 5 = 19 bytes of CDC-specific class data inside the Communications interface. Plus the 8-byte IAD ahead of it and the 9-byte Communications interface descriptor itself, you've already spent 36 bytes before you've described a single endpoint.

4. The four class-specific control requests

CDC ACM defines a small set of class requests that ride EP0, all with bmRequestType.type = CLASS (1) and bmRequestType.recipient = INTERFACE (1). wIndex carries the Communications interface number (almost always 0x0000). The host issues these after SET_CONFIGURATION, often without the device firmware doing anything — Zephyr's CDC ACM class handles them all internally and exposes the results as Zephyr UART state.

The four request codes (PSTN 1.20 §6.3):

bRequestNamebmRequestTypewValuewLengthDATA stage
0x20SET_LINE_CODING0x2107OUT, 7-byte line-coding struct
0x21GET_LINE_CODING0xA107IN, 7-byte line-coding struct
0x22SET_CONTROL_LINE_STATE0x21DTR/RTS bits0none (2-stage)
0x23SEND_BREAK0x21duration ms0none (2-stage)

4a. SET_LINE_CODING / GET_LINE_CODING (0x20 / 0x21)

These two carry a 7-byte struct in the DATA stage:

struct cdc_acm_line_coding {
    uint32_t dwDTERate;      // baud — little-endian, 32-bit
    uint8_t  bCharFormat;    // 0 = 1 stop bit, 1 = 1.5, 2 = 2
    uint8_t  bParityType;    // 0 None, 1 Odd, 2 Even, 3 Mark, 4 Space
    uint8_t  bDataBits;      // 5, 6, 7, 8, 16
} __packed;

The host issues SET_LINE_CODING when the userspace tool calls something like tcsetattr() (Linux) or SetCommState() (Windows) on the tty / COM. The device is supposed to act on the new settings.

Why "act on" usually means "ignore"

There is no actual UART out the back of a USB virtual serial port. The "115200 8N1" the user typed in their terminal program describes a non-existent physical link — the bits are USB packets, not edges on a TXD pin at a particular bit time. SET_LINE_CODING on a CDC ACM device is therefore advisory: the device is free to store the values and report them back on GET_LINE_CODING, but it cannot meaningfully apply them. The exception is when the firmware bridges CDC ACM to a real UART on the MCU's GPIOs — in that case SET_LINE_CODING means "reconfigure the hardware UART to these settings," and a wrong baud breaks the bridge. Zephyr's CDC ACM class stores the values and exposes them via uart_line_ctrl_get(dev, UART_LINE_CTRL_BAUD_RATE, ...) in case the application wants to honour them, but the class itself does nothing with them.

4b. SET_CONTROL_LINE_STATE (0x22) — the DTR/RTS bits

This request has no DATA stage — both control bits live in wValue:

wValue bit 0  →  DTR (Data Terminal Ready)   — host is "ready to talk"
wValue bit 1  →  RTS (Request To Send)       — host is "ready to receive"
wValue bits 2-15 → reserved (must be zero)

RS-232 used DTR and RTS as flow-control signals; on a CDC ACM device they're vestigial. But one of them — DTR — has become the universal "the host has opened the port" signal. When a userspace program calls open("/dev/ttyACM0", ...), the kernel's CDC ACM driver issues SET_CONTROL_LINE_STATE with wValue = 0x0003 (DTR + RTS) before returning from open(). When the program closes the port, the driver issues SET_CONTROL_LINE_STATE with wValue = 0x0000.

This is the basis of every "wait for host" pattern you'll see in CDC ACM firmware — including Zephyr's:

// Pseudocode — see usb_10 §5 for the real Zephyr API
device_init();
usbd_enable();
LOG_INF("Waiting for DTR...");
wait_until(DTR_was_asserted);    // first host open()
LOG_INF("Host attached.");
// Now it's safe to send greeting / start streaming

Without the gate, the device might start streaming bytes the moment USB enumeration completes — but the host's userspace hasn't called open() yet, so the tty's read buffer doesn't exist yet, and the kernel either discards the data or holds it in a small FIFO that promptly overflows. The DTR gate is how the firmware says "wait until somebody is listening."

macOS is the strictest about this

Linux's cdc_acm driver buffers ~64 KB of pre-open RX data per device by default. macOS buffers much less and is more likely to drop early bytes. Windows is mixed (depends on the USB-serial filter chain). If your firmware sends a greeting line on boot and you see it on Linux but not macOS, the DTR gate is the fix.

4c. SEND_BREAK (0x23)

Same shape as SET_CONTROL_LINE_STATE but wValue is the duration of the break condition in milliseconds. wValue = 0xFFFF means "until I send another SEND_BREAK with wValue=0." Few devices implement this; the ACM functional descriptor's bmCapabilities bit 2 declares support. If you set bit 2 in the descriptor and the host issues SEND_BREAK, you must handle it — even if "handle" means "log it and do nothing."

5. The interrupt IN — SerialState notifications

The Communications interface has exactly one endpoint: an interrupt IN at MPS = 8 or 16 bytes, with bInterval = 10ms on Full-Speed. Its job is to carry asynchronous device-to-host notifications about line state. The most common notification is SerialState (0x20), which is a 10-byte header plus a 2-byte UART state bitmap:

Offset Size Field            Notes
 0     1   bmRequestType    0xA1   (D2H | class | interface)
 1     1   bNotificationType 0x20  (SERIAL_STATE)
 2     2   wValue           0x0000
 4     2   wIndex           Communications iface number (LE)
 6     2   wLength          0x0002  (2 bytes of body follow)
 8     2   UART state bitmap (LE):
              bit 0  bRxCarrier    (DCD asserted upstream)
              bit 1  bTxCarrier    (DSR)
              bit 2  bBreak        (break received)
              bit 3  bRingSignal   (RI asserted)
              bit 4  bFraming      (framing error)
              bit 5  bParity       (parity error)
              bit 6  bOverRun      (overrun)

Most ACM devices never send a SerialState notification in their lifetime. The hosts mostly don't care. But the endpoint must exist (a CDC ACM Communications interface with zero endpoints is malformed; the spec says one notification EP), the host enumerates it, the kernel allocates a URB for it, and it sits NAKing forever. That NAK has a cost — about 64 bytes of bus bandwidth per interval (~1 KB/s on FS) — but it's an unavoidable feature of the class shape.

The Zephyr CDC ACM class translates a couple of UART line-ctrl writes into SerialState notifications: uart_line_ctrl_set(dev, UART_LINE_CTRL_DCD, 1) sets bit 0 and pushes a notification; same for DSR (bit 1) and BREAK (bit 2 — distinct from receiving a host SEND_BREAK). This is mostly useful when the firmware is bridging to a real UART and wants to surface the upstream line-state changes to host software that knows how to consume them — gdbserver, modem managers, etc.

6. The bulk pair — where the bytes actually go

The Data interface (class 0x0A, subclass 0x00, protocol 0x00) has exactly two endpoints: a bulk IN and a bulk OUT, both at MPS = 64 on Full-Speed (MPS = 512 on High-Speed). No functional descriptors, no class requests, nothing fancy. Every byte the host writes to /dev/ttyACM0 arrives on bulk OUT. Every byte the firmware writes ends up on bulk IN.

Host write to tty:
   userspace write(fd, "hello\n", 6)
     → kernel CDC ACM driver
       → URB on bulk OUT (EP 0x01), 6 bytes
         → device sees 6 bytes in its RX FIFO

Host read from tty:
   userspace read(fd, buf, 1024)
     → kernel poll completed URBs on bulk IN (EP 0x81)
       → URBs filled by device sending 64B at a time, until ZLP or short packet

6a. The ZLP rule

This is the one bulk-endpoint rule that bites everybody on their first CDC ACM project: when a transfer's length is an exact multiple of the endpoint's MPS, the receiver doesn't know it's over until either (a) it gets a short packet (less than MPS) or (b) it gets a zero-length packet (ZLP).

Sender writes 64 bytes (== 1 × MPS):
   → IN/OUT packet 1: 64 bytes
   → ?? — receiver doesn't know if more is coming
   → Sender must follow with a ZLP to signal end-of-transfer
     → IN/OUT packet 2: 0 bytes (the ZLP)
   → Receiver knows the transfer is complete.

Sender writes 100 bytes:
   → IN/OUT packet 1: 64 bytes
   → IN/OUT packet 2: 36 bytes (a short packet)
   → No ZLP needed — the short packet itself signals end-of-transfer.

For a CDC ACM device on FS where MPS = 64: every "send N bytes" operation where N is a multiple of 64 must be followed by a ZLP. Zephyr's CDC ACM class handles this for you when you write through the UART API — the class tracks the last transfer's length and enqueues a ZLP when needed. If you were writing a raw class, you'd have to do it yourself. See Ch 13 (Endpoint buffers, MPS, ZLPs) for the deep version of this rule.

7. Duplex — full-duplex UART, half-duplex wire, full-duplex endpoints

UART is genuinely full-duplex at the silicon: separate TX and RX shift registers in the peripheral, separate TXD and RXD pins out, no contention between the two directions ever. USB 2.0's wire — the D+/D− differential pair — is genuinely half-duplex: only one transmitter (host or device) drives the bus at any instant. That mismatch is the source of a recurring "how can CDC ACM bridge a full-duplex UART through a half-duplex wire?" question. The answer is that the question is comparing layers that aren't talking about the same thing.

At the endpoint layer — where the CDC ACM class spec lives, and where your firmware writes its uart_fifo_fill() / uart_fifo_read() calls — USB is fully duplex. Bulk IN and bulk OUT are independent FIFOs from the device's point of view. The firmware doesn't pick one or the other; it pushes RX bytes into the bulk IN ring and pulls TX bytes out of the bulk OUT ring continuously and concurrently, with no mutex between the two directions. The bridge is a pair of pipes:

            ┌─────────── device side ───────────┐    ┌─── host side ───┐

UART RX ISR ──► rx_ring ──► usbd_ep_enqueue(bulk IN)  ────► /dev/ttyACM0 read()
                                                       (wire)
host write() ──► /dev/ttyACM0 ◄──── usbd_ep_enqueue(bulk OUT) ──► tx_ring ──► UART TX ISR

Neither side waits on the other. The UART peripheral has its own RX FIFO; the USB controller has its own bulk-IN and bulk-OUT endpoint FIFOs; the rings in between absorb timing skew. If you stop the host's reader thread, the device's bulk-IN endpoint NAKs (no data accepted), the rx_ring fills, and eventually the UART overruns — but the opposite direction (bulk OUT → UART TX) continues running unimpeded. The two directions are decoupled.

7a. What the half-duplex wire does to the model

At the wire layer, the host scheduler time-multiplexes IN and OUT transactions on the single differential pair. On Full-Speed the frame is 1 ms; on High-Speed the microframe is 125 µs. Inside each frame the host issues a sequence of IN tokens and OUT tokens (and SOFs and the occasional control token), each one followed by its data and handshake packets. From a single endpoint's point of view, that endpoint sees a transaction targeted at it whenever the host decides to schedule one — typically every frame for a bulk endpoint with data pending, with the host re-polling on NAK and moving on.

The wire being half-duplex therefore has no impact on whether the model is duplex — it has impact only on aggregate bandwidth. If your IN endpoint wants 9 Mbps and your OUT endpoint wants 9 Mbps on the same FS bus, the wire's 12 Mbps physical ceiling forces them to share the budget. That's what §8 quantifies.

Three layers, three duplex states
  • Application: CDC ACM looks full-duplex. read() and write() on the same fd do not contend.
  • Endpoint: Bulk IN and bulk OUT are independent. The class spec and the device firmware see two parallel pipes.
  • Wire: D+/D− carries one direction at a time. The host scheduler interleaves IN and OUT transactions; on FS the interleaving granularity is well below 1 ms, on HS below 125 µs.

Confusion arises when people transplant the wire-layer fact ("half-duplex bus") onto one of the upper layers ("therefore CDC ACM can't bridge a full-duplex UART"). The transplant is invalid — the endpoint layer rebuilds duplex on top of the half-duplex wire, the same way switched Ethernet rebuilds it on top of CSMA/CD's collision domain.

7b. Where the model leaks

The duplex abstraction is leaky in three places worth knowing about:

  • Back-pressure asymmetry. A stalled host reader fills the device's rx_ring, which back-pressures the UART RX, which (without hardware RTS) eventually overruns. A stalled host writer just means there's nothing on bulk OUT — the device's TX side starves cleanly. The two failure modes look completely different in logs, but they both originate as "the host stopped pumping bytes."
  • No flow control bridges through. CDC ACM has SET_CONTROL_LINE_STATE's RTS bit, but the host driver doesn't translate the device's "I'm full" state into RTS toward the user's tty layer — most user programs see only blocking write(). Symmetrically, a device's UART driving a hardware target with XON/XOFF flow control has to handle the flow-control bytes itself; CDC ACM is byte-transparent and won't surface them as anything special.
  • Latency, not bandwidth, breaks tight loops. Even on an idle bus, host-side bulk URB scheduling can introduce 1–5 ms gaps (Windows is worst). A request-response protocol expecting < 1 ms turnaround over CDC ACM will see jitter that has nothing to do with the duplex model and everything to do with kernel scheduling. The fix is HID interrupt endpoints or vendor bulk with very deep host-side URB queues — not anything in the duplex axis.

8. Bandwidth budget — what UART rates fit a Full-Speed bulk pair

Full-Speed USB 2.0 is 12 Mbps on the wire. Of that, the bulk-transfer payload you actually get is much less, because every transaction carries token, data, and handshake packets with their own SYNC / PID / CRC overhead, plus bit-stuffing, plus inter-packet gaps. The math matters because the RP2350's USB peripheral is Full-Speed only (Ch 14 §2), so every byte across every endpoint on an edevkit shares one ~1 MB/s pipe. Sustained UART at any reasonable baud is fine; sustained UART plus SWO bulk plus CMSIS-DAP plus a second CDC ACM is where the budget gets interesting.

8a. Per-transaction overhead on Full-Speed

A single bulk IN or OUT transaction at MPS = 64 has three packets on the wire — a token from the host, a data packet (either direction), and a handshake (typically ACK from the receiver). Adding bit-stuffing and inter-packet gaps:

PacketBytesBit-times on wire
Token (SYNC + PID + ADDR + EP + CRC5 + EOP)3~35
Data (SYNC + PID + 64 B payload + CRC16 + EOP)67~547
Handshake (SYNC + PID + EOP)1~19
Inter-packet gaps + bit-stuffing—~60
Total per 64-byte transaction—~660

A Full-Speed frame is 1 ms = 12,000 bit-times. So you fit ~18 bulk transactions per frame, total, shared across every endpoint on every interface on the device.

8b. Single-direction ceiling

If only one direction is busy: 18 × 64 B = 1152 B/ms ≈ 1.15 MB/s ≈ 9.2 Mbps payload. This is the often-quoted "FS bulk wire-rate," and is what one-direction stream benchmarks see. Same number applies to bulk-OUT alone. The 800 KB/s figure cited in §10 below is the same physical ceiling derated for OS-side overhead (URB allocation, tty layer, scheduling) on a typical real host.

8c. Bidirectional ceiling — the half-duplex wire bites here

When both directions push data, the 18 transactions/frame budget is split. The host scheduler tries to alternate roughly fairly — typically ~9 IN + ~9 OUT per frame on a quiet bus. That works out to:

9 × 64 B = 576 B/ms = ~576 KB/s per direction (simultaneous)
                    = ~5.7 Mbaud UART payload per direction (10 bits/byte: 1 start + 8 data + 1 stop)

That's the theoretical ceiling. Real stacks lose to:

  • NAKs. Host polls; device says "no data yet"; the transaction burns wire time for zero payload.
  • Short-packet padding. UART trickling 1 byte at a time forces a 1-byte bulk packet — full ~660-bit overhead for 8 bits of payload, ~80× worse than a full-MPS packet.
  • ZLPs after exact-multiple-of-MPS transfers (§6a). One extra empty transaction per multiple-of-64 transfer.
  • Host scheduling jitter. Windows in particular can park bulk URBs for 1–8 ms gaps under load. Linux is best; macOS is in between.
  • Other endpoints on the device. Every NAK from every interrupt and bulk endpoint on every interface eats into the same 18-transaction budget — including the SerialState interrupt-IN endpoint that mostly just NAKs (§5).

Rule of thumb: derate by 3–4× for "I don't want to think about it." That puts a safe bidirectional sustained ceiling at ~1.5–2 Mbaud per direction on a stock CDC ACM bridge.

8e. High-Speed USB 2.0 — for completeness

On HS (480 Mbps), the microframe is 125 µs and bulk MPS is 512 B. You fit ~13 bulk transactions per microframe — call it ~53 MB/s aggregate. Real single-direction CDC ACM throughput on HS is 30–40 MB/s; bidirectional ~25 MB/s each. In UART terms that's roughly 200 Mbaud per direction — the UART peripheral, the level shifters, and the target's pin slew rate will fail decades before HS USB does. If you can put a CDC ACM bridge on HS instead of FS, you stop having to think about bandwidth at all.

RP2350 is Full-Speed only (Ch 14 §2), so on edevkit hardware the FS analysis above is the operative one. The host's HS hub upstream of the device doesn't help — it just shifts the scheduling tightness to the hub's translator.

8f. edevkit-specific — the shared FS pipe

The v1 design puts all of these on one FS pipe at once:

  • CMSIS-DAP v2 bulk pair (debug commands + SWO trace)
  • Target log UART CDC ACM
  • Dev-protocol UART CDC ACM
  • SMP / mcumgr CDC ACM (OTA, shell)

That's three CDC ACM functions plus a vendor bulk pair, all sharing the same 18-transaction/ms budget. Each function has its own bulk pair; the host's USB stack arbitrates across them. The 800 KB/s number quoted in §10 for a single CDC ACM pipe collapses fast when three of them are streaming at once. This is why the edevkit firmware uses back-pressure-aware producers on SWO and the protocol sniffer rather than per-channel rate caps — when one channel saturates, all channels degrade gracefully instead of one channel silently dropping.

9. How the host loads a driver against CDC ACM

The whole point of declaring the right class triple is that the OS finds and loads a stock driver without any user intervention. Each major OS does it slightly differently:

Linux — cdc_acm

The kernel's USB core walks the device's interfaces, finds Interface 0 with class 0x02/0x02/any, and probes the cdc_acm driver against it. The driver creates a /dev/ttyACMx tty (where x is a sequential index per device) and a sysfs entry at /sys/class/tty/ttyACMx/. dmesg shows the binding:

cdc_acm 1-2:1.0: ttyACM0: USB ACM device

The 1.0 is "interface 0 of device 1-2." The Data interface (1.1) is silently claimed by the same driver via the Union descriptor — the kernel knows that Comms iface 0 has Data iface 1 as a subordinate. Userspace opens /dev/ttyACM0 like any tty; the tcgetattr / tcsetattr calls become GET/SET_LINE_CODING control transfers under the hood.

macOS — AppleUSBCDCACMData

The IOKit USB stack matches on class/subclass and loads the kernel extension that ships with the OS. The device appears as a pair of nodes in /dev/:

/dev/tty.usbmodemXXXXX     # callout-mode
/dev/cu.usbmodemXXXXX      # call-in mode (the one you want)

The XXXXX is derived from the device's iSerial string descriptor — which is why per-unit serials (Ch 07) matter on macOS. Two identical devices without per-unit serials get the same node path and only one will work.

Windows — usbser.sys

Since Windows 10 build 1709 (Fall Creators Update, Oct 2017), usbser.sys auto-loads against any device whose composite IAD or interface declares CDC ACM. The device appears under "Ports (COM & LPT)" in Device Manager as COMn. Pre-1709 Windows required an .inf file pointing at usbser.sys; vendors shipped one but it was always a thin wrapper around the OS-bundled driver. As of 2026, the .inf dance is a museum exhibit — every supported Windows version auto-loads.

10. Where CDC ACM falls short

  • Throughput. §8 has the full budget analysis. Short version: on FS you get ~800 KB/s real throughput out of one CDC ACM pipe one-way, dropping to ~1.5–2 Mbaud per direction safely sustained when both directions are live, and lower still when multiple class functions share the bus. On HS, ~30 MB/s. Above that you need vendor-class bulk transfers (Ch 21) or larger MPS — CDC ACM's 64/512 MPS is fixed by the class spec.
  • Latency. Linux's cdc_acm coalesces small writes into 64-byte URBs with up to 1ms of latency; macOS and Windows do similar. If you need < 1ms host-to-device latency, CDC ACM is the wrong tool.
  • Reliability of small writes. Many host programs write byte-at-a-time and expect each byte to arrive promptly. That works if the host stack flushes after each write() — most do — but you can't rely on it. Defensive firmware treats incoming data as a stream and parses out frame boundaries.
  • Composite limits. macOS used to balk at composite CDC + anything-else devices unless the IAD was exactly right. As of macOS 11+ this is fine, but composite samples (Ch 19, usb_20) still occasionally surface stack-specific quirks.
  • No bidirectional flow control. RTS/CTS doesn't actually flow-control anything — they're just wValue bits on a 2-stage control request, with no wire-level effect. If the device's bulk OUT FIFO is full, it has to NAK on the bulk OUT until the application drains — which manifests as the host's write() blocking, not as RTS deasserting. That's usually fine but it surprises people coming from real RS-232.

Common pitfalls

Symptom → Fix

"Device enumerates but no /dev/ttyACMx appears on Linux." The class-triple sentinels are wrong. Run lsusb -v and check: device descriptor must be 0xEF / 0x02 / 0x01, IAD must precede the Communications interface with bFunctionClass=0x02 / bFunctionSubClass=0x02, Communications interface must be bInterfaceClass=0x02 / bInterfaceSubClass=0x02. Any one of those wrong and the kernel's USB core won't probe cdc_acm against the device.

Symptom → Fix

"Greeting line shows up on Linux but not macOS." Classic missing DTR gate — see §4b. Firmware sends the greeting before the host has opened the port, Linux's larger pre-open buffer hides the bug, macOS doesn't. Move the greeting to after DTR is asserted.

Symptom → Fix

"Host write of exactly 64 bytes never completes." Missing ZLP on the bulk-OUT path. 64 bytes is exactly one FS MPS; the receiver waits for either a short packet or a ZLP to know the transfer is done. Either pad to 65 bytes or send a ZLP after the 64-byte payload (§6a). Zephyr's CDC ACM class handles this on the IN side automatically; on the OUT side it's the host's responsibility, and host drivers do it correctly — so this usually surfaces only in custom userspace tools that bypass the kernel tty layer.

Symptom → Fix

"Two identical edevkits both enumerate but only one is visible on macOS." Missing or duplicate iSerial. macOS keys its /dev/cu.usbmodemXXXXX path off the serial-number string descriptor. If two devices have the same iSerial (or both have iSerial = 0), the second one collides. Per-unit serials (Ch 07, usb_03) are mandatory on macOS for fleet use.

Symptom → Fix

"screen hangs on connect on macOS but works on Linux." Often tty.usbmodem vs cu.usbmodem — the tty. node blocks on DCD; the cu. ("call-up") node doesn't. Always connect to /dev/cu.usbmodemXXXXX, not /dev/tty.usbmodemXXXXX, on macOS.

Symptom → Fix

"Bytes arrive corrupted on Windows when baud is set to something other than 115200." The firmware is ignoring SET_LINE_CODING (correctly — see §4a callout) but the firmware also looks like it's transforming the data somehow. Almost always this is the userspace tool on the host doing line-ending translation (CR/LF mangling) based on the assumed terminal type, not USB at all. Confirm with raw byte capture (USBPcap) before blaming the device.

Symptom → Fix

"Device works once but the second plug-in shows up as a new COM port number on Windows." Windows ties COM-port assignment to the device's USB instance ID — which includes the iSerial when one is present. With a per-unit serial (Ch 07) the same device always gets the same COM port. Without one, Windows treats every (port × hub-position) combination as a new device and burns COM numbers indefinitely.
