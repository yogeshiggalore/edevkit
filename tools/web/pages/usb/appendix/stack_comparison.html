USB Stack Comparison

TinyUSB vs Zephyr USB-Device-Next — what each one is, where each one wins, and why edevkit1 picked UDN.

Audience

Read this if you're new to embedded USB and trying to choose a stack, or if you're new to edevkit1 and wondering why the design doc says "Don't use TinyUSB". The decision is locked for edevkit1, but the trade-offs are real on either side — this page explains them honestly so you can make your own call on your own project.

1. Why this appendix exists

If you Google "Raspberry Pi Pico USB", almost every result uses TinyUSB. If you Google "Zephyr USB", you find half a dozen overlapping pages about a legacy usb_* API and a newer usbd_* API and it's not obvious which is which. So it's reasonable to land at edevkit1 and ask: "Why didn't you just use TinyUSB like the pico-sdk samples do?"

This appendix answers that. It's not a TinyUSB hit-piece. TinyUSB is excellent software — it powers thousands of shipping products. It's just not the right fit for what edevkit1 does. The rest of this page makes that concrete.

2. TinyUSB at a glance

TinyUSB is a portable, RTOS-agnostic USB stack written by Ha Thach and maintained by Adafruit, Raspberry Pi, Espressif, and others. It targets MCUs without an OS — pico-sdk, Arduino, bare-metal projects — and ships as a single git submodule you build into your firmware.

AspectTinyUSB
LanguageC (C99), no C++
Build systemCMake, plain Make, or hand-rolled — drop into any project
RTOSOptional — runs bare-metal; hooks for FreeRTOS, RT-Thread, Mynewt
MCU support~25 families: RP2040/RP2350, STM32 (all), nRF5x/53, ESP32-S2/S3/P4, SAMD/E/L, Kinetis, EFM32, MAX32, LPC, MSP432, …
Built-in classesCDC, HID (boot & report), MSC (BBB), MIDI, Audio (UAC1+2), Video (UVC), DFU, vendor, network (NCM/ECM/RNDIS), Bluetooth HCI
Composite devicesYes — but you hand-write the descriptor byte arrays
Host stackYes (TinyUSB Host) — same API, runs on OTG-capable parts
ConfigurationPreprocessor macros in tusb_config.h
Memory modelStatic, no malloc required
LicenseMIT

The TinyUSB mental model

You include the headers, define your descriptors as const uint8_t arrays, implement a handful of tud_*_cb() callbacks (tud_cdc_rx_cb, tud_hid_get_report_cb, etc.), call tud_task() from your main loop or a thread, and you're done. There is no concept of a "device object" that you register interfaces with — descriptors are bytes, classes are callbacks, and the configuration is whatever you typed into tusb_config.h.

This is genuinely beautiful for a single-purpose device. A pure CDC ACM "USB serial" on RP2040 with TinyUSB is about 200 lines of code and 12 KB of flash.

3. Zephyr USB-Device-Next at a glance

Zephyr USB-Device-Next (UDN, the usbd_* API under subsys/usb/device_next/) is the second-generation USB device stack in Zephyr RTOS. It replaces the legacy usb_* stack — which is still present but feature-frozen and slated for removal once UDN reaches feature parity. As of Zephyr 4.x, UDN is the default and recommended choice for new development.

AspectZephyr USB-Device-Next
LanguageC (C11), uses Zephyr's macro DSL heavily
Build systemCMake via west / sysbuild — part of the Zephyr workspace
RTOSRequired — UDN is a Zephyr subsystem, uses kernel threads, msgq, work queues
MCU supportWhatever has a Zephyr UDC driver: RP2040/RP2350, STM32, nRF5x/52/53, NXP i.MX/Kinetis, SAM, ESP32-S series (WIP), native_posix (for tests)
Built-in classesCDC ACM, CDC ECM/NCM, HID, MSC, DFU, MIDI 2.0, Bluetooth HCI, BTH HCI, vendor
Composite devicesYes — declarative: each function is an instance you register on a device object
Host stackSeparate subsystem (USB-Host-Next, still maturing)
ConfigurationKconfig + devicetree, validated at build time
Memory modelStatic; kernel objects own their own storage
LicenseApache-2.0 (with Zephyr)

The UDN mental model

You instantiate a USB device context with USBD_DEVICE_DEFINE(), declare each configuration with USBD_CONFIGURATION_DEFINE(), and then register class instances on it — CDC ACM, HID, vendor, etc. — each pulled in from devicetree by enabling Kconfig symbols. Descriptors are generated from those definitions; you never hand-author the byte stream. You call usbd_init() once, then usbd_enable(), and the stack runs on its own threads.

USBD_DEVICE_DEFINE(edevkit_usbd, DEVICE_DT_GET(DT_NODELABEL(zephyr_udc0)),
                   USBD_VID, USBD_PID);

USBD_DESC_LANG_DEFINE(lang);
USBD_DESC_MANUFACTURER_DEFINE(mfr, "edevkit");
USBD_DESC_PRODUCT_DEFINE(prod, "edevkit1 debug probe");

USBD_CONFIGURATION_DEFINE(cfg_1, USB_SCD_SELF_POWERED, 250, &cfg_str);

int main(void) {
    usbd_add_descriptor(&edevkit_usbd, &lang);
    usbd_add_descriptor(&edevkit_usbd, &mfr);
    usbd_add_descriptor(&edevkit_usbd, &prod);
    usbd_add_configuration(&edevkit_usbd, USBD_SPEED_FS, &cfg_1);
    /* register class instances declared elsewhere */
    usbd_register_class(&edevkit_usbd, "cdc_acm_0", USBD_SPEED_FS, 1);
    usbd_init(&edevkit_usbd);
    usbd_enable(&edevkit_usbd);
    return 0;
}

It's more ceremony than TinyUSB for a hello-world serial port. It scales much better for everything else — see §6.

5. Footprint & resource cost

The honest numbers, measured on rpi_pico2/rp2350a/m33 with -Os, single-config FS device:

ConfigurationTinyUSB (flash / RAM)UDN + Zephyr kernel (flash / RAM)
Hello-world CDC ACM~14 KB / ~2.5 KB~46 KB / ~8 KB
CDC + HID composite~22 KB / ~4 KB~54 KB / ~10 KB
edevkit1 full stack (CMSIS-DAP v2 + 2× CDC + SMP + vendor)est. ~60 KB / ~14 KB plus the RTOS you bring~120 KB / ~22 KB including kernel, shell, logging, MCUboot client
Apples-to-oranges, on purpose

TinyUSB's flash number is just USB. UDN's flash number is USB plus everything else you also got "for free" — preemptive scheduler, logging subsystem, shell, watchdog, MCUboot client. If you build a comparable feature set on top of TinyUSB, the gap narrows to roughly 25–35 KB. That's a real cost, but the RP2350B has 4 MB of XIP flash and 520 KB of SRAM — it's not the constraint.

6. Composite-device authoring

This is where the two stacks feel most different in practice. edevkit1 ships five interfaces in one configuration:

  • CMSIS-DAP v2 (vendor-specific, bulk IN + OUT)
  • Target-log UART (CDC ACM, IAD-grouped)
  • Test UART (CDC ACM, IAD-grouped)
  • SMP server (vendor-specific, bulk IN + OUT)
  • Sniffer drain (vendor-specific, bulk IN)

TinyUSB way

You hand-author the configuration descriptor as one contiguous uint8_t array. Every interface, every endpoint, every IAD — bytes you typed yourself. You renumber endpoints by hand if you reorder. You compute wTotalLength by hand. You add the IAD descriptors at the right offsets by hand. tusb_config.h grows a wall of CFG_TUD_CDC=2 CFG_TUD_HID=0 CFG_TUD_VENDOR=3-style macros, and each vendor interface gets its own tud_vendor_*_cb() callback set distinguished by an interface number you tracked manually.

This works. People do it. It is the highest-error-density code in the project, and every reordering produces a one-line bug that doesn't show up until enumeration on a specific OS.

UDN way

Each interface is a separate Kconfig-enabled instance:

# prj.conf
CONFIG_USBD_CDC_ACM_CLASS=y
CONFIG_USBD_VENDOR_CLASS=y   # CMSIS-DAP v2, SMP, sniffer

/* devicetree overlay */
cdc_acm_target_log: cdc_acm_target_log {
    compatible = "zephyr,cdc-acm-uart";
};
cdc_acm_test:       cdc_acm_test       { compatible = "zephyr,cdc-acm-uart"; };

/* app code */
usbd_register_class(&edevkit_usbd, "cdc_acm_0", USBD_SPEED_FS, 1);
usbd_register_class(&edevkit_usbd, "cdc_acm_1", USBD_SPEED_FS, 1);
usbd_register_class(&edevkit_usbd, "dap_v2",    USBD_SPEED_FS, 1);
usbd_register_class(&edevkit_usbd, "smp_bulk",  USBD_SPEED_FS, 1);
usbd_register_class(&edevkit_usbd, "sniffer",   USBD_SPEED_FS, 1);

Endpoint numbering, interface numbering, IAD insertion, wTotalLength calculation — all done by UDN at registration time. Reorder freely. Add a sixth class. Add a second configuration. The descriptor is correct by construction.

Why this matters for edevkit1

edevkit1's composite topology is going to evolve as v1 ships and v2 plans grow (logic analyzer, more sniffers, possible WebUSB endpoint). Each change is a one-liner under UDN; it would be a 50-line audit under TinyUSB.

7. RTOS integration

edevkit1 is not a single-purpose device. Multiple subsystems run concurrently with hard-ish latency requirements:

  • CMSIS-DAP command/response loop (sub-millisecond turnaround)
  • SWO ITM frame pump (back-pressure-aware, never drop)
  • Two UART bridges (independent Vio domains)
  • I²C and SPI protocol engines (PIO-driven, DMA-fed)
  • YAML emulator engine (responds to PIO IRQs)
  • SMP server (low priority but must always respond)

Each of these is naturally a thread or work-queue item with priorities, message queues, and synchronization. With TinyUSB you bring an RTOS yourself (FreeRTOS is the common pairing) and then bridge TinyUSB's callbacks to RTOS primitives. With UDN, the stack is those primitives. CDC RX is a ring_buf. Vendor bulk completion is a k_sem. State changes are usbd_msg on a work queue. There's no impedance mismatch because there's no boundary.

Concretely: UDN delivers events on a Zephyr work queue. Your handler runs in thread context with full kernel API access — k_msgq_put() to the protocol engine, LOG_INF() to the log subsystem — without ever wondering whether you're in IRQ context.

9. Ecosystem & maturity

TinyUSB

  • Adoption: every Adafruit CircuitPython board, every Raspberry Pi Pico example, MicroPython, Arduino-Pico, Espressif's USB samples, countless one-person hardware projects.
  • Worked examples: massive. Want a UVC webcam on RP2040? There's an example. Want HID + CDC + MSC composite? Three blog posts.
  • Failure mode: examples are the docs. When the example diverges from your case, you're reading the source. Class drivers have varying quality — CDC and HID are pristine; some others are "works for the example that ships with them."
  • Velocity: steady, community-driven, conservative on breaking changes.

Zephyr USB-Device-Next

  • Adoption: Nordic Connect SDK, NXP MCUXpresso, Antmicro reference designs, Memfault, Golioth. Production grade where Zephyr already lives.
  • Worked examples: smaller catalog, mostly in zephyr/samples/subsys/usb/. Each sample is a working, twister-tested project — but you won't find a UVC example because UVC isn't there yet.
  • Failure mode: API churn. The 3.7 → 4.0 → 4.1 transitions all renamed or restructured a few symbols. The USB Book exists partly because the gotchas are easier to write down than to discover via git blame.
  • Velocity: faster than TinyUSB right now — UDN is being actively built out toward feature parity with the legacy stack and beyond.
UDN is still maturing

This is the honest case against UDN. If you start a project on UDN today, expect to hit one or two undocumented edge cases per quarter where the answer is in a Zephyr PR rather than a doc page. edevkit1 mitigates this by writing the USB Book — every gotcha we hit becomes a chapter entry. See G1–G7, G8–G10, G15, G16–G18, G19–G22.

10. When to pick which

If you're starting a new firmware project today, here's an honest decision tree:

Pick TinyUSB when

  • You're targeting bare-metal or pico-sdk and don't want an RTOS.
  • Your device is single-purpose: one CDC, or one HID, or one MSC.
  • Footprint matters — you have ≤256 KB flash or ≤32 KB RAM.
  • You need UVC, UAC1/2, or a class UDN doesn't have yet.
  • Your team already knows pico-sdk and won't get paid to learn Zephyr.
  • You want the largest pool of online worked examples.

Pick Zephyr USB-Device-Next when

  • Your device is composite (≥3 interfaces) or will become composite.
  • You need MCUboot / DFU / SMP / OTA as a first-class feature.
  • You want a real RTOS with preemption, priorities, and SMP — not a main-loop with callbacks.
  • You need shell, logging, settings, LittleFS, networking, BLE — and you want them to integrate, not collide.
  • You're going to ship multiple firmware variants from one codebase (devicetree + Kconfig pays for itself).
  • You're comfortable absorbing some API churn in exchange for forward velocity.

Genuinely a coin-flip

  • Dual CDC ACM serial bridge with no other features — both are fine. TinyUSB is faster to write; UDN is easier to extend later.
  • HID + one vendor interface — both work; pick by team familiarity.

11. Why edevkit1 picked UDN

The decision in one sentence

edevkit1 is a composite, OTA-updatable, multi-protocol developer kit running real-time engines under preemptive scheduling — exactly the workload UDN is designed for and exactly the workload TinyUSB would have us rebuild around.

Five concrete reasons, ranked by how load-bearing each is:

  1. MCUboot + SMP OTA is non-negotiable.

    edevkit1 ships as a sealed device — there's no exposed SWD pin on the user side. Field updates have to be brick-proof, which means MCUboot's swap-and-revert flow with image confirmation. MCUboot's SMP server, the matching mcumgr host tool, and Zephyr's bootloader integration are one ecosystem. Wiring TinyUSB DFU 1.1 to MCUboot is possible but means: (a) write the DFU class glue, (b) implement slot management ourselves, (c) implement the confirmation-window timer ourselves, (d) write our own host tool because mcumgr won't speak DFU 1.1. That's ~2,000 lines of brittle code we don't write in Zephyr.

  2. The composite topology is the product.

    The five-interface device (CMSIS-DAP + 2× CDC + SMP + vendor sniffer) is what edevkit1 is. Under UDN every interface is an independent module behind a Kconfig; under TinyUSB they're all entries in one hand-rolled desc_configuration[] array. The composite is the place we'll most often change scope as v1 ships and v2 plans firm up. We want adding an interface to be a Kconfig flip, not a descriptor-array audit.

  3. SMP dual-core with watchdog, shell, logging, LittleFS — already running.

    edevkit1 needs all of these. They're built, tested, documented, and integrated in Zephyr. Picking TinyUSB means picking not Zephyr, which means rebuilding all of this against pico-sdk + FreeRTOS, where each piece is a separate library with its own conventions. The substrate has more leverage than the USB stack alone.

  4. Host side already speaks Zephyr.

    mcumgr, smpclient, probe-rs's Zephyr support, the Zephyr Shell — all run unchanged against UDN. The Tauri host GUI talks to standard Zephyr endpoints. Choosing TinyUSB would mean re-implementing the device-side of every protocol our existing host tools already speak.

  5. Forward velocity outweighs catalog size.

    UDN's catalog of worked examples is smaller than TinyUSB's. Acknowledged. But UDN's velocity is higher and its gotchas are writeable up — the USB Book exists precisely because the gotchas are systematic enough to enumerate. TinyUSB's gotchas are scattered across thousands of one-off projects and not written down anywhere central.

Trade-offs we explicitly accept

  • Heavier minimum footprint — ~46 KB for hello-world CDC vs ~14 KB. Non-issue on RP2350B (4 MB XIP flash).
  • Steeper onboarding — devicetree + Kconfig + west are unfamiliar to anyone arriving from pico-sdk. Mitigated by the USB Book.
  • API churn — UDN signatures and macros are still settling. Mitigated by pinning Zephyr to a known release in west.yml and updating deliberately.
  • Smaller class catalog — no UVC/UAC today. Not in v1 scope.

Re-open this decision only if

  • edevkit1 firmware size approaches the 512 KB slot limit, and profiling shows UDN + kernel are the dominant cost.
  • Zephyr UDN is announced as deprecated or unmaintained (extremely unlikely — it's the official direction).
  • A future product variant truly needs UVC or UAC2 and UDN still doesn't ship them.
  • We discover a class of bugs in UDN that hits edevkit1's workload and is not fixable upstream within one release cycle.

None of those is true today. Decision locked. See edevkit1.html §15 — Decision log.

12. Migration notes (if we ever had to)

Recorded for completeness, not because we plan to. If a future incident forces a migration from UDN to TinyUSB, here's what would have to change:

  • Replace USBD_DEVICE_DEFINE + class registrations with a hand-written desc_configuration[] array; one descriptor byte per row, IAD inserted manually.
  • Re-implement each class:
    • CDC ACM × 2 → CFG_TUD_CDC = 2 plus tud_cdc_rx_cb(), plus instance-aware RX routing.
    • Vendor (CMSIS-DAP, SMP, sniffer) → CFG_TUD_VENDOR = 3 plus three callback sets distinguished by interface number.
  • Replace UDN's work-queue events with TinyUSB callbacks (tud_mount_cb, tud_suspend_cb, etc.); wire each to whatever thread/queue stays.
  • Replace SMP server with hand-rolled DFU 1.1 glue + custom host tool (or keep SMP but speak it over a raw vendor endpoint).
  • Re-author the WinUSB / MS OS 2.0 descriptors as raw bytes — UDN has helpers for these; TinyUSB makes you compose them yourself.
  • Audit endpoint numbering globally; TinyUSB doesn't auto-assign.
  • Move logging/shell/settings off Zephyr's subsystems onto whatever substitute we adopt.

Estimated effort: 6–8 weeks at the firmware level, plus an unknown amount of host-side rework. The estimate exists so anyone proposing the migration has to justify against it.
