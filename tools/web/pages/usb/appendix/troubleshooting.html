CMSIS-DAP bulk transfers fail through cascaded USB hubs on macOS

Symptom: A CMSIS-DAP v2 debug probe (e.g., Raspberry Pi Debug Probe) enumerates correctly with probe-rs list, but the first bulk transfer (e.g., probe-rs download or probe-rs info) fails:

Error: Failed to open probe: Failed to open the debug probe.
Caused by:
    0: An error which is specific to the debug probe in use occurred.
    1: Error handling CMSIS-DAP command Info.
    2: Error in the USB access.

The probe works fine if plugged directly into the Mac's USB port or a single hub, but fails behind two or more cascaded hubs.

Root causes (three independent, additive factors):

  1. USB 2.0 Transaction Translator (TT) jitter cascading. A CMSIS-DAP v2 device runs full-speed (12 Mbps). The Mac's USB port is high-speed (480 Mbps). Each hub between them runs a TT to bridge the speed gap. Each TT adds approximately 1–2 ms of jitter to packet scheduling. CMSIS-DAP v2's command/response loop expects sub-millisecond turnaround time. With two cascaded TTs, jitter can exceed the probe firmware's timeout on the very first Info command. (Note: probe-rs list works because it uses descriptor reads cached by the kernel, not bulk transfers.)
  2. macOS IOUSB interface-binding race. When a device is 2+ hub tiers deep, the macOS IOUSB driver hands libusb a device handle (allowing enumeration), but claim_interface() on the bulk interface fails because the interface binds to a different device-tree node than the one libusb originally resolved. probe-rs surfaces this as "Error in the USB access."
  3. V_BUS droop through passive hubs. Cheap passive USB-C hubs deliver 4.6–4.8 V (below the 5.0 V nominal). Each downstream passive hub drops a further 100–200 mV at its passthrough switch. At the second hub, V_BUS can sag below 4.5 V. The RP2040 inside the CMSIS-DAP probe (which runs the probe firmware) misses USB packets when V_BUS sags during bulk-transfer current spikes, compounding the timeout from cause #1.

Fix (in priority order):

  1. Plug the probe directly into the Mac's USB port if possible (eliminates all three causes).
  2. If a hub is required, plug the probe into a single active USB hub (one with its own power supply). Active hubs have buffered TTs with predictable latency and clean 5.0 V output.
  3. Avoid cascading hubs (Mac → hub 1 → hub 2 → probe). If unavoidable, ensure each hub is powered (not passive) and has dedicated 5 V supply.
  4. On macOS, if the above doesn't help, try a different USB host controller. Some Macs have multiple controllers; switching physical ports can help (e.g., move from a Thunderbolt 3 dock's USB 3.1 hub to a native USB-A port if available).

RP2350-E12 — DPRAM corruption when clk_sys ≤ 48 MHz

Symptom: An RP2350-based device appears to enumerate cleanly but transfers corrupt mid-stream, or the device randomly drops off the bus during sustained traffic. RTT may show BUFF_STATUS events on endpoints with stale or partial data. Affects only RP2350 silicon — RP2040 doesn't have this errata.

Root cause: The RP2350 USB block's DPRAM at 0x50100000 is dual-ported between clk_sys (CPU side) and clk_usb (controller side, fixed 48 MHz). The dual-port arbitration logic assumes clk_sys > clk_usb. If clk_sys is at or below 48 MHz while the USB controller is enabled, DPRAM accesses can return stale data or silently drop writes — producing apparently-random transfer corruption. RP2350 datasheet §12.7.3.1, p. 1144.

Fix: Keep clk_sys ≥ 50 MHz any time MAIN_CTRL.CONTROLLER_EN = 1. Zephyr's default RP2350 board init runs clk_sys at 150 MHz, so this is a paper concern unless you've intentionally under-clocked the system. If you need to drop clk_sys below 50 MHz for power, disable USB first (MAIN_CTRL.CONTROLLER_EN = 0 or MAIN_CTRL.PHY_ISO = 1), then drop the clock; reverse on resume.

See Ch 14 §7 — Errata for the full datasheet quote and context.

RP2350 USB never enumerates after porting RP2040 driver code

Symptom: A USB driver that worked on RP2040 produces no enumeration on RP2350 — host sees no device at all, no IRQs fire, no line-state events. Port that works on every other MCU family fails specifically on RP2350.

Root cause: RP2350 added a new MAIN_CTRL.PHY_ISO bit (bit 2) at register offset USBCTRL_REGS_BASE + 0x040 that doesn't exist on RP2040. The reset value is 1 (PHY isolated from the digital core). RP2040 driver code never touched this bit because it didn't exist; if you copy that driver verbatim to RP2350, the PHY stays isolated and nothing works. RP2350 datasheet §12.7.2 calls this out as the one breaking change from RP2040: "You must clear the MAIN_CTRL.PHY_ISO bit at startup and after power down events."

Fix: In your driver init path, before any other USB activity, write 0 to bit 2 of MAIN_CTRL:

// Clear PHY_ISO before enabling controller. RP2350 only.
hw_clear_bits(&usb_hw->main_ctrl, USB_MAIN_CTRL_PHY_ISO_BITS);

Zephyr's udc_rpi_pico.c already does this; you'd only hit this when porting a non-Zephyr stack. See Ch 14 §6 for the full RP2350-vs-RP2040 delta.
