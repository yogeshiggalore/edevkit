Inside RP2350 — the USB hardware subsystem

Eight chapters of generic USB land on a chip. This chapter opens the chip and shows what's actually there — the four hardware blocks that together implement everything Parts I and II describe, plus the three channels by which a Cortex-M33 core talks to them.

You'll learn
  • The four hardware blocks: controller, PHY, DPRAM, PLL_USB — and exactly what each one does.
  • The three CPU↔USB channels (MMIO, shared DPRAM, IRQ) — including why there's no DMA path on RP2350.
  • The register map: USBCTRL_REGS_BASE = 0x50110000, USBCTRL_DPRAM_BASE = 0x50100000, USBCTRL_IRQ = 14, the bit layouts you actually touch.
  • What's new in RP2350's USB block vs RP2040 — including the one bit that bites every port (MAIN_CTRL.PHY_ISO).
  • The single RP2350 USB errata (RP2350-E12) — and the one-line workaround.

1. Three quick answers

If you're skimming, this whole chapter answers three questions:

QuestionAnswerSection
Is USB separate hardware? Yes — a discrete digital + analog IP block on the chip's bus fabric. Runs independently of the Cortex-M33 cores; a halted CPU can still receive a SETUP packet. §2, §3
How does the CPU talk to it? Three channels: MMIO registers (control/status, at 0x50110000), shared DPRAM (endpoint buffers, at 0x50100000), and interrupts (one IRQ line, vector 14). No DMA path — the CPU does the data movement. §4
What does the silicon contain? USB controller (digital FSM) + USB PHY (analog D+/D- driver/receiver) + 4 KB DPRAM + dedicated PLL_USB (48 MHz). All on-die — no external USB chip needed. §3

Two clock domains, one shared memory, one IRQ line. Everything inside the dashed box runs at clk_usb = 48 MHz; everything outside runs at clk_sys. The DPRAM is dual-ported — both sides can read and write simultaneously, with hardware arbitrating in the rare case of an exact-cycle collision.

Note — clk_sys constraint (RP2350-E12)

The dual-port arbitration only works when clk_sys > clk_usb. If you somehow run clk_sys ≤ 48 MHz while USB is active, DPRAM access timing breaks silently. RP2350-E12 (datasheet §12.7.3.1) is the only RP2350 USB errata, and the workaround is one line: keep clk_sys ≥ 50 MHz. Zephyr's default RP2350 board init runs clk_sys at 150 MHz, so this is a paper concern unless you go out of your way to under-clock for power.

3. The four pieces

3a. USB Controller — the digital FSM

The controller is a single state machine that does everything Ch 03 describes in hardware. It receives bytes from the PHY, identifies the PID, validates CRCs, recognises SETUP packets, routes them into the DPRAM, and sends ACK/NAK/STALL responses within the bus turn-around window (~7.5 bit-times at FS — too tight for any software response). On the TX side it generates TOKEN and DATA packets, computes CRCs, and clocks them out through the PHY. It manages the per-endpoint DATA0/DATA1 toggle automatically; firmware never thinks about it.

The controller can run as either a USB device or a USB host — the silicon supports both. Mode is selected by MAIN_CTRL.HOST_NDEVICE:

BitFieldMeaning
MAIN_CTRL @ USBCTRL_REGS_BASE + 0x040 (datasheet §12.7.5, Table 1198)
31SIM_TIMINGReduced timings for FPGA simulation. Reset = 0. Leave alone in real silicon.
2PHY_ISOIsolates the PHY from the digital core. Reset = 1 (isolated). Software MUST clear this before any USB traffic — see §6.
1HOST_NDEVICE0 = device mode, 1 = host mode. Reset = 0 (device).
0CONTROLLER_ENMaster enable. Reset = 0.

In device mode, the controller exposes 16 IN + 16 OUT endpoints (EP0 through EP15 in each direction); EP0 is always the control endpoint. In host mode, you get a single configurable EPx pair plus up to 15 hardware-polled interrupt-endpoint slots — enough to drive a hub of HID devices without per-poll software involvement (datasheet §12.7.1.1.1, §12.7.1.1.2).

Almost everything from Ch 03 happens entirely inside this block. Your firmware does not see PIDs, NRZI, or stuffed bits — it sees endpoint events and buffer pointers.

3b. USB PHY — the analog block

The PHY is the only analog block in the USB subsystem. Three jobs:

  1. Differential drive and receive on D+/D-. Output swing is rail-to-rail on the 3.3 V supply; receivers report J/K/SE0/SE1 line states up to the digital line-state-detect logic, which resolves them into bus-reset / suspend / resume / data events.
  2. NRZI encode/decode and bit-stuffing. The USB 2.0 wire format from Ch 02 §NRZI happens here. Firmware writes plain bytes into the DPRAM; the PHY translates them into the line transitions on the wire. (The "0 toggles, 1 holds" rule and the stuffed-bit-after-six-ones convention are PHY responsibilities.)
  3. On-die pull-ups (device mode) for FS speed signaling. The USB 2.0 spec mandates 1.5 kΩ ± 5 % on D+ for FS; RP2350 ships two programmable pull-up strengths, neither of which exactly matches the spec target:
SettingResistanceUSB 2.0 spec
SIE_CTRL.RPU_OPT = 0 (reset default)~1.2 kΩBelow the 1.425 kΩ floor — softer than spec.
SIE_CTRL.RPU_OPT = 1~2.3 kΩAbove the 1.575 kΩ ceiling — harder than spec.

In practice every host accepts both — the spec window is conservative, and real hosts treat any idle-state pull above their threshold as a connect event. The only time it matters is when you're qualifying against a strict compliance tester, in which case you need an external resistor and you set both internal options to whichever flips your D+ voltage furthest off the rail.

The 27 Ω external series resistors are not optional. Datasheet §1.2.2 requires a 27 Ω series resistor on each of USB_DP and USB_DM:

Quote from datasheet §1.2.2

"A 27 Ω series termination resistor is required on each pin, but bus pullups and pulldowns are provided internally."

This is a real difference from many other MCU USB blocks — RP2350 does not include the series-impedance termination on-die. Omit them on a custom PCB and you'll see eye-pattern violations and intermittent enumeration failures, especially with longer cables.

Pin assignment on the QFN-80 package (RP2350B, the part edevkit1 uses): USB_DP on pin 48, USB_DM on pin 46, USB_OTP_VDD on pin 51 (datasheet Table 2). Pins 46/48 can be repurposed as GPIO27 if USB is unused — that's new in RP2350; on RP2040 they were USB-only.

3c. DPRAM — the shared mailbox

4 KB of dual-ported SRAM at 0x50100000. Both the CPU (through clk_sys) and the USB controller (through clk_usb) access it through independent ports — no contention, no mailbox protocol, just shared memory with hardware arbitration in the (very rare) case of exact-cycle conflict. The first 256 bytes hold endpoint control structures; the remaining 3,840 bytes hold endpoint data buffers (datasheet §12.7.3.7, p. 1146).

Buffer control word (32-bit) — drives every transfer

This is the register firmware writes to "submit" a transfer. Two halves: bits 31:16 control buffer 1 (the second buffer in a double-buffered endpoint), bits 15:0 control buffer 0:

Bit(s)FieldMeaning
31FULL_1Buffer 1 contains valid data. Set by CPU on IN, by controller on OUT.

The transfer protocol from a firmware perspective is short: write your bytes into the buffer at the offset declared by the endpoint's BUFFER_BASE_OFFSET, set LENGTH_n, set PID_n (toggle), set FULL_n (for IN — controller sets it for OUT), set AVAILABLE_n. The controller eventually clears AVAILABLE_n and raises BUFF_STATUS in the IRQ. Zephyr's UDC driver wraps all this — your application code never touches these bits — but knowing the shape explains every "transfer hangs" symptom.

Symptom → Fix

"Transfer hangs after the first packet of a multi-buffer sequence." The AVAILABLE_n bit must be written after the rest of the buffer-control word has settled, because the dual-port arbiter can race a single-cycle write across both clock domains. The RP2350 ships with LINESTATE_TUNING.DEV_BUFF_CONTROL_DOUBLE_READ_FIX enabled by default which closes this race; don't disable it unless you have a very specific reason. (Datasheet §12.7.2.2.)

3d. PLL_USB — the dedicated 48 MHz clock

USB Full-Speed needs exactly 48 MHz (12 Mbps × 4-sample oversample). RP2350 has a dedicated PLL — separate from PLL_SYS — whose output is fixed at 48 MHz when fed the standard 12 MHz crystal. PLL_USB is also used by the ADC and the UART boot ROM, but the system clock can be anything PLL_SYS supports (up to 150 MHz on RP2350) without affecting USB.

This separation is why USB keeps working when you change the CPU clock (e.g., dropping clk_sys to save power between bursts). The catch — RP2350-E12 — is in §7.

Zephyr's RP2350 board init configures PLL_USB automatically. Application code does not normally touch it. The one time you would: implementing low-power suspend, where you disable PLL_USB to save the ~2 mA it consumes, and re-enable on bus resume.

4. How the CPU talks to the USB subsystem

Three channels — and an important non-channel.

4a. MMIO registers (0x50110000)

4b. Shared DPRAM (0x50100000)

Already covered in §3c. From the CPU's perspective it's just memory — load and store at byte / halfword / word granularity. The endpoint control words are at fixed offsets; the buffers are where you choose to put them, declared via each endpoint's BUFFER_BASE_OFFSET. There is no malloc — buffer placement is part of your driver's static layout.

4c. The single IRQ line — vector 14

The entire USB block raises one NVIC line, USBCTRL_IRQ = vector 14 on the Cortex-M33 NVIC table (datasheet §3.2 System Interrupts). The ISR reads INTS to dispatch — bus reset, SETUP arrived, buffers ready, error — services it, and write-1-clears the source bit. This is how Zephyr's USB stack stays responsive without polling: a SETUP at 1 µs into a 1 ms frame fires IRQ 14 immediately; the ISR runs the dispatcher; user code wakes via Zephyr's work queue.

If you ever set CONFIG_USBD_LOG_LEVEL_DBG=y in a sample's prj.conf and watch RTT, the storm of SETUP, BUFF_STATUS, SOF events you see is the ISR dispatching on these INTS bits.

4d. The non-channel: DMA

RP2350's general-purpose DMA controller has channels for SPI, I²C, ADC, PIO, UART — but no DREQ signal for USB. Endpoint data movement is CPU-driven: when BUFF_STATUS fires, the ISR (or a downstream worker thread) memcpys bytes between the DPRAM buffer and an application ring buffer. For the bandwidths FS USB delivers (~1 MB/s sustained), this is fine; the CPU spends well under 5 % of one core on data movement at that rate.

This is a real difference from RP2040 → some other MCU families. STM32H7's USB OTG_HS, for instance, has dedicated DMA. RP2350 chose silicon area over DMA throughput, on the well-founded assumption that 1 MB/s isn't worth a dedicated channel. Ch 27 revisits this when the topic is high-throughput bulk transfers.

The other consequence of "no DMA": SET_ADDRESS handling is firmware-driven. The controller does not auto-parse the SETUP payload — when a SET_ADDRESS(N) arrives, your ISR reads the SETUP buffer at DPRAM 0x00, extracts the new address, and writes it to ADDR_ENDP before completing the status stage. The exact ordering rule is: the host's status-stage IN token uses the old address; firmware must update ADDR_ENDP after the controller has acked the status stage but before the host's next SETUP. Zephyr's UDC driver handles this; you only need to know it exists when something else has gone wrong (see Ch 05).

5. Where Zephyr's UDC driver fits

This is the one place where everything else in the book stops being portable. Application code calling usbd_register_class() and usbd_enable() works the same on STM32, nRF52, ESP32, RP2040, and RP2350. Underneath that API, a per-MCU UDC (USB Device Controller) driver translates the portable calls into the actual register writes. On RP2350 that driver lives at $ZEPHYR_BASE/drivers/usb/udc/udc_rpi_pico.c — it knows 0x50110000, it knows the bit at position 26 of the buffer-control word means AVAILABLE_1, it services USBCTRL_IRQ.

This is why usb_01's main.c contains zero RP2350 register references. That's all hidden in udc_rpi_pico.c. If you ever need to debug a "the host sees us at the wrong address" problem, that file is where the breakpoint goes.

Reading udc_rpi_pico.c alongside this chapter is faster than reading the datasheet end-to-end. The driver is ~1500 lines and uses the exact field names (BUFF_STATUS, SIE_CTRL, EP_CTRL) that appear in §3 above.

6. What's new in RP2350 vs RP2040

The USB IP is largely the same — register addresses, endpoint count, DPRAM size, IRQ vector all carry over from RP2040 unchanged. Datasheet §12.7.2 lists ten enhancements. The ones that affect firmware:

FeatureWhat it isCompatibility
PHY isolation control MAIN_CTRL.PHY_ISO (bit 2) — new in RP2350. Powers down the PHY analog block while keeping the digital core powered. Breaks RP2040 software if not handled. Reset value is 1 (isolated). Software MUST clear it at startup before any USB activity.
USB pin GPIO mux USB_DP and USB_DM can be repurposed as GPIO27 when USB is unused. Additive. Default behaviour matches RP2040 — pins are USB unless explicitly remapped.
NAK-poll bulk-stop (host) NAK_POLL.STOP_EPX_ON_NAK + INTS.EPX_STOPPED_ON_NAK. Hardware can stop a polled bulk transfer when the device NAKs, instead of busy-polling. Additive. Useful when implementing a USB host driver for unbuffered devices.
Multi-hub fix LINESTATE_TUNING.MULTI_HUB_FIX — corrects inter-packet timeouts across long hub chains. Default off. Enable if your application sees enumeration failures behind two-or-more hub tiers (see appendix → cascaded hubs).
Device-suspend wake fix LINESTATE_TUNING.DEV_LS_WAKE_FIX — wake on any bus activity (K or SE0), not just K. Default ON in RP2350. Improves robustness vs RP2040 corner case.
DPRAM double-read fix LINESTATE_TUNING.DEV_BUFF_CONTROL_DOUBLE_READ_FIX — closes the dual-port arbitration race that needed 3-nop spacing on RP2040. Default ON. Removes the need for __NOP() __NOP() __NOP() sequences around AVAILABLE bit writes.
EP0 stop-on-short SIE_CTRL.EP0_STOP_ON_SHORT_PACKET (bit 19) — auto-stops EP0 on a short packet during data stage. Default off. Enables a cleaner state machine on devices with non-standard control-transfer flows.
Device SM watchdog DEV_SM_WATCHDOG register + interrupt source. Resets the device state machine if it wedges. Default off. Last-ditch defence against firmware bugs that leave the FSM in an unrecoverable state.
Device RX-error quiesce LINESTATE_TUNING.DEV_RX_ERR_QUIESCE — stop device RX on first error to avoid double-error IRQs. Optional. Useful for stress testing.
SIE RX chatter fix LINESTATE_TUNING.SIE_RX_CHATTER_SE0_FIX — handles hub-EOP-followed-by-valid-token sequences. Default ON. Closes a hub-interaction corner case.
RP2040 errata fixed RP2040-E2 (endpoint abort), E5 (bus reset), E3/E4 (host buffer flags), E15 (bus-error hang during IN) — all silicon-fixed. Pure improvement; no software workarounds needed.
The one bit that bites every port

If you're moving a working RP2040 USB driver to RP2350 and the device just won't enumerate — no IRQs, no line-state events — you forgot to clear MAIN_CTRL.PHY_ISO. The PHY is isolated by default at reset (= 1), and on RP2040 there was no such bit, so existing driver code doesn't touch it. Datasheet §12.7.2 quote: "You must clear the MAIN_CTRL.PHY_ISO bit at startup and after power down events." This is the single change that breaks otherwise-portable code.

7. Errata — RP2350-E12

Exactly one USB-related errata exists in the RP2350 datasheet:

RP2350-E12 — clk_sys must be > 48 MHz when USB is active

The DPRAM dual-port arbitration assumes clk_sys > clk_usb. If clk_sys ≤ 48 MHz while the USB controller is enabled, DPRAM accesses can return stale data or drop writes — silently. Symptom: enumeration appears to work but transfers corrupt, or the device randomly drops off the bus mid-transfer.

Workaround: keep clk_sys ≥ 50 MHz while USB is enabled. The Zephyr default of clk_sys = 150 MHz is well above the threshold; you'd only hit this if you intentionally under-clock for power, in which case disable USB first (MAIN_CTRL.CONTROLLER_EN = 0) before dropping the clock.

Datasheet §12.7.3.1, p. 1144.

That's the entire RP2350 USB errata list. Comparable RP2040 had five distinct USB errata (E2, E3, E4, E5, E15) — RP2350 silicon fixed all of them. Net: RP2350's USB block is one of the cleanest IP blocks in the chip from an errata standpoint.

8. Practical implications

  1. Endpoint count is generous. 16 IN + 16 OUT + EP0 in device mode. Most samples use 2–4 endpoints; only a multi-bulk-IN composite (think usb_25) starts feeling the limit, and even there the harder constraint is DPRAM space, not endpoint slots.
  2. DPRAM is the real budget. 4 KB total, 256 B reserved for endpoint control words, 256 B used by EP0's two 64 B buffers and their wrapping, leaving 3,584 B of headroom for application endpoints. With 64 B bulk packets, that's 56 buffers — well above any composite device's needs. The exception: isochronous endpoints with 1023 B MPS, where you can use up two-thirds of DPRAM with a single double-buffered iso stream.
  3. No external USB chip required. The PHY is on-die. Your PCB needs only USB-C connector + 27 Ω series resistors + a TVS array for ESD (recommended; not specified in the datasheet but standard practice). No USB redrivers, no level shifters.
  4. USB on its own clock. You can change clk_sys arbitrarily (above 48 MHz, per E12) without affecting USB. Useful for power management.
  5. PIO-based USB exists but isn't needed here. The pico-pio-usb project lets a PIO state machine bit-bang a second USB interface. Useful when you want a USB host AND a USB device on the same chip; not relevant for edevkit1, which uses the native peripheral as a device only.
  6. Power: PLL_USB consumes ~2 mA. Disable it during deep suspend; re-enable on bus resume. The Zephyr PM subsystem handles this if CONFIG_PM is enabled.

9. Verification milestone

The proof that all four hardware pieces from §3 are working in concert is one RTT log line you've already seen if you've run usb_01:

[00:00:00.323,000] <inf> usbd_core: Actual device speed 1

The 1 here is USBD_SPEED_FS. For that single number to appear, every block had to do its job:

  • The controller received BUS_RESET via INTS, dispatched it to the driver, completed the reset state machine.
  • The PHY drove the FS pull-up high (after firmware set SIE_CTRL.PULLUP_EN) and detected the host's J/K signaling.
  • The DPRAM received the host's first SETUP packet (GET_DESCRIPTOR(Device)) at offset 0x00, and EP0's IN buffer at 0x100 contained the device descriptor bytes the firmware loaded there.
  • PLL_USB was running at exactly 48 MHz the whole time — without that, the controller can't time-recover the host's bits and would never have seen the SETUP at all.

Find that line in your RTT and you've verified the whole chapter.

10. Going deeper

  • RP2350 datasheet § 12.7 (USB), pp. 1141–1181 — the full register and bitfield reference. RP-008373-DS-2.
  • RP2350 datasheet § 1.2.2 (Pin descriptions table) — pin 46 / 48 / 51 USB pin assignments + the 27 Ω external-resistor requirement.
  • RP2350 datasheet § 3.2 (System Interrupts) — full NVIC vector table including USBCTRL_IRQ = 14.
  • Zephyr RP2350 UDC driver: $ZEPHYR_BASE/drivers/usb/udc/udc_rpi_pico.c. The actual register-programming code; reading it is faster than the datasheet for any "how is X implemented in firmware" question.
  • Raspberry Pi Hardware Design Guide: github.com/raspberrypi/hardware-design-guide — recommended USB-C + RP2350 PCB layouts, including 27 Ω placement and ESD diode footprints.
  • USB 2.0 Specification, §7.1.5 and §7.1.6 — the spec context for the on-die pull-up and the bit-stuffing rules the PHY implements.
