Control transfers in depth

EP0 is the only endpoint that's always open in both directions, always there from the moment a device is reset, and always parsed with the same 8-byte Setup format. Every other USB transfer is data; control transfers are conversation.

You'll learn
  • The 8-byte Setup packet — every bit position, every field, every meaning.
  • The three control-transfer shapes (3-stage IN, 3-stage OUT, 2-stage no-data) and why the STATUS stage always flips direction.
  • STALL on EP0 — what it means, when it happens, and why it auto-clears without intervention.
  • Standard vs class-specific vs vendor-specific requests — and which level of the firmware handles each.
  • Why you can pack up to 4 bytes of payload into the Setup packet itself and skip the DATA stage entirely.

1. What EP0 is, and what makes it special

Every USB device has at least one endpoint: endpoint 0. EP0 is reserved by the spec for control transfers and is never enumerated in the descriptor tree — it's implicitly there from reset. Two properties make EP0 unique:

  • Bidirectional in one identity. Other endpoints have a direction bit in their address (0x01 is OUT, 0x81 is IN — these are two endpoints sharing a number). EP0 has no direction bit; the host decides direction per-transfer via the Setup packet.
  • Always available, even before configuration. Bulk / interrupt / isochronous endpoints come alive only after SET_CONFIGURATION. EP0 is alive from the bus reset onward — that's how the host can issue GET_DESCRIPTOR before knowing the device's address or class.

The trade-off: EP0 is slow. bMaxPacketSize0 is 8/16/32/64 (Full-Speed) and there's no concept of streaming — every transfer is bounded by the SETUP/STATUS framing. Control transfers are for configuration and command, not bulk data.

2. The 8-byte Setup packet

Every control transfer begins with a SETUP token followed by an 8-byte DATA0 packet on EP0 OUT. The 8 bytes have a fixed layout (USB 2.0 §9.3, Table 9-2):

OffsetSizeFieldMeaning
01bmRequestType3 bit-fields packed: direction, type, recipient. See §2a.
11bRequestThe request code itself. Standard, class, or vendor-defined depending on the type field in bmRequestType.
22wValueRequest-specific 16-bit value, little-endian. For GET_DESCRIPTOR it's (type << 8) | index; for SET_CONFIGURATION it's the config value; for a vendor request, whatever you defined.
42wIndexRequest-specific 16-bit value, LE. For an INTERFACE-recipient request, this is the interface number; for ENDPOINT-recipient, the endpoint address; for DEVICE-recipient, often a langid (e.g., for GET_DESCRIPTOR(STRING)) or zero.
62wLengthNumber of bytes the host expects to transfer in the DATA stage, LE. 0 means "no DATA stage" — the transfer is 2-stage.

That's the entire wire-level shape of a control request. Every standard request, every class request, every vendor request: same 8 bytes, different field values.

2a. bmRequestType bit layout

bit:    7        6 5      4 3 2 1 0
        ─        ─ ─      ───────────
       direction type     recipient

direction: 0 = host → device (OUT)    1 = device → host (IN)
type:      0 = standard               1 = class

3. The three transfer shapes

After the SETUP packet, the rest of the transfer is determined entirely by two values: direction (bit 7 of bmRequestType) and wLength. They combine into exactly three shapes:

3a. 3-stage host-IN (wLength > 0, direction = 1)

sequenceDiagram
  participant H as Host
  participant D as Device

  H->>D: SETUP (8 bytes)
  Note over D: parse bRequest, prepare response
  D-->>H: DATA in (1..N packets, total ≤ wLength)
  H->>D: STATUS out (ZLP)
  Note over H,D: Transfer complete — host received data, device knows it landed

Used for GET_DESCRIPTOR, GET_STATUS, GET_CONFIGURATION, and any vendor / class request that returns data. The device fills the DATA stage with up to wLength bytes; STATUS is a zero-length OUT from the host saying "I got it."

3b. 3-stage host-OUT (wLength > 0, direction = 0)

sequenceDiagram
  participant H as Host
  participant D as Device

  H->>D: SETUP (8 bytes)
  H->>D: DATA out (1..N packets, total = wLength)
  Note over D: process the payload
  D-->>H: STATUS in (ZLP)
  Note over H,D: Transfer complete — device received data, host knows it landed

Used for SET_DESCRIPTOR (rare), class-specific bulk-of-data setters (CDC SET_LINE_CODING), and any vendor request that ships a payload. The host sends wLength bytes after SETUP; STATUS is a zero-length IN from the device saying "consumed."

3c. 2-stage no-data (wLength = 0)

sequenceDiagram
  participant H as Host
  participant D as Device

  H->>D: SETUP (8 bytes; payload in wValue/wIndex)
  D-->>H: STATUS in (ZLP)
  Note over H,D: Transfer complete — payload was small enough to fit in Setup

Used for SET_ADDRESS, SET_CONFIGURATION, CLEAR_FEATURE, SET_FEATURE, and any request whose payload fits in the 4 bytes (wValue + wIndex) of the Setup packet itself. STATUS is always a zero-length IN — even though "direction" was OUT, status flips direction because there's no DATA the device needs to ACK in reverse.

Why STATUS always flips direction

The STATUS stage exists so the recipient of the DATA stage can signal "done." In a 3-stage IN transfer, the host receives DATA, so the host sends the STATUS ZLP. In a 3-stage OUT, the device receives DATA, so the device sends STATUS. In a 2-stage transfer, there's no DATA; STATUS is always an IN ZLP regardless. This isn't arbitrary; it's USB 2.0 §8.5.3, and it's the rule that lets the host and device synchronise on transfer completion without an explicit length-acknowledged field.

4. wLength is the host's upper bound, not the actual size

For 3-stage IN transfers, wLength is an upper bound on what the host is willing to accept — not a contract that the device must return exactly that many bytes. Two cases:

  • Device has fewer bytes than wLength requested: device returns what it has; if the response length is less than wLength AND is an exact multiple of bMaxPacketSize0, the device appends a ZLP (zero-length packet) so the host knows the transfer is done. Otherwise, the short packet itself signals end-of-transfer.
  • Device has more bytes than wLength requested: device returns exactly wLength, no more. The host explicitly bounded the transfer; truncation is the host's intent.

For 3-stage OUT transfers, wLength is what the host declared it will send — but it can short-packet the DATA stage early, terminating before wLength bytes are sent. The device's handler should always trust the actual received-byte count (the buf->len field in Zephyr) over setup->wLength.

G17 / G18 from usb_04

The "use buf->len, not setup->wLength" rule is one of the three new gotchas introduced in usb_04 §5. Both directions have the same problem — trusting the host's declared length over what actually arrived on the wire opens you up to reading uninitialised memory or padding garbage onto the wire.

5. STALL — the third response (after ACK and NAK)

A device has three ways to respond to a Setup packet:

ResponseMeaningHost behaviour
ACK"Got it; will handle."Proceeds to DATA / STATUS stage.
NAK"Got it but busy; retry."Resends the SETUP after a short delay.
STALL"Cannot handle this request."Aborts the transfer; control pipe is "halted."

STALL is the wire-level error signal. Common causes:

  • Unknown bRequest (device's vendor-class handler doesn't recognise the request code).
  • wValue / wIndex out of range (e.g., GET_DESCRIPTOR(STRING, 99, ...) when only 6 strings exist).
  • Request type mismatch (e.g., a CLASS-recipient request to a device that's pure vendor).
  • Deliberate — usb_04's STALL_ME request returns -ENOTSUP from the class callback specifically to demonstrate STALL handling.

5a. EP0 STALL auto-clears

This is the most important property of EP0 STALL, per USB 2.0 §8.5.3.4:

EP0 STALL auto-clears on the next SETUP

Unlike STALLs on bulk or interrupt endpoints (which require an explicit CLEAR_FEATURE(ENDPOINT_HALT) from the host to recover), EP0 STALLs clear automatically the moment the next SETUP token arrives. The host doesn't have to do anything special — it just issues its next request and the device responds normally.

This is why STALL on EP0 is cheap. Returning STALL for an unrecognised vendor request is the correct behaviour; the host gets an error from its libusb_control_transfer() call, but the pipe is healthy for the next request.

5b. How Zephyr signals STALL from a class callback

The class API has two control callbacks: control_to_host (for IN requests) and control_to_dev (for OUT requests). Both return int. The convention is:

ReturnMeaningWire
0Handled successfully.Core completes the transfer with ACK/data.
-ENOTSUPNot mine.Core sends STALL on EP0. Auto-clears on next SETUP.

Returning 0 for a request you didn't actually handle (e.g., your switch statement's default case) is silently wrong: the core ACKs the transfer, the host moves on thinking everything succeeded, and the bytes the host expected never arrive. Always return -ENOTSUP for the cases you don't handle.

6. Three families of control requests

The type bits of bmRequestType (bits 5–6) split control requests into three families. Each is handled at a different level of the firmware:

FamilyType bitsDefined byHandled byExample
Standard00USB 2.0 §9.4USB core, transparentlyGET_DESCRIPTOR, SET_CONFIGURATION
Class01Class spec (HID, CDC, MSC, …)The class driver (if device declares that class)HID GET_REPORT, CDC SET_LINE_CODING
Vendor10YouYour control_to_* callbacksusb_04 GET_UPTIME

The 11 standard requests (USB 2.0 Table 9-3) are handled by the Zephyr core without your involvement — that's how every prior usb_* sample worked even though no callbacks were defined. Class requests fire only if you declared a recognised class in bInterfaceClass; usb_02/03/04 use 0xFF (vendor specific), so class requests never reach the device. Vendor requests are the open territory — you define both the request codes and the response logic.

6a. Routing — who gets which request

Within Zephyr's class API, control requests are routed by recipient (bits 0–4 of bmRequestType), not by type:

RecipientwIndex carriesRoutes to
DEVICE (0)0, or langid for some standard requestsPer-bRequest device vreq table (if registered); else device-level vendor handler.
INTERFACE (1)Interface numberClass owning that interface, via control_to_*.
ENDPOINT (2)Endpoint addressClass owning that endpoint, via control_to_*.
OTHER (3)Implementation-definedRare; mostly hub class.

This is why a vendor class that declares its requests as recipient = INTERFACE needs to filter on RequestType.type inside its callback: the core delivered the request because the recipient matched, not because the type matched. Standard or class requests with the same recipient would land in the same handler. G16 in usb_04 §5 shows the one-line guard:

if (setup->RequestType.type != USB_REQTYPE_TYPE_VENDOR) {
    return -ENOTSUP;
}

That kicks the request back to the core, which routes it to standard-request handling or returns STALL — exactly what we want.

Common pitfalls

Symptom → Fix

"My vendor request times out with no handler ever firing." Recipient mismatch: host used CTRL_RECIPIENT_DEVICE but device routes only INTERFACE, or wIndex doesn't match the interface number. The core silently drops requests that don't match any registered handler. Add a LOG_INF at handler entry to confirm dispatch; if it never fires, the routing's wrong.

Symptom → Fix

"libusb_control_transfer() returns success but my callback never ran." Almost certainly a GET_STATUS or other standard interface-recipient request that fell through to the core's default standard-request handler. The core handled it without your callback. If you intended your vendor handler to fire, check the bmRequestType.type bits — you probably sent 0x81 (standard) instead of 0xC1 (vendor).

Symptom → Fix

"After STALL, subsequent requests also fail." Shouldn't happen on EP0 — STALL auto-clears on next SETUP (§5a, USB 2.0 §8.5.3.4). If it persists, suspect (a) the host stack got confused (replug to reset), or (b) the request handler crashed (check RTT for fault output).

Symptom → Fix

"3-stage IN request returns N bytes, host expected M and got short." Working as designed — wLength is an upper bound (§4). If the host needs exactly M bytes, the device must produce M bytes; the core won't pad on your behalf.

Symptom → Fix

"3-stage OUT handler reads garbage past the intended payload." Using setup->wLength instead of buf->len as the byte count (§4, G18). Host short-packeted; only buf->len bytes are valid; reading further is uninitialised memory.

Symptom → Fix

"2-stage request expects 2 bytes of data in DATA stage." wLength > 0 makes a request 3-stage automatically. To use a 2-stage transfer with payload in the Setup packet, set wLength = 0 and pack the data into wValue (16 bits) or wValue + wIndex (32 bits). No DATA stage will occur.
