Device, configuration, interface, endpoint

The four-level tree that every USB device speaks — and the one request that reads the entire tree in a single transfer.

You'll learn
  • The four descriptor types and how they nest into a tree.
  • Why a single GET_DESCRIPTOR(configuration) returns the configuration descriptor and every interface and endpoint descriptor that follows.
  • What wTotalLength measures, and why it's the single field most likely to be miscalculated.
  • The bit-level layout of bEndpointAddress and bmAttributes — direction, EP number, transfer type, sync, usage.
  • The difference between the device-level class triple and the interface-level class triple, and which one drivers actually look at.

1. The descriptor hierarchy

A USB device describes itself with a nested tree of descriptors. Each descriptor is a small fixed-format byte blob; the first two bytes are always bLength (size in bytes) and bDescriptorType (what kind of descriptor this is). The tree has exactly four levels:

Three facts to internalise before going further:

  1. Exactly one device descriptor per device. No alternatives, no plurals. It carries vendor / product / class identifiers and tells the host how many configurations exist.
  2. One configuration is "active" at a time. Multi-config devices exist (USB-Ethernet adapters often ship one config for "Ethernet-only" and another for "Ethernet + serial"), but only one is selected at any moment. 90% of devices, including usb_02, ship a single configuration.
  3. Endpoints belong to interfaces, not devices. An interface is a logical function (e.g., "serial port", "mass storage"); its endpoints are the data pipes that function uses. Endpoint 0 is implicit and shared by every interface — it's the control pipe and is never listed in any descriptor.
Why this matters

Drivers bind to interfaces, not to devices. A single physical USB device can have multiple interfaces, each loaded by a different driver — that's how a USB headset can simultaneously present an audio interface (claimed by the audio driver) and an HID interface for the volume buttons (claimed by the HID driver) without anybody coordinating.

2. The device descriptor

18 bytes. Read in two control transfers during enumeration (see Ch 05): first 8 bytes to discover bMaxPacketSize0, then the full 18 bytes after addressing. The bytes usb_02 returns:

OffsetHexFieldMeaning
0x0012bLength18 — the entire device descriptor is exactly 18 bytes.
0x0101bDescriptorTypeDEVICE (0x01).
0x0200 02bcdUSBUSB 2.0.0 in BCD, little-endian (0x0200). Required to be 2.x even for a Full-Speed-only device.
0x0400bDeviceClass0 — "see interface descriptors." Forces the host to look one level deeper for class info.
0x0500bDeviceSubClass0 — meaningful only when bDeviceClass ≠ 0.
0x0600bDeviceProtocol0 — same.
0x0740bMaxPacketSize064 — max packet size for EP0 (control pipe). Valid values for FS: 8, 16, 32, 64.
0x0809 12idVendor0x1209 (pid.codes open-hardware umbrella). LE on wire so bytes are 09 12.
0x0AE2 EDidProduct0xede2 for usb_02. LE.
0x0C00 01bcdDevice0x0100 — device revision. Increment to bust the macOS string-descriptor cache when iterating during development.
0x0E00iManufacturerString index 0 → no manufacturer string. (Ch 07 turns this on.)
0x0F00iProduct0 → no product string.
0x1000iSerialNumber0 → no serial number string.
0x1101bNumConfigurations1 — single config. The host iterates 1..bNumConfigurations to discover all configs.

The first byte of every descriptor type is bLength and the second is bDescriptorType. That two-byte prefix lets the host parse a flat byte stream into a tree without needing a separate length-prefixed structure — every descriptor is self-delimiting.

3. The configuration descriptor

9 bytes — and yet the entire sub-tree (every interface and endpoint below this config) is read in a single GET_DESCRIPTOR(configuration) transfer. That's because of wTotalLength:

OffsetHexFieldMeaning
0x0009bLength9 — this descriptor alone is 9 bytes…
0x0102bDescriptorTypeCONFIGURATION (0x02).
0x0220 00wTotalLength32 — total bytes for this config + every interface + every endpoint that follows. For usb_02: 9 (config) + 9 (iface) + 7 (ep_out) + 7 (ep_in) = 32.
0x0401bNumInterfaces1 interface in this configuration.
0x0501bConfigurationValue1 — the value the host passes to SET_CONFIGURATION to activate this config. Configurations are 1-indexed (0 means "unconfigured").
0x0600iConfigurationString index 0 → no config string. Ch 07 turns this on.
0x07A0bmAttributesBit 7 = 1 (reserved, must be set); bit 5 = 1 (Remote Wakeup); bit 6 = 0 (not self-powered).
0x0832bMaxPower50 — in 2 mA units. 50 × 2 = 100 mA, the default per USB 2.0 §9.6.3.

3a. wTotalLength — the one field most likely to break

When the host issues GET_DESCRIPTOR(configuration, wLength=0xFFFF), the device returns wTotalLength bytes — the configuration descriptor itself, immediately followed by every interface descriptor, followed by every endpoint descriptor (and any class-specific descriptors interspersed). The host parses the resulting byte stream into a tree using each descriptor's bLength prefix.

This means wTotalLength must equal the sum of every descriptor byte the device emits in the response. Get it wrong by even one byte and one of two failures happens:

  • Too small → host parses only the first wTotalLength bytes; trailing endpoints are silently dropped. Symptom: device enumerates, but pipes don't open.
  • Too large → host waits for bytes that never arrive, the IN transfer times out. Symptom: device "won't enumerate."
Zephyr writes wTotalLength for you

In Zephyr USB-Device-Next, wTotalLength is computed by the core at usbd_init() time by walking the class's fs_desc[] array and summing each descriptor's bLength. You initialise the field to 0 in USBD_CONFIGURATION_DEFINE and the core fills it in. Hand-rolling a USB stack? Compute this yourself, get it wrong, and you'll spend a day debugging.

4. The interface descriptor

OffsetHexFieldMeaning
0x0009bLength9.
0x0104bDescriptorTypeINTERFACE (0x04).
0x0200bInterfaceNumber0 — interfaces are 0-indexed within a configuration.
0x0300bAlternateSetting0 — default alternate setting. Multi-alt interfaces (e.g., audio bandwidth modes) use higher values; not relevant for usb_02.
0x0402bNumEndpoints2 — number of endpoints this interface uses, not counting EP0 (which is shared).
0x05FFbInterfaceClass0xFF — Vendor Specific. No standard driver loads.
0x06FFbInterfaceSubClass0xFF — vendor-defined.
0x07FFbInterfaceProtocol0xFF — vendor-defined.
0x0800iInterfaceString index 0 → no interface name. Ch 07 turns this on.

The interface descriptor is the level where drivers bind. Linux's usb-storage looks for any interface with class 0x08; the HID core looks for class 0x03; Windows' WinUSB matches on a custom 0xFF with a particular GUID. A device's "personality" is decided by the interface class triple, not the device class triple.

4a. Alternate settings

A single interface number can appear multiple times in the descriptor tree, distinguished by bAlternateSetting. Each alternate carries its own endpoint set. The host issues SET_INTERFACE(alt) to switch between them — typically used by isochronous classes (audio, video) to swap between idle (zero endpoints, zero bandwidth) and active (1+ endpoints, reserved bandwidth) modes. usb_02 has only the default alt setting; multi-alt examples land in later samples.

5. The endpoint descriptor

7 bytes — the smallest descriptor type — but the highest bit-density:

EP OUT (bulk, address 0x01)
0x0007bLength7.
0x0105bDescriptorTypeENDPOINT (0x05).
0x0201bEndpointAddressEP1 OUT. Bit 7 = 0 (OUT), bits 0–3 = 1 (EP number). See §5a.
0x0302bmAttributesBulk. See §5b for the bit decoding.
0x0440 00wMaxPacketSize64 bytes (LE). Max payload per packet on this EP.
0x0600bInterval0 for bulk and isochronous; for interrupt EPs this is the polling interval in (FS) ms or (HS) 125-µs microframes.

5a. bEndpointAddress bit layout

bit:    7    6 5 4    3 2 1 0
        ─    ─────    ───────
       DIR  reserved  EP number (0..15)

DIR = 1 → IN  (device → host)
DIR = 0 → OUT (host → device)

Two endpoint descriptors sharing the same EP number but different directions form an IN/OUT pair at the controller level. They consume one hardware endpoint slot on most controllers (including the RP2350's), so pairing IN and OUT on the same number is the common idiom for bulk-data classes (see usb_02 — EP1 OUT and EP1 IN).

5b. bmAttributes bit layout

bit:    7 6     5 4         3 2          1 0
        ─ ─     ─ ─         ─ ─          ─ ─
       reserved usage     sync           transfer type

Transfer type (bits 0-1):
  00 = Control     (only used for EP0, never declared in descriptors)
  01 = Isochronous
  10 = Bulk        ← usb_02 uses this
  11 = Interrupt

Sync (bits 2-3) — only meaningful for isochronous:
  00 = No sync   01 = Async    10 = Adaptive   11 = Sync

Usage (bits 4-5) — only meaningful for isochronous:
  00 = Data      01 = Feedback 10 = Implicit feedback data

For bulk and interrupt endpoints, bits 2–7 are reserved-zero. Only bits 0–1 carry information. The rich sync/usage encoding exists for isochronous audio/video endpoints — covered in Ch 25.

5c. wMaxPacketSize — the size of one packet, not one transfer

USB transfers are made of multiple packets. wMaxPacketSize caps the size of an individual packet on the wire; a larger transfer is split into ceil(transfer_size / wMaxPacketSize) packets, terminated by a short packet (or a ZLP if the transfer is an exact multiple of wMaxPacketSize). For Full-Speed bulk endpoints, only 8, 16, 32, 64 are legal; 64 is the standard pick because anything smaller leaves throughput on the table.

6. How the host walks the tree

The device descriptor is read first (during enumeration; see Ch 05). To discover the rest of the tree, the host issues a single control transfer:

sequenceDiagram
  participant H as Host
  participant D as Device

  Note over H,D: After SET_ADDRESS and partial device-descriptor read
  H->>D: GET_DESCRIPTOR(configuration, index=0, wLength=9)
  D-->>H: 9 bytes — config descriptor only
  Note over H: Reads wTotalLength = 32 from the response
  H->>D: GET_DESCRIPTOR(configuration, index=0, wLength=32)
  D-->>H: 32 bytes — config + iface + ep_out + ep_in concatenated
  Note over H: Parses byte-by-byte using each bLength prefix

Two-phase to avoid blind big reads: first 9 bytes to learn the real size, then the full wTotalLength. Some hosts skip the first phase and request a large fixed value (e.g., 255) up front; the device truncates if it has less, so both approaches work.

The full byte-stream the host receives in the second transfer is:

byte  meaning
  0    bLength = 9   (configuration descriptor begins)
  1    bDescriptorType = 0x02 (CONFIGURATION)
  ...
  8    end of configuration descriptor
  9    bLength = 9   (interface descriptor begins — host knows it's an interface from byte 10)
 10    bDescriptorType = 0x04 (INTERFACE)
  ...
 17    end of interface descriptor
 18    bLength = 7   (endpoint descriptor begins)
 19    bDescriptorType = 0x05 (ENDPOINT)
  ...
 24    end of EP OUT
 25    bLength = 7   (next endpoint)
 26    bDescriptorType = 0x05
  ...
 31    end of EP IN — done (wTotalLength = 32)

This is why every descriptor's first byte is bLength: it makes the byte stream walkable without an external length array. The host advances by the current descriptor's bLength, reads the next descriptor's first two bytes, dispatches on bDescriptorType.

7. Class triples — device-level vs interface-level

Both the device descriptor (§2) and the interface descriptor (§4) carry a (class, subclass, protocol) triple. Which one drivers consult depends on the value of the device-level bDeviceClass:

bDeviceClassMeaningDrivers consult
0x00"See interface descriptors"Interface class triple — one driver per interface.
0xEF + subclass 0x02 + proto 0x01Interface Association (composite)Interface class triple, but interfaces are grouped via IADs (Ch 19).
0x09HubDevice-level — hub class is special-cased.
Other (e.g., 0x02 CDC)Device is a single-function device of that classDevice-level. Rare in modern devices.

The overwhelming convention is bDeviceClass = 0x00 and let each interface declare its own class. This lets multi-function (composite) devices have a CDC interface and an HID interface in the same configuration without conflict. usb_02 follows the convention.

Zephyr's default is not the convention

Zephyr USB-Device-Next defaults the device-level triple to 0xEF / 0x02 / 0x01 (IAD) on every device, even single-interface ones. Most hosts accept this without complaint, but lsusb output will read "Miscellaneous Device" instead of "Vendor Specific Device" until you call usbd_device_set_code_triple(&ctx, USBD_SPEED_FS, 0, 0, 0) between usbd_init and usbd_enable (see the Zephyr USBD reference).

Common pitfalls

Symptom → Fix

"Device shows in lsusb but no endpoints are listed." Either bNumEndpoints on the interface descriptor is 0, or wTotalLength on the configuration is too small to include the endpoint descriptors. Re-count wTotalLength: it's the byte-sum of every descriptor in the sub-tree, not just the configuration descriptor itself.

Symptom → Fix

"GET_DESCRIPTOR(configuration) never returns." wTotalLength is larger than the bytes you actually emit. The host waits for more bytes that the device will never send; the IN transfer times out. Recount; if you're hand-rolling a USB stack, this is the most common bug in the first week.

Symptom → Fix

"Two endpoints with the same address show as one." EP IN and EP OUT with the same number (e.g., 0x01 and 0x81) are two distinct endpoints, not duplicates — they share the EP number, not the address. lsusb prints them separately. If a tool collapses them, it's a tool bug.

Symptom → Fix

"bmAttributes looks right but the EP behaves wrong." Check the full byte, not just bits 0–1. A stray bit in 4–5 turns "bulk" into "bulk with isochronous-usage", which most hosts handle gracefully but some sniffers misdecode. Use 0x02 for plain bulk OUT/IN, 0x03 for plain interrupt, 0x01 for isochronous with no sync.

Symptom → Fix

"bMaxPower is reported in mA but my 50 shows as 100 mA." Correct — the field is in 2 mA units (USB 2.0 §9.6.3). A value of 250 means 500 mA, the FS bus maximum. Setting it to 100 requests 200 mA.
