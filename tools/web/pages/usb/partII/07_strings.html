String descriptors & per-unit identity

Strings sit outside the descriptor tree, fetched by index in a separate transfer — and that gap is what makes localised product names, runtime-derived serials, and "Vendor Specific Device → human readable name" all possible.

You'll learn
  • Why strings are not embedded in the configuration response — and the transfers that fetch them instead.
  • The LANGID descriptor at index 0, and why every host refuses to read anything else until it has been served.
  • UTF-16LE encoding on the wire vs ASCII7 in the firmware source, and why mixing them silently corrupts non-ASCII characters.
  • Late-binding string indices — the iface descriptor declares "iInterface = N" but the bytes at index N are decided at boot, not compile time.
  • How to derive a per-unit, globally-unique serial from the silicon's chip ID with zero per-unit production work.

1. Strings sit outside the tree

When the host fetched the configuration descriptor in Ch 06, it got 32 bytes — the configuration, the interface, and both endpoint descriptors, concatenated. Notice what it did not get: not a single byte of human-readable text.

That's by design. Strings are stored separately. The device descriptor has three index fields (iManufacturer, iProduct, iSerialNumber); the configuration descriptor has one (iConfiguration); the interface descriptor has one (iInterface). Each is a 1-byte index into the device's string-descriptor table — fetched separately, one string at a time, with the request:

GET_DESCRIPTOR(STRING, index, langid)

The reasons for this separation:

  • Localisation. One device can ship the same product name in English, French, Japanese, etc. — the host picks the langid it wants from a list the device advertises.
  • Variable size. Strings can be arbitrarily long. Inlining them into the configuration response would make wTotalLength dependent on language, complicating descriptor parsing.
  • Lazy fetch. Hosts only read strings they need to display. A driver that only needs the class/subclass triple to bind never fetches the iProduct string at all.
Mental model

The descriptor tree is the structural blueprint — fixed-size fields, indices, types. The string table is the presentation layer — variable-size text, fetched on demand, optionally localised. Indices in the tree point into the table, but the table is fetched separately.

2. Index 0 — the LANGID descriptor

Every other string fetch requires a langid argument. Where does the host learn which langids the device supports? From the descriptor at index 0, which has a special format: instead of a UTF-16LE string, it's an array of 2-byte language IDs.

OffsetHexFieldMeaning
0x0004bLength4 — two bytes of header + one 2-byte langid. (For multi-language devices: 2 + 2*N.)
0x0103bDescriptorTypeSTRING (0x03) — same type byte as every other string descriptor.
0x0209 04wLANGID[0]0x0409 — English (United States). LE on wire so bytes are 09 04.

USB 2.0 §9.6.7 reserves index 0 for this LANGID descriptor and requires the host to read it before reading any other string. Without LANGID at index 0, requests like GET_DESCRIPTOR(STRING, 1, 0x0409) have no defined behaviour — the host can't know whether the device supports the langid it's asking for.

For the great majority of devices, single-language English-US (0x0409) is plenty. Multi-language devices simply append more 2-byte langids to the array; the host picks the one matching its locale.

Skipping LANGID is the most common string-descriptor bug

If a device doesn't register a LANGID descriptor at index 0, every string fetch fails. Different hosts respond differently: Linux logs and gives up, Windows shows the device as "Unknown Device", macOS displays whatever it cached previously. The symptom is "names don't show up"; the cause is always index 0.

3. UTF-16LE on the wire

String descriptors at indices 1+ are UTF-16LE on the wire, regardless of how they're stored in firmware. The byte layout for a string "edevkit" (7 ASCII chars):

OffsetHexFieldMeaning
0x0010bLength2 + 2 × 7 = 16. Header + one UTF-16 code unit per character.
0x0103bDescriptorTypeSTRING (0x03).
0x0265 00UTF-16LE 'e'0x0065 little-endian.
0x0464 00UTF-16LE 'd'0x0064.
0x0665 00UTF-16LE 'e'
0x0876 00UTF-16LE 'v'
0x0A6B 00UTF-16LE 'k'
0x0C69 00UTF-16LE 'i'
0x0E74 00UTF-16LE 't'

Two consequences for firmware:

  1. String length on the wire is 2 × (character count). A 32-character interface name like "Vendor (descriptor demo, no I/O)" takes 66 bytes on the wire (2 header + 64 UTF-16LE).
  2. Non-ASCII characters require multi-byte UTF-16LE encoding. An em-dash (U+2014) is 14 20 on the wire; a naïve "widen each byte to 2 bytes" implementation produces 14 00, which is U+0014 (a control character), not the em-dash. Many USB stacks (Zephyr included) provide an "ascii7" path that does the naïve widening only; non-ASCII content needs pre-encoded UTF-16LE bytes.
G12 — wire bytes are UTF-16LE, source is ASCII7

Zephyr's USBD_DESC_PRODUCT_DEFINE("My—Product") with an em-dash compiles cleanly but emits wrong bytes on the wire. Stick to printable ASCII7 (0x20–0x7E) for string literals, or hand-craft a UTF-16LE byte array and disable the ascii7 flag on the descriptor node. The G12 defense in usb_03 §5 has the implementation details.

4. Indices are runtime-assigned, not hard-coded

The interface descriptor declares iInterface = 4. The configuration descriptor declares iConfiguration = 5. Where do those numbers come from?

They come from registration order. The stack assigns string indices in the order usbd_add_descriptor() is called: the first call gets index 0, the second gets index 1, and so on. So if your firmware registers strings in this order:

1. LANGID            → idx 0  (mandatory; spec)
2. Manufacturer      → idx 1
3. Product           → idx 2
4. Serial            → idx 3
5. Interface         → idx 4   ← stored in iface.iInterface
6. Configuration     → idx 5   ← stored in cfg_desc.iConfiguration

… then the iface descriptor's iInterface field needs to carry 4 and the configuration's iConfiguration needs to carry 5. But neither value is known at compile time — they depend on the runtime order of usbd_add_descriptor calls.

4a. The late-binding moment

sequenceDiagram
  participant App as main()
  participant Core as USB core
  participant Wire as Wire (post-enumeration)
  Note over App: iface.iInterface = 0 at compile time
  App->>Core: usbd_add_descriptor(&lang)        idx 0
  App->>Core: usbd_add_descriptor(&mfr)         idx 1
  App->>Core: usbd_add_descriptor(&product)     idx 2
  App->>Core: usbd_add_descriptor(&serial)      idx 3
  App->>Core: usbd_add_descriptor(&iface_str)   idx 4
  Note over App: ← Patch: iface.iInterface = usbd_str_desc_get_idx(&iface_str)  // = 4
  App->>Core: usbd_add_configuration(&cfg)      stack auto-adds cfg_str → idx 5
  App->>Core: usbd_init()                       validates the tree
  Note over Wire: Host issues GET_DESCRIPTOR(iface)
  Wire-->>Wire: iInterface byte = 4
  Note over Wire: Host issues GET_DESCRIPTOR(STRING, 4, 0x0409)
  Wire-->>Wire: "Vendor (descriptor demo, no I/O)"

The descriptor blob's iInterface byte is patched after registration, before usbd_init validates the tree. This pattern is the "late binding" of USB strings: the structural descriptor and the string it references are connected at runtime, not at compile time. Reorder the usbd_add_descriptor calls and the index byte changes — but if it's read from usbd_str_desc_get_idx() and patched into the struct, the firmware Just Works regardless of order.

iConfiguration is the easier sibling

Zephyr's USBD_CONFIGURATION_DEFINE(cfg, ..., &cfg_str) takes a pointer to the config-string descriptor as its fourth argument. usbd_add_configuration() registers cfg_str on your behalf and writes iConfiguration into the cfg descriptor automatically. You handle iInterface by hand; the stack handles iConfiguration for you. Don't try to register cfg_str twice — see G15 in usb_03 §5.

5. Per-unit identity — the serial number

The iSerialNumber field exists so that two physically-identical USB devices can be distinguished on the same host. Without unique serials, your OS (especially Windows) may assign the same COM port number or mount point to whichever device plugged in first, leading to surprising behaviour when you have two boards on one machine.

5a. Where unique numbers come from

Three common sources, ordered by quality:

SourceQualityNotes
Silicon chip IDBestMost modern MCUs ship a 64-bit unique ID laser-fused at manufacture. RP2350 exposes this through hwinfo_get_device_id(). Globally unique, board-independent, zero per-unit production work.
External EEPROM / one-time-programmable regionGoodRequires a production step to write. Adds component cost but lets you re-flash firmware without losing identity.
Random at first boot, persisted to flashWorkableCheapest. Random number space large enough to avoid collisions in any practical population (≥ 64 bits). Requires writable flash and a persistence layer (LittleFS, raw page).

5b. The EDV-XXXX-YYYY-ZZZZ format

The production edevkit1 firmware uses a 12-hex-character serial derived from the first 6 bytes of the RP2350 chip ID, formatted in three 4-char groups for readability:

EDV-A1B2-C3D4-E5F6
└┬┘ └─────┬─────┘
"EDV"    first 6 bytes of hwinfo_get_device_id(), formatted as %02X
prefix   per chip

18 characters total, 38 bytes on the wire after UTF-16LE encoding. The "EDV" prefix is a courtesy to humans grepping for edevkit serials in logs; the host doesn't care about the format and treats the entire string as opaque.

Why not all 64 bits?

USB string descriptors are limited to 256 bytes total (one byte for bLength). At 2 bytes per UTF-16LE character, that's ~125 characters maximum, far more than 16 hex characters would need. The 6-byte (48-bit) prefix is plenty unique to the practical population of any one product — the birthday-paradox collision probability with 1M devices and 48-bit serials is < 0.002%. Twelve hex characters also fits on a printed sticker.

6. Host-side caching

Operating systems cache string descriptors aggressively, keyed on the tuple (VID, PID, bcdDevice). The cache survives reboots; macOS in particular is famous for serving stale strings well after a device's firmware has been reflashed.

The intended pattern is: bump bcdDevice by one whenever you ship a string change. Hosts treat any tuple change as a new device, drop the cached strings, and re-fetch. During iterative development this is a one-byte change in USBD_DEVICE_DEFINE; in production firmware it's the device-revision field and bumps for any wire-visible change.

Symptom: rebuilt firmware, host still shows old strings

macOS, Windows, even some Linux distros cache. Increment bcdDevice. If you can't (because production firmware is shipping that exact value), the workarounds are sudo killall -HUP USBHostBSDClient on macOS, replug + driver reinstall on Windows, or echo 0 > /sys/bus/usb/devices/.../authorized; echo 1 > ... on Linux.

Build this

Sample: firmware/samples/usb/usb_03/ · design doc

The sample registers all six standard string descriptors (LANGID + Manufacturer + Product + Serial + Interface + Configuration), derives the serial at boot from hwinfo_get_device_id(), and applies the late-binding pattern from §4 to keep iInterface in sync with whatever index the stack assigns. After flashing, lsusb -v -d 1209:ede3 on Linux should print human-readable text at every level of the descriptor tree — and the serial will be unique to your specific board.

Common pitfalls

Symptom → Fix

"Manufacturer / Product / Serial all show as empty or '(error)'." LANGID descriptor at index 0 is missing or not registered first (§2). Every other string fetch depends on LANGID having been served. Confirm USBD_DESC_LANG_DEFINE(lang_desc) is the first usbd_add_descriptor() call.

Symptom → Fix

"iInterface shows as 0 on the host even though I registered an interface string." The late-binding patch (§4a) didn't run, or ran after usbd_init (which validates the tree). Order must be: register strings → query indices → patch descriptors → call usbd_init.

Symptom → Fix

"Product string truncates mid-word on Linux but works on macOS." The string literal contains a non-ASCII character (em-dash, accented letter) and the firmware's ASCII7 widening path produced wrong UTF-16LE bytes (§3, G12). Replace with ASCII7 substitutes (— → -) or hand-encode UTF-16LE.

Symptom → Fix

"Two boards show the same serial." Either (a) hwinfo_get_device_id() returned a board-default constant — possible on emulators or unconfigured boards but rare on real RP2350 silicon, or (b) snprintf formatting is masking the chip-ID bytes (bug in the format string). Log the raw id[0..7] bytes via RTT and compare them across boards first.

Symptom → Fix

"lsusb -v shows the strings but they disappear after replug." Almost always macOS — a cache invalidation race in USBHostBSDClient. Bump bcdDevice in firmware, replug, and the strings will stick.
