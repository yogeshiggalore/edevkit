QSPI — Dual / Quad / Octal SPI, end-to-end

A focused reference on the multi-IO extensions of SPI — the same four-wire bus widened to two, four, or eight data lanes so that a single SPI flash can feed an MCU's instruction stream at hundreds of megabytes per second through five tiny pins. This document covers the full dimensional matrix of lane modes (the "x-y-z" notation), the command / address / dummy / data phase split that makes the modes interoperable, mode bytes and continuous-read, XIP (execute-in-place), DDR / DTR (data on both clock edges), QPI vs Quad-I/O, octal flash, HyperRAM, and how edevkit1's PIO-based QSPI engine drives a master, plays a slave, or sniffs a live bus up to 50 MHz / 200 MT/s with one connector and three PIO state machines.

Audience. A firmware engineer who knows plain SPI (see proto_spi.html) and now needs to bring up a quad-flash boot path, drive a HyperRAM cache, decode a QSPI capture from a vendor's reference board, or use the edevkit1 QSPI tester to emulate a flash chip while developing the host driver. By the end you should be able to read any modern serial-flash datasheet's "fast read" matrix, choose the right lane mode from a part's spec, debug the universal "boots in 1-1-1, hangs in 1-4-4" symptom, and configure the edevkit1 QSPI engine for whichever role your bench setup needs.

Sister docs. Read proto_spi.html first if you are not yet comfortable with CPOL/CPHA, nCS, MOSI/MISO, and the four single-IO modes — almost everything here builds on that foundation. For the underlying RP2350 PIO state machines that implement the bus, see platform_pio.html. For sister protocols on the same edevkit1 connector see proto_i2c.html and proto_uart.html. What "QSPI" actually means — and what it doesn't

QSPI is one of the most overloaded acronyms in embedded systems, and that ambiguity is the cause of an enormous amount of bring-up grief. The letter Q stands for Quad, but the term gets used for at least four distinct things by different vendors and different communities:

  • QSPI as a wire-level protocol. A serial bus that uses one clock, one chip-select, and up to four bidirectional data lines, allowing 4 bits to be transferred per clock. This is the meaning that matters at the protocol level. Every modern serial-flash datasheet from Winbond, Macronix, ISSI, Micron, GigaDevice, Cypress, and the rest builds on this. This is the meaning used everywhere in this document.
  • QSPI as a peripheral name (Microchip, ST, NXP, etc.). A specific on-chip controller IP that speaks the wire protocol above, usually with built-in support for XIP, DMA, and a "linear address window" so the flash can be read like memory. STM32's QUADSPI peripheral, NXP LPC's SPIFI, Microchip's QSPI controller, and similar are all "QSPI" in this sense, but they have non-trivial differences in how they program the command / address / dummy / data phases.
  • QSPI as a Cypress trademark (now Infineon). Originally a marketing name for Cypress's "QSPI High Speed" flash family, where the IP and the wire format share branding. Many people use "QSPI" as if it were a Cypress invention; it isn't.
  • QSPI as "anything not single-IO". Casual usage where someone says "QSPI" but actually means dual-IO, or octal, or PSRAM, or HyperRAM. This is sloppy but extremely common in conference talks and chat threads.

Throughout this doc, when we say QSPI without qualification we mean the wire-level Quad-IO protocol. When we mean the broader family — single, dual, quad, octal — we say multi-IO SPI. The single-IO subset is documented in proto_spi.html; here we focus on the dual-, quad-, and octal-lane extensions.

Mental model QSPI is just SPI with the data wires pluralised and made bidirectional. The clock is the same. The chip-select is the same. CPOL=0, CPHA=0 still applies (almost universally for flash; CPOL=1, CPHA=1 is also accepted by most parts). What changes is: (a) more wires moving data per clock, and (b) every transaction grows a structured "phase" header — command, then address, then dummy cycles, then data — because the slave needs to know when to switch the data wires from "master is driving" to "slave is driving" without anyone explicitly toggling a direction line. There is no MOSI / MISO; there is only IO0..IO3 (or IO0..IO7), all of which can be driven by either side at different times.

Why multi-IO exists — the bandwidth-vs-pin-count problem

By the late 2000s, single-IO SPI flash had hit a hard ceiling. The fastest commodity parts could be clocked at 80–100 MHz, which works out to about 10–12 MB/s of read throughput — enough for boot data and configuration, but nowhere near enough to feed a Cortex-M-class MCU's instruction stream from external flash without massive caching. Meanwhile, parallel NOR flash, the alternative, demanded 20–32 wires (a full address bus plus a 16-bit data bus plus control), which was unaffordable on a small MCU package and ate critical PCB area.

The industry's answer, around 2009–2011, was to widen SPI rather than parallelise it. The insight was simple: two of the four SPI wires (MOSI and MISO) are already data wires, just driven from opposite sides. If we make all data wires bidirectional, change which side drives them depending on the phase of the transaction, and add two more pins (originally the "WP#" write-protect and "HOLD#" hold pins, which were vestigial on most parts), we get four data wires for the cost of two extra package pins. Quadruple the throughput. No new clock pin. Same nCS.

That's the entire idea behind QSPI. Octal SPI just extends the same trick to eight data lanes, which requires a slightly larger package but is otherwise the same protocol idea.

Throughput numbers above include the fixed protocol overhead (command + address + dummy cycles) for a 1 KiB sequential read, which is the typical XIP burst size. Random-access throughput is much lower because the protocol overhead is paid per access — see XIP.

Sanity check Multi-IO is a throughput trick, not a latency trick. The first byte of any QSPI transaction takes the same number of SCK edges as it would in single-IO (often more, because of mode bytes and dummy cycles). What you save is on the bytes after the first one. For random 4-byte reads, single-IO at 100 MHz can be faster than QSPI at 50 MHz because the protocol overhead dominates. Multi-IO pays off for sustained reads (XIP, DMA bursts, framebuffer fills).

Why bidirectional matters — and why it's harder

In single-IO SPI, MOSI and MISO are unidirectional. Each side knows whose turn it is to drive at all times: master always drives MOSI, slave always drives MISO. There's no contention and no need to coordinate "switch direction now".

In multi-IO SPI, the same physical wires change direction mid-transaction. The master drives IO0..IO3 during the command and address phases, then both sides need to tristate simultaneously, and then the slave starts driving IO0..IO3 during the data-read phase. If the timing of the handover is wrong — if the master keeps driving for one too many cycles, or the slave starts too early — both sides briefly drive opposite levels on the same wire. This is bus contention. It corrupts the first data byte, wastes power, and on extreme cases (faster edges, tighter rise times) can damage the I/O cell.

The protocol's solution is the dummy cycle phase, which we'll explore in detail below. The whole purpose of dummy cycles is to give both master and slave a deterministic number of "nobody drives" SCK edges between the master-drives phase and the slave-drives phase. The slave datasheet specifies the minimum required count, the master programs at least that many, and the wires are guaranteed to be at high-Z during the handover. Getting the dummy-cycle count wrong is the single most common QSPI bring-up bug.

Pull-ups Most QSPI flash datasheets recommend an external 10 kΩ pull-up on each of IO0..IO3, even though the chip has internal pull-ups on IO2/IO3. Reason: during the brief tristate windows (between phases, and when nCS is high), the data wires would otherwise float, and capacitive coupling from the toggling SCK would inject noise into adjacent traces. External pull-ups bias the wires to a known state during these windows. edevkit1 includes 10 kΩ pull-ups on IO0..IO3 by default, switchable in software for tests where you want to verify the slave's tristate behaviour explicitly.

The x-y-z notation — command / address / data lanes

Every modern serial-flash datasheet describes its supported "fast read" modes using a three-number notation:

  x - y - z
  │   │   └── number of data lanes used during the data phase
  │   └────── number of address lanes used during the address phase
  └────────── number of command lanes used during the command phase

So 1-4-4 means: command sent on 1 lane (single-IO), address sent on 4 lanes (quad-IO), data transferred on 4 lanes (quad-IO). And 4-4-4 means everything is on 4 lanes — including the command itself, which is what distinguishes QPI from Quad-IO.

The notation has one quirk: when a part supports DDR for some phase, the lane number is followed by a D. So 8D-8D-8D is "octal lanes, double data rate, all phases". This is the format that the Macronix Octa-Flash datasheets and JEDEC SFDP use.

Why 1-4-4 is the workhorse

The vast majority of QSPI bring-ups settle on 1-4-4 mode with command 0xEB ("Quad I/O Fast Read") because:

  • The command is sent on a single lane, matching the slave's power-on state. No "QPI mode enable" sequence is required. The same flash that boots the MCU can later be reprogrammed using single-IO commands without ever leaving 1-4-4.
  • Once the address phase starts, the bus is on 4 lanes — so the address (3 bytes for 24-bit, 4 bytes for 32-bit) finishes 4× faster than in 1-1-4. For random-access reads of 32-byte or smaller chunks, this saves enough time that 1-4-4 beats 4-4-4 even though 4-4-4 sends the command faster.
  • The mode-byte mechanism (see below) lets you skip the command on subsequent transactions, giving QPI-like throughput for random-access reads without ever entering QPI mode.

If you are bringing up a board for the first time, configure your flash driver for 1-4-4 with 6 dummy cycles, 24-bit addressing — that's the universal "works on every part" combo for read-only XIP at 50–80 MHz. Tune from there.

Throughput: 1 bit per SCK. With command 0x0B (FAST_READ) the master adds 8 dummy cycles between address and data so the slave can prefetch from its flash plane. With command 0x03 (READ) there are zero dummy cycles, but the SCK is capped at ~25 MHz on most parts.

1-1-2 — Dual Output Read (cmd 0x3B)

SCK   _─_─_─_─_─_─_─_─  ... (4 edges per byte)
IO0   ─ tristated by master (was MOSI in cmd/addr) ─    a6 a4 a2 a0   b6 b4 b2 b0
IO1   ─ tristated by master                          ─   a7 a5 a3 a1   b7 b5 b3 b1
                            ↑ slave starts driving here (after dummy cycles)

Throughput: 2 bits per SCK. The slave splits each byte across IO0 (even-indexed bits, msb first) and IO1 (odd-indexed bits). Because IO0 was MOSI moments earlier, the master must explicitly tristate IO0 before the data phase or contention occurs.

1-2-2 — Dual I/O Fast Read (cmd 0xBB)

cmd phase     IO0:  c7 c6 c5 c4 c3 c2 c1 c0  ─ master drives, IO1..3 tristated
addr phase    IO0:  a23 a21 a19 ... a1        ┐ master drives both IO0 and IO1
              IO1:  a22 a20 a18 ... a0        ┘
mode byte     IO0:  m7 m5 m3 m1                master drives both
              IO1:  m6 m4 m2 m0
data phase    IO0:  d6 d4 d2 d0  ...           slave drives both
              IO1:  d7 d5 d3 d1  ...

The 8-bit address is split across two lanes for a 4-cycle address phase (was 24 cycles in 1-1-2). The command stays on a single lane so a "fresh" master can recognise the part. Mode byte is the same width as the address.

1-1-4 — Quad Output Read (cmd 0x6B)

cmd phase    IO0: c7..c0           ─ single-IO command
addr phase   IO0: a23..a0          ─ single-IO 24-bit address (24 SCK edges)
dummy        IO0..3: tristate      ─ master release, slave preparing
data phase   IO0: d4 d0  d4 d0...   ┐ slave drives 4 lanes
             IO1: d5 d1  d5 d1...   │
             IO2: d6 d2  d6 d2...   │
             IO3: d7 d3  d7 d3...   ┘
                  └─byte 0─┘└byte 1┘

Throughput: 4 bits per SCK during data. The address phase is still single-IO so there's no need for the slave to be in QPI mode. Most parts ship with this mode enabled out of the box; some require the QE (Quad Enable) bit in the status register to be set first — see the pitfalls section.

1-4-4 — Quad I/O Fast Read (cmd 0xEB) — the workhorse

cmd phase      IO0: c7..c0                    ─ 1 lane × 8 bits = 8 SCK edges
addr phase     IO0..3: a23..a0 split           ─ 4 lanes × 6 nibbles = 6 SCK edges
mode byte      IO0..3: m7..m0 split            ─ 4 lanes × 2 nibbles = 2 SCK edges
dummy          IO0..3: tristate                ─ typ. 4 cycles
data phase     IO0..3: d7..d0 split, byte after byte

A complete 4-byte random read costs 8 (cmd) + 6 (addr) + 2 (mode) + 4 (dummy) + 8 (data) = 28 SCK edges. At 80 MHz that is 350 ns — quite a bit better than the 320 ns of single-IO at 100 MHz once you account for the eight dummy cycles single-IO needs in fast-read mode. For longer reads, 1-4-4 wins decisively.

4-4-4 — QPI (cmd 0xEB / 0x0B / etc., all on 4 lanes)

cmd phase     IO0..3: c7..c0 split  ─ 2 SCK edges (was 8)
addr phase    IO0..3: a23..a0 split ─ 6 SCK edges
dummy         IO0..3: tristate      ─ typ. 6 cycles
data phase    IO0..3: byte after byte

QPI mode requires the slave to be told "you are now in QPI mode" via a one-time command (typically 0x38 on Winbond, 0x35 on Macronix). After that, all commands accepted by the slave are sent on 4 lanes — including the very command (0xFF or vendor-specific) that exits QPI mode. This means a host that crashes mid-QPI cannot recover by sending single-IO commands until it issues the QPI-exit command on 4 lanes. See the pitfalls section for the recovery dance.

8-8-8 — OPI / Octal SPI

cmd phase     IO0..7: c7..c0   ─ 1 SCK edge per byte
addr phase    IO0..7: 32-bit address ─ 4 SCK edges
dummy         tristate ─ typ. 16 cycles (because slave clocks are higher)
data phase    IO0..7: byte after byte ─ 1 SCK edge each

Octal mode is symmetrical to QPI but on 8 lanes. The dummy phase is much longer in clock count because the SCK frequency is 2–3× higher (133–200 MHz), so the slave's flash-plane access time corresponds to more SCK cycles. Most octal parts also support DDR (data on both edges), at which point the wire format is referred to as 8D-8D-8D and a separate DQS strobe is required for timing — see DDR.

QPI vs Quad-I/O — the subtle but breaking difference

This is the single most common point of confusion in QSPI bring-up, so it gets its own section.

Quad-I/O (1-4-4)QPI (4-4-4)
Command width1 lane (8 SCK edges)4 lanes (2 SCK edges)
Slave must be in special mode?No — slave's default state.Yes — must issue Enter-QPI command (e.g. 0x38) first.

If your XIP boot is unreliable Bring up your flash in 1-4-4 first. Verify it works at low clock and high clock. Only then experiment with QPI. The performance delta between 1-4-4 and 4-4-4 for typical read-mostly XIP is single-digit percent. The pain of a half-bricked QPI flash that you can no longer talk to is enormous. Many vendor reference drivers default to 1-4-4 for exactly this reason; the extra QPI-mode dance is opt-in.

Command, address, mode, dummy, data — the five phases

Every multi-IO transaction has up to five phases. Each phase has a defined width (in cycles) and a defined lane count (1, 2, 4, or 8). Transitions between phases happen without nCS toggling — the master and slave use the SCK count alone to know which phase is active.

PhaseDriven byWidthPurpose
1. Commandmaster8 bits, on 1 or 4 or 8 lanesTells the slave what kind of operation follows. Values are vendor-defined but largely standardised — 0x03 / 0x0B / 0x6B / 0xEB / 0x9F etc. are universal.
2. Addressmaster24 or 32 bits, on N lanesLinear byte address into the slave's address space. 24-bit addresses (16 MiB) are the legacy default; 32-bit (4 GiB) is required for parts ≥ 32 MiB. The slave needs to be told which mode to expect — there are explicit "Enter 4-byte mode" commands or per-command 4-byte variants.
3. Mode bytemaster8 bits, on N lanes (only in 1-4-4 / 4-4-4)Optional. A "magic byte" that, if it matches a vendor-defined pattern, tells the slave "skip the command on the next transaction" (continuous read). See below.
4. Dummynobody0–32 SCK cycles, all lanes tristateGives the slave time to fetch from its flash plane after seeing the address. Also gives both master and slave time to relinquish drive on the data lanes before the slave starts driving them. Count is part-specific and frequency-dependent — see below.
5. Dataslave (read) or master (program)N bytes, on M lanesThe actual payload. Continues for as long as nCS stays asserted. The slave keeps incrementing its internal address pointer for as long as you keep clocking SCK — there's no "end of data" signal, you just deassert nCS when done.

Dummy cycles — what they are, why they vary, how to count them

A dummy cycle is a SCK edge during which neither side drives any data lane. Its count is how the master tells the slave: "please don't start replying yet — I'm giving you N cycles to look up the data." It also gives both sides a guaranteed handover window where the data lanes are tristated.

Why the count varies

Three things determine the right dummy-cycle count:

  1. The slave's flash-plane access time. A NOR-flash plane takes a fixed wall-clock time to fetch a row (typically 30–80 ns). At higher SCK frequencies, that fixed time corresponds to more SCK cycles. A part rated 80 ns at 100 MHz needs 8 SCK cycles of dummy; at 50 MHz, only 4.
  2. The lane count. Quad-IO returns 4 bits per SCK, so the slave's internal data pre-fetch must finish 4× faster (or the dummy must absorb the difference). Octal at 8 bits/SCK puts the most pressure here.
  3. The mode byte. Some parts let you trade dummy cycles for a longer mode byte phase — the mode byte effectively counts as part of the dummy. On a Macronix MX25R for example you can configure 4 dummy + 2 mode-byte cycles or 6 dummy + 0 mode-byte cycles for the same total.

How the count is communicated

The dummy-cycle count is not on the wire. There is no field in any command that says "this many dummies". Instead, the master and slave are pre-agreed on the count, and the agreement is established in one of three ways:

  • Datasheet-default. Cmd 0xEB on a W25Q always uses 6 dummy cycles. Hardcoded. The host driver knows this from the part's datasheet.
  • Status-register configurable. Many parts have a "Latency Code" or "Dummy Cycle Count" field in a configuration register. The host writes the desired count (e.g. 4 / 6 / 8) once during initialisation, and from that point on the slave uses that count for the relevant fast-read commands.
  • SFDP (Serial Flash Discoverable Parameters). A JEDEC-standard parameter table at a fixed offset in the flash that the host can read in single-IO mode. Each entry describes a supported mode and its required dummy count. A well-written generic flash driver reads SFDP at startup and configures itself with no part-specific code.
Practical tip When porting a flash driver to a new part, the very first thing to verify is "does my dummy-cycle count match what the part wants at this SCK frequency?" If it doesn't, you'll see shifted data — the right bytes, but offset by N nibbles. 0xDEADBEEF reading as 0xDDEEAADD is the giveaway: the master is sampling the lanes one cycle too early or late, so each byte gets misaligned. Increase or decrease the configured count by 2 (one nibble pair) and re-test.

The mode byte — continuous read and the "stay in fast mode" trick

The mode byte is a small but crucial optimisation. The idea: most of the time, when a host issues a fast-read command, the very next transaction is going to be another fast-read at the next address. So why pay for the 8-bit command phase every time?

The mode byte solution: after the address phase of a 1-4-4 / 4-4-4 / 8-8-8 fast read, the master clocks an extra 8 bits of "magic value" onto the lanes. If the value matches a vendor-defined pattern (typically 0x20 or 0xA5 for Winbond, 0xA0 for Macronix), the slave enters continuous-read mode. While in continuous-read, the master can deassert nCS, immediately reassert it, and skip directly to the address phase — the slave assumes it's another fast-read with the same opcode.

To exit continuous-read, the master clocks a non-magic value in the mode byte slot of the next transaction. The slave reverts to "command-required" mode on the transaction after that.

Why this matters for XIP Without continuous-read, every random-access read pays an 8-cycle command phase. With it, sequential reads (the common case during code execution) pay only address + dummy + data. For a Cortex-M cache-line fill of 32 bytes at 80 MHz 1-4-4: command-required is 8+6+4+64 = 82 SCK edges, continuous-read is 6+4+64 = 74 — about 10% faster. Across a long boot it adds up.

Many MCU QSPI controllers expose a "send mode byte automatically" toggle that does exactly this without driver-side bookkeeping. STM32's QUADSPI calls it "Send Instruction Mode Cycles Only Once"; NXP's FlexSPI calls it "Continuous Read Mode". Both implement the same wire-level behaviour.

A common bring-up trap

If your QSPI driver sets the mode byte correctly but doesn't also issue an explicit "exit continuous-read" sequence at the start of every cold transaction (most don't), the slave can be left in continuous-read mode after a host crash. Subsequent reads will fail because the host now sends a command phase, but the slave is interpreting those bits as if they were an address. Symptom: the very first XIP fetch after a soft reset returns garbage; subsequent fetches work because the slave eventually exits continuous-read on its own after a few non-matching mode bytes.

Workaround: the standard "soft reset" sequence for an unknown-state QSPI flash is to clock a long stream of 0xFFs on a single lane (this exits any continuous-read mode and any QPI mode), then issue cmd 0x66 (Enable Reset) followed by 0x99 (Reset Device). After a few hundred microseconds, the part is in a known state. Every robust driver does this on every boot.

XIP — execute-in-place from serial flash

Execute-in-place means the MCU's instruction cache fills directly from the QSPI flash, treating it as if it were memory-mapped. There is no separate "load image into RAM" step at boot. The CPU asks for an instruction at address 0x10000400; the QSPI controller, configured for 1-4-4 with a mode byte, autonomously generates the right wire-level transaction; the data flows back; the cache is filled; the CPU executes the instruction. All of this is invisible to software.

XIP is what makes "boot from external flash on a small MCU" practical. Without it, you'd need enough on-chip RAM to hold a copy of the entire firmware, which is unaffordable on the kind of small parts that pair with QSPI flash. With it, RP2350 (264 KB SRAM) can run firmware that is many megabytes in size — most of it executing directly from the W25Q128 boot flash at modern speeds.

Cost model

An XIP cache miss has a fixed protocol overhead and a per-byte payload cost. Roughly:

  • First fetch (cold cache, after reset): 8 SCK cmd + 6 SCK addr + 2 SCK mode + N SCK dummy + (cache-line-bytes × 2) SCK data. At 50 MHz with a 32-byte cache line and 6 dummy cycles, that's (8+6+2+6+64)/50e6 = 1.7 μs. Equivalent to ~250 wasted M0+ cycles per cold miss.
  • Subsequent fetches (warm, with continuous-read enabled): 6 SCK addr + 2 SCK mode + N SCK dummy + 64 SCK data = 1.5 μs at 50 MHz, 0.96 μs at 80 MHz, 0.6 μs at 133 MHz. About 100 wasted CPU cycles per miss.
  • Hot path (cache-resident): zero. XIP miss only happens on cache eviction. The deeper your cache and the better your code locality, the less you pay.

This is why the RP2350 boot ROM goes to such lengths to enable XIP cache: the cache is the difference between "boots in 50 ms" and "boots in 800 ms".

XIP vs copy-to-SRAM

If your firmware is small enough to fit in RAM, copy-to-SRAM at boot is faster after the first second. The cost is paid once at boot (a single sequential read of the entire firmware) and then there are zero misses ever after. For larger firmware, XIP wins because there's no "load whole thing" step — you only fetch what you actually run.

edevkit1's MCUboot loader runs from XIP (it lives in slot 0), but performance-critical PIO setup code is marked __attribute__((section(".time_critical"))) which the linker places in SRAM. Inner USB and CMSIS-DAP loops are also pulled into SRAM. Everything else stays in XIP.

DDR / DTR — data on both clock edges

SPI's normal "single data rate" (SDR) clocks one bit per lane per SCK edge, but only the rising or only the falling edge — never both. Double data rate (DDR), also called double transfer rate (DTR) by Cypress and "dual-edge" by others, doubles throughput by clocking on both edges. A 100 MHz DDR bus moves 200 megabits per lane per second.

For QSPI in DDR mode, the wire-level changes are:

  • The slave drives the data lanes such that a new value is valid on each edge. Setup/hold tightens to ~½ of SDR's window.
  • The master samples on each edge instead of on every other edge.
  • A data strobe (DQS or RWDS) becomes mandatory above ~100 MHz DDR. The DQS pin acts as a slave-driven clock that the master uses to time-align its sampling: the master ignores its own SCK for sampling purposes and instead uses DQS, which automatically tracks the slave's actual response time including PCB flight delay. Without DQS, you can't run reliably above ~80 MHz DDR even on a perfectly-laid-out 4-layer board.

When DDR pays off

DDR is mandatory for sustained read throughput above 100 MB/s. Below that, it's a wash: the increased timing tightness eats into your margin and forces shorter PCB traces, more careful impedance control, and (above 100 MHz) the extra DQS pin on the package. For the average MCU + QSPI flash board, SDR at 80–100 MHz on 1-4-4 is the sweet spot. DDR matters mainly for octal-DDR flash on Cortex-M7 / -A class systems where you're trying to hit 200+ MB/s.

Octal SPI & HyperBus — when four lanes still aren't enough

Octal-IO doubles the lane count over QSPI: eight bidirectional data wires plus SCK, nCS, and (for DDR) DQS. The protocol is otherwise identical to QSPI — same five phases, same x-y-z notation, same mode-byte trick. Octal flash typically peaks at 200 MHz DDR / 400 MB/s, which is competitive with parallel NOR but at a fraction of the pin count.

Major octal-flash families:

  • Macronix MX25LM / MX66LM — JEDEC standard "Octa-Flash". Modes 8-8-8 and 8D-8D-8D. SCK up to 200 MHz DDR.
  • Cypress / Infineon S26 / S28 Semper — uses HyperBus protocol, which is octal-DDR with a slightly different command/address structure.
  • Micron MT35X xSPI / Winbond W25M — JEDEC OPI compliant.

HyperBus — Cypress's octal-DDR variant

HyperBus (originally Cypress, now Infineon) is an 11-pin variant of octal-DDR that bundles command and address into a single 6-byte "command/address" header, then transfers data in DDR mode. It uses a bidirectional RWDS pin for both read-data-strobe and read-data-valid signalling. HyperBus is wire-incompatible with JEDEC OPI, even though the pin count is similar — different command encodings, different timing diagrams, different pin definitions.

HyperBus is most often used for HyperRAM (a self-refreshing pseudo-static RAM behind the same protocol) rather than HyperFlash, because the use case for octal-DDR external RAM is much stronger than for octal-DDR flash. See below.

QSPI / Octal PSRAM — RAM behind a serial bus

Pseudo-static RAM is a cheap, dense DRAM that hides its refresh state inside the chip and exposes itself as a flat byte-addressable memory. Modern PSRAM is sold in QSPI and Octal-DDR variants — same wire-level protocol as flash, but with vastly different commands and behaviour:

  • No erase. Writes go directly to the requested address. No 4KB / 64KB sector boundaries.
  • No "program time". Writes complete in the SCK cycle they finish on. (Compare: NOR flash takes ~2 ms to program a page.)
  • Limited burst length. Most QSPI PSRAM chips can only do bursts up to a configurable boundary (typically 1024 bytes) before the master must deassert nCS to let the chip refresh internally. The master is responsible for splitting long transactions; if it doesn't, data corrupts silently after the boundary.
  • Higher latency than SRAM, lower than DRAM. Typical first-byte access ~100 ns; burst rate ~50–100 MB/s on QSPI, ~200 MB/s on Octal-DDR.

edevkit1 does not include external PSRAM (RP2350's 520 KB SRAM is enough for v1), but the design doc reserves PIO state machines for v2 PSRAM support.

Speed bands and the long-cable problem (revisited)

The single-IO discussion in proto_spi.html#speed applies here, but the math gets tighter in proportion to the lane count. The same 1 ns of round-trip flight time that's tolerable at 50 MHz single-IO is borderline at 50 MHz quad-IO and fatal at 50 MHz octal-DDR.

Reasons:

  • In quad-IO mode the slave is simultaneously driving four lanes. Skew between those four lanes is a new failure mode that doesn't exist in single-IO. Crosstalk between adjacent IOs becomes meaningful at 50+ MHz.
  • The dummy phase needs to absorb both the fixed flash-plane access time and the round-trip flight time. Any cable that adds significant flight delay forces the firmware to use a higher dummy count, which directly subtracts from peak throughput.
  • DDR mode tightens setup/hold to ½ of SDR. A board that runs 100 MHz SDR fine may collapse at 50 MHz DDR if signal integrity isn't excellent.

The edevkit1 QSPI engine — PIO1 SM1 + SM2 + sniffer

edevkit1 implements QSPI not via a hardware QSPI peripheral (RP2350 has only one, and it's dedicated to the boot flash) but via a pair of PIO state machines on PIO1, with a third state machine standing by for sniffer mode.

Master mode — the wire shapes

The QSPI master state machine is a single .pio program with a few side-set channels. Conceptually:

.program qspi_master_1_4_4
.side_set 1                             ; SCK is sideset

start:
    pull   block        side 0          ; pull cmd-byte from FIFO, SCK low
    out    pins, 1      side 0  [3]     ; place IO0 = cmd bit, hold
    nop                 side 1  [3]     ; rising edge — slave samples
    ; ... (similar for the other 7 cmd bits)

addr_phase:
    pull   block        side 0          ; pull a 24-bit address word
    out    pins, 4      side 0  [3]     ; place top nibble on IO0..3
    nop                 side 1  [3]     ; clock high — slave samples
    out    pins, 4      side 0  [3]     ; next nibble
    nop                 side 1  [3]
    ; ... (6 nibbles total for 24-bit addr)

dummy:
    set    pindirs, 0   side 0  [3]     ; tristate IO0..3
    set    x, 5         side 0          ; 6 dummy cycles
dummy_loop:
    nop                 side 1  [3]
    jmp    x--, dummy_loop side 0 [3]

data_phase:
    set    pindirs, 0   side 0          ; lanes still tristated
data_loop:
    nop                 side 1  [3]     ; rising edge
    in     pins, 4      side 1          ; sample top nibble
    nop                 side 0  [3]     ; falling
    in     pins, 4      side 0          ; sample bottom nibble
    push   block        side 0          ; emit byte to RX FIFO
    jmp    data_loop    side 0

This is a sketch, not the actual production .pio source — see app/pio/qspi.pio in the firmware tree for the real version, which handles 1-4-4, 4-4-4, address widths of 24 and 32 bits, configurable dummy counts, and the mode byte. The dispatcher is in src/proto/qspi.c.

Achieving 50 MHz with a 150 MHz core

RP2350 runs at 150 MHz default. Each PIO instruction takes one cycle. To produce a 50 MHz SCK, the master program needs three core cycles per SCK period — one for the instruction that toggles SCK low, one for the data-out instruction, one for the SCK-high. That's exactly what the side-set delays in the sketch above achieve: [3] on the data-out instruction means "hold for 3 extra cycles", giving a 4-cycle phase × 2 = 8 cycles per SCK = 18.75 MHz at 150 MHz. Reducing the delay to [1] gives 50 MHz. The configurability of these delays is how the firmware exposes a runtime SCK-frequency knob without recompiling the .pio program.

Code — Zephyr API, JEDEC ID, fast-read 1-1-4

Initialisation

#include <zephyr/kernel.h>
#include <edevkit/qspi.h>

Reading the JEDEC ID — the universal sanity check

JEDEC ID (cmd 0x9F) works on every JEDEC-compliant flash and PSRAM at single-IO mode 0, regardless of any other configuration. It returns 3 bytes: manufacturer ID, memory type, capacity code.

static int read_jedec_id(uint8_t out[3])
{
    struct edevkit_qspi_xfer xfer = {
        .cmd_value     = 0x9F,
        .cmd_lanes     = 1,
        .addr_lanes    = 0,                   /* no address phase */
        .mode_lanes    = 0,
        .dummy_cycles  = 0,
        .data_lanes    = 1,
        .data_dir      = EDEVKIT_QSPI_DIR_RX,
        .data_buf      = out,
        .data_len      = 3,
    };
    return edevkit_qspi_xfer(qspi, &xfer);
}

/* expected output for a Winbond W25Q128: { 0xEF, 0x40, 0x18 }
 *   0xEF = Winbond manufacturer
 *   0x40 = SPI flash family
 *   0x18 = 2^24 bytes = 16 MiB
 */

A 1-4-4 quad-I/O fast read at 50 MHz

static int fast_read_quad(uint32_t addr, uint8_t *buf, size_t len)
{
    struct edevkit_qspi_xfer xfer = {
        .cmd_value     = 0xEB,                /* QIOR */
        .cmd_lanes     = 1,
        .addr_value    = addr & 0xFFFFFF,     /* 24-bit address */
        .addr_lanes    = 4,
        .addr_bits     = 24,
        .mode_value    = 0xFF,                /* non-magic — disables continuous-read */
        .mode_lanes    = 4,
        .mode_bits     = 8,
        .dummy_cycles  = 4,                   /* W25Q @ 50 MHz */
        .data_lanes    = 4,
        .data_dir      = EDEVKIT_QSPI_DIR_RX,
        .data_buf      = buf,
        .data_len      = len,
    };
    return edevkit_qspi_xfer(qspi, &xfer);
}

The driver's job: assemble these phase parameters into a 32-bit "wire descriptor" that PIO1 SM1 understands, push it into the SM's TX FIFO, then DMA the data buffer to/from the SM's RX FIFO. Latency: ~6 μs per transaction setup, dominated by the kernel call and DMA descriptor-write — the actual wire transaction at 50 MHz is much faster for small reads.

Emulator mode — pretending to be a W25Q flash

edevkit1 can play the role of a QSPI flash chip on its protocol header, replying to every command a real flash would. The use case: you're developing host-side firmware that drives a flash chip, but you don't have a board with the flash chip on it yet. Plug in edevkit1, point your driver at the protocol header, and it sees a "real" flash chip — JEDEC ID, fast read, page program, sector erase, all of it.

The emulator is implemented as a YAML profile that the firmware loads from /lfs/profiles/flash/<part-name>.yaml. Built-in profiles ship for:

  • Winbond W25Q128JV (16 MiB, 1-4-4 supported, QPI supported)
  • Macronix MX25R6435F (8 MiB, low-power, slightly different command set)
  • ISSI IS25LP128F (16 MiB, alt-vendor reference)
  • GigaDevice GD25Q64 (8 MiB, popular in low-cost designs)

Custom parts can be added by writing a YAML file describing the supported commands, dummy counts, status-register layout, and address space. The emulator backs the address space with the data flash (W25Q01JV, 128 MiB) so emulating a 16 MiB flash leaves plenty of room for capture buffers.

Sniffer mode — passively decoding a live QSPI bus

Sniffer mode is the trickiest of the three roles, because there are five wires to monitor (SCK, nCS, IO0..3) at up to 50 MHz, and you need to decode the phase boundaries based on the protocol — there's nothing on the wire that says "we're now in the dummy phase".

The PIO program implements a state-driven decoder:

  1. While nCS is high: do nothing, idle.
  2. On nCS falling edge: start sampling. Reset phase counter to 0.
  3. Sample IO0..3 on each rising SCK edge. Push 4-bit nibbles into a ring buffer in SRAM.
  4. Maintain a per-transaction state machine that tracks: command byte (8 cycles, 1 lane), address (configurable: 1 / 2 / 4 lanes, 24 / 32 bits), mode byte (configurable, 0 / 8 cycles), dummy (configurable, 0–32 cycles), data (until nCS rises).
  5. The host CLI provides the phase configuration ("decode as 1-4-4 with 4 dummy cycles"); the sniffer applies it. Without configuration, the sniffer captures the raw cycle-by-cycle nibble stream and lets the host post-process.

The host CLI (edevkit qspi sniff) renders the captured stream as a transaction list:

[0.000 ms] CMD 0x9F (RDID)         resp: EF 40 18           (W25Q128 detected)
[0.012 ms] CMD 0x06 (WREN)
[0.018 ms] CMD 0xEB QIOR @0x000000  (4 dummy)  resp: 00 00 00 00 ...    32 bytes
[0.032 ms] CMD 0xEB QIOR @0x000020  (4 dummy)  resp: A5 5A A5 5A ...    32 bytes
...
Sniffer gotcha At 50 MHz quad-IO, the sniffer is moving 25 MB/s of raw nibble data into SRAM. At a USB CDC of ~1 MB/s sustained, the host can't keep up with anything more than a 100-ms burst. The CLI defaults to capture mode — record into the 128 MB data flash, then upload offline. Live streaming is for low-rate buses (sensor I²C, slow SPI) only.

Universal recovery sequence When all else fails — flash is in an unknown state, you don't know if it's in QPI, continuous-read, 4-byte mode, or all three:
  1. Toggle nCS high for at least 1 ms.
  2. Clock at least 64 SCK edges of 0xFF on 8 lanes (covers any of single / dual / quad / octal commands).
  3. Issue 0x66 on 8 lanes, then 0x99 on 8 lanes (Enable Reset + Reset Device — works in all modes because the slave decodes whichever bits land on whichever lane it's currently configured for).
  4. Wait 100 μs. The part is now in factory-default state: single-IO, 24-bit address, 8 dummy cycles for fast-read, no continuous-read.
This wins ~95% of "I bricked my flash" bring-up cases.
