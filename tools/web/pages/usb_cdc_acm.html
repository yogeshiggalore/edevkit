USB CDC-ACM — the "make my device look like a serial port" class

A focused reference on USB CDC-ACM (Communication Device Class, Abstract Control Model subclass) — the class definition that lets a USB device show up as /dev/ttyACM0 on Linux, /dev/cu.usbmodem* on macOS, or COMx on Windows. It is the de-facto serial-over-USB protocol: every USB-to-UART bridge chip (FTDI's CDC variants, Silabs CP2102N CDC mode, WCH CH340 CDC mode) and every native-USB MCU stack (STM32, nRF52/53/54, RP2040/RP2350, ESP32-Sx, SAM-Dx, Teensy, Pico-SDK, TinyUSB, Zephyr USB-Device-Next, embassy-usb) implements it.

Audience. Firmware engineers exposing a debug log, AT-command channel, or generic data pipe over USB; host-side developers writing tools that talk to such a port; integrators chasing a "my port works on Linux but not Windows" bug. By the end of this doc you should be able to read a CDC-ACM descriptor block, debug a host-stalls-after-64-bytes ZLP issue, write a udev rule that stops ModemManager from grabbing a developer-test port, and understand exactly how edevkit1 exposes two CDC-ACM bridges in its composite descriptor — one for the target's log UART, one for the developer-test UART — without confusing the host driver.

Not covered here. The general USB endpoint / configuration / descriptor model lives in usb_overview.html; the CMSIS-DAP v2 vendor-class side of edevkit's composite device lives in usb_cmsis_dap.html; the UART hardware behind the two CDC-ACM bridges lives in proto_uart.html. The full v1 design rationale (why two CDC-ACM instances, why separate Vtgt/Vio domains) lives in edevkit1.html.

What CDC-ACM is, and why everyone uses it

The Communications Device Class (CDC) is one of the standard USB device classes — class code 0x02 — defined originally by the USB Implementers Forum in 1999 (USB Class Definitions for Communication Devices, rev 1.1) and updated as USB CDC 1.2 in 2007. CDC was an umbrella for "anything that looks like a communications endpoint": telephone modems, ISDN terminal adapters, Ethernet bridges, ATM adapters, and yes, garden-variety serial ports. Each kind got its own subclass. Today, only a handful of those subclasses are common in the wild:

Subclass codeNameWhere you see it
0x02Abstract Control Model (ACM)The "USB serial port" everyone means when they say "CDC". Modems, USB-UART bridges, debug-log channels.
0x06Ethernet Control Model (ECM)USB Ethernet adapters on Linux/macOS. Windows prefers NCM.
0x0DNetwork Control Model (NCM)Higher-throughput USB Ethernet (RNDIS replacement, used by Apple iPhone tethering).
0x0EMobile Broadband Interface Model (MBIM)4G/5G modems with structured AT replacement.

This doc is about the first row. Abstract Control Model means "abstract" in the sense that the device is pretending to be a Hayes-compatible modem — it accepts SET_LINE_CODING ("set baud rate / parity / stop-bits / data-bits"), SET_CONTROL_LINE_STATE (DTR/RTS), and reports SERIAL_STATE notifications (DCD/DSR/RI/Frame/Parity/Overrun). The host driver then exposes a tty/COM-port abstraction to userspace, and any tool that knows how to open a serial port — screen, minicom, picocom, PuTTY, the Arduino IDE serial monitor, your firmware's printf consumer, your AT-command parser — works without modification.

key idea CDC-ACM is the contract a USB device signs that says "treat me like a serial port and the OS will hand userland a tty/COM device." The class itself is thin — a couple of descriptors and a small set of control requests — but its leverage is enormous, because every serial-port tool ever written immediately works against it. That is why native-USB MCUs implement CDC-ACM directly (no UART chip behind it), and why USB-to-UART bridge chips (FTDI's CDC mode, CP2102N CDC mode, CH340 CDC mode) emulate it: it lets the chip ship with zero driver install on every modern host OS.

Why not a vendor-specific class?

You absolutely can roll your own — TinyUSB, libusb on the host, and you're done. FTDI famously did this for decades: their FT232/FT2232 family historically used vendor class with proprietary FTDIBUS drivers (ftdi_sio on Linux, FTDIVCP on Windows). The reasons people no longer do this for new designs:

  • Driver hell on Windows. Anything not in the standard CDC class needs a signed INF, and historically a kernel-mode driver. Microsoft made this dramatically worse in 2014–2017 with the WDF driver-signing tightening. CDC-ACM dodges this entirely — see §Cross-platform behaviour.
  • Linux already ships the driver. The kernel's cdc-acm module has been in mainline since 2.4, gets near-zero churn, and works on every distro from Alpine to Ubuntu LTS. A vendor protocol means writing/maintaining your own kernel module.
  • macOS gatekeeper. macOS has shipped a CDC driver in IOUSBHostFamily since 10.7. A custom kext is now nearly impossible to ship without a notarised, paid-developer kernel-extension entitlement; DriverKit is a partial workaround. CDC-ACM is the path of least resistance.
  • Userspace fallbacks. Even if you do go vendor-class, libusb works on Linux and macOS but is awkward on Windows (you need WinUSB + a paired INF). And once you've solved that, you've reinvented CDC-ACM.

The single legitimate reason to skip CDC-ACM is ultra-high throughput or ultra-low latency — a logic analyser at 480 Mbps doesn't fit through CDC-ACM's modem-shaped descriptors gracefully (you'd want isochronous or your own bulk-only protocol). For everything else — debug logs, AT commands, register-poke shells, firmware updates over a text protocol — CDC-ACM is the right tool.

Class structure — two interfaces tied by IAD

An ACM "function" is not a single USB interface. It is a pair: a communications-class interface that carries control + notifications, and a data-class interface that carries the actual byte stream. The reason this two-interface split exists is historical: in 1999, modems were a fully realised communications device with its own protocol stack on top of the data link, and the spec wanted to separate "modem control plane" from "modem data plane" cleanly. We're stuck with the two-interface design forever, because CDC-ACM never broke compatibility.

InterfacebInterfaceClassbInterfaceSubClassbInterfaceProtocolEndpointsCarries
Communications (control)0x02 (CDC)0x02 (ACM)0x01 (AT cmds, V.250) — or 0x00 for "no protocol"1 × interrupt IN (notifications) + the default control endpoint EP0SET_LINE_CODING, SET_CONTROL_LINE_STATE, GET_LINE_CODING, SEND_BREAK, SERIAL_STATE notifications
Data0x0A (CDC-Data)0x000x001 × bulk IN, 1 × bulk OUTThe bytes that read() / write() on the host see

Two interfaces means two interface descriptors in the configuration descriptor, and on a composite device — anywhere your USB device has more than one function, e.g. CDC + HID, or CDC + CDC, or CDC + vendor — the host now has a problem. Which two interfaces belong together? The host can guess by adjacency (interface N and N+1 with classes 0x02 and 0x0A look like an ACM pair), but guesses are unreliable and Microsoft refused to ship that heuristic. The solution USB-IF added in 2003 is the Interface Association Descriptor (IAD).

The Interface Association Descriptor

An IAD is a tiny 8-byte descriptor that sits in the configuration descriptor immediately before the first interface of a function and says "the next N interfaces are one logical function, treat them together." For CDC-ACM:

// Interface Association Descriptor — bind the two CDC-ACM interfaces
// 0x08, 0x0B,             // bLength=8, bDescriptorType=IAD (0x0B)
// bFirstInterface,        // index of the Communications interface
// 0x02,                   // bInterfaceCount = 2 (Comm + Data)
// 0x02, 0x02, 0x01,       // function class/sub/proto = CDC / ACM / AT-V.250
// 0x00                    // iFunction = no string

static const uint8_t iad_cdc_acm[] = {
    8, 0x0B,
    /* bFirstInterface  */ 0x00,
    /* bInterfaceCount  */ 0x02,
    /* bFunctionClass   */ 0x02,   /* CDC                 */
    /* bFunctionSubClass*/ 0x02,   /* ACM                 */
    /* bFunctionProtocol*/ 0x01,   /* AT-V.250 (optional) */
    /* iFunction        */ 0x00
};

For an IAD to be recognised by Windows, the device descriptor must declare itself a Multi-Interface Function device:

// Device descriptor fields that enable IAD-aware enumeration on Windows:
bDeviceClass     = 0xEF;   // Miscellaneous Device Class
bDeviceSubClass  = 0x02;   // Common Class
bDeviceProtocol  = 0x01;   // Interface Association Descriptor

If you forget that triple, Windows will refuse to use IADs and will fall back to per-interface class matching, which on a CDC + HID composite means both sub-functions break. This is the most common composite-device bring-up bug — see §Composite-device gotchas.

Don't forget On a single-function device with only one CDC-ACM, the IAD is technically optional: Windows will still match by class adjacency. The moment you add a second interface (any second interface — another CDC, an HID, a vendor-class CMSIS-DAP), you must use IADs and set bDeviceClass = 0xEF / 0x02 / 0x01. Composite descriptors without IADs are the single largest cause of "works on Linux but Windows says Code 10".

Functional descriptors — Header, CM, ACM, Union

Inside the Communications-Class interface descriptor, between the interface descriptor itself and its endpoint descriptor, the spec inserts a sequence of functional descriptors (descriptor type 0x24, "CS_INTERFACE"). These describe extra metadata that doesn't fit into the standard interface descriptor — for ACM, four of them are mandatory:

SubtypeNameWhat it declares
0x00HeaderThe CDC spec revision the device claims to implement (bcdCDC, e.g. 0x0120 for CDC 1.20).
0x01Call ManagementWhether the device handles call-management commands itself, and which interface (Comm or Data) carries them. For pure data-pipe ACM, almost always bmCapabilities = 0x00 and bDataInterface = <data-iface>.
0x02Abstract Control ManagementWhich ACM-class requests the firmware actually implements. Bitmap: bit0 = comm-feature, bit1 = line-coding/line-state/serial-state, bit2 = send-break, bit3 = network-connection-notif. Most "USB-as-serial-port" firmwares set bmCapabilities = 0x02 (line coding + line state + break) or 0x06 (also break).
0x06UnionBinds the Communications interface to its Data interface — duplicates information already in the IAD, but predates the IAD and is still required for older hosts. Lists bMasterInterface (= Comm) and bSlaveInterface[0] (= Data).

Concrete byte-level example

This is the full configuration-descriptor fragment for a one-CDC-ACM function, suitable for a single-function device. Bytes are decimal/hex annotated; this is what your USB capture (Wireshark with usbmon, or USBlyzer on Windows) will literally show after enumeration:

0x02, 0x02, 0x01,           // CDC / ACM / AT-V.250

//   EP=0x02 (OUT2), Bulk, wMaxPacketSize=64 (FS), bInterval=0

//   Endpoint: bulk IN — device -> host
0x07, 0x05, 0x82, 0x02, 0x40, 0x00, 0x00,
//   EP=0x82 (IN2), Bulk, wMaxPacketSize=64 (FS), bInterval=0

Total: 66 bytes for one CDC-ACM function. On a Full-Speed device the bulk endpoints carry wMaxPacketSize = 64; on High-Speed (480 Mbps), they're typically 512; on USB 3 SuperSpeed they max at 1024. The interrupt-IN notification endpoint is conventionally 16 bytes — that's what the Zephyr USB-Device-Next CDC-ACM implementation uses, and what every host driver expects.

descriptor parsing tip When debugging a non-enumerating CDC device, dump the device's descriptor block and walk it byte by byte. lsusb -v -d <vid>:<pid> on Linux is excellent — it parses every functional descriptor, names every field, and flags malformed lengths. On macOS, system_profiler SPUSBDataType shows a coarser tree but covers most fields. On Windows, USBView (free, ships with the Windows Driver Kit) or usbtreeview.exe (Uwe Sieber's free tool) are both more revealing than Device Manager.

Class-specific control requests

All ACM-class requests target the Communications interface (not the Data interface), and travel over the default control endpoint EP0 with bmRequestType.type = Class and recipient = Interface. The Setup packet's wIndex field is the interface number of the Communications interface; on multi-CDC composite devices that's how the host disambiguates which CDC instance the request is for.

bRequestNameDirectionwValuewLength / payload
0x20SET_LINE_CODINGOUT07 bytes (see below)
0x21GET_LINE_CODINGIN07 bytes
0x22SET_CONTROL_LINE_STATEOUTbitmap (DTR=bit0, RTS=bit1)0
0x23SEND_BREAKOUTbreak duration in ms (0xFFFF = continuous, 0x0000 = stop)0

The seven-byte payload of SET_LINE_CODING / GET_LINE_CODING is the "modem-control state" the host wants to apply (or read back), packed as little-endian:

struct cdc_acm_line_coding {
    uint32_t dwDTERate;     // baud rate, bits-per-second (e.g. 115200 = 0x1C200)
    uint8_t  bCharFormat;   // 0=1 stop, 1=1.5 stop, 2=2 stop bits
    uint8_t  bParityType;   // 0=none, 1=odd, 2=even, 3=mark, 4=space
    uint8_t  bDataBits;     // 5, 6, 7, 8, or 16
} __attribute__((packed)); // exactly 7 bytes

The full request list and field semantics live in USB CDC PSTN 1.2 §6.3. The four above are the ones a USB-serial firmware always implements; the rest of PSTN (V.24 vendor signals, ringer, SET_LINE_PARMS, RECORD_PHONE_NUMBER) is modem-only and almost universally ignored on data-pipe ACM.

The "ignore SET_LINE_CODING" trick

Here's the part nobody tells you. On a USB-to-UART bridge chip (FT230X CDC mode, CP2102N CDC mode, CH340 CDC mode), SET_LINE_CODING matters: the chip uses the seven-byte payload to physically reprogram its UART block (BAUD divisor, parity, frame format). If the host opens the port at 9600 8-N-1 and the device has been wired to a target that expects 115200 8-N-1, you have to actually obey SET_LINE_CODING or the wire-side baud will be wrong.

On a native-USB MCU exposing a debug log over CDC-ACM, there is no real UART behind the interface. The bytes go directly from the bulk OUT endpoint into a ring buffer, and from a ring buffer out the bulk IN endpoint. The "baud rate" is meaningless — USB Full-Speed bulk runs at ~10 Mbps regardless of what the host sets. So the firmware accepts the seven bytes, optionally stores them so GET_LINE_CODING returns what the host last set (otherwise some host drivers complain), and does nothing else.

stash, don't act The standard pattern in TinyUSB, Zephyr's usbd_cdc_acm, embassy-usb's cdc-acm, and the Pico-SDK is: maintain a 7-byte buffer per CDC instance, write the SET payload into it on receipt, return it on GET, and otherwise ignore the values. The host is happy because GET round-trips correctly; the firmware is happy because nothing has to be reconfigured. This is what edevkit1's data-pipe CDC-ACM bridge does. The UART-bridge CDC-ACM instances do obey SET_LINE_CODING and reprogram the underlying RP2350 UART — see §edevkit1's two CDC-ACM bridges.

DTR / RTS via SET_CONTROL_LINE_STATE

SET_CONTROL_LINE_STATE (request 0x22) is interesting because it is the way the host signals "I have opened the port" or "I have closed the port". The bitmap in wValue:

  • bit 0 — DTR (Data Terminal Ready). Asserted when host opens the tty. Conventionally used by Arduino-style firmware as a "host is connected, you can start sending output now" signal.
  • bit 1 — RTS (Request To Send). Asserted similarly. Some bootloaders use the DTR/RTS toggle pattern as a "reset into bootloader" hint (e.g. Arduino's classic 1200 bps + DTR auto-reset).

On a native-USB CDC-ACM device, you receive the request and can use it as an "is the host's terminal actually attached?" gate. A common firmware pattern: do not write log output to the bulk IN endpoint until DTR has been asserted at least once after enumeration. Otherwise the device's TX ring buffer fills with bytes that the host driver never drained, the device blocks on flow control inside its own TX queue, and the user wonders why the first 4 KB of boot output is missing.

SEND_BREAK

SEND_BREAK is rarely used on data-pipe CDC-ACM (no real UART means no real break condition), but on a UART-bridge CDC-ACM you do generate a real line-break of the requested duration. wValue == 0xFFFF means "hold break until told to stop"; wValue == 0x0000 means "stop the break"; any other value is "send a break of N milliseconds". proto_uart.html covers the wire-side semantics.

The notification endpoint and SERIAL_STATE

The Communications interface has one endpoint of its own: the interrupt IN notification endpoint, conventionally 16 bytes max, polled at ~16 ms. The host is supposed to read it and act on the notifications it carries. In practice the only ACM notification anyone uses is SERIAL_STATE (notification code 0x20), which carries the modem-status bitmap:

BitNameMeaning
bit 0bRxCarrier / DCDData Carrier Detect — modem connected to remote.
bit 1bTxCarrier / DSRData Set Ready.
bit 2BREAKThe other end is currently asserting a break condition.
bit 3RINGPhone is ringing — yes, really.
bit 4FRAMINGUART framing error detected.
bit 5PARITYUART parity error detected.
bit 6OVERRUNRX overrun — bytes were dropped.

The packet on the wire is 10 bytes: an 8-byte SETUP-shaped notification header (bmRequestType=0xA1, bNotification=0x20, wValue=0, wIndex=<comm-iface>, wLength=2) followed by 2 bytes of the bitmap above (little-endian).

required, not optional The notification endpoint is required by the spec and must be present in your descriptor block. It does not have to actually emit notifications. Every shipping CDC-ACM firmware on the planet stubs this endpoint — it sits there configured but silent — and the host is fine with that; it just polls and gets NAKs forever. If you implement it, the most useful payload is to forward UART-level FRAMING / PARITY / OVERRUN errors from a UART-bridge CDC-ACM instance, so the host can correlate "byte N looks corrupted" with "the wire framing was bad here". screen, minicom, etc. ignore it; tools like grabserial and bespoke instrument tooling sometimes consume it.

Bulk pipe data flow and the ZLP rule

The interesting part: how a write(fd, "hello\n", 6) on the host turns into bytes in the device's bulk-OUT ring buffer, and how a read(fd, buf, N) retires bytes from the device's bulk-IN endpoint.

      host userspace           kernel cdc-acm           USB controller         device firmware
      ───────────────          ──────────────          ────────────────         ────────────────
                       write(fd, buf, n)
   write("hello\n",6) ────────────────────▶ kernel ring buffer
                                                ─────────────▶  bulk-OUT URB
                                                                     ──────────▶ EP_OUT (bulk)
                                                                                    │
                                                                                    ▼

                                                                                    ▼
                                                                            (app code reads
                                                                             the bytes here)

                                                                ◀─────── bulk-IN URB
                                            kernel rx ring   ◀───────────
   read(fd, buf, n)  ◀──────────── kernel returns up to n

The host queues a bulk-OUT URB ("USB Request Block") of size S. The USB controller breaks S into one or more packets of size at most wMaxPacketSize (64 on Full-Speed, 512 on High-Speed). The device receives those packets and runs its OUT-endpoint callback; the bytes go into a ring buffer; the application's uart_poll_in / cdc_acm_rx_cb handler drains them.

The Zero-Length Packet (ZLP) rule

This is the most subtle, most-bitten-by-newcomers part of CDC-ACM. The USB host stack uses one of two heuristics to decide "the bulk transfer is complete":

  1. The transfer length received so far equals the URB size requested. (Easy.)
  2. A short packet arrived — i.e. a packet smaller than wMaxPacketSize, including a zero-length packet. (This is the "the device decided to stop talking now" signal.)

The trap: if your transfer happens to be an exact multiple of wMaxPacketSize, then condition 1 alone fires only when the host queued an URB of exactly that length. A read on the host side typically asks for a much larger buffer (read(fd, buf, 4096)), and the kernel queues an URB of 4096 bytes hoping to fill it. If the device sends exactly 64 bytes and stops — without a follow-up short packet — the host URB does not retire. The read blocks. Output appears to "stick".

The fix: after every bulk-IN transfer whose length is a non-zero multiple of wMaxPacketSize, the device must send a Zero-Length Packet on the same endpoint. The ZLP is itself a "short packet" (zero is less than 64), so it satisfies condition 2 above and the URB retires.

// Pseudocode for a CDC-ACM TX worker on the device side:
void cdc_acm_tx_pump(struct cdc_acm *c) {
    while (ring_buf_used(&c->tx) > 0) {
        size_t n = MIN(ring_buf_used(&c->tx), wMaxPacketSize);
        usb_ep_write(c->ep_in, buf, n);
        // If we just sent a full packet AND the buffer is now empty,
        // we owe the host a ZLP to retire its URB.
        if (n == wMaxPacketSize && ring_buf_used(&c->tx) == 0) {
            usb_ep_write(c->ep_in, NULL, 0);   // ZLP
        }
    }
}
classic bug Symptom: "every 64-byte (or 512-byte) printf hangs on macOS / sometimes on Linux / always on Windows; if I print one extra byte, it works." Cause: missing ZLP. Fix: send a ZLP after a max-packet-sized terminal transfer. This is true even if your data just happens to be exactly 64 bytes — silly cases like printf("[%-62s]\n", short_msg) are a 64-byte newline-terminated string. macOS is the most aggressive at retiring URBs only on short-packet boundary; Windows usbser.sys is similar. Some HALs (Zephyr's USBD_NEXT, embassy-usb) handle ZLP automatically; STM32 HAL Cube CDC infamously does not, and the workaround is either to set UX_DEVICE_CLASS_CDC_ACM_WRITE_AUTO_ZLP=1 (Azure RTOS USBX) or to wrap your transmit and append a ZLP yourself.

CDC-specific ZLP edge case

Zephyr's CDC-ACM has a quirk worth knowing: rather than emit a ZLP, it sometimes prefers to send two short packets. The reason is that some CDC-ACM host drivers historically interpreted a ZLP as end-of-transmission in the modem-style sense — i.e. they'd close the port. The "two-short-packets" pattern (e.g. send 60 bytes, then 4 bytes, instead of 60 + 4 + ZLP) is safer on those legacy hosts. In 2026 you almost never see this driver-side bug, but if you're hunting "Linux gets the bytes but macOS thinks the channel closed", that's the territory.

Composite-device gotchas (multiple CDC, IAD, usbser.sys)

Once you put more than one function on a USB device, you're in composite-device territory and the OS-driver story gets more interesting.

IAD ordering rule

The IAD must come immediately before the first interface of the function it describes, and the function's interfaces must be contiguous. You cannot interleave interfaces from two functions. A correct composite layout for two CDC-ACM + one CMSIS-DAP vendor-class function:

Configuration descriptor
├── IAD #1                       (bFirstInterface=0, bInterfaceCount=2)
│   ├── Interface 0: CDC Comm   (CDC-ACM #1 control)
│   │   └── EP 0x81 INT          notification
│   └── Interface 1: CDC Data   (CDC-ACM #1 data)
│       ├── EP 0x01 OUT bulk
│       └── EP 0x82 IN bulk
├── IAD #2                       (bFirstInterface=2, bInterfaceCount=2)
│   ├── Interface 2: CDC Comm   (CDC-ACM #2 control)
│   │   └── EP 0x83 INT
│   └── Interface 3: CDC Data   (CDC-ACM #2 data)
│       ├── EP 0x03 OUT bulk
│       └── EP 0x84 IN bulk
└── (no IAD — vendor class is one interface)
    └── Interface 4: vendor    (CMSIS-DAP v2)
        ├── EP 0x05 OUT bulk
        └── EP 0x85 IN bulk

Endpoint numbers are unique device-wide (max 15 OUT + 15 IN on FS); interface numbers are unique configuration-wide. The bookkeeping is your job — the USB stack will let you ship a descriptor with two interfaces both claiming endpoint 0x82 IN and the device will silently misroute.

usbser.sys auto-loads on Windows 10/11

One of the best things to happen to USB serial in the last decade: as of Windows 10, the in-box usbser.sys driver auto-loads for any compliant CDC-ACM function via compatible-ID matching. Specifically, any interface whose bInterfaceClass = 0x02 / bInterfaceSubClass = 0x02 matches the compatible ID USB\Class_02&SubClass_02 and gets usbser.sys as the function driver — with no INF file shipped by the device.

On older Windows (7, 8, 8.1), you needed a vendor INF that referenced usbser.sys by name (typically derived from mdmcpq.inf) and had to be installed via Device Manager or a setup package. The shift to in-box auto-loading is documented at Microsoft Learn — USB Serial Driver (Usbser.sys). There are two specific traps:

  • Device-class triple matters. If your device descriptor declares bDeviceClass = 0x02 but bDeviceSubClass != 0x02, Windows will not auto-load usbser.sys. The fix on a single-CDC device: leave bDeviceClass = 0x00 ("see interface descriptors"). On a composite, set bDeviceClass = 0xEF / 0x02 / 0x01 (Multi-Interface Function with IAD).
  • Composite interfaces still need IADs. Even with auto-loading, on a multi-function device usbser.sys only attaches to interfaces grouped under an IAD. Without an IAD, Windows enumerates each interface separately and the CDC pair gets split across two stack instances, neither of which works.

Endpoint budget

Every CDC-ACM instance burns 3 endpoints: 1 interrupt IN + 1 bulk IN + 1 bulk OUT. On RP2350 USB hardware (Full-Speed device, 16 endpoints in each direction including EP0), you have headroom for 4–5 CDC-ACM instances before you run out — but the practical limit is bandwidth and latency, not endpoints. edevkit1 ships with two CDC-ACM bridges + one CMSIS-DAP vendor pair, totalling 8 endpoints in each direction, well within the budget.

Cross-platform behaviour — Linux, macOS, Windows

The same descriptor block produces three different host behaviours. Here's what to expect (and what to fix) on each.

Linux

Kernel driver: drivers/usb/class/cdc-acm.c — built into every distro kernel, has been mainline since 2.4. Auto-binds to any interface with class/sub 02/02.

Device node: /dev/ttyACMn where n is the next free index. Multiple CDC-ACM instances on a single composite get separate node numbers (e.g. /dev/ttyACM0 and /dev/ttyACM1). Node numbering is not stable across reboots if you have other devices that auto-load — use udev symlinks for stable names: /dev/edevkit-target-log, /dev/edevkit-test-uart.

Permissions: tty devices belong to group dialout (Debian/Ubuntu) or uucp (Arch/Fedora). Add yourself: sudo usermod -aG dialout $USER. Logout/login required.

The ModemManager problem

ModemManager is a system service that probes any newly-attached CDC device to see if it's a real cellular modem. Probing means: it grabs the port for ~2–10 seconds at attach time, sends AT commands at it, and watches for valid responses. If you happen to be a non-modem CDC-ACM device that doesn't speak AT, ModemManager will spew bytes into your developer-test UART, your firmware-update protocol, and your debug log, and the user sees garbage at attach time and "device or resource busy" if they try to open() the port during that window.

Fix: blacklist your VID/PID via a udev rule. Drop a file at /etc/udev/rules.d/79-edevkit.rules:

# /etc/udev/rules.d/79-edevkit.rules — keep ModemManager off edevkit
# Replace 1209/edev with the actual edevkit VID/PID once assigned.
SUBSYSTEM=="usb", ATTRS{idVendor}=="1209", ATTRS{idProduct}=="edev", \
    ENV{ID_MM_DEVICE_IGNORE}="1"

# Also: a stable-name symlink so users don't chase /dev/ttyACM index churn
SUBSYSTEM=="tty", ATTRS{idVendor}=="1209", ATTRS{idProduct}=="edev", \
    ATTRS{interface}=="edevkit Target Log",  SYMLINK+="edevkit-target-log"
SUBSYSTEM=="tty", ATTRS{idVendor}=="1209", ATTRS{idProduct}=="edev", \
    ATTRS{interface}=="edevkit Test UART",   SYMLINK+="edevkit-test-uart"

# Group access: both CDC ACM ports go to dialout
SUBSYSTEM=="tty", ATTRS{idVendor}=="1209", ATTRS{idProduct}=="edev", \
    GROUP="dialout", MODE="0660"

Reload: sudo udevadm control --reload-rules && sudo udevadm trigger. Replug the device. Verify with udevadm info --query=property --name=/dev/ttyACM0 and look for ID_MM_DEVICE_IGNORE=1.

Detail: ID_MM_DEVICE_IGNORE must be set on the USB device (the matched parent), not on the tty subsystem. The rule above does that by matching SUBSYSTEM=="usb" with ATTRS{...} — the matched parent is the USB-device node, the env var propagates to its children. See ModemManager's "modem filter" reference for the full taxonomy.

macOS

Kernel driver: AppleUSBCDCACM.kext, in-box since OS X 10.7 Lion. Auto-binds to class/sub 02/02.

Device nodes: two per CDC-ACM instance — /dev/cu.usbmodem<serial><N> and /dev/tty.usbmodem<serial><N>. The cu ("call-up") variant is the call-out side; the tty variant is the call-in side. The naming convention dates to old serial-modem days when "tty" devices blocked on open() waiting for DCD before letting userspace through.

  • Use /dev/cu.usbmodem* for everything — that's what screen, minicom, the Arduino IDE, and 99% of tools default to. open() returns immediately.
  • Avoid /dev/tty.usbmodem* on a non-modem CDC-ACM device. open() on the tty variant blocks indefinitely waiting for DCD to assert, because that's how a fax modem worked in 1995. Most CDC-ACM firmware never asserts DCD, so your tool hangs.

The serial number suffix (usbmodemSERIAL1, usbmodemSERIAL3) is derived from the device's iSerialNumber string. If your firmware doesn't ship a serial-number string descriptor, macOS falls back to a synthetic name based on the USB topology (location ID), which changes when the user plugs into a different USB port — leading to "my port name moves around". Always ship a real iSerialNumber string.

Multiple CDC-ACM instances on one composite get incrementing tail digits — e.g. /dev/cu.usbmodemEDV12341 and /dev/cu.usbmodemEDV12343. Note the odd increments — interface number 0 → suffix 1, interface 2 → suffix 3, etc. macOS appends the Communications-interface number, not a sequence index. This is a real surprise the first time you see it.

No driver install, no kext signing, no DriverKit needed for plain CDC-ACM. The kext is already loaded.

Windows

Kernel driver: usbser.sys, in-box since Windows XP, but auto-loaded only since Windows 10 (build 1507+). On 7/8/8.1 you need a vendor INF that references usbser.sys via mdmcpq.inf.

Device node: COMx where x is assigned by the system (5, 7, 23, …, you don't get to predict it). Multiple CDC-ACM instances on one composite get separate, usually-non-adjacent COM numbers. List them with PowerShell:

# From an elevated PowerShell prompt:
Get-PnpDevice -Class Ports | Format-Table FriendlyName, InstanceId
# Or, for current COM mapping:
[System.IO.Ports.SerialPort]::GetPortNames()

Stable names: Windows assigns COM numbers based on the iSerialNumber string — same serial number, same COM after replug. Different serial → fresh COM. (This is why a USB hub with multiple identical FT232RLs without unique iSerial gets a new COM every replug.) Always ship a real iSerialNumber.

What can go wrong

  • "Code 10" — device cannot start. Almost always missing IAD on a composite, or wrong device-class triple (forgot 0xEF / 0x02 / 0x01).
  • Device shows as "Unknown USB Device" in Device Manager. Descriptor parse failure — usually a malformed descriptor length or an interface with bNumEndpoints mismatched to actual endpoint descriptors. USBView from the WDK shows you the parser's complaint.
  • Port works in HyperTerminal-equivalents but not in your tool. Almost always missing ZLPs — see §ZLP rule. Windows is the strictest about ZLP enforcement.
  • "Driver downgrade" warning. If you previously installed a vendor INF for the same VID/PID, Windows pins usbser.sys's ancestor INF; you may need to uninstall the device with "delete the driver software" before auto-load works.

edevkit1's two CDC-ACM bridges

edevkit1 exposes two CDC-ACM functions in its USB composite descriptor, plus the CMSIS-DAP v2 vendor function. The two CDC-ACM instances bridge to two physically and electrically separate UARTs on the RP2350B:

CDC-ACM instanceBridges toVoltage domainPin map (RP2350B)Purpose
"Target Log"RP2350 hardware UART0Vtgt (auto-sensed from target)GPIO 0 (TX) / GPIO 1 (RX), level-shifted via LVC8T245Capture log/printf output from the target chip being debugged. Voltage matches the target's IO so you can connect to a 1.8V or 3.3V SoC without burning anything.
"Test UART"RP2350 hardware UART1Vio (selectable: 1.8 or 3.3 V)GPIO 2 (TX) / GPIO 3 (RX), level-shifted via LVC8T245Developer-test channel for poking at AT-modem-emulation profiles, sending arbitrary protocol frames, scripting against an external UART peripheral.

Why two instances on the same device, instead of one with a per-message tag? Because each end-user terminal tool wants its own port. The user's screen /dev/edevkit-target-log 115200 session and their edevkit test-uart send "AT+CGSN\r" command must not interfere with each other; the OS-level tty isolation does this for free. Multiplexing in firmware would either need a custom client or it would tag-eat the user's bytes.

The full design rationale (independent voltage domains, level-shifter direction control, why we don't just expose a single multi-baud-rate UART) lives in edevkit1.html §6.6 and §7.2. The actual UART hardware behaviour, baud-rate matrix, and break/framing semantics live in proto_uart.html.

   OUT: EP1, EP3, EP5                   (bulk×2, dap×1)

Three things to notice in this layout:

  1. SWO trace rides EP 0x86 — a bulk-IN inside the CMSIS-DAP vendor function, not a third CDC-ACM. This is deliberate: the host's probe-rs / pyOCD matches CMSIS-DAP by interface string and reads SWO from the configured bulk endpoint. Exposing SWO as a CDC-ACM would tempt people to point screen at it; the bytes wouldn't make sense (they're an ITM-encoded multi-stream trace), the host driver wouldn't know what to do, and you'd lose synchronisation with the live debug session. See usb_cmsis_dap.html.
  2. Both CDC-ACM instances obey SET_LINE_CODING fully. Unlike a pure data-pipe CDC-ACM (which can stash-and-ignore — see §stash, don't act), edevkit1's two bridges do have a real RP2350 UART behind them. When the host sets 115200 8-N-1, the firmware reprograms UART0 / UART1 BAUD divisors. SEND_BREAK becomes a real wire-break.
  3. iInterface strings are part of the API. Host-side udev rules (Linux), system_profiler filtering (macOS), and PowerShell Get-PnpDevice queries (Windows) all use these strings to find the right port without hard-coding tty/COM numbers. Don't change them lightly — host scripts and the edevkit GUI/CLI match on them.
design contract edevkit1 will never reuse the iInterface strings "edevkit Target Log" and "edevkit Test UART" for anything else, in any future revision. They are part of the host-detection contract: tools rely on them to disambiguate the two ports without race-prone /dev/ttyACM* guessing. New CDC-ACM functions added in v2 (or a v1.x update) must use new, distinct interface-string names.

Bandwidth and latency notes

USB Full-Speed (which is what RP2350 USB-Device hardware tops out at — there's no High-Speed PHY in the chip) gives you a theoretical 12 Mbps, and a practical ~10 Mbps usable bulk throughput after framing, error correction, and bus-scheduling overhead. Split across multiple bulk endpoints and shared with the host's other USB devices, you typically see 5–8 Mbps per CDC-ACM instance in real benchmarks.

Wire baudBytes/sec%-of CDC-ACM bandwidthComment
9 600960<0.1%Trivial.
115 20011 520~1%Standard debug-log baud — a rounding error on USB.
921 60092 160~7%Common "fast" UART speed; CDC-ACM has plenty of room.
1 500 000150 000~12%RP2350 UART hardware ceiling for synchronous reception; still well within CDC-ACM.
3 000 000300 000~24%Two channels at this rate would saturate; one is fine.

So bandwidth is essentially never the issue with CDC-ACM — even saturating two UART bridges plus running CMSIS-DAP and SWO at the same time stays inside a Full-Speed bus, with margin.

Latency floor

What CDC-ACM is not good at is single-byte latency. The fundamental floor for "host writes one byte, device sees it" is 1 frame = 1 ms on Full-Speed (a "frame" in USB-speak is the bus-scheduling slot the host carves into 1 ms intervals). On High-Speed it drops to 125 µs (microframe), but RP2350 doesn't have HS. Practical measurements with a logic analyser:

ScenarioTypical RTTNotes
Host write 1 byte, device echoes 1 byte, host reads it (FS, idle bus)~2 ms1 ms host-OUT + 1 ms host-IN, both rounded up to the next bus frame.
Same, but host driver has tcsetattr buffering enabled10–20 msHost kernel TX-coalesces in 10ms windows by default. Disable with cfmakeraw() + VMIN=1, VTIME=0.
Same, but Windows usbser.sys with default INF16 msDefault poll interval; fix with SerialEnumerator=N or by registering a NTW_LATENCY_TIMER override (rare).
Same, but ZLP missing from device∞ (hangs)See §ZLP rule.

Bottom line: CDC-ACM is a high-throughput, moderate-latency channel. It is great for streaming kilobytes of log output; it is mediocre for "tap a key and watch it echo at sub-millisecond". If you need echo-cancelling local-edit semantics, those belong on the host side (every terminal emulator does this); if you need genuine low-latency single-character control (tight register-poke shells, REPLs that need crisp feedback), it's tolerable but not great. For ultra-low-latency streaming you'd reach for a vendor-class bulk pipe with smaller, host-tuned URBs.

Measuring round-trip latency

The cheapest accurate measurement: bridge the device's TX to its own RX (literally short the two CDC-ACM data lines on the wire side, or enable an internal echo path), and time a thousand 1-byte ping-pongs from the host:

// host-side, Python
import serial, time, statistics
p = serial.Serial("/dev/edevkit-target-log", 115200, timeout=0.1)
samples = []
for _ in range(1000):
    t0 = time.perf_counter_ns()
    p.write(b"x"); p.flush()
    p.read(1)
    t1 = time.perf_counter_ns()
    samples.append((t1 - t0) / 1000)   # microseconds
print(f"p50={statistics.median(samples):.0f}us  "
      f"p95={statistics.quantiles(samples, n=20)[18]:.0f}us  "
      f"p99={statistics.quantiles(samples, n=100)[98]:.0f}us")

On Linux + edevkit1, expect p50 ~1.5 ms, p95 ~3 ms, p99 ~6 ms with a quiet host bus. macOS is similar; Windows tends to be 2–3× slower at p99 due to driver scheduling.

Where to learn more

Primary sources first, then practical writeups:

  1. USB CDC 1.2 specification (usb.org) — the canonical class definition. Sections 4 (descriptors), 5 (control requests), 6 (notifications) are the ones to read; the rest is modem-era detail.
  2. USB PSTN 1.2 specification (usb.org) — the subclass document that owns the ACM-specific part. §6.2.2 lists every ACM control request; §6.5 lists the SERIAL_STATE notification format. This is where SET_LINE_CODING and the line-coding struct live.
  3. USB Interface Association ECN — defines the IAD descriptor, the device-class triple 0xEF/0x02/0x01, and Microsoft's enumeration rules.
  4. Microsoft Learn — USB Serial Driver (Usbser.sys) — official documentation on auto-loading via compatible-ID, the device-class triple requirement, and INF construction for older Windows.
  5. Linux kernel — drivers/usb/class/cdc-acm.c — the canonical CDC-ACM host driver. Reading the source is faster than any tutorial; the comments around acm_probe() and acm_tty_open() explain edge cases the spec is silent on.
  6. ModemManager docs — "modem filter" reference and the common udev tags page. If you've ever fought "ModemManager keeps grabbing my port", these are the primary sources.
  7. Zephyr USB-Device-Next CDC-ACM — official docs + single-CDC sample + composite sample. The composite sample is the closest thing to edevkit1's two-CDC-ACM layout in the upstream Zephyr tree, and is the right starting point when bringing up the firmware.
  8. Michael Stapelberg, "Linux and USB virtual serial devices (CDC ACM)" (2021) — practical writeup on udev rules, ModemManager interaction, stable naming, and access permissions. The single best blog post on the subject; everything in §Linux OS behaviour above is a condensed version of his findings.
  9. argonblue, "Much ado about nothing: USB zero-length packets" — exhaustive walkthrough of the ZLP rule, why hosts behave the way they do, and a reproducer. Read this if §the ZLP rule isn't clicking.
  10. USBView (Windows WDK) and Uwe Sieber's USB Device Tree Viewer — descriptor parsers that show you exactly what the host enumerator sees. When debugging a non-enumerating composite, these are the first tools to reach for.
  11. Sister docs. usb_overview.html for the underlying USB endpoint / transfer / URB model; usb_cmsis_dap.html for the CMSIS-DAP v2 vendor-class function that shares the composite descriptor with the two CDC-ACM bridges; proto_uart.html for the wire-side UART hardware behind the bridges; edevkit1.html for the system-level rationale of why the kit is shaped the way it is.

If you read only one external thing: the argonblue ZLP article. The ZLP rule is the difference between a CDC-ACM implementation that ships and one that "works on my machine but the user reports random hangs". Get it right once and you'll never re-debug it.
