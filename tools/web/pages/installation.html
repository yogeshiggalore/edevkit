edevkit · onboarding · installation guide

From-scratch developer installation guide

Everything a new developer needs to set up a working edevkit environment on Linux, macOS, or Windows — from a fresh OS install to building firmware, host tools, hardware artifacts, and the enclosure. The current product is edevkit1 (RP2350B + Zephyr); the install steps below set you up for it, and the same env will carry forward to future generations. Cross-references the edevkit1 design spec at edevkit1.html.

Estimated total time: 1.5–3 hours of mostly-unattended downloads, depending on your network and CPU. About 25–35 GB of disk usage when fully provisioned (Zephyr workspace + SDK + Rust + KiCad + libraries).

You'll be ready to: build & sign the firmware via west sysbuild, flash it via SWD or BOOTSEL, build the Tauri host GUI and Rust CLI, run KiCad ERC/DRC, regenerate enclosure STLs, and run the unit-test suite.

What is edevkit1, and what does this guide install?

edevkit1 is an open-source, USB-powered single-board embedded development tool that combines an ARM + RISC-V debug probe (CMSIS-DAP v2), SWO trace, two UART bridges, an I²C protocol tester, an SPI/DSPI/QSPI tester, a YAML-driven device emulator, a Tauri GUI + Rust CLI on the host, and OTA firmware updates over MCUboot. The full v1 spec lives at documents/edevkit1.html; this guide does not assume you've read it, but you'll want to skim it before writing code.

Hardware platform: RP2350B (Cortex-M33 dual-core, 12 PIO state machines, QFN-80, 48 GPIO) running Zephyr RTOS 4.x with MCUboot. The host is Tauri (Rust + web view) plus a Rust CLI. Hardware is designed in KiCad 9; enclosure in CadQuery or FreeCAD 1.x. Open-source tools only — no proprietary CAD or closed-source toolchains anywhere in the build path.

This guide installs everything you need for all five of those subsystems (firmware, host, hardware, enclosure, tools) on a fresh OS. You can skip the steps for subsystems you won't touch: see the step-coverage table at the start of step 1.

Greenfield note As of writing, the edevkit/ repo is mostly the design spec — firmware/, host/, hardware/, enclosure/ are placeholders or empty. This install guide brings the environment to a state where, when those subsystems land, you can build them immediately. If a step references a file that doesn't yet exist (e.g. firmware/west.yml), the install still completes — you just won't be able to use that subsystem until the code lands.

Pick your OS path

This guide supports the three major OS families. Pick one — the rest of the guide branches on this choice. Recommendations are based on what's most-tested in the embedded Zephyr / Tauri / KiCad ecosystem, not on personal taste.

OSRecommendationWhy
Linux (Ubuntu 24.04 LTS or Debian 12)first-classZephyr is developed primarily on Linux. KiCad, sigrok, openocd, probe-rs all native. Easiest USB device permissions story (udev rules).
macOS (Sonoma 14+ / Sequoia 15+, Apple Silicon or Intel)first-classHomebrew makes most installs trivial. No driver work for USB CDC/HID devices. probe-rs and pyOCD work fine.
Windows 11 (with WSL2 + Ubuntu)recommendedRun all the Zephyr/host tooling inside WSL2 (Ubuntu); use usbipd-win to forward USB devices to WSL2 for flashing. KiCad runs natively on Windows.
Windows 11 (native, no WSL2)supported with caveatsZephyr SDK and west work natively, but several smaller tools (sigrok-cli, smpclient builds) are smoother on Linux/WSL2. Documented in step 1 below.
Linux Fedora / Arch / openSUSEsupportedSubstitute the package manager (dnf / pacman / zypper). Most package names are identical or near-identical to the Ubuntu list.
Linux on ARM (e.g. Raspberry Pi 5)supportedZephyr SDK ships aarch64 host toolchains. KiCad runs (slowly).
Windows on ARMnot supportedZephyr SDK doesn't currently ship Windows-ARM host toolchains. Use WSL2-on-ARM with the Linux SDK instead.
Windows users We strongly recommend WSL2 + Ubuntu over native Windows for the firmware and host work. Native Windows is documented for completeness, but you'll fight more environmental quirks. The single thing you'll launch natively even on a WSL2 box is KiCad (it's faster as a native Windows app than through WSL2's GUI forwarding).

Assumptions and conventions in this guide

Different developers organise their disks differently. To keep this guide copy-pasteable for everyone, every command below references five environment variables that you set once at the start. Pick the layout that fits your habits, set the vars, and the rest of the guide works without further adjustment.

The five path variables

VariableWhat it points atWhy it's separate
EDEV_HOMEThe "wrapper" directory that holds everything project-related on your machine. Optional in flatter layouts; convenient when you want one folder to back up or move.Convenience root only — never referenced by tools, only used to compose the other vars below.
EDEVKIT_REPOYour local clone of the edevkit git repo (this is the dir that contains firmware/, hardware/, host/, tools/).The only "tracked" path. Everything else is gitignored or lives outside.
EDEV_VENVThe Python virtual environment that holds west, pyocd, kibot, etc.Hundreds of MB of pip packages — must stay outside the repo so it doesn't pollute git status.
ZEPHYR_WORKSPACEThe west workspace (zephyr/, modules/, bootloader/, etc.).6–10 GB of upstream code; outside the repo so it never lands in git status and can be shared across multiple Zephyr projects on the machine.
ZEPHYR_SDK_INSTALL_DIRThe Zephyr SDK toolchains (arm-zephyr-eabi-gcc and friends).One install per machine, shared across all Zephyr projects. Standard env var that Zephyr's CMake auto-detects.

Pick a layout

Three patterns are all valid. The relationships are what matter (venv and workspace OUTSIDE the repo); the exact paths are user choice. Layout A is recommended if you don't have a preference — it keeps everything project-related under one wrapper folder, and matches what's actually used on the maintainer's machine.

Layout A · wrapper-dir recommended

One wrapper directory holds the venv + the cloned repo as siblings. The Zephyr workspace lives at a sibling location (one level higher). Easy to back up, easy to move, easy to find everything.

~/projects/temp/edevkit/        ← EDEV_HOME (wrapper)
├── edev/                       ← EDEV_VENV     (Python venv)
└── edevkit/                    ← EDEVKIT_REPO  (git repo)
~/projects/temp/zephyrproject/  ← ZEPHYR_WORKSPACE
~/zephyr-sdk-1.0.1/             ← ZEPHYR_SDK_INSTALL_DIR

Set the env vars (Linux / macOS / WSL2):

export EDEV_HOME="$HOME/projects/temp/edevkit"
export EDEVKIT_REPO="$EDEV_HOME/edevkit"
export EDEV_VENV="$EDEV_HOME/edev"
export ZEPHYR_WORKSPACE="$HOME/projects/temp/zephyrproject"
export ZEPHYR_SDK_INSTALL_DIR="$HOME/zephyr-sdk-1.0.1"

Windows PowerShell equivalent:

$env:EDEV_HOME              = "$HOME\projects\temp\edevkit"
$env:EDEVKIT_REPO           = "$env:EDEV_HOME\edevkit"
$env:EDEV_VENV              = "$env:EDEV_HOME\edev"
$env:ZEPHYR_WORKSPACE       = "$HOME\projects\temp\zephyrproject"
$env:ZEPHYR_SDK_INSTALL_DIR = "$HOME\zephyr-sdk-1.0.1"
Layout B · flat siblings

Repo, venv, and Zephyr workspace are all parallel folders directly under ~/projects/. No wrapper dir. Simple, but loses the "everything in one place" property of Layout A.

~/projects/
├── edevkit/                    ← EDEVKIT_REPO  (git repo)
├── edev/                       ← EDEV_VENV
└── zephyrproject/              ← ZEPHYR_WORKSPACE
~/zephyr-sdk-1.0.1/             ← ZEPHYR_SDK_INSTALL_DIR

Set the env vars (Linux / macOS / WSL2):

export EDEV_HOME="$HOME/projects"
export EDEVKIT_REPO="$EDEV_HOME/edevkit"
export EDEV_VENV="$EDEV_HOME/edev"
export ZEPHYR_WORKSPACE="$EDEV_HOME/zephyrproject"
export ZEPHYR_SDK_INSTALL_DIR="$HOME/zephyr-sdk-1.0.1"
Layout C · fully separate

Venv and workspace go directly into $HOME; the repo lives wherever you keep code. Useful if you already have a Zephyr workspace at ~/zephyrproject/ from another project and want to reuse it.

~/edev/                         ← EDEV_VENV
~/zephyrproject/                ← ZEPHYR_WORKSPACE
~/projects/edevkit/             ← EDEVKIT_REPO  (or anywhere else)
~/zephyr-sdk-1.0.1/             ← ZEPHYR_SDK_INSTALL_DIR

Set the env vars (Linux / macOS / WSL2):

export EDEVKIT_REPO="$HOME/projects/edevkit"
export EDEV_VENV="$HOME/edev"
export ZEPHYR_WORKSPACE="$HOME/zephyrproject"
export ZEPHYR_SDK_INSTALL_DIR="$HOME/zephyr-sdk-1.0.1"
# EDEV_HOME unused in this layout
Persist the variables export only lasts for the current shell. To keep them across new terminals, append the same lines to ~/.bashrc (bash), ~/.zshrc (zsh), or your PowerShell $PROFILE. Step 13 covers a more complete shell-helpers block that includes these.

Other conventions

  • Path constraints: avoid spaces, non-ASCII characters, or paths that exceed Windows MAX_PATH (260 chars). On Windows native, prefer C:\proj\edevkit over C:\Users\Your Name\Documents\edevkit — Zephyr/CMake/west compose paths several levels deep and bump into MAX_PATH limits.
  • Shell: bash or zsh on Linux/macOS; PowerShell 7 (not 5.x) for native Windows. Inside WSL2, your default Ubuntu bash.
  • Privilege: any command starting with sudo needs administrator/root; everything else runs as your normal user. On Windows, "Run PowerShell as Administrator" when noted.
  • Code blocks: when a step has different commands per OS, you'll see three colored cards labeled Linux macOS Windows. Run the one that matches your OS.
  • Variables vs. literal paths: every command below uses $EDEVKIT_REPO / $EDEV_VENV / $ZEPHYR_WORKSPACE instead of hard-coded paths. If you forgot to export, the command will fail with a confusing "no such file"; echo $EDEVKIT_REPO first to confirm it's set.

Step 1 — System prerequisites

1~10–25 min · network-bound

Install the OS-level toolchain (compilers, build tools, Python, git)

These are the base tools every later step depends on. Most OSes ship some of them; we install the missing ones now in a single batch.

Linux · Ubuntu / Debian 
sudo apt update
sudo apt install -y \
    git curl wget xz-utils file ca-certificates \
    build-essential cmake ninja-build gperf ccache dfu-util device-tree-compiler \
    python3 python3-dev python3-venv python3-pip python3-setuptools python3-tk \
    qemu-system-arm qemu-system-misc \
    libsdl2-dev libmagic1 \
    libusb-1.0-0-dev libudev-dev pkg-config \
    unzip zip

Fedora equivalent: sudo dnf install -y git curl wget xz file ca-certificates @development-tools cmake ninja-build gperf ccache dfu-util dtc python3 python3-devel python3-pip python3-tkinter qemu-system-arm SDL2-devel file-libs libusbx-devel systemd-devel pkgconf-pkg-config

Arch equivalent: sudo pacman -S --needed git curl wget xz file ca-certificates base-devel cmake ninja gperf ccache dfu-util dtc python python-pip tk qemu-arch-extra sdl2 file libusb pkgconf

macOS

Install the Xcode Command Line Tools and Homebrew first (skip if already installed):

# 1) Xcode Command Line Tools (clang, make, git)
xcode-select --install        # GUI installer pops up; takes ~5 min

# 2) Homebrew
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# 3) make sure brew is on the PATH (Apple Silicon vs Intel paths differ)
#    On Apple Silicon:
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zprofile
eval "$(/opt/homebrew/bin/brew shellenv)"
#    On Intel:
echo 'eval "$(/usr/local/bin/brew shellenv)"' >> ~/.zprofile
eval "$(/usr/local/bin/brew shellenv)"

Now the system deps:

brew install \
    cmake ninja gperf ccache qemu dtc libmagic libusb \
    python3 \
    git curl wget xz

macOS ships Python 3 with the Command Line Tools, but the Homebrew Python is the one Zephyr's tooling expects to find on PATH. Don't fight it — let brew win.

Windows · WSL2 (recommended)

Enable WSL2 and install Ubuntu:

# In an elevated PowerShell:
wsl --install -d Ubuntu-24.04

# Then reboot. After reboot, an Ubuntu terminal will pop up and ask you to
# create a username + password. From this point on, run all Linux commands
# from inside this Ubuntu shell — exactly as in the Linux card above.

After Ubuntu starts, follow the Linux · Ubuntu/Debian instructions above for the package install. Everything except step 8 (KiCad) and step 10 (OrcaSlicer) is done from inside WSL2.

Windows · native (no WSL2)

Install the package manager scoop (no admin required, easier than Chocolatey for our needs):

Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
Invoke-RestMethod -Uri https://get.scoop.sh | Invoke-Expression
scoop bucket add main
scoop bucket add extras
scoop bucket add versions

Then the system deps:

scoop install git curl wget 7zip
scoop install cmake ninja gperf dtc python python311
# python311 ensures you have a 3.11+ interpreter even if scoop's default 'python'
# moves on. Zephyr supports 3.10+; we lock at 3.11 for predictability.

# Visual C++ Build Tools (for Rust later, and for Zephyr host builds)
# Open https://visualstudio.microsoft.com/visual-cpp-build-tools/ and install
# "Desktop development with C++" — about 6 GB. No way around it for native
# Windows.

Why scoop over Chocolatey: scoop installs to your user dir, doesn't need admin, and updates atomically. Chocolatey is fine if you already use it — substitute choco install for the package names.

Verify step 1

git --version           # ≥ 2.30
python3 --version       # 3.10+, 3.11 or 3.12 ideal
cmake --version         # ≥ 3.20
ninja --version         # any
gperf --version         # any
dtc --version           # any
ccache --version        # any (optional speedup)

Step 2 — Clone the repository

2~1 min

Get the source tree

Clone via HTTPS (or SSH if you have keys configured). The path matters less than the lack of spaces and special characters.

Make sure $EDEVKIT_REPO (and $EDEV_HOME if your layout uses one) are exported — see Pick a layout.

# create the parent dir, then clone into $EDEVKIT_REPO
mkdir -p "$(dirname "$EDEVKIT_REPO")"
git clone https://github.com/<org>/edevkit.git "$EDEVKIT_REPO"
cd "$EDEVKIT_REPO"
ls -la
# expected top-level layout (per design spec §2):
#   hardware/    enclosure/    firmware/    host/    documents/
#   tools/       docs/         README.md     CLAUDE.md (locally — not committed)

Windows PowerShell: substitute $env:EDEVKIT_REPO for $EDEVKIT_REPO and use New-Item -ItemType Directory -Force ... instead of mkdir -p.

Path constraint Avoid paths with spaces, non-ASCII characters, or that exceed Windows MAX_PATH (260 chars). On Windows native, prefer C:\proj\edevkit over C:\Users\Your Name\Documents\edevkit — Zephyr/CMake/west compose paths several levels deep and bump into MAX_PATH limits.

Step 3 — Python environment (venv + west)

3~3 min

Create an isolated Python environment, then install west into it

Zephyr's tooling is mostly Python (west, twister, the build infrastructure). Installing into a venv keeps Zephyr's pinned package versions away from any other Python projects on the machine. Do not install west globally — always use a venv.

One command does it all Use the tools/setup_venv.sh helper. It (1) creates the venv at $EDEV_VENV, (2) installs west + the docs-hub deps, (3) appends a guarded source line to the venv's activate so the shell helpers (edev_* commands) load automatically every time you enter the venv, and (4) runs west zephyr-export if your workspace is initialised. Idempotent — safe to re-run.
Linux · macOS · WSL2 
# layout vars must already be exported (see "Pick a layout"):
bash "$EDEVKIT_REPO/tools/setup_venv.sh"

# now activate — helpers come along for the ride:
source "$EDEV_VENV/bin/activate"

# verify:
edev_help
Windows · native 
& "$env:EDEVKIT_REPO\tools\setup_venv.ps1"

# activate:
. "$env:EDEV_VENV\Scripts\Activate.ps1"

# verify:
edev_help

If Activate.ps1 fails with an execution-policy error, run Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser once.

One venv for everything The setup script installs west AND the docs-hub server's deps (fastapi, uvicorn, jinja2) into the same $EDEV_VENV. After step 4 finishes (west packages pip --install), this venv will also hold pyocd, imgtool, pyelftools, and ~100 other Zephyr dependencies — about 750 MB total. One activation, every Python tool the project needs.
Helper-loading model — Pattern C The setup script appends a single guarded line to the venv's activate file: 
if [ -n "${EDEVKIT_REPO:-}" ] && [ -f "$EDEVKIT_REPO/tools/env_helpers.sh" ]; then
    source "$EDEVKIT_REPO/tools/env_helpers.sh"
fi
The actual helpers stay in the repo, so git pull always updates them without re-touching activate. If you ever rm -rf "$EDEV_VENV" and re-create it, just re-run setup_venv.sh. The full reference for every edev_* helper lives at shell_helpers.html.

Manual fallback (if you'd rather not run the helper script)

Show manual steps → 
# Linux / macOS / WSL2:
mkdir -p "$(dirname "$EDEV_VENV")"
python3 -m venv "$EDEV_VENV"
"$EDEV_VENV/bin/pip" install --upgrade pip wheel
"$EDEV_VENV/bin/pip" install west
"$EDEV_VENV/bin/pip" install -r "$EDEVKIT_REPO/tools/web/requirements.txt"

# Manually wire the helpers into activate (Pattern C):
cat >> "$EDEV_VENV/bin/activate" <<'EOF'

# --- edevkit helpers ---
if [ -n "${EDEVKIT_REPO:-}" ] && [ -f "$EDEVKIT_REPO/tools/env_helpers.sh" ]; then
    source "$EDEVKIT_REPO/tools/env_helpers.sh"
fi
EOF

Always re-activate the venv in new terminals

The venv only "exists" inside a shell that's sourced its activate script. Every new terminal needs:

# Linux / macOS / WSL2:
source "$EDEV_VENV/bin/activate"

# Windows native:
& "$env:EDEV_VENV\Scripts\Activate.ps1"

You can shorten this to one word — edev_activate — once you've added a tiny shim in your shell rc that sets the layout vars and calls activate. Step 13 covers the rc setup.

With the venv activated, the helper edev_doc launches the local hub at http://127.0.0.1:8765/ in your default browser:

Step 4 — Zephyr workspace

4~20–60 min · 6–10 GB · network-bound

Initialize the Zephyr west workspace and pull all modules

The Zephyr workspace contains Zephyr itself, third-party HALs (Pico-SDK, MCUboot, MbedTLS, etc.), and Zephyr's Python build helpers. The west.yml at the top of firmware/ pins exact revisions of everything. This step is the largest single download in the whole install.

Workspace lives OUTSIDE the project repo Do not run west init inside $EDEVKIT_REPO. The Zephyr workspace is a multi-GB tree of unrelated upstream code (Zephyr itself, MCUboot, HALs, MbedTLS) that does not belong in the project's git history. The project's CMake files locate Zephyr at build time via the ZEPHYR_BASE environment variable that west zephyr-export sets; the workspace can sit anywhere on disk. 
# ensure the venv is activated (step 3) so `west` is on PATH

# initialise the workspace at the location you exported in $ZEPHYR_WORKSPACE:
west init "$ZEPHYR_WORKSPACE"

# pull every module. The --narrow / --depth=1 trick keeps the clone shallow.
# Trade-off: shallow clones save ~3 GB but lose git history of HALs.
cd "$ZEPHYR_WORKSPACE"
west update --narrow -o=--depth=1

# expose Zephyr's CMake package so out-of-tree apps (like this project) can find it:
west zephyr-export

# install Zephyr's Python deps (pyelftools, anytree, pykwalify, etc.) into the venv:
west packages pip --install
Sharing with other Zephyr projects If you already have a Zephyr workspace from another project on this machine, skip west init and reuse it — just point $ZEPHYR_WORKSPACE at the existing path. Make sure west zephyr-export has been run in that workspace and that ZEPHYR_BASE resolves to its zephyr/ subdir. One workspace per machine saves ~6–10 GB across projects.
If the project later ships its own west manifest When $EDEVKIT_REPO/firmware/west.yml lands (pinning Zephyr revision + project-specific HALs), reinit the workspace pointed at that manifest: 
rm -rf "$ZEPHYR_WORKSPACE"                       # or move it aside
west init -l "$EDEVKIT_REPO/firmware"            # uses the project's west.yml
cd "$EDEVKIT_REPO/firmware"
west update --narrow -o=--depth=1
west zephyr-export
Even with a project-local manifest, the bulk of the workspace (zephyr/, modules/, etc.) still lives outside the repo via west's topdir mechanism — only the west.yml file itself sits in the repo.

Verify step 4

cd "$ZEPHYR_WORKSPACE"
west list                              # should print zephyr, mcuboot, hal_rpi_pico, mbedtls, …
west boards | grep -i rpi_pico2        # rpi_pico2 (with /rp2350a/m33 etc. variants) should appear
echo "$ZEPHYR_BASE"                    # set by `west zephyr-export` — should be $ZEPHYR_WORKSPACE/zephyr

Step 5 — Zephyr SDK (toolchains)

5~10 min · 2–10 GB

Install the cross-compilers Zephyr uses to build for ARM (and other targets)

The Zephyr SDK is a self-contained bundle of cross-compilers, GDB, and host tools. We use SDK 1.0.1 at the time of writing; any 0.17.x or newer 1.x release that's compatible with your Zephyr version will work. The SDK is not Python — install once per machine, share across projects.

The full SDK is ~10 GB across all architectures. You only need arm-zephyr-eabi for edevkit1, which is ~700 MB. Use the -t flag to filter.

You already exported $ZEPHYR_SDK_INSTALL_DIR in the layout step (e.g. $HOME/zephyr-sdk-1.0.1). The install command below points west sdk at the parent dir of that target via the -b flag, so the SDK lands exactly where the env var expects.

Linux · WSL2 
# west sdk handles the download and unpack:
west sdk install -b "$(dirname "$ZEPHYR_SDK_INSTALL_DIR")" -t arm-zephyr-eabi
# ~700 MB; drop the `-t` flag to install all archs (~10 GB)

# Persist the env var across new shells:
echo 'export ZEPHYR_SDK_INSTALL_DIR="'"$ZEPHYR_SDK_INSTALL_DIR"'"' >> ~/.bashrc
macOS 
west sdk install -b "$(dirname "$ZEPHYR_SDK_INSTALL_DIR")" -t arm-zephyr-eabi

echo 'export ZEPHYR_SDK_INSTALL_DIR="'"$ZEPHYR_SDK_INSTALL_DIR"'"' >> ~/.zprofile

On Apple Silicon, the SDK ships native arm64 binaries (no Rosetta required). If you see "cannot be opened because Apple cannot check it for malicious software" on first run, allow it once via System Settings → Privacy & Security → Open Anyway.

Windows · native 
# In PowerShell, with the venv activated:
west sdk install -b (Split-Path $env:ZEPHYR_SDK_INSTALL_DIR) -t arm-zephyr-eabi

# Persist the env var for your user (adjust the path if you used a different location):
[Environment]::SetEnvironmentVariable(
    "ZEPHYR_SDK_INSTALL_DIR",
    $env:ZEPHYR_SDK_INSTALL_DIR,
    "User")
# Restart any terminals after this for the new var to be picked up.

Verify step 5

ls "$ZEPHYR_SDK_INSTALL_DIR/gnu/arm-zephyr-eabi/bin/"
# should list arm-zephyr-eabi-gcc, -gdb, -objdump, etc.

"$ZEPHYR_SDK_INSTALL_DIR/gnu/arm-zephyr-eabi/bin/arm-zephyr-eabi-gcc" --version

Step 6 — pioasm (RP2350 PIO assembler)

6~5 min

Build the RP2350 PIO assembler from source

The RP2350's PIO blocks (12 state machines) are what make it a viable debug probe + protocol tester at speeds the M33 cores can't bit-bang. PIO source files (.pio) are written in a small assembly language and compiled to a C array embedded in firmware. Zephyr does not bundle the assembler. No package manager (brew/apt/scoop) ships it as a standalone binary — we build from source.

cd "$EDEVKIT_REPO/tools"
git clone --depth 1 https://github.com/raspberrypi/pico-sdk-tools.git
cd pico-sdk-tools/pioasm
mkdir -p build && cd build
cmake .. -G Ninja
ninja
# resulting binary: $EDEVKIT_REPO/tools/pico-sdk-tools/pioasm/build/pioasm

# add it to PATH for your venv. Append to your shell rc OR to step 13's helpers file:
echo 'export PATH="$EDEVKIT_REPO/tools/pico-sdk-tools/pioasm/build:$PATH"' \
    >> ~/.bashrc       # zsh: ~/.zshrc
source ~/.bashrc

On Windows native, build the same way using the "x64 Native Tools Command Prompt for VS" (so MSBuild + Ninja can find the C++ compiler), then add $env:EDEVKIT_REPO\tools\pico-sdk-tools\pioasm\build to PATH via System Properties → Environment Variables.

Verify step 6

which pioasm                # …/pico-sdk-tools/pioasm/build/pioasm

Step 13 — Shell helpers & daily workflow

13~2 min

Verify the helpers load, and learn the every-terminal recipe

You'll re-run the same build/flash/test commands hundreds of times. The repo ships convenience helpers (edev_build, edev_flash, edev_create_app, …) that already got wired into your venv by Step 3 — setup_venv.sh. Nothing more to install — this step just confirms it works and shows the daily-use recipe.

The wiring chain is:

  1. Your shell rc (~/.zshrc, ~/.bashrc, or PowerShell $PROFILE) exports the four layout vars from Pick a layout.
  2. Activating the venv (source "$EDEV_VENV/bin/activate") auto-sources $EDEVKIT_REPO/tools/env_helpers.sh, because Step 3 appended a guarded source line to activate.
  3. The helpers are now functions in your shell. Full reference for every helper at shell_helpers.html.

Daily workflow — every new terminal

Linux · macOS · WSL2 (bash / zsh) 
# open a new terminal — your rc auto-loads the layout vars.
# then, one line:
source "$EDEV_VENV/bin/activate"

# verify:
edev_help
Windows · native (PowerShell) 
# open a new PowerShell — $PROFILE auto-loads the layout vars.
# then, one line:
. "$env:EDEV_VENV\Scripts\Activate.ps1"

# verify:
edev_help

That's the whole loop. The "(edev)" prompt prefix is your cue that west, pyocd, imgtool, and every edev_* helper are on PATH.

When you need to refresh something

You changed…Do this
Your shell rc (added a layout var, fixed a typo) Open a new terminal — or in the current one: deactivate 2>/dev/null; source ~/.zshrc; source "$EDEV_VENV/bin/activate"
$EDEVKIT_REPO/tools/env_helpers.sh itself (e.g. after git pull) Just re-source it — no need to leave the venv: source "$EDEVKIT_REPO/tools/env_helpers.sh"
Recreated the venv (rm -rf "$EDEV_VENV" + new python -m venv) Re-run setup_venv.sh — that re-appends the auto-source block to the new activate
Want the helpers in every shell, not just inside the venv Append source "$EDEVKIT_REPO/tools/env_helpers.sh" to your shell rc (Pattern B; see shell_helpers.html for trade-offs)
Want every new terminal to auto-activate the venv Append source "$EDEV_VENV/bin/activate" to your shell rc — convenient, but slows shell startup ~50–100 ms and the venv is then live in unrelated shells too

USB permissions (Linux udev rules)

On Linux, USB devices are owned by root by default. Without rules, probe-rs / pyOCD / mcumgr will fail with "permission denied" or "device not found." Install rules once per machine:

# Add yourself to the plugdev group:
sudo usermod -aG plugdev $USER
# Log out and back in for the group change to take effect.

# probe-rs udev rules:
sudo curl -fsSL \
    https://probe.rs/files/69-probe-rs.rules \
    -o /etc/udev/rules.d/69-probe-rs.rules

# RP2350 BOOTSEL mode (for UF2 drag-and-drop) — file as 70-rp2350.rules:
sudo tee /etc/udev/rules.d/70-rp2350.rules >/dev/null <<'EOF'
SUBSYSTEM=="usb", ATTR{idVendor}=="2e8a", ATTR{idProduct}=="000f", MODE="0666", GROUP="plugdev"
EOF

# Reload:
sudo udevadm control --reload-rules
sudo udevadm trigger

Re-plug the device after this; permissions should now be correct.

USB passthrough on WSL2

WSL2 doesn't see USB devices by default — they're attached to the Windows kernel. Use usbipd-win to forward devices on demand.

# In Windows (admin PowerShell, one-time):
winget install usbipd

# List USB devices visible to Windows:
usbipd list
# Note the BUSID of the device you want to forward, e.g. 1-7

# Bind it (one-time per device):
usbipd bind --busid 1-7

# Attach it to WSL2 (each session, after replug):
usbipd attach --wsl --busid 1-7

# Inside WSL2, verify:
lsusb               # should show the device

# To release it back to Windows:
usbipd detach --busid 1-7

Add the udev rules from the section above inside your WSL2 Ubuntu — same as a native Linux install.
